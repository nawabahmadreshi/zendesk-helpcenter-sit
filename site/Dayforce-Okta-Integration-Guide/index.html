Dayforce / Okta Integration Guide

← Back to index

This guide applies to the latest public version of the integration (version 3.3.0).

Overview

The Aquera platform allows your organization to scale complex data management by automating data integration across resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

In this integration, any users created, updated, or terminated in Dayforce are automatically updated in Okta based on the defined schedule. The integration also supports the writeback of business email addresses and federated IDs from Dayforce to Okta.

To identify user records across both systems for user sync from Dayforce to Okta, the integration uses the attribute configured in the Source Field for Correlation parameter in Dayforce and matches it to the attribute configured in the Target Field for Correlation parameter in Okta as a correlation attribute.

Key Functions

The Dayforce / Okta Integration performs the following functions:

  • Synchronizes user records from Dayforce to Okta.
  • Creates new users in Okta when they are added in Dayforce.
  • Updates existing Okta users when their information changes in Dayforce.
  • Deactivates or deletes Okta users when they are no longer active in Dayforce.
  • Generates usernames and email IDs for users in Okta based on configured policies.
  • Updates business email addresses and federated ID's from Okta to Dayforce (writeback).
  • Supports both full synchronization and incremental synchronization to process only changed records.

Prerequisites

The following are the prerequisites:

  • Admin Access to Okta: You must have admin access to your Okta instance. You can visit the Okta portal to get access to an Okta instance.
  • Admin Access to Dayforce: You must have admin access to your Dayforce instance.
  • Dayforce Field-Level Access: The specific field-level permissions must be enabled for the service account to manage user information in Dayforce. For more details, refer to the Configuring Field-Level Access in Dayforce section.
  • Enable XaaS access: You must open a support case with Okta to enable XaaS in your Okta instance.
  • Source Priority for Username and Email Update: In case of username and email update for Okta profile, ensure that the username and email attributes in Okta are mastered by the correct system. Refer to the Source Priority section for more details.
  • Aquera Tenant Account: You must have a tenant account in the Aquera Admin portal. Visit https://admin.aquera.io to register for a tenant account. Contact your Aquera account manager to get Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io
  • Configure Application Connectors in the Aquera Admin Portal: You need to configure the following application connectors in the Aquera Admin Portal:
    • Dayforce: Refer to the Dayforce Configuration Guide for configuration details.
    • Okta: Ensure that you configure the Okta (version 11) or Okta XaaS (Anything-as-a-Source) connector in the Aquera Admin portal.

For details on the limitations of Okta's Anything-as-Source (XaaS) solution, refer to the Okta documentation.

Note: The inactive and terminated users will not be integrated from Dayforce to Okta.

Source Priority

If your use case requires updating username and email for Okta profile, ensure that these attributes in Okta are mastered by the correct system. Perform the following steps to configure the source priority:

  1. In Okta Admin Console, navigate to Directory > Profile Editor and select the User Profile.
  2. On the Profile Editor page, scroll down to the Attributes section.
  3. Find username attribute and click the info (i) icon.
  4. Set the Source Priority as follows:
    • Select Inherit from Profile Source if you want Dayforce to be the source of truth to provide and maintain the attribute value.
    • Select Inherit from Okta if the attribute must be managed in Okta and then pushed to Dayforce. In this case, Okta becomes the source of truth.
  5. Repeat the same steps for the email attribute and set the appropriate source priority based on your requirements.

Integration Setup

This section outlines how the integration must be configured to push user-related data from Dayforce to Okta.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Select Dayforce / Okta.
  5. Enter a name and a description for the integration.
  6. Select your Dayforce and Okta connector instances.
  7. Click Continue. The Configuration page is displayed to configure the integration.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Enable Features

The following sections are displayed under the Enable Feature tab:

Integration Scenarios

The Integration Scenarios section enables you to customize the integration settings according to your specific use case. It has the following parameters:

  • Enable User Management: Enable this toggle button to synchronize user's data from Dayforce to Okta. When enabled, the User Management tab and the following parameter are displayed for further configuration:
    • Select User Operations to Perform: Select one or more operations from the dropdown menu to determine when user accounts should be created, updated, or deactivated during provisioning. 
  • Enable Write Back: Enable this toggle button to write back the user's business email address and federated ID from Okta to Dayforce. When this toggle button is turned on, the WriteBack tab is displayed to configure further integration settings.
  • Generate Unique Username/Email Using AI Widget: Enable this toggle button to generate a unique username and email address using the AI widget. When this toggle button is turned on, the Policy tab is displayed to configure further integration settings.
  • Create Custom Attributes in Okta: Enable this toggle button to create Dayforce custom attributes in Okta. When this toggle button is turned on, the Custom Attribute tab is displayed to configure further integration settings.

Manage User

The Manage User section enables you to customize the integration settings according to your specific use case. It has the following settings:

Correlation Settings

Configure the following correlation settings:

Note: All parameters marked with a red asterisk (*) are mandatory.

  • Dayforce Correlation Attribute*: Select the Dayforce attribute from the dropdown menu to distinctly find the associated Okta user record. For example, select Employee Number.
  • Okta Correlation Attribute*: Select the Okta attribute from the dropdown menu to distinctly find the associated Dayforce user record. For example, select Employee number.

User Settings

Configure the following user settings:

Note: All parameters marked with a red asterisk (*) are mandatory.

  • Limit Import: Enter the maximum number of user records to be imported into Okta during each integration execution. Once this limit is reached, no additional records will be processed. For example, if you enter 10, only the first ten user records will be imported into Okta.
  • Source Filter: Select a filter type from the dropdown menu. Based on your selection, only employee records that satisfy the selected condition are imported. The following are the available options:
    • All Employees: Select this option to filter and import all the employees into Okta from Dayforce.
    • All Full-Time Employees: Select this option to filter and import all full-time employees into Okta from Dayforce.
    • Filter on Employee Numbers: Select this option to filter employees by their employee numbers. When selected, the following parameter is displayed:
      • Employee Numbers: Enter the user's employee numbers that must be imported into Okta from Dayforce.
    • Filter on Employment Statuses: Select this option to filter employees by their employment statuses. When selected, the following parameter is displayed:
      • Employment Statuses: Select one or more employment statuses from the dropdown menu. Only employees with these statuses will be imported into Okta from Dayforce.
    • Filter with Selected Attribute: Select this option to filter employees by their attribute value. When selected, the following parameters are displayed:
      • Attribute Name: Select an attribute from the dropdown menu by which the employees must be filtered. The available options are Employee XRef Code and Department XRef Code.
      • Operator: Select an operator from the dropdown menu. By default, the value is Equals to.
      • Values: Enter a value for the selected attribute and operator in the fields above. Only employees whose selected attribute matches the specified value will be filtered and imported into Okta from Dayforce.
    • Custom: Select this option if you want to define a condition that determines which employees are imported into Okta from Dayforce.
      • Custom Source Filter: Enter a JavaScript expression that defines which employees will be filtered and imported into Okta from Dayforce.
  • Dayforce Server Time Zone*: Enter the Dayforce server UTC offset in ±HH:MM format, such as -04:00 or +05:30. This value is used to convert dates and timestamps between UTC and Dayforce server time for imports. For example, if the value is set to -04:00, the integration treats the Dayforce server as four hours behind UTC. A Dayforce server local time of 2026-01-20T10:00:00 is interpreted as 2026-01-20T14:00:00.000Z in UTC.
  • Properties to Fetch for Dayforce Employees: Select one or more properties from the dropdown menu. Only attributes related to the selected properties are included in the integration.
  • Include Manager's Full Details: Select this checkbox to include the user's manager's full details in the integration.

Create Settings

This section enables you to define when users should be created in Okta from Dayforce.

Note: This section is displayed only when the Create option is selected in the Select User Operations to Perform parameter. All parameters marked with a red asterisk (*) are mandatory.

  • Limit Create: Select an option from the dropdown menu to decide whether to limit user creation in Okta from Dayforce. The following are the available options:
    • Yes: Select this option if you want to apply a limit on user record creation. The following parameter is displayed only when this option is selected.
      • Create Limit*: Enter the maximum number of user records that can be created in Okta from Dayforce during a single integration run. For example, if you enter 10, only the 10 user records that satisfy the condition provided in the Create Condition parameter will be updated.
    • No: Select this option if you do not want to apply a limit on user record creation. If this option is selected, all user records that satisfy the condition provided in the Create Condition parameter below will be created in Okta.
  • Create Condition: Select a condition from the dropdown menu to determine when users must be created in Okta from Dayforce. The following are the available options:
    • Always Create Employees: Select this option to create all user records in Okta from Dayforce.
    • Only Create Active Employees: Select this option to create all active user records in Okta from Dayforce.
    • Custom: Select this option to define a custom condition for creating user records in Okta from Dayforce. Only user records that satisfy this condition will be created in Okta from Dayforce.
      • Custom Create Condition*: Enter a JavaScript expression that defines the create condition.

Time Aware Onboarding

Configure the following time-based onboarding settings:

Note: All parameters marked with a red asterisk (*) are mandatory.

  • Onboarding Date Offset Type: Select an onboarding date offset type from the dropdown menu. The following are the available options:
    • Same Day: Select this option to create the user in Okta on the same day as the user's onboarding date.
    • Create Before Onboarding Date: Select this option to create the user in Okta before the user's actual onboarding date. The following parameter is displayed only when this option is selected.
      • Onboarding Offset days*: Enter the number of days before the actual onboarding date when a user should be created.
    • Create After Onboarding Date: Select this option to create the user in Okta after the user's actual onboarding date. The following parameter is displayed only when this option is selected.
      • Onboarding Offset days*: Enter the number of days after the actual onboarding date when a user should be created.
  • Attribute To Consider For Onboarding Date*: Select an attribute from the dropdown menu whose value will be used as the onboarding date. For example, if Hire Date is selected, its value is used as the user’s onboarding date.
  • Preferred Time Zone for Onboarding: Select a time zone from the dropdown menu used to evaluate the onboarding date and trigger onboarding actions. The following are the available options:
    • Dayforce Server Timezone: Select this option to use the Dayforce server timezone to determine when onboarding actions are executed. When selected, the following parameter is displayed to configure the time of day when the onboarding action should be triggered. 
      • Onboarding Time Of The Day*: Enter the time of day when onboarding actions should be triggered based on the Dayforce server time zone. By default, the value is 00:00.
    • Based on Employee Work Location: Select this option to use the employee's work location timezone to determine when onboarding actions are executed. When selected, the following parameter is displayed to configure the time of day when the onboarding action should be triggered.
      • Onboarding Time Of The Day*: Enter the time of day when onboarding actions should be triggered based on the employee work location. By default, the value is 00:00.
    • Based on Employee Home Location: Select this option to use the employee's home location timezone to determine when onboarding actions are executed. When selected, the following parameter is displayed to configure the time of day when the onboarding action should be triggered.
      • Onboarding Time Of The Day*: Enter the time of day when onboarding actions should be triggered based on the employee home location. By default, the value is 00:00.
    • Custom: Select this option to use the custom timezone to determine when onboarding actions are executed. When selected, the following parameter is displayed to configure the time of day when the onboarding action should be triggered.
      • Custom Onboarding Time Of The Day with Time Zone*: Enter the onboarding time with a UTC offset in HH:mm±HH:mm format, such as 18:00-04:00 or 09:00+05:30. Seconds are optional. The integration uses this custom time and offset to calculate the effective UTC onboarding processing time. If the value is blank or invalid, the integration falls back to the configured onboarding time and Dayforce server timezone.

WriteBack

The following sections are displayed under the WriteBack tab:

Note: This section appears only if the Enable WriteBack parameter is enabled in the Enable Features tab.

Manage Writeback

The Manage Writeback section allows you to customize writeback scenarios from Okta to Dayforce based on your specific use case.

General Settings

This section allows you to define which set of Okta users should be considered for writeback to Dayforce.

  • Application: Select a type from the dropdown menu to determine which Okta users should be considered for writeback to Dayforce. The supported values are Okta Directory Users and Custom Identity Source Users.
    Okta Directory Users: Selecting this option will include all users in the Okta directory for writeback from Okta to Dayforce.
    Custom Identity Source Users: Selecting this option includes only users from the custom application created in Okta for writeback to Dayforce. The integration identifies the application based on the ID configured in the Identity Source ID tenant attribute of the Okta connector, and only users from that application are considered for writeback.

Configure the following writeback correlation settings:

Note: All parameters marked with a red asterisk (*) are mandatory.

  • Okta Correlation Attribute*: Select the Okta attribute to distinctly find the associated Dayforce user record. For example, select roles.
  • Dayforce Correlation Attribute*: Select the Dayforce attribute to distinctly find the associated Okta user record. For example, select Login ID.

Writeback Settings

Configure the following settings:

Note: All parameters marked with a red asterisk (*) are mandatory.

    • Yes: Select this option if you want to apply a limit on user record writeback. The following parameter is displayed only when this option is selected.
      • Import Limit*: Enter the maximum number of user records to be imported into Dayforce during each integration execution. Once this limit is reached, no additional records will be processed. For example, if you enter 10, only the first ten user records will be imported into Dayforce.
    • No: Select this option if you do not want to apply a limit on user record writeback. If this option is selected, all user records that satisfy the condition provided in the Source Filter parameter will be updated in Dayforce.
  • Source Filter*: Select a filter from the dropdown menu by which the users should be filtered in Okta. The following are the available options:
    • All Users: Select this option to write back all users from Okta to Dayforce.
    • Employee Number To Fetch: Select this option to filter the employees by their employee numbers.
      • Employee Numbers: Enter one or more user employee numbers. Only the users with these employee numbers are considered for write-back from Okta to Dayforce.
    • Filter on Employment Statuses: Select this option to filter the employees by their employment statuses.
      • Employment Statuses: Enter one or more employment statuses from the dropdown menu. Only the users with these employment statuses are considered for write-back from Okta to Dayforce.
    • Filter with Selected Attribute: Select this option to filter the employees by an attribute. When selected, the following parameters are displayed for further configuration.
      • Attribute Name: Select an attribute from the dropdown menu. The available options are Employee Login and Employee Email.
      • Operator: Select an operator from the dropdown menu. The available option is Equals To.
      • Values: Enter one or more attribute values for the selected attribute in the Attribute Name parameter.
        Users with all the selected attribute values will be filtered and considered for write-back from Okta to Dayforce.
    • Custom: Select this option to define a custom condition that determines which users are selected for write-back.

FAQs

This section provides answers to the frequently asked questions related to the Dayforce / Okta Integration.

Question 1: Does Aquera support defining percentage-based threshold limits during synchronization?
Answer: Currently, the integration does not support threshold limits by percentage during synchronization. However, it does support threshold limits based on user count, which can be configured in the User Life Cycle section for create, update, and deactivate operations. For details, refer to the User Life Cycle section.

Question 2: Can Aquera generate alerts when a high volume of user records are modified within a sync cycle?
Answer: No, Aquera does not generate alerts for high-volume user record modifications. However, supports sending notification alerts based on the transaction types selected in the Notify Only When Below Transaction Happens parameter in the Notifications section. For details, refer to the Notifications section.

Question 3: Is there a way to require manual approval or confirmation before processing bulk attribute updates?
Answer: Currently, the integration does not support a manual approval or confirmation procedure before processing bulk attribute updates.

Question 4: Can we preview or simulate attribute changes before applying them?
Answer: Yes, you can generate a report that displays all the data that would be changed without actually making those changes. For this, enable the Report-Only Mode parameter in the Global Settings section.
