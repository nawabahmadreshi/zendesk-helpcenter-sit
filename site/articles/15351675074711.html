Overview

Microsoft 365 is the productivity cloud designed to help users pursue passion and run businesses. More than just apps like Word, Excel, and PowerPoint, Microsoft 365 brings together best-in-class productivity apps with powerful cloud services, device management, and advanced security in one, connected experience.

As an identity provider, Microsoft 365 lets organizations manage user logins and permissions in one place for Microsoft services, third-party apps, and custom solutions. With features like single sign-on (SSO), multi-factor authentication (MFA), conditional access, and user lifecycle management, it offers secure and easy access for users.

This guide helps users create a Microsoft 365 connector on the Aquera platform.

For further details on the Microsoft 365 application, visit the Microsoft 365 website.

NOTE 1: Follow this guide to configure the Microsoft 365 connector with OAuth2 flow. Refer to the Microsoft 365 App Registration Configuration Guide to configure the Microsoft 365 connector with Client Credentials flow.
NOTE 2: The version 21 of this connector is compatible only with version 5 of the User Integration. If you are using an older version of the User Integration, contact Aquera support team to upgrade it to version 5.

Notes:

  • Access to these resources is through the Microsoft Graph API endpoint. Microsoft Graph is a RESTful web API that enables you to access Microsoft Cloud service resources, such as Users, Groups, and Mailboxes.
  • You cannot update the attributes of a user in Entra ID if those attributes are synced from Active Directory using the Microsoft Entra Connect. You can update the attributes of a user account only if the source of those attributes is Entra ID.
  • The resource types Teams, Shared Mailbox, App Roles, Channels, and Privileged Identity Management are managed through the resource type Groups (/Groups).
  • While retrieving channels, the channel's owner will not be included in the response.
  • You can add or remove a user from a shared channel only if the userType of the user is Member, and the user must be active.
  • You can add or remove a user from a private channel if the userType of the user is Member or guest, and the user must be active.
  • You cannot add or remove users from a standard channel.
  • When you want to set up multiple Aquera Integrations in an incremental mode where Entra ID is the source/target application, make sure that you create multiple instances/endpoints of the Microsoft 365 connector to be used in different Integrations. A single instance of the connector must not be used in multiple Integrations. Otherwise, incremental imports may not work as expected.
  • It is recommended to set the frequency of incremental synchronization runs to less than 25 days. Otherwise, a full import of users will be performed instead of incremental imports.

Prerequisites

The following are the prerequisites:

  • Service Account: A service account/user account with appropriate roles is needed for authentication and configurations in the Microsoft portal. You can use the same service account or create a new service account to manage PowerShell Services as well.
    Use one of the following approaches if your use case requires managing PowerShell Services:
    When MFA is enabled When MFA is not enabled

    If MFA is enabled for the service account, you must use Certificate-Based Authentication (CBA) to manage PowerShell Services. In this case, a service account to manage PowerShell Services is not needed.

    If MFA is not enabled, a service account is needed to manage PowerShell Services, and the following tenant attributes must be configured in the connector:

      • Admin's Username
      • Admin's Password

    Note: You can also use the Certificate-Based Authentication (CBA) to manage PowerShell Services, even when MFA is not enabled. In this case, a service account to manage PowerShell Services is not needed. 
    Refer to the Service Account Setup in Microsoft 365 guide to learn how to set up a service account.
    To know more about the PowerShell features and their related attributes, refer to the Manage PowerShell Services section.
  • Application for Certificate-Based Authentication: To use app-only or certificate-based authentication, you must register an application in Microsoft Entra ID. Refer to the Application for Certificate Based Authentication section for further details.
    Note: If your use case requires managing PowerShell Services and if MFA is enabled for the service account, you must use this authentication method.
    In this case, the following tenant attributes must be configured in the connector:
      • Exchange Online Authentication Certificate
      • App ID of Registered Application
      • Domain provided by Microsoft
      • Password provided while certificate export
  • PnP Management Shell Application: If your use case requires features related to OneDrive and SharePoint, you must configure the PnP Management Shell Application in the Microsoft Azure Portal using PowerShell commands.
    In this case, you must configure the PnP Management Shell Application ID tenant attribute.
    However, if you are using Certificate-Based Authentication, you do not need to configure the PnP Management Shell application. In this case, you must configure the App ID of Registered Application tenant attribute instead of the PnP Management Shell Application ID tenant attribute in the connector to use features related to OneDrive and SharePoint.
  • User Consent Setting for API Authorization: Ensure that the user consent setting allows you to authorize the connector in the Aquera Admin portal. Refer to the User Consent Setting section for more details.

Supported Extensions

Extensions can be added to the user, group, organization, device, and application resources. Only 100 extension values across all types and all applications can be written to any single Azure AD resource.

For more details on extensions, refer to the Microsoft documentation.

The following extension types are supported:

Open Extensions

Open extensions provide an easy way to directly add untyped properties to a resource in Microsoft Graph. These are represented by the openTypeExtension resource.

Any open extension added to a resource shows up in the extensions navigation property, which is derived from the extension abstract type. Each extension has an extensionName property which is the only pre-defined, writable property for all extensions along with your custom data.

For detailed information, refer to the Microsoft documentation.

The connector supports open extensions through the tenant attribute Extended Schema. Refer to the Aquera Tenant Attributes section for more details. A maximum of two open extensions are supported for a user.

Note: If you select the create external ID checkbox in the Aquera Tenant Attributes section, it is considered an open extension attribute.

Schema Extensions

Schema extensions allow you to define a schema to extend and add strongly-typed custom data to a resource type. The custom data appears as a complex type on the extended resource.

The connector supports schema extensions through the tenant attribute Schema Extensions Custom Attribute IDs. Refer to the Aquera Tenant Attributes section for more details.
To add Schema extensions, you must have schema extension IDs.
Perform the following steps to obtain the schema extension IDs:
  1. Log in to https://developer.microsoft.com/en-us/graph/graph-explorer using your Microsoft 365 credentials.
  2. On the left menu pane, expand Extensions.
  3. Click GET available schema extensions and select the required ones, which support targetTypes User.
  4. Since it is a paged response, copy the value of the @odata.nextLink attribute available in the response and paste it into the URL section of Graph Explorer.
  5. Run the query again.
  6. Once the response is found, check for the required ID/attribute. Follow the same steps until you reach the last page where the @odata.nextLink attribute will not be present.
Note: Existing customers, who need this feature to be enabled, have to get re-authorized.

For detailed information on schema extension, refer to the Microsoft documentation.

Application Extensions

Application extensions enable you to control and access custom properties of an application resource that is registered in Azure.

The connector supports application extensions through the tenant attribute Application ID for extension properties. Refer to the Aquera Tenant Attributes section for more details.

To add or modify application extensions, you must have application IDs. Refer to the Obtaining Application ID section for more details on obtaining the application ID of your application.

These extension attributes can be added and modified only through APIs and not through the Azure application UI.

Notes:

  • Extension attributes of string data type only are supported.
  • The name of the application extension attributes must be entered in the format, extension_<applicationId>_<attributeName>. For example, extension_d038e19b0ac14ab295da0a19b7812a34_SAMPLEATTRIBUTE
  • The application ID in the extension attribute name must not contain hyphen (-). Also, it is recommended to have the extension attribute names starting with an alphabet.

Connector Configuration

Perform the following steps to configure the Microsoft 365 connector:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Applications and click Add Application.
  3. Search for Microsoft 365 and select the Microsoft 365 icon.
  4. Enter the values for the following fields under the Configuration tab:
    • Basic Details:
      • Display Name: Enter a display name for your application.
      • Description: Enter a description for your application.
      • Source Owner: Enter valid email IDs of the source owners.
      • Subscribe to error notification: Enable the toggle button to send notification emails to source owners for errors in the user data transactions. Click Edit Preference to select the error types for which you want to receive notifications. After you configure your preference, you will receive error notifications only for the selected error types. By default, all error types are selected.
      • Pick your protocol: Select the protocol from the dropdown list based on your requirement. SCIM is the default value.
      • Copy this URL: This field displays the base URL of the connector API endpoint. The URL is populated based on your selected protocol. Copy and paste this URL while provisioning to the identity service provider or performing OpenAPI operations, if supported.
      • Authorization: This is the bearer token for authentication. Refer to the Authentication section for further details.
    • Tenant Attributes: Refer to the Aquera Tenant Attributes section for details.
  5. Click Create to add the Microsoft 365 connector.

Refer to the Getting Started with Connectors guide to learn more about connector setup including different tabs of the connector configuration page.

Refer to the Test Connection for Microsoft 365 Connector guide to learn more about the Test Connection feature.

    $mycert | Export-Certificate -FilePath mycert.cer

    # get bytes from the certificate

  • Send auto-reply.
  • Include mailbox status in the response.
  • Include mailbox size in user response.
  • Include shared mailbox as groups during group import.

To allow the connector to use PowerShell, you must have a service account that has a valid license with the Exchange Online and Exchange Online Archiving plans enabled. For example, E3.
Note: If you are using Certificate-Based Authentication, a service account is not needed to manage PowerShell services.

You must have Exchange Administrator role along with SharePoint administrator role assigned to the service account to allow the connector to use PowerShell.

The Exchange Administrator role is required for Exchange related PowerShell scripts, such as conversion of mailbox and managing mail-enabled security groups.
The SharePoint Administrator role is required for SharePoint related PowerShell scripts, such as providing OneDrive access to the user's manager.

To manage PowerShell services, when MFA is enabled, you must use Certificate Based Authentication. In that case, you must configure the following tenant attributes:

  • Exchange Online Authentication Certificate
  • App Id of Registered Application
  • Domain provided by Microsoft
  • Password provided while certificate export

However, if MFA is not enabled for the service account, following tenant attributes are mandatory to manage PowerShell services:

  • Admin's Username
  • Admin's Password

Notes:

  • To allow OneDrive access to a user's manager, the service account must have the SharePoint plan enabled along with the Exchange Online and Exchange Online Archiving plans.
  • You must configure the PnP Management Shell Application ID tenant attribute if you want to use features related to OneDrive and SharePoint. On-Demand Provisioning For Dynamic Groups

When a member is assigned to or removed from a dynamic group in Azure AD, the change will be reflected in the target application automatically when the automatic provisioning is triggered in Azure AD. In order to reflect the change immediately in the target application, you must use the Provision on demand feature in Azure AD.

Note: In Azure AD dynamic groups, members are assigned based on a rule, and not manually.

The following is the procedure to perform the on-demand provisioning in Azure AD for adding or removing users in the target application group:

  1. Select your application in Azure AD, and then go to the Provisioning configuration page.
  2. Select Provision on demand.
  3. Search for the dynamic group and then select the user. To select the user:
    • If the user is removed from the group, click View all users and then search for the user account and select it.
    • If the user is assigned to the group, click View members only and then search for the user account and select it.
  4. Select Provision at the bottom of the page. After the successful on-demand provisioning of a user, a window similar to the following is displayed:

    The change will be reflected immediately in the target application.

Note: Any user assignment or removal in the target application groups will not be synced back to Azure AD.

Configure Azure AD certificate-based authentication

Users can sign in to the Microsoft Azure portal using one of the following options:

  • Using the username and password of an administrator.
  • Using an x.509 certificate.

Azure Active Directory (Azure AD) certificate-based authentication (CBA) enables organizations to allow users to authenticate with X.509 certificates created by their Enterprise Public Key Infrastructure (PKI) for applications and browser sign-in. This feature enables organizations to adopt phishing-resistant modern passwordless authentication by using an x.509 certificate. 

The x.509 certificate is validated against the user account and if successful, they sign in.

Perform the following steps to configure certification authority (CA) in the Microsoft Azure portal:

  1. Sign in to the Microsoft Azure portal as an administrator.
  2. Navigate to Azure Active Directory > Security > Certificate authorities.
  3. Click Upload. Upload a .cer file.
  4. Select Yes if the CA is a root certificate, otherwise, select No.
  5. Set the HTTP internet-facing Certificate Revocation List URL and Delta Certificate Revocation List URL. Click Add.

Perform the following steps to enable the certificate-based authentication (CBA) in the Microsoft Azure portal:

  1. Sign in to the Microsoft Azure portal as an Authentication Policy Administrator.
  2. Navigate to Azure Active Directory > Security > Authentication methods > Policies.
  3. Select the authentication method as Certificate-based authentication.
  4. Toggle to turn on the Enable button.
  5. Include Target by clicking All users, or Select groups to select specific groups. Click Save.

The users who are enabled for certificate-based authentication will be able to sign in using the X.509 certificate. 

For more information on certificate-based authentication, refer to the Microsoft documentation.

FAQs

Question 1: How does the connector determine the active status of a user?

Answer: The active status of a user is determined based on the Account enabled setting on the user's profile page. If this setting is enabled for a user, the user will be considered active.
Note: If you are using an older version of the Microsoft 365 application where the Account enabled setting is not available, the active status of a user is determined based on the Block sign in setting on the user's profile page. If this setting is disabled for a user, the user will be considered active.

Question 2: Are the behaviour of IAN attributes and their corresponding tenant attributes the same?

Answer: The behaviour of IAN attributes and their corresponding tenant attributes are different. The IAN attribute, such as shareMailboxWithManager can be used for active or inactive users. However, the corresponding tenant attribute will work only if a user is deactivated.

In other words, the IAN attributes are independent of active/inactive, whereas the user must be deactivated for the tenant attributes to work.

Question 3: Can I whitelist an IP or a range of IPs of a server to allow service account login only from the whitelisted IPs?

Answer: Yes, you can whitelist the IPs of the server where the PS scripts are running. This will allow service account login only from the whitelisted IPs.

Question 4: How do you verify that the PowerShell commands are using certificate details or the service account?

Answer: If you have provided only service account details, the PowerShell commands use service account only. If both service account details and certificate-based authentication details are provided, you can find out which credentials are used with the help of Microsoft Entra ID audit logs.

Question 5: How to validate the basic permissions while using certificate-based authentication for PowerShell related activities?

Performing OpenAPI Operation

Using an API Tool

You can perform OpenAPI operations using an OpenAPI tool as follows:

  1. Copy the OpenAPI URL from Copy this URL field on the connector configuration page. 
  2. Add /invoke at the end of the copied URL.
  3. Paste the URL in an API tool.
  4. Provide the required credentials for authentication.
  5. Pass the operation ID, parameters, and properties in the request body.
    Note: The parameters/properties vary depending on the type of operation.
  6. Run the URL using the POST method.

The following is the format to be passed in the request body:

{
"operationId": <<Operation ID>>,
"parameters": {<<parameter name>>: <<parameter value>>},
"requestBody": {<<property name>>: <<property value>>}
}

Using Workflows

You can perform OpenAPI operations using Workflows. Contact the Aquera support team for configuring Workflows.

Using Scripts

You can perform OpenAPI operations using Scripts. Refer to the Integration Using Scripts guide for details on how to set up an integration using scripts.

OpenAPI operations can be invoked using the following format:

<AppVariableName>.<OpenAPI_MethodName>({parameterName: parameterValue})

Here, <AppVariableName> is the name of the variable that you have configured in the script to refer to the provisioned connector, and the <OpenAPI_MethodName> is the name of the OpenAPI method to invoke the corresponding OpenAPI operation. 

Provide a comma-separated list of parameter names with parameter values enclosed in curly braces.

For further details on how to invoke OpenAPI methods using scripts, refer to the Working With Integration Scripts guide.
