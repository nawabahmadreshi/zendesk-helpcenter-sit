Workflow User Guide

Overview

Aquera provides a solution to set up a Workflow of tasks to be performed on the data across applications without writing scripts. The data can be users, groups, or tickets based on the application's supported resource types. You can perform various tasks on the data, including reading, creating, updating, deleting, and performing general tasks such as sending emails and creating service requests.

For example, you can set up a Workflow to achieve an automatic IT onboarding process for a user from an HR system to multiple applications.

This guide explains how to create, configure, and execute a Workflow. The Workflow internally creates JavaScript code based on the configurations. When the Workflow runs, this script gets executed. You can set the Workflow to be triggered by an event, schedule, form, or another Workflow.

Prerequisites

The following are the prerequisites:

  • The user should have an Owner, Superadmin, Admin, or Synchronization Editor role to create, edit, delete, and execute Workflows.
    Notes:
    • Users with Readonly, Application Administrator, or Basic roles will not be able to view the Workflows module in the Aquera Admin portal.
  • Configure the required connectors before creating a Workflow in the Aquera Admin portal. Refer to the respective connector configuration guides for further details.

Create Workflow

You can create a Workflow using one of the following options:

  • Create Workflow using a Template
  • Create your own Workflow
  • Create Workflow by Importing an Existing Workflow

Create Workflow using a Template

Templates are pre-built and configurable Workflow for common identity automation use cases. You can use the templates as-is or modify them to match your organization's needs.

Perform the following steps to view the Workflow templates:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Workflow on the left navigation panel and click Create Workflow at the top-right corner of the page. This page displays the Workflow templates.

You can filter the templates based on the following categories:

  • Application Filter: In the Applications dropdown list, select the applications of your choice, then the Create Workflow page displays the Workflow templates that use the selected applications. 
  • Use Case Filter: You can search templates based on your use case by providing the use case in the Search box.

Perform the following steps to create a Workflow using a template: 

  1. You can select a template from the Create Workflow page based on your requirements. 
    For this use case, we are selecting the template Create ticket for each new hire in HR System. This template retrieves new users from an HR system whose hire date is today, then creates a ticket for each user in an ITSM system, and sends email notifications to the respective users.
  2. Enter a name and a description for the Workflow in the Create New Workflow window. Ensure to provide a unique Workflow name. 
  3. For this use case, select an HRIS connector and an ITSM connector in the APPLICATION fields. The application variable names are pre-populated in the NAME field by default. If required, you can modify the application variable names. These application variable names are used in the Workflow to refer to the respective connectors.
  4. Click Continue to set up the Workflow.
    The following Workflow is displayed according to the selected template. The template automatically adds all the required elements and prefills the required fields. You can then modify the Workflow by adding more elements and configuring the fields as per your requirement. 
  5. Click Create at the top right corner of the Workflow page to create the Workflow.

Create your own Workflow

Perform the following steps to create your own Workflow:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Workflow and click Create Workflow at the top-right corner of the page.
  3. Click Blank to create a new Workflow.
  4. Enter a name and a description for the Workflow in the Create New Workflow window. Ensure to provide a unique Workflow name.
  5. You can also click the + Add Application button to add an application to be used in the Workflow. For adding an application, select the required connector in the APPLICATION field and enter a variable name for the application in the NAME field to refer to the connector in the Workflow. 
    Note: For details on the naming convention for the application variable names, refer to the General Rules for Names section.
  6. Click Continue to set up the Workflow. You can then add elements to the Workflow and configure the fields as per your requirements.
  7. Click Create at the top right corner of the Workflow page to create the Workflow.

Create Workflow by Importing an Existing Workflow

Perform the following steps to create a Workflow by importing an existing Workflow:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Workflow and click Create Workflow at the top-right corner of the page.
  3. Click Import to import an existing Workflow.
  4. Select the JSON file of the Workflow that you would like to import.
    Note: To export the JSON file of a Workflow, navigate to Workflow, click the vertical ellipsis of the Workflow in the list or card view, and click Export.
  5. The Workflow name and description are pre-populated by default in the Create New Workflow window. If required, you can update the Workflow name and description. Ensure to provide a unique Workflow name.
  6. Select the required connector in the APPLICATION field. The application variable names are pre-populated in the NAME field by default. If required, you can modify the application variable names. These application variable names are used in the Workflow to refer to the respective connectors. 
    Note: For details on the naming convention for the application variable names, refer to the General Rules for Names section.
  7. Click Continue to set up the Workflow. The Workflow is imported successfully and then if required, you can modify the Workflow by adding more elements and configuring the fields as per your requirement.
  8. Click Create at the top right corner of the Workflow page to create the Workflow.

Workflows Listing Page

When you navigate to the Workflows module, you can view the list of all Workflows created by the users in the tenant. You can search for a Workflow from the search bar. The following are the types of views supported for listing the Workflows:

  1. List View: The Workflows are listed in the form of a list of rows.


  2. Card View: The Workflows are listed in the form of cards.

The List and Card views display the following details:

  • Execution Icon: The icon to start or stop a Workflow execution. You can click the play icon to start a Workflow execution. You can click the stop icon to stop a Workflow execution that is already running.
  • Toggle Button: The state of the Workflow indicating enabled or disabled. You can turn on the toggle button to enable the Workflow so that the Workflow can be executed. You can turn off the toggle button to disable the Workflow so that the Workflow can never be executed.
  • NAME: The name of the Workflow. Click on the Workflow name to navigate to the corresponding Workflow.
  • TRIGGER: The trigger state of the Workflow:
    • For scheduled triggers, it indicates the schedule of the Workflow execution.
    • For event triggers, it indicates if it will be triggered by a Synchronization event, Application event, or Form event.
    • For Workflow triggers, it indicates that it will be triggered by Another Workflow.
    • When no trigger is set for a Workflow, it indicates that the trigger is not set.
  • APPLICATIONS: The applications configured in the Workflow.
  • HISTORY: The execution history of the recent five Workflow runs in the form of color bars. The right-most color bar represents the most recent Workflow run. The following are the supported colors:
    • Green: Indicates a successful execution.
    • Red: Indicates a failure in execution.
    • Blue: Indicates execution in progress.
    • Grey: Indicates a canceled execution or not executed.
  • AUTHOR: The email address of the Workflow's creator.
  • DATE CREATED: The creation date of the Workflow.
  • Vertical Ellipsis: Click the vertical ellipsis that displays the following options:
    • Duplicate: Click Duplicate to create a copy of the Workflow.
    • Export: Click Export to download the JSON file of the Workflow. This JSON file can later used to migrate a Workflow from one tenant to another tenant using the Import option. 
    • Delete: Click Delete to delete the Workflow.

Sorting Workflows

You can sort Workflows in the list and card view pages. By default, Workflows are sorted based on the Workflow names in ascending order. You can sort Workflows in the list and card view pages either based on the Workflow name or the Date Created. 

To sort based on the Workflow names, click the up arrow icon or the down arrow icon next to the NAME column to sort Workflow names in descending or ascending order respectively.

To sort based on the Date Created, click the up arrow icon or the down arrow icon next to the DATE CREATED column to sort Workflows based on the latest or the earliest creation dates respectively.

Workflow Editor

The following are the sections in a Workflow editor:

  • Components Panel
  • Workflow Canvas
  • Configuration Panel

Locate Steps

The Locate Steps option displays all the steps in the Workflow for easy navigation. You can click a step name to navigate to the respective step in the Workflow. You can also search for a step name in the search bar and click the step name to jump to the respective step in the Workflow.

Components Panel

The Components panel on the left of the Workflow page consists of items to set up a sequence of tasks in the Workflow. You can search for an operator, helper, or application name in the search bar.

The following are the sections in the Components Panel:

The Operators section has the following items that control the flow of tasks in the Workflow:

Item

Description

If-else

Performs a set of tasks when the condition is true and performs a different set of tasks when the condition is false.

Cases

Performs different sets of tasks based on different values of an attribute or expression evaluation.

For-each

Performs a set of tasks for an array of objects.

While

Performs a set of tasks as long as a specific condition is true.

The Helpers section has the following items that help accomplish Workflow tasks:

Item

Description

Prints a message in the log when the Workflow runs.

Sets values to variables in the Workflow.

Object 

Sets values to objects in the Workflow.

Pauses the Workflow execution for the specified seconds.

Stores data that can be used across multiple Workflows in the same tenant.

The Invoke section has the following items to invoke forms, Workflows, or Synchronizations:

Description

Applications

The Applications section displays the connectors you have added to the Workflow. You can drag and drop applications from the Applications section to the Workflow to perform the required operations (SCIM or OpenAPI) supported by the connector.

You can click + Add Application to add an application to be used in the Workflow. Then, select the required connector from the Application field and enter a variable name for the connector in the Name field to refer to the connector in the Workflow. Click Add to include the connector in the Applications section.

Workflow Canvas

Workflow Canvas is where you create the Workflow to set up a sequence of tasks. The Workflow consists of elements that are connected using arrows.

Elements

An element is an individual task in the Workflow. 

Configuration Panel

The configuration panel appears to the right of the Workflow page when you click an element. This panel lets you configure an element. Refer to the Workflow Elements Configuration Guide guide for further details. 

Workflow Trigger

Workflows can be triggered based on a schedule, an event, or another Workflow. Navigate to the Start Workflow configuration panel > Trigger section to set a trigger for the Workflow.

Select one of the following options from the Triggered By dropdown to trigger the Workflow:

  • Schedule: The Workflow can be triggered at specific intervals based on the schedule.
  • Event: The Workflow can be triggered when specific events occur in an application such as user data changes.
  • Another Workflow: The Workflow can be triggered (invoked) by another Workflow.

Set a Schedule Trigger

You can set a schedule to run the Workflow at specific intervals. Select a Basic schedule or an Advanced schedule depending on the frequency by which the Workflow should be run.

Basic Schedule

You can set a basic schedule based on the following frequencies:

  • 15 minutes
  • 30 minutes
  • 1 hour
  • 4 hours
  • 6 hours
  • 12 hours
  • 24 hours

Note: If you select Never, the Workflow never runs automatically. You can manually run the Workflow when required by selecting the Execute button at the top right of the Workflow page.

Advanced Schedule

You can set an advanced schedule based on the following frequencies:

  • Never: The Workflow never runs automatically.
  • Continuous: Select Continuous to run the Workflow continuously. 
    Note: There can be a delay of a maximum of 5 minutes to start the next instance of the Workflow after the previous instance ends.
  • Minutes: Select Minutes to schedule the Workflow to run every X minutes, where you should select X minutes from the Repeat Every dropdown list. The available options are from 15 to 60 minutes.
  • Hourly: Select Hourly to schedule the Workflow to run every X hours, where you should select X hours from the Repeat Every dropdown list. The available options are from 1 to 24 hours.
  • Daily: Select Daily to schedule the Workflow to run every X days at a specific time, where you should select the day frequency from the Repeat Every dropdown list and select the time from the Starting at(Time) dropdown list. You can set up a maximum of 8 daily schedules. The available options for the days are from 1 to 6. The Starting at(Time) that you select here is the UTC time.
  • Weekly: Select Weekly to schedule the Workflow to run on specific days of the week at a specific time, where you should select the days in the Repeat Every field and select the time from the Starting at dropdown list. The Starting at time that you select here is the UTC time.

Advanced setting: You can set the start and end dates during which the schedule will be run. You can select a date using the calendar option in the Schedule Start Date and Schedule End Date fields.

Set an Event Trigger

You can configure the Workflow to run when specific events occur in an application. An event will be triggered when there is a data change in an application such as a new user creation, existing user modification, or user deactivation. The application or the synchronization sends the events that trigger the Workflow.

The following are the fields to be configured for event-based Workflow:

  • Source Type: Select the source type such as Synchronization / Application / Form from which an event will be sent. For details on Synchronization events, Application events, and Form events, refer to the Events section.
  • Source Name: Select the name of the source that sends the events which can be the name of the synchronization, connector, or form.
  • Event Name: Select the name of the event from the dropdown list. For the supported event names, refer to the Events section. For details on the request object, refer to the Accessing Event Request in Workflows guide.
  • Schedule Event Execution: Set a schedule for the execution of the event trigger. The available options are the same as the Advanced Schedule.
  • Advanced Setting: You can set the start and end dates during which the events will be triggered. You can select a date using the calendar option in the Schedule Start Date and Schedule End Date fields.

Events

The following are the supported events:

Synchronization Events

The Workflow is triggered when a User Synchronization creates, updates, or deactivates a user in the target application configured in the User Synchronization.

The following are the supported synchronization events:

  • User Created by Synchronization: The Workflow is triggered when a user is created in the target application configured in a User Synchronization. The Workflow is triggered for each target user creation and the Workflow receives a request object of the created user.
    To receive the event notification, configure the New User Synchronization Hook in the User Synchronization. 

  • User Updated by Synchronization: The Workflow is triggered when a user is updated in the target application configured in a User Synchronization. The Workflow is triggered for each target user update and the Workflow receives a request object of the updated user.
    To receive the event notification, configure the Update User Synchronization Hook in the User Synchronization.

  • User Deactivated by Synchronization: The Workflow is triggered when a user is deactivated in the target application configured in a User Synchronization. The Workflow is triggered for each target user deactivation and the Workflow receives a request object of the deactivated user.
    To receive the event notification, configure the Deprovision User Synchronization Hook in the User Synchronization.

Application Events

The Workflow is triggered when there is a data change in an application. A list of application events is shown in the Event Name field if the connector selected in the Source Name field supports the Events resource type. For example, ADP Enterprise HR is one of the connectors that supports the Events resource type.

When the selected event occurs in the application, the Workflow is triggered and receives a request object of the event.

Note: You may have to configure the events for which the Workflow needs to be triggered in the respective application's UI. Refer to the respective connector configuration guides for details on the events supported by an application.

Workflow also supports the following general Application Event:

  • Users Modified Since Event: The Workflow is triggered when there is a user modification in an application. The Users Modified Since option will be shown in the Event Name field only if you have selected Application as the Source Type and the connector selected in the Source Name supports incremental import of users.
    For this event to be listed in the Workflow, the connector should support filtering users by the meta.lastModified attribute. Refer to the respective connector configuration guides for further details.
    The Workflow is triggered for each user modified after the meta.lastModified date and receives a request object of the modified user.

Form Events

A Form Event is an event received by a Workflow when a form recipient clicks on an action button in the form.

You can create and configure a form in a Workflow to be sent to form recipients. The form recipients receive an Aquera form URL to their email addresses and the recipient can fill in the form details and click on an action button in the form. In the create form Workflow, the action button on the form is mapped to a Workflow that processes the form data.

When the form recipient clicks on the action button on the form, the respective Workflow that is mapped to the action button is triggered. The triggered Workflow receives a request object containing the details filled in by the form recipient.

Multiple Events

Workflow supports multiple events. You can either add multiple Synchronization Events or add multiple Application Events of the same application in a Workflow.

When you configure multiple Synchronization Events in the Workflow, the same Workflow is triggered for those specific configured Synchronization Events. For example, if you have configured the User Created by Synchronization event and the User Updated by Synchronization event, then the Workflow is triggered whenever a new user is created or an existing user is updated in the target application of the User Synchronization. The Workflow is triggered for each target user created or updated and the Workflow receives a request object of that user. You need to configure the Synchronization Hooks in the User Synchronization to receive the event notifications. 

 

When you configure multiple application events in the Workflow, the same Workflow is triggered for those specific configured application events. For example, if you have configured Okta application events user.account.update_profile and user.lifecycle.activate and also configured general application event Users Modified Since then, the Workflow is triggered when a user's profile is updated, a user is activated, or a user is modified in the Okta application. The Workflow is triggered for each application event and the Workflow receives a request object of that event.

You can enable the Condition checkbox to write conditions for Application events. Only when the condition is satisfied, the Workflow will be triggered. Set another Workflow as the Trigger

You can trigger (invoke) the current Workflow from another Workflow using the Invoke Workflow element. For example, suppose you have two Workflows named Workflow A and Workflow B and for your use case, you need to trigger (invoke) Workflow B from Workflow A. Then, set the Triggered By option in Workflow B as Another Workflow. Next, in Workflow A, add and configure the Invoke Workflow element, and in the Select Workflow field select Workflow B to invoke it. You can set a Schedule Trigger for Workflow A. Then, when Workflow A runs on a scheduled basis, Workflow A triggers (invokes) Workflow B.

Workflow Definition

You can define input parameters, a return, and variables for a Workflow.

  • Input Parameters: Input Parameters are the inputs to the current Workflow passed from another Workflow or Synchronization.
  • Return: Return specifies the return value from the current Workflow.
  • Variables: Variables store values that can be used within the current Workflow.

Perform the following steps to define a new input parameter, a return, or a variable in a Workflow:

  1. Navigate to the Start Workflow configuration panel > Definition section.
  2. Click + New Parameter /+ New Return / + New Variable to define a new input parameter / new return / new variable respectively.
  3. Provide a name for the new input parameter / new return / new variable.
  4. Select a data type in the Type field. The supported data types are Boolean, Number, String, and Object.
  5. Enter an initial value in the Initialization field if required.
    • For Boolean, Number, and String types, you can provide a default value for the input parameter / return / variable in the Initialization field. You can also write JavaScript code by clicking the Switch to Expression icon.
    • For Object type, you can define the object by importing an existing schema or defining a new schema. For further details, refer to the following sections:
      • Import Schema
      • Define New
  6. After providing the required details, click Add to add the new input parameter / new return / new variable.
  7. Click Save Changes at the top-right corner of the Workflow page to save the changes to the Workflow. 

Perform the following steps to modify an input parameter, a return, or a variable in a Workflow:

  1. Navigate to the Start Workflow configuration panel > Definition section.
  2. Click the vertical ellipsis next to the name.
  3. Click Edit to modify the input parameter / return / variable.
  4. After making the required modifications, click Update to update the new changes.
  5. Click Save Changes at the top-right corner of the Workflow page to save the changes to the Workflow. 

 Notes:

  • In the Trigger section, if the Triggered By option is selected as Schedule, there will be no parameters and return value.

  • In the Trigger section, if the Triggered By option is selected as Event, the request object ($request) will be the input parameter and there will be no return value. You can click Show Attributes to view all the attributes of the request object.

  • For details on the naming convention for the parameter names, return names, and variable names, refer to the General Rules for Names section.

  • You can add any number of parameters and variables to the Workflow. However, a Workflow can have only one return.

  • You cannot modify/delete the attributes of the application schema on the Select Existing Schema page, the Define New Schema page, and the Edit Schema page.

Import Schema

Perform the following steps to define the object by importing an existing schema from an application:

  1. Navigate to the Start Workflow configuration panel > Definition section.
  2. Provide a name for the new input parameter / new return / new variable.
  3. Select Object data type in the Type field.
  4. Click Import Schema under the Object Definition.
  5. Click a resource object under the APPLICATION OBJECTS to select a schema to be used for the object definition.
    Note: If you have not added any applications in the Applications section of the Workflow, you cannot select any existing schema. However, you can define a new schema for the object.
  6. Click the Use button to select the schema.
  7. Click Add to save the object definition.
  8. Click Save Changes at the top-right corner of the Workflow page to save the changes to the Workflow.

Perform the following steps to change the imported schema to a new schema:

  1. Navigate to the Start Workflow configuration panel > Definition section.
  2. Click the vertical ellipsis next to the Object name.
  3. Click Edit to modify the input parameter / return / variable.
  4. Click Edit Schema under the Object Definition to change the schema.
  5. Click a resource object under the APPLICATION OBJECTS to select a new schema to be used for the object definition.
  6. Click the Use button to select a new schema.
  7. Click Update to update the new changes to the object definition.
  8. Click Save Changes at the top-right corner of the Workflow page to save the changes to the Workflow. 

Define New

Perform the following steps to define a new schema for the object:

  1. Navigate to the Start Workflow configuration panel > Definition section.
  2. Provide a name for the new input parameter / new return / new variable.
  3. Select Object data type in the Type field.
  4. Click Define New under the Object Definition to define your new schema.
  5. On the Define New Schema page, you can add attributes to create your schema. Click Add Attribute and select one of the following types: Boolean, String, Object, Number, Predefined, or resource objects of applications.
  6. Provide a name for the attribute. For details on the naming convention for the attribute names, refer to the General Rules for Names section.
  7. You can select the List checkbox to specify an attribute as an array of objects. For example, you can set emails, phone numbers, addresses, or roles as a List.
  8. You can select the Mandatory checkbox to specify an attribute as required.
  9. You can select the permission for an attribute as Read Only, Write Only, and Read & Write.
    Note: If an attribute is selected as Read Only, you cannot set a value for this attribute in the Set Object.
  10. For an attribute of type object, click the Add Attribute in the object row to add sub-attributes. For example, you can define givenName and familyName as sub-attributes for the name attribute.
  11. For an attribute of type predefined, click the Set Value to set predefined values. Provide each predefined value and press enter. Then, click Done. For example, you can define a type sub-attribute for emails and set work and home as predefined values.
  12. After adding or modifying the attributes for the object schema, click Define and Use.
  13. Click Add to save the object definition.
  14. Click Save Changes at the top-right corner of the Workflow page to save the changes to the Workflow.

Perform the following steps to modify the schema:

  1. Navigate to the Start Workflow configuration panel > Definition section.
  2. Click the vertical ellipsis next to the Object name.
  3. Click Edit to modify the input parameter / return / variable.
  4. Click Edit Schema under the Object Definition to change the schema.
  5. In the Edit Schema page, you can add, modify, or delete the attributes of the schema.
  6. After making the required changes for the schema, click Define and Use.
  7. Click Update to update the new changes to the object definition.
  8. Click Save Changes at the top-right corner of the Workflow page to save the changes to the Workflow. 

Tables in Workflow

You can add tables in a Workflow that define a name-value pair to create custom attributes and use it in Workflows.

Perform the following steps to add a table in a Workflow:

  1. Navigate to the Start Workflow configuration panel > Tables section.
  2. Click + New Table to create a new table.
  3. Provide a table name and description.
    Note: For details on the naming convention for the table names, refer to the General Rules for Names section.
  4. Each row in the table corresponds to a key-name and key-value pair. You can add rows to the table using one of the following options:
    • Add Row: Click Add Row to manually add a row by providing a key-name and a key-value.
    • Import CSV: Click Import CSV to upload rows to the table using a CSV file. The CSV file should have 2 columns with headings NAME and VALUE.
      Notes:
      • The key-name within a table should be unique. The key-name can include a combination of letters and numbers.
      • The key-name supports all the special characters except the following:
        • / (forward slash)
        • \ (backslash)
        • " " (double quotes)

  5. Click Add at the bottom of the panel to create the table.

  6. Click Save Changes at the top-right corner of the Workflow page to save the table to the Workflow.

The following is a sample table:

You can perform the following actions on tables by clicking the vertical ellipsis displayed next to a table entry in the Tables section:

  • Edit: Click Edit to modify a table. After you make changes to the table, click Update at the bottom of the panel to update the table. Ensure to also click Save Changes at the top-right corner of the Workflow page to save the table to the Workflow.
  • Export: Click Export to export a table. This will download the table in a CSV file format.
  • Delete: Click Delete to delete a table.

A Table can be accessed in a Workflow using the following formats:

  • To specify the key name defined in a table, enclose it within double quotes:
    <table_name>["<key_name>"]
    Example: CountryCode["United States"]
  • To specify a variable name or an application's schema attribute name as the key name, do not enclose it in double quotes:
    <table_name>[<key_name>]
    Example: CountryCode[<userObject>.addresses[work].country]

Note: Auto suggestions for the key names of tables are supported in the Expressions Editor. For example, you can access using CountryCode.Germany. Auto suggestions will not work when the key names contain spaces and special characters.

Setup the Workflow

To set up a sequence of tasks in the Workflow, you should add elements and configure the elements in the Workflow.

You can perform the following actions to set up a Workflow:

Add an Element

To add an element to the Workflow, drag the required item from the Components section and drop the item on the + icon of the Workflow.                     

For example, as shown in the following image, drag the Log item from the Components section, hover the mouse over the + icon, and drop the Log item when a blue line appears over the + icon.

Configure an Element

To configure an element in the Workflow, click on the element that will open the configuration panel on the right of the Workflow page where you can provide the required configurations. After configuring the element, you can click the collapse icon to close the configuration panel.

The fields marked with a red asterisk (*) are mandatory. If you do not configure the mandatory fields for an element, an alert icon will be shown on the element and the message Incomplete Configuration will be displayed when you hover the mouse pointer over the element.

For further details on the different element configurations, refer to the Workflow Elements Configuration Guide guide.

Delete an Element

To delete an element in the Workflow, hover the mouse over the element and click the Delete icon. A Delete Element confirmation message will pop up asking you to confirm deleting the element.

If you are deleting an operator element, a Delete Element confirmation message will pop up asking you to confirm deleting the element and its child elements.

If you are deleting an element where the output variable name is used in the elements below it, the configuration panel of the dependent elements will show an error message Referred variables do not exist.

Drag and Drop an Element within the Workflow

You can drag and drop existing elements within the Workflow. Drag the required element in the Workflow, hover the mouse on the + icon, and drop the element when a blue line appears over the + icon.

Reset the Zoom level of the Workflow Canvas

You can click the + or - buttons at the bottom of the Workflow canvas to zoom in and out of the Workflow. You can also drag the Workflow on the Workflow canvas to navigate to an element. You can click the Reset button to bring the Workflow back to its original position and size. 

User Actions on the Workflow

The following are the UI elements to perform user actions on Workflows:

  • Execute: Click Execute at the top-right corner of the Workflow page to manually run the Workflow.
  • Bell Notification Icon: Click the  icon to view a list of configuration panel notifications for which you have not configured the mandatory configuration fields. Click an item from the list to display the respective configuration panel.

  • Vertical Ellipsis: Click the vertical ellipsis at the top-right corner of the Workflow page to view the following options:

    • Settings: Click Settings to display the Settings Configuration panel. The panel lists options to set the log level and execution mode of the Workflow.

      The following are the supported global log levels: 
      Log Level Description
      Default

      Prints the messages of the Print Log elements in the Workflow logs.
      Info

      Prints the following in the Workflow logs:

      • The messages of the Print Log elements.
      • Each step of the Workflow execution.
      Verbose

      Prints the following in the Workflow logs:

      • The messages of the Print Log elements.
      • Each step of the Workflow execution.
      • The input and output values of each element.
      Note: The messages of the Logs section of each element are printed in the Workflow logs based on the Log Level setting of the element. For further details, refer to the section Configure Log Levels for Elements. 

      The following are the supported execution modes: 
      Execution Mode Description
      Normal If any step in the Workflow execution fails, it prints the error messages in the logs and the Workflow execution continues.
      Exception If any step in the Workflow execution fails, it prints the error messages in the logs and the Workflow execution stops.          
    • Latest Logs: Click Latest Logs to view the logs of the most recent execution of the Workflow on the log pane. Click Goto Logs in the log pane to navigate to the Logs module.
      Notes:
      • Workflow logs will have the name FlowBuilder in the NAME column of the Logs module.
      • The RESPONSE DATA column of the Workflow logs indicates each step of the Workflow execution. This can be observed when the Workflow takes a longer time to execute. 

    • Delete: Click Delete to delete the Workflow. 

Appendix

General Rules for Names

Follow these rules for constructing names for variables, parameters, and returns: 

  • The name should be unique within the Workflows module - across application variable names, and the variable names, input parameter names, and return names defined in Workflows.  
  • Only letters, digits, underscores, and dollar signs are permitted in names. Any other special characters are not allowed.
  • Names are case-sensitive.
  • A letter, an underscore (_), or a dollar sign ($) must be the first character in a name.
  • Spaces are not allowed in names.
  • JavaScript keywords and reserved words should not be used as names. 
  • The following Aquera's reserved keywords should not be used as names:
    IAN, ENT, GIAN, currentStatus, statusCode, statusMessage, now, and today.
