Overview

This user guide explains how you can set up a Synchronization orchestration in Aquera Admin.

In Synchronization orchestration, every user record that is newly created or modified in the source application is updated to the target application using a PUT operation, as per the defined orchestration schedule.

Setting Up the Synchronization Orchestration

This procedure describes how to set up the Synchronization orchestration. To explain the process, we consider BambooHR as the Source application and Okta as the Target application. 

Prerequisites

Make sure that the source (BambooHR) and target (Okta) applications are added and configured correctly in Aquera Admin, before creating the orchestration. Refer to the corresponding configuration guides for details.

Creating the Orchestration

  1. Log in to the Aquera Admin console.
  2. Navigate to Orchestrations and click Create Orchestration. The Add Orchestration page is
    displayed. 
  3. Enter a Name for the orchestration, for example, BambooHR_Okta.
  4. Select Scheduled tab 
  5. Select Synchronization 

  6. Click Done. The Add Orchestration page is displayed with the fields to configure the orchestration. Also, additional tabs are displayed on the page.

The following procedures are involved in the Synchronization orchestration setup:

  • Configuration
  • Mapping of Attributes
  • Including Source Attribute Values Using Tables
  • Setting Group Rules
    • Using Basic Condition
    • Using Expression
  • Scheduling the Orchestration Run

Mapping of Attributes

  1. Click the Attribute Map tab.

    The default mapping of SCIM attributes from source to target application is displayed on the page. The source attributes are listed on the left side and the target attributes on the right side of the page.

    The mapping simplifies the process of executing Create, Update, and Delete operations, as the changes made to a source attribute will automatically be reflected in the mapped target attribute when the orchestration runs. The source attribute has to be edited in accordance with the target attribute. 

    The following actions can be performed in this tab, as per the requirements:

    • Search for a user from the source application by entering the userId or userName attribute value to get a preview of the values that are mapped to the target attributes.

      The user data is loaded corresponding to each source attribute. This can be done to ensure that the mapping is done in such a way that the values from the source are mapped to the right target SCIM attributes. 

    • Search and filter Core, Enterprise, and custom (IAN) attributes from the target attributes list by the attribute name. 

    • Set the right mapping option for a source attribute.

      You can specify if the mapping of an attribute is to be done for create and update operations, for create operation only, or for update operation only. You can also choose not to map a source SCIM attribute at all.

    • Click Add Row to Map at the end of the attribute mapping list to add a new row to set an attribute mapping.
  2. Click Save Changes after the source to target attribute mapping is completed.

Using Expressions in Attribute Map

When you set the attribute mapping, you can also write expressions to perform few operations on the attribute values from the source system.  These expressions can contain variables, attribute names, methods, and functions. Aquera provides few inbuilt methods and functions that can be used in the expressions. 

You can either write the expression directly in the source attribute field or click the Expand icon in the field to write a long expression.

The Edit Expression window is displayed where you can write a long expression.

When the orchestration runs, the methods and functions in the expressions are performed on that source attribute's values and the resulting values are updated in the target application.

For details on the keywords, inbuilt methods and functions provided by Aquera, refer to Orchestration - Expressions.

Including Source Attribute Values Using Tables

Tables can be used to create custom source attributes, and POST (sync) them to the target application. Tables are used when a source attribute is empty or a particular data is not sent from the source, and the user wants to update a value to the corresponding target attribute.

Scenario 1

If the hire date of employees is not available in the source attribute and the user wants the hire date to be posted in the target application for all those employees, you can add a table containing the hire date value in the orchestration setup.

  1. Click the Tables tab. The following page is displayed.

  2. Click Add Table. The Add Table window is displayed.

  3. Enter the table name and description as shown in the above screenshot.

    The table inputs can be entered either manually as rows or by uploading a CSV file.

    • To enter manually, the user can click Add Row. 
    • To upload the table values using a CSV file, click Upload CSV File. The orchestration supports only .csv file format and the file should have only two columns.
      A sample CSV file is attached for your reference. You may edit this file to add the required details and upload it.

    In this scenario, we add a table entry by clicking Add Row. The hireDate table is created with the column name as empHireDate and the hire date value as 2020-09-09.

  4. Click Add. The new table is listed on the page.

  5. Click Save Changes.

If you want the hire date of all employees to be updated with the value from the table, the expression for the source attribute in the Attribute Map must be given in the format,
TABLE.<table_name>[<column name>]

TABLE.hireDate[empHireDate]

In this case, the hire date of all employees will be updated in the target system with the same date mentioned in the table.

Scenario 2

If the hire date of a particular employee is not available in the source attribute and the user wants the hire date to be posted in the target application, you can add a table in the orchestration setup containing the unique employee number and the corresponding hire date.

Setting Group Rules

You can setup rules to manage groups in the target system. The rules are applied to the groups set in the Group Management Restriction List field on the Configuration tab.

  1. Click the Group Rules tab.

  2. Click Add Rule.The Add Rule pop-up window is displayed.

  3. Enter a name for the rule.
  4. Set the rule condition in the IF section using one of the following methods:
    • Using Basic Condition
    • Using Expression
  5. Click Add to save the rule.

Using Basic Condition

  1. Select Use Basic Condition.
  2. Select a source user attribute that must be used as the variable in the IF conditional statement.

  3. Select one of the following operators or methods to be used in the conditional statement.

     

    Option Description
    Equals Determines if the attribute value equals to a specific value in the condition
    Not Equals Determines if the attribute value is not equal to a specific value
    Contains Determines if the attribute value contains a specific string
    Less Than Determines if the attribute value is less than a specific value
    Greater Than Determines if the attribute value is greater than a given value
    Starts With Determines if the attribute value starts with a specific string
    Ends With Determines if the attribute value ends with a specific string
  4. Enter the attribute value against which the condition has to be validated in the Value field.
  5. In the THEN section, enter the group name to which the rule has to be applied. The user satisfying the condition will be added to this group.

    Note: 

    If user records already exist in a group and some of those records do not meet the rule criteria specified for the group during the orchestration run, those user records are removed from the group.

Using Expression

You can write javascript expressions to set the rules for updating group-related data. Expressions can use all of the helper functions as well as user attributes in source and destination applications. SOURCE must be used in an expression to represent an attribute in the source application and DEST to represent a destination attribute. For example, SOURCE.userName, SOURCE[ENT].organization, DEST.title, SOURCE[IAN].empStatus, and so on.

For details on the keywords, inbuilt methods and functions provided by Aquera, refer to Orchestration - Expressions.

  1. Select Use Expression.
  2. Enter the condition as an expression in the IF section.

  3. In the THEN section, enter the group name(s) to which the rule has to be applied. The user satisfying the condition will be added to the group.

    Note: 

    If user records already exist in a group and some of those records do not meet the rule criteria specified for the group during the orchestration run, those user records are removed from the group.

All the added rules are displayed as a list on the Group Rules page.

You can add more rules, and modify or delete an existing rule, as required.

Scheduling the Orchestration Run

You can set the frequency by which the orchestration must run.

    1. Click the Schedule tab.

      You can either select a basic schedule or set an advanced schedule.

      Basic Schedule

      You can select a frequency by which the orchestration must run, from a pre-defined list of frequencies ranging from 15 minutes to 24 hours.

      Note: If you select Never, the orchestration never runs.

      Advanced Schedule

      • Select Never if you do not want the orchestration to run on a scheduled basis.
      • Select Minutes to schedule the orchestration to run every X minutes, where you need to select X from a drop-down list.

       

      • Select Hourly to schedule the orchestration to run every X hours, where you need to select X from a drop-down list.

      • Select Daily to schedule the orchestration to run every X days at a specific time, where you need to select X and the time from drop down lists. You can set up to a maximum of 5 daily schedules, if required.

      • Select Weekly to schedule the orchestration to run on specific days of a week at a specific time.

    2. Click Save Changes to save the orchestration setup.

Filtering Data

This section details the syntax of filter expressions that can be used in the orchestration. It provides information on the various operators that can be provided in the filter expression and the functionality of each operator.

Each expression must contain an attribute name followed by an attribute operator and an optional value.

Multiple expressions may be combined using logical operators. Furthermore, expressions can be grouped together using parenthesis ().

Attribute Operators

The following table lists the various attribute operators that can be used in a filter expression:

Operator Description Behaviour and Example
eq equal

The attribute and operator value must be identical to meet the filter criterion.

userType eq "Admin"
ne not equal

The attribute and operator value must not be identical to meet the filter criterion.

userType ne "Guest"
co contains

The entire operator value must be a substring of the attribute value for a match.

userName co "@domain.com"
sw starts with

The entire operator value must be a substring of the attribute value, starting at the beginning of the attribute value.

urn:ietf:params:scim:schemas:ian:2.0:User:employeeID sw "A"
ew ends with

The entire operator value must be a substring of the attribute value, matching at the end of the attribute value.

userName ew "@domain.com"
pr present
(has value)

If the attribute has a non-empty or non-null value, or if it contains a non-empty node for complex attributes, there is a match.

urn:ietf:params:scim:schemas:ian:2.0:User:departmentNumber pr
gt greater than

If the attribute value is greater than the operator value, there is a match. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTime types, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

urn:ietf:params:scim:schemas:ian:2.0:User:employeeID gt 100
ge greater than or equal to

If the attribute value is greater than or equal to the operator value, there is a match. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTime types, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

urn:ietf:params:scim:schemas:ian:2.0:User:employeeID ge 100
lt less than

If the attribute value is less than the operator value, there is a match. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTime types, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

urn:ietf:params:scim:schemas:ian:2.0:User:employeeID lt 100
le less than or equal to

If the attribute value is less than or equal to the operator value, there is a match. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTime types, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

urn:ietf:params:scim:schemas:ian:2.0:User:employeeID le 100

 

Logical Operators

The logical operators are used to combine multiple expressions. The following table lists the various logical operators that can be used in a filter expression:

Operator Description Behaviour and Example
and Logical AND

The filter is a match only if both the expressions (criteria) are met.

userType ne "Guest" and urn:ietf:params:scim:schemas:ian:2.0:User:employeeID pr
or Logical OR

The filter is a match if either of the expressions (criteria) is met.

userType ne "Guest" or urn:ietf:params:scim:schemas:ian:2.0:User:employeeID gt 100
not NOT function

The filter is a match if the expression evaluates to false.

userType eq "Admin" and not (userName co "example.com" or urn:ietf:params:scim:schemas:ian:2.0:User:employeeID le 100)

Note: not takes precedence over and, which takes precedence over or.

Grouping Operators

Operator Description Behaviour and Example
() Precedence Grouping

Boolean expressions may be grouped using parentheses to change the standard order of operations, that is, to evaluate logical or operators before logical and operators.

userType eq "Admin" and not (userName co "example.com" or urn:ietf:params:scim:schemas:ian:2.0:User:employeeID le 100)
[] Complex attribute filter grouping

Complex filters are supported where expressions must be applied to the same value of a parent attribute specified immediately before the left square bracket ("["). The expression within square brackets ("[" and "]") must be a valid filter expression based upon sub-attributes of the parent attribute. Nested expressions may be used.

emails[type eq "work" and value co "@example.com"] or ims[type eq "xmpp" and value co "@foo.com"]

 

Using Data Values

The filter expression may include optional values. The following table shows how values of different data types must be used in a filter expression.

Type of Data Usage of Value in Filter Expression
Text Use the value within double quotes. For example, "John"
Number Use the value without quotes. For example, 100
Date and Time Use the value within double quotes. For example, "2011-05-13T04:42:34Z"
Boolean Use true or false without any quotes. For example, true
Null value Use null without quotes.
