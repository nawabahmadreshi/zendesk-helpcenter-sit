Overview

You can build an integration template using the Integration Studio. In an integration template, you can add various parameters to allow end users to configure their integration preferences. As an integration developer, you need to write JavaScript to perform tasks on the data across applications. Depending on the resources supported by the application, the type of data that the integration handles can vary - it can be users, groups, tickets, and so on. You can perform various tasks on the data, including reading, creating, updating, deleting, and performing general tasks such as sending emails and creating service requests. You can set up the integration to run either on a specific schedule or based on the data events in an application. When the integration runs, the data will be synchronized across various applications and systems.

This user guide explains how you can build an integration using Integration Studio in the Aquera Admin portal.

The script supports SCIM functions and open API methods supported by an application, utility functions, and log functions. It is possible to access the schema attributes of an application in the script. If tables are configured in the integration, you can access tables in the script. For further details on the functions supported in the script and how to work with the scripts, refer to the Working With Integration Scripts guide.

Prerequisites

Configure the required connectors correctly in the Aquera Admin portal before creating an integration. Refer to the corresponding configuration guides for details on configuring the required connectors. 

Build an Integration Template

Perform the following steps to build an integration template:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel, which displays the list of integrations created by the user. 
  3. Click Create Integration.
  4. Select Integration.
  5. Enter a name and a description for the integration.
  6. Click Continue. A new page is displayed to configure the integration.
  7. Click Create to start building the integration template.

Integration Panels

An integration page displays the following panels:

  • Configuration
  • Script
  • Flow Chart
  • Documentation
  • Assistant
  • Summary

To view a panel, click the icon at the top right portion of the integration page and select the panel that you would like to view on the integration page.

You can rearrange the panels by dragging and dropping them on the integration page. Click the icon next to the name of a panel to drag and drop the panel into the desired position on the integration page.

Each panel includes an Expand icon and a Close icon at its top right corner.

 

  • When you click the Expand icon on a panel, the panel expands and hides all the other panels on the page. In the Expanded mode, you can click the Collapse icon to return to the multi-panel view.
  • When you click the Close icon, the panel will be closed. However, you cannot close all the panels; at least one panel will always remain open.

Note: You can view the Script, Flow Chart, Documentation, Assistant, and Summary panels on the integration page only if you are an Aquera employee and if those features are enabled in your tenant. You can switch to the customer view by clicking the Hide Code option from the ellipsis icon at the top-right corner of the page. To switch back to the Aquera employee view (with the script panel), click Show Code.

Configuration Panel

The Configuration panel allows you to add the necessary applications, add configuration parameters, create attribute maps between different applications, set up a trigger, and add tables to be used in the script.

Applications

The Applications section allows you to add applications to perform the operations required for the integration. You can add multiple applications as needed. Ensure to add at least one valid application for successful integration. 

Note 1: If no application is configured in the integration, the Applications sub-group will not be displayed in the configuration panel for end users.

Note 2: You can change the default name, Applications, to another name, if required, by navigating to the Sub Group section in the Configuration panel.

Perform the following steps to add an application:

  1. Click the + icon next to the Applications to display the Add Application pop-up window.
  2. Select the required connector instance from the Application drop-down.
  3. Enter a unique variable name for the connector in the Variable field. This name will be used to refer to the connector in the script.
    Note: The variable name must be unique within the script configured for a specific integration. It can contain letters and numbers but must start with a letter, cannot be a JavaScript keyword, and should not include spaces or special characters.
  4. Enter a display name for the connector in the Display Name field.
  5. Select whether you want to use the SCIM methods, Open API methods, or both in this integration from the Protocol field, based on your requirements. Based on the selection, the methods are available for use in the integration script.
    Note: The Protocol field is displayed only if the selected application connector supports both SCIM and Open API protocols.
  6. Click the Add button to add the connector to be used in the integration.

Refresh Application

After adding applications to the integration, if any changes have been made to the application schema, you must use the Refresh icon adjacent to the Applications section to retrieve the most recent schema for all applications.

Additionally, you have the option to click the ellipsis icon next to each application and select the refresh icon to retrieve the latest schema for individual applications.

Persistent Variables

You can use persistent variables to store data that can be used across multiple integrations configured in a tenant. The values in the persistent variables are retained even after the execution of an integration is completed. Those values can be used or modified in successive executions of the same integration and in other integrations that use those persistent variables.

The following is a sample use case to explain the use of a persistent variable:

Assume you have configured two integrations, and you want these integrations to run in such a way that only the incremental data of users after the previous execution of an integration needs to be processed in another integration.

In this scenario, you can create a persistent variable to store the date and time of execution of the integration. The same persistent variable can be used in a different integration to use the execution date/time of the other integration, for obtaining the incremental data. The execution date and time stored in the persistent variable is modified in every run of these integrations, and that value will be retained for use and modification by successive integration runs.

Add a Persistent Variable

Perform the following procedure to add a persistent variable:

  1. Click the + icon in the Persistent Variables section. The Add Persistent Variable pop-up window is displayed.
  2. Select one of the following options:
    • Create New Variable: Select this option if you want to create a new persistent variable. Proceed to the next step.
    • Select Existing Variable: Select this option if you want to use an existing persistent variable that you have created in another integration. Select the persistent variable from the drop-down list and click Add to add it to the current integration.
  3. Enter the following details:
    • Variable Name: Enter a unique name for the new persistent variable. The variable name must be unique across all the integrations configured in the tenant. This variable name will be used in the integration scripts to update the values.
    • Display Name: Provide a display name for the persistent variable.
    • Description: Provide a short description for the persistent variable.
    • Variable Type: Select the data type of the persistent variable from the drop-down list.
    • Value:  Provide the initial value of the persistent variable.
  4. Click Add. The persistent variable is added to the integration. 
    The persistent variable is also added to the global list of persistent variables available across all integrations in the tenant:

    Refer to the Viewing the Global List of Persistent Variables section for details.

Note: To modify the details of a persistent variable, click the ellipsis icon next to the persistent variable and select Edit. You can modify the display name, description, and value of the persistent variable. However, you cannot change the variable name.

Unlinking a Persistent Variable from an Integration

If you do not want to use a persistent variable that is added to an integration, you have the option to unlink it from the integration.

  1. Click the ellipsis icon next to the persistent variable and select Unlink. A confirmation pop-up window is displayed.
  2. Click Unlink to confirm. The persistent variable is now not available for use in the integration.

Viewing the Global List of Persistent Variables

You can view the list of all the persistent variables configured across all the integrations in your tenant.

  1. Navigate to the Integrations module. The list of all integrations configured in the tenant is displayed.
  2. Click the ellipsis icon at the top-right corner of the integrations listing page and select Persistent Variables. A panel is displayed with the list of all the persistent variables available in the tenant.

    For each persistent variable, the description and the current value of the variable are displayed in the panel.

Deleting a Persistent Variable

You can delete a persistent variable only if the variable is not associated to any integrations. If you want to delete a persistent variable that is linked to an integration, unlink the persistent variable from that integration by following the Unlinking a Persistent Variable from an Integration section and then delete the variable.

  1. Click the Delete icon next to a persistent variable on the Persistent Variables panel. A pop-up window is displayed for confirmation.
  2. Click Delete to confirm. 
    Note: If the persistent variable is associated to an integration, an error message is displayed.

Parameters

The Parameters section lets you define settings for the integration by adding configurable fields. These fields store values as variables, which you can then use in your integration script to customize how the integration behaves.

It also provides an option to organize these parameters as tabs (groups), subgroups within tabs, or sections. You can even create parameters without any grouping.

Note 1: For end users, the Parameters subgroup will be displayed under the Configuration tab only if atleast one parameter is added in this subgroup. Else, the Parameters subgroup will not be displayed.

Note 2: You can change the default name, Parameters, to another name, if required, by navigating to the Sub Group section in the Configuration panel.

   log.info('Raw parameter value: ' + JSON.stringify(groupOfParamValues));

Trigger

You need to set up a trigger for executing the integration. You can set up the integration to execute based on an event or a schedule.

Event Trigger

The integration will be triggered when specific events occur in an application. An event notification will be sent from the application when there is a data change in the application, such as the creation of a new user, modification of an existing user, activation of a user, addition of a user to a group, or removal of a user from a group.

Note: The events for which this integration needs to be triggered may need to be configured in that application. 

Perform the following steps to set up an event trigger:

  1. Click the + icon next to the Trigger to display the Add Trigger window.
  2. Select the Trigger Type as Event.
  3. Select an application from the Select Source drop-down list. A URL is generated for the integration.
  4. Copy the URL and paste it into the appropriate event setup section of the application. This is the connector's URL to which all the configured events from that application will be sent for processing.
  5. Click Create Token to generate a token for authentication purposes.
  6. Copy this token by clicking the copy icon next to the generated token. The token needs to be stored securely and may have to be configured in that specific application for authentication purposes.
  7. Click Save to save the event trigger setup.

For details on how to use events in the scripts, refer to the Working With Integration Scripts guide.

The integration will be triggered automatically based on a scheduled time-frequency set. 

Perform the following steps to set up a schedule trigger:

  1. Click the + icon next to the Trigger to display the Add Trigger window.
  2. Select the Trigger Type as Schedule.
  3. Select one of the following schedule options depending on the frequency by which an integration should be run.
    • Basic
    • Advanced

Disable or Enable Scheduled Runs

The Schedule Enabled toggle controls whether the integration runs automatically based on the configured schedule.

The toggle can have the following states:

  • ON: When the toggle is ON, the integration runs automatically based on the configured schedule. The integration is triggered at the defined time and frequency without manual intervention
  • OFF: When the toggle is OFF, the integration does not run automatically on a schedule. However, you can still execute the integration manually whenever required.

Notes:

  • Turning off the schedule does not stop an integration that is already running
  • If an integration with the schedule turned off is used in any integration hook, it will not be triggered

Example:

Assume an integration is configured in the New User Integration Hook to run when a new user is created. If the schedule for that integration is turned off, the integration will not be triggered when the event occurs. This is because the integration is not enabled for automatic execution.

For further details on the schedule setup, refer to the Integration Schedule Setup Guide.

Tables

Tables enable you to update specific data of a resource (for example, User attributes) in a target application during integration runs when that data is missing in the source application from which the resource data are retrieved.

Tables store data as key-value pairs and can be added to integrations to synchronize these values with another application. Tables are particularly used when an application attribute is empty or particular data is not provided by an application, and you want to update an attribute value in another application.

Add Tables in Integration

Perform the following steps to add a table in an integration using scripts:

  1. If you are accessing the customer view, navigate to your integration and click the Tables tab and click Add Table.

    Or,
    If you have the code view, then click the Add icon next to the Tables section.

    The Add Table pop-up window is displayed.

     

  2. Enter a unique Table Name,  a unique Variable Name, and a Description for the new table.
    Note: The table variable name can include a combination of letters and numbers. It should always start with a letter. It should not have only numbers. It should not contain any spaces or special characters and is case-sensitive. 
  3. Add rows to tables in one of the following ways:  
    • Add Row: To enter rows manually, click Add Row at the bottom of the Add Table window and enter a key name and a key value. Each row in the table corresponds to a key-name and key-value pair.
    • Upload CSV File: To upload rows to the table using a CSV file, click Upload CSV File. Aquera's Integration supports only the .CSV file format and the file should have only two columns. The column headings in the CSV file must be Name and Value.
      A sample CSV file is attached for your reference. You may edit this file to add the required details and upload it. The CSV file can have data of any number of rows.
      Notes:
      • The key name should be unique. The key name can include a combination of letters and numbers.
      • The key name and key value support all the special characters except the following:
        • / (forward slash) 
        • \ (backslash)
        • " " (double quotes)
      • When the key value contains a comma, the data next to it is considered the next value.
      • If the key value contains comma-separated values, ensure there are no spaces between the values.
      • Do not enclose the key values in double quotes.
  4. Click the Add button at the bottom of the Add Table window to add the table.
  5. Close the table window and click Save Changes at the top right of the integration page to save the table to the integration.
    Note: When a new table is added or a table is modified, then save the changes and do a page reload to access the table in the integration.

Setting Table Accessibility for End Users

You can set whether a table has to be readonly or hidden for an end users. The readonly or hidden properties can be set only for a table created in the "Show code" view. 

Perform the following procedure to set the acessibility options:

  1. Click Edit next to the table that you have created. The Update Table pop-up window is displayed.

    By default,  end users can see and update the table data.
  2. If required, you can set the end user accessibility for the table as follows:
    • Select Read Only if you want the end users to see the table data but not modify it.
    • Select Hidden if you want to prevent the end users from viewing the table data. The integration scripts can still use the table data.
      Note: The Read Only and Hidden options will be available only if the table was added in the "Show Code" view. If the table was added from the customer view, these two options will not be available.
  3. Click Save to save the changes.

Integration Details

The Integration Details section will appear only when the Configuration panel is shown. This panel displays the following:

  • Execution Icon: The icon to start or stop an integration execution. You can click the execute icon to start an integration execution. You can click the stop icon to stop an integration execution that is already running.
  • Execution History: The execution history color bars represent the status of the most recent five integration runs. The history color bar moves from right to left for every integration run and the right-most color bar represents the most recent integration run. The following are the supported color codes:
    • Green: Indicates the integration run is a success.
    • Red: Indicates the integration run is a failure or incomplete.
    • Grey: Indicates the integration has never run or the integration run is canceled.
    • Blue: Indicates a currently running integration.
    • Yellow: Indicates the integration run has errors.
  • APPLICATIONS: Displays the applications that are part of the integration. You can click the + icon to add new Applications. Refer to the Applications section for further details.
    When multiple applications are associated with an integration, only a limited number of application icons are displayed in the integration list screen. Additional applications are represented as +N More, where N indicates the number of additional applications. When you hover over the application icons or the +N More indicator, the view expands to display all associated application icons.
  • AUTHOR: The name of the user who created the integration.
  • DATE CREATED: The creation date of the integration.

Target Attributes

The target attributes are automatically populated based on the SCIM or OpenAPI target resource schema defined in the selected attribute map. The mapping list displays only those target attributes whose mutability is set to readWrite or writeOnly, as per the schema.

Source Attributes

The format in which the source attributes must be configured in the mapper varies based on the attribute mapper version:

For Attribute Mapper version 1:

The source attribute must be provided in the format <appName>.<resourceName>.<attributeName>. For example, app1.Employee.employeeNumber, where app1 is the application variable name, Employee is the name of the resource defined in the Attribute Map, and the employeeNumber is the name of an attribute in the Employee resource, as defined in the schema. Auto-suggestions assist you in selecting the right resource name and the attribute name from the source attribute fields, as shown below.

Note: By default, the source and target attributes having the same names are mapped.

For Attribute Mapper version 2:

The source attribute name must be provided in the format, <attributeName>. For example, employeeNumber, which is defined in the schema.

Mapping Options

For each mapping, you can configure how the source attribute must be mapped to the target attribute by selecting one of the following options from the dropdown list between the source and target attributes:

  • Map on Create and Update: The target attribute is updated during both create and update operations
  • Map on Create: The target attribute is updated only during the create operation.
  • Map on Update: The target attribute is updated only during the update operation.
  • Don't Map: The target attribute will never be updated.

Search by Attribute Names

You can search for a source or target attribute by providing the attribute name in the Search Attributes field at the top right corner of the attribute mapping list.

Modify Attribute Mapping via JSON

You can add or modify the source to target attribute mappings from the mapping list page by enabling the Edit JSON option at the top right corner of the page.

  1. Toggle the Edit JSON option to ON. The attribute mapping is displayed in the JSON format on the page.
  2. Modify the mapping as per the requirement and save the changes.
  3. Toggle the Edit JSON option to OFF to return to the attribute mapping list view.

Refresh Attribute Mapping

A Refresh icon is available at the top right corner of the attribute mapping list page.

Whenever any changes are made to the SCIM or OpenAPI schema definition in the Attribute Map configuration, you must click the Refresh icon to populate the latest set of attributes. Else, the changes will not be reflected on the attribute mapping list page.
If any attributes have been removed in the schema, a pop-up window is displayed with the list of those attribute mappings when you click refresh.

You can select the attributes that must be removed from the mapping. By default, all the attributes are selected for removal. Click Remove Selected to remove the selected attributes from the mapping.

Using Parameters in Attribute Map

When you write expressions in the Source Attribute/Expression column in the Attribute Map tab, you can access the values of the parameters configured in your script-based integration.

Use the following format to access the parameter value:

PARAMETER.<parameter_variable_name>

For example, PARAMETER.hireDate

For parameter fields that support multiple selections (arrays), the format must be as follows:
PARAMETER.<parameter_variable_name>[<index>]. For example, PARAMETER.statuses[0]

Script Panel

The Script panel allows you to write a script to set up the integration. The script is based on JavaScript and the script supports application SCIM functions, utility functions, log functions, and open API methods. 

As you type the lines of code, the script editor automatically suggests the code for you. If you want to use the code suggested by the editor, you must tap the tab key. 

To invoke an application function, use the following format in the script:

<App_VariableName>.<functionName>

For example, okta.importAllUsers()

For details on the functions supported in the script and how to work with the scripts, refer to the Working With Integration Scripts guide.

Assistant Panel

The Assistant panel generates scripts based on the inputs provided. You can open the Assistant Panel in one of the following ways:

  • Click the icon  and then select the Assistant option
  • Double-click on any part of the script in the script panel, and then click the Chat option that pops up

You can input your requests in English in the Enter prompt here text field and press the Enter key. The AI Assistant analyzes your inputs and creates a step-by-step plan outlining the logic based on your prompt. It then generates the JavaScript code that you can use in the script editor. Soon after the code, a brief explanation of the script is generated for your understanding. 

Note 1: For simple and straightforward use cases, the Assistant panel may generate only the JavaScript and the descriptive explanation of the script. The step-by-step logic may not be generated.

Note 2: To stop the AI Assistant while it is processing, click the Stop button next to the prompt field.
