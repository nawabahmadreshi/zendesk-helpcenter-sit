Integrations Using Workflows

← Back to index

Overview

This guide explains how you can set up Integration using Workflows. You can set up a Workflow of actions to be performed on users' data across applications without actually writing a script. The various actions that can be performed on user data are import, create, update, delete, and so on. For example, you can achieve an automatic user onboarding process from an application to multiple applications with a single Workflow setup. You can add multiple applications to the Workflow, as required. A JavaScript is internally created based on the Workflow that you have set up and when the Workflow runs, this script is executed.

You can set up the Workflow to run based on a schedule or based on the events (user data changes) in an application.

Prerequisites

Make sure that the application connectors are added and configured correctly in the Aquera Admin console before creating the Workflow. Refer to the corresponding configuration guides for details on configuring the required connectors. 

Creating a Workflow

Perform the following steps to create a Workflow:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel which displays the list of integrations created by the user. 
    Note: The number of integration records displayed on this page is based on the display size of the user's screen.
  3. Click Create Integration.
  4. Select Workflow.
  5. Enter a name and a description for the workflow.
  6. Click Continue. A new page is displayed to set up the workflow.
  7. Click Create to create the workflow.

Setting a Trigger for the Workflow

You can set one of the following triggers for a Workflow:

Event Trigger

The Workflow is triggered when specific events occur in an application.

Note: The events for which this Workflow needs to be triggered may need to be configured in that application. 

  1. In the Add Trigger section, select Event and then select an application in the Select Source drop-down list. The Workflow will be executed when a specific event (for example, user creation) occurs in the selected application.

    When you click Create to create the Workflow, a URL is generated for the event.
    You must copy this URL and paste it in the appropriate event setup section of the application. This is the URL to which all the configured events from that application will be sent for processing. 
  2. Click Create Token to generate a token for authentication purposes. This token has to be copied and stored securely. It may have to be configured in that specific application for authentication purposes.


  3. Click Save Changes to save the event trigger setup.

Schedule Trigger

The Workflow runs based on the schedule set for the Workflow. For details on the schedule setup, refer to the Integration Schedule Setup Guide.

Setting Up a Workflow

You can set up a Workflow containing a sequence of tasks that must be executed when the Workflow runs. The tasks can be any operations that must be performed on the user objects in various applications and general tasks such as sending emails, creating service requests, and so on. The following options are available to set up the Workflow:

  • Actions
  • Flows
  • Functions
  • Variables

Each of these options is explained in the following sections.

Actions

The Actions option enables you to add the required applications to the Workflow. You can also add the various methods (operations) and the Open API methods that need to be performed on users. 

Adding Applications

You must add applications that must be part of the Workflow.
Note: Ensure to add at least one valid connector for the Workflow to run successfully.

  1. Click the Actions icon  and click Add Applications. A pop-up window is displayed as follows:

    Note: Click the Collapse icon to collapse the left panel and vice-versa.
  2. In Application Name, search for the application to be added to the Workflow and select the required application.
  3. In the Variable field, enter the variable name that must be used in the Workflow to refer to the application. The script that will be generated in the backend will have this variable to refer to the application.
  4. Click Add to add the application to the Actions pane. You can add as many applications as required.
  5. Click Save Changes at the top right corner of the page to save the application details.
    Notes:
    • You can edit the application variable name or change the application using the Edit option next to an application. If you change the application, the application methods added for the old application will remain in the Workflow as it is. If any application methods that are not supported by the new application remain in the Workflow, then a warning icon is displayed for those application methods. The Workflow will fail if you do not remove those highlighted application methods from the Workflow.
    • You can delete an application from the Workflow using the Delete option next to an application. You cannot delete an application if it is used in the Workflow. You must first remove any actions specific to an application from the Workflow before deleting that application.

Adding and Configuring Application Methods

You can add the application methods for performing the required operations on user objects within an application.  For example, Import All Users, Create User, Update User, Find User by Id, and so on. 

  1. Expand an application on the Actions panel. The list of all application methods (operations) and open API methods supported by that application is displayed.
  2. Expand an application on the Actions pane and then drag and drop the required method (action) to the Workflow editor area. This action is now part of the Workflow.

Refer to the following section for details on the standard SCIM methods supported in an application. Refer to the Dragging and Dropping Workflow Items section for details.

Supported SCIM Methods

The following table lists the standard SCIM methods (actions) that can be added to the Workflow and the corresponding descriptions.

Note 1: Currently, group operations (for example, Add Groups, Retrieve Groups, and so on) are not supported in the Workflow.

Note 2: Any input values must be provided as follows in the Workflow:

  • Alphanumeric values must be provided within either single quotes or double quotes. For example,  "abc123" or 'abc123'
  • Numeric values can be provided without quotes. You can even enclose the numerical values within single quotes or double quotes. For example, 1234567, "1234567", or '1234567'
  • Any variable names and attribute names must be provided without quotes. For example, variablex, oktaUser.active.

Note 3: All input fields can be expanded to open in an editor window. When you move the cursor over an input field, the option to expand the field is displayed.

  • Only the attributes that are readWrite can be updated for a user. 
  • Only readWrite, writeOnly, and immutable attributes are populated for selection.

  • The ID of the user or the variable that holds the ID
  • password string to be set for the user.

Note 2: Make sure that the password meets any password policy/complexity rules that are set for the application. 

Note 3: The application connector must support setting passwords for users.

You must enter the ID of the user to be activated.

You must enter the following details:

  • The attribute such as the ID or the userName of the user to be added to the group. The attribute must be a unique value to identify the user.

You must enter the following details:

  • The display name or ID of the group from which a user has to be removed.
  • The ID, the userName, or the attribute name containing the ID/username of the user to be removed from the group. 

Note: To delete any action from the Workflow, click the ellipsis icon and select Delete.

Flows

Flows include the following conditional statements and iterations that control the flow of operations included in the Workflow:

  • IF
  • IF ELSE
  • FOR EACH
  • WHILE

You can drag and drop these items to the Workflow editor area. Refer to the Dragging and Dropping Workflow Items section for details on dragging and dropping items from the panel to the Workflow editor area.

Each flow is explained in detail in the following table:

Flow Description
IF

The IF statement is used to set a condition.

You can set the actions to be executed based on whether the condition evaluates to true (Yes) or false (No).

In this scenario, a particular message is printed only if the username contains "example.com".

You can also set the condition using expressions. If you want to use expressions, set the mode to Expression.

By default, Condition mode is selected.

You can add an ELSE condition to any IF statement. After adding the ELSE condition to the IF statement, it works the same as the IF ELSE statement. To add the ELSE condition to the IF statement, click the vertical ellipsis adjacent to IF as shown in the below image, and select Add Else option.

You can also remove the ELSE condition from any IF or IF ELSE statement by selecting the Remove Else option.
IF ELSE

The IF ELSE statement is used to set two conditions.

You can set the actions to be executed based on whether the condition evaluates to true (Yes) or false (No).

In this scenario, if a user's title is admin then that user is added to Group 1 in Okta, else if the user's title is not admin then that user is added to Group 2 in Okta.

You can remove the ELSE condition from any IF ELSE statement by selecting the Remove Else option. After removing the ELSE condition from the IF ELSE statement, it works the same as the IF statement.
FOR EACH

The For Each statement is used to execute a function or a block of code once for each item in an array.

Enter the variable name to represent each item in the array and enter the name of the array that contains the list of items.

In this scenario, each user in the oktaUsers_1 array is checked to see if the user's email contains "example.com".
WHILE

The WHILE statement is used to repeatedly execute a function while a specific condition holds true.

In this scenario, as long as the value of the attribute active is true, users will be created in Okta. Once the value of the attribute active changes, the WHILE loop is exited. 

Note: You can expand or collapse the actions within an IF, IF ELSE, FOR EACH, or WHILE block using the + or the - icon. In the collapse mode, the number of actions within a block will be displayed next to the + icon.

Functions

The Functions list provides the standard javascript functions that can be added to the Workflow.

You can drag and drop the required functions to the Workflow editor area. Refer to the Dragging and Dropping Workflow Items section for details on dragging and dropping items from the panel to the workflow editor area.

The following table lists the functions available in the panel.

Function Description
Log

Prints a message in the log when the Workflow runs.

Enter the message to be printed in the field. It can contain any string, variables, and any specific user attribute(s).
Email

Sends an email when the Workflow runs.

You can enter the list of email IDs in the To, Cc, Bcc, and From fields, and set the email subject and body.

The From parameter enables you to specify the sender email ID on behalf of which emails will be sent to users. This is an optional parameter. To set up the sender email ID in Aquera’s service, you must first send a request to support@aquera.com. When the email ID is set up, a verification email will be sent to that email ID. That email user must click the verification link included in the email to allow Aquera to send emails on behalf of that user.
For more information about email verification and DKIM settings, see Email Verification and DKIM Settings.  
Set persistent variable

To set a value to a persistent variable, provide the name of the persistent variable within double quotes and enter the value for it. For more information about persistent variables, refer to the Persistent Variables section.
Get persistent variable

To get the value from a persistent variable, enter the name of the persistent variable from which its current value has to be retrieved. The name of the persistent variable must be provided within double quotes. You can use a persistent variable that is defined in another integration.

You must also enter the name of the variable to store that value.

For more information about persistent variables, refer to the Persistent Variables section.
Remove persistent variable

To remove a persistent variable, enter the name of the persistent variable to be removed. The name of the persistent variable must be provided within double quotes. After this persistent variable is removed and if this removed persistent variable is used in any other integration, then that integration will fail.

For more information about persistent variables, refer to the Persistent Variables section.
Set Response

Set the status code and the body for the response.

 

Persistent Variables

This section provides detailed information on how persistent variables work.

A persistent variable is a variable that can be used across various integrations in your tenant. The value of a persistent variable will be retained even after the integration is executed, and can be used and modified in successive executions of the same integration and in other integrations.

The following is a sample use case to explain the use of the persistent variables.

Assume that multiple scheduled integrations are configured in your tenant. Now, you want these integrations to run in such a way that only the incremental data of users after the previous execution date/time of an integration need to be processed in another integration.

In this scenario, we can set a persistent variable in an integration to store the date and time of execution of that integration. The same persistent variable can be used in a different integration to use the run date/time of the other integration, for obtaining the incremental data. The execution time in the persistent variable keeps changing for every run of these individual integrations and that value will be retained for use and modification by the successive integrations.

Scope of Variables

If a variable is created within an IF or FOR EACH code block, the scope of the variable is only within that block. It cannot be used or referenced anywhere else in the Workflow.
Note: The scope of a persistent variable is across all integrations within a tenant. A persistent variable that is set in an integration can be used in other integrations within the same tenant.

Dragging and Dropping Components to a Workflow

To create the Workflow, you must drag the required components from the left panel and drop them into the Workflow editor area. The tasks will be executed in the order they are placed in the Workflow.

The first component in the Workflow is always the trigger that you have set for the Workflow - an event or the schedule.  

For details on the trigger setup, refer to the Setting a Trigger for the Workflow section.

When you drag an item from the panel to the Workflow editor area, a grey area with a dotted borderline is displayed in the Workflow editor area to indicate the placeholder of the item to be dropped.

You can place the item in the appropriate placeholder, as per your requirement. When you drop the item, it will be added to the Workflow and a number is assigned to it as per its sequence. The arrow indicates the flow of tasks in the Workflow.

A connecting line is displayed after the last item in the Workflow. When a new component is added after that item in the Workflow, the connecting line is changed to an arrow, which indicates the direction of the flow of tasks.

When you want to place an item that must be executed within a flow (IF, For Each), it must be placed one level inside the flow. You can click the + or - icon to expand or collapse the levels within it.

In the above scenario, the IF statement is executed within the For Each flow. You can add any number of such nested flows (levels) to the Workflow, as required. The placeholders are highlighted accordingly when you drag the items to the flow.

Accessing User Attributes Passed from a User Integration

You can execute a Workflow version 3 from a User Integration version 3. To enable this feature, you need to configure the following settings in a User Integration:

  • New User Integration Hook
  • Update User Integration Hook
  • Deprovision User Integration Hook

For further details on these settings, refer to the User Integration Setup Guide.

When a User Integration runs, an event will be passed from the User Integration to the Workflow. This event will contain the user object when a user is successfully created, updated, or de-provisioned. 

For example, when a User Integration runs, if a new user is successfully created, then the Workflow set in the New User Integration Hook can access the user attributes of the created user from the event.

The following table describes how to access a user attribute that is passed from a User Integration:

Description Format Example
Access a source user attribute event.source.<attributeName>
  • Core attribute:
    event.source.active
  • Complex multivalued attribute:
    event.source.emails.find(email => email.type == 'work').value
  • ENT attribute:
    event.source[ENT].department
  • IAN attribute:
    event.source[IAN].hireDate
Access old target user attribute event.target.old.<attributeName>
  • Core attribute:
    event.target.old.active
  • Complex multivalued attribute:
    event.target.old.emails.find(email => email.type == 'work').value
  • ENT attribute:
    event.target.old[ENT].department
  • IAN attribute:
    event.target.old[IAN].hireDate
Access new target user attribute event.target.new.<attributeName>
  • Core attribute: event.target.new.active
  • Complex multivalued attribute:
    event.target.new.emails.find(email => email.type == 'work').value
  • ENT attribute:
    event.target.new[ENT].department
  • IAN attribute:
    event.target.new[IAN].hireDate

User Actions on an integration

You can perform actions on an integration such as execute, upgrade, copy, modify, delete, export, import, view script, set log levels, and view logs. For further details, refer to the User Actions on Integrations guide. 

Sample Use Case 

This section describes how to set up a Workflow for the following use case:

All active users from the BambooHR system must be created in Okta. 

Prerequisite:

  • Configure the BambooHR and Okta applications in the Aquera Admin portal.

Set up a Workflow as follows:

  1. Create a new Workflow with the trigger set as Schedule.
  2. Add the BambooHR and Okta applications to this Workflow.
  3. Drag and drop the Import All Users action from the BambooHR app, to import all the users from BambooHR.
  4. Traverse through the list of imported users and search for any active user, using the For Each and IF flows.
  5. Create active users in Okta.
  6. Print the username of the created user in the log.
