User Integration Setup Guide - UKG Ready to Active Directory

← Back to index

Overview

This guide covers the end-to-end tasks involved in setting up an orchestration to sync users in UKG Ready with Active Directory.

The following tasks are involved in setting up the orchestration from UKG Ready to Active Directory:

  • Initial Preparation
  • Configuring Applications in Aquera Admin
  • Orchestration Setup
  • Running Orchestration

Initial Preparation

Aquera Admin Console

  1. Your IT admin must register as an Aquera Admin.
  2. Once registered successfully, Aquera creates a tenant for the IT Admin and provides access to it.

Active Directory

Determine the architecture to be followed for setting up communication between Aquera and the Windows Active Directory Domain Controller. Refer to this Aquera guide for detailed information on the different architectural approaches.

Based on the selected approach, the following procedures may have to be performed:

  • A Windows Server must be ready to deploy Aquera Agent onto the server where Domain Controller is present. For instructions on installing Aquera Agent, refer to the Installation and Configuration Guide.
  • A firewall must be set up to allow communication with Aquera.
  • The static IP address of Aquera needs to be whitelisted. Contact the Aquera Support team.

To allow Aquera to make updates to Active Directory (for creating, updating, and deleting users), LDAPS communication to Active Directory needs to be set up, which requires a TLS connection. Active Directory Certificate Services must be enabled for this. To install Active Directory Certificate Services, see docs.microsoft.com.

Configuring Applications in Aquera Admin

The following applications must be added and configured in the Aquera Admin console:

  • UKG Ready. Refer to the UKG Ready Configuration Guide for details.
  • Active Directory. Refer to the Active Directory Configuration Guide for details.

Orchestration Setup

This section details how the orchestration must be configured to allow data push from UKG Ready to Active Directory. 

It also describes the various analyses to be performed and the important points to be considered before setting up the orchestration.

Considerations

  • The naming convention for the User Principal Name (UPN) in the Active Directory
  • The naming convention for the SAM Account Name in the Active Directory
  • Fields currently populated in the Active Directory
  • Work email can be retrieved from the UKG Ready system or be generated by Aquera orchestration and updated back to the system.
  • The criteria for matching a user record from UKG Ready with that in the Active Directory.

General Analysis Steps

Before setting up the orchestration, the following analyses need to be done:

  • Fill out the mapping template attached with this guide for mapping UKG Ready user attributes to Active Directory user attributes. The most common attributes are mapped in the attached template. You can add more attribute maps to the sheet as per your requirement.
  • Expression language can be used in the mapping to create UPNs and SAM Account Names without duplicates or errors. In the expressions, you can use inbuilt methods such as toNormalizedDiacritics() and toSAMAccountName() for this purpose. You can also use the keywords such as %ATTEMPT, %USEQ, SEQUENCENUMBER to avoid duplicates. For details, refer to the Orchestration - Expressions Guide.
  • Define the logical OU mapping rules for user containers.

You can analyze the users in UKG Ready as well as in Active Directory using one of the following ways:

  • Downloading the CSV file containing the list of all users through the Data tab on the Application page. For details, refer to the User Reports Guide.
  • Using Script orchestration. For scripts, refer to the Appendix.

Building Orchestration

This document describes how to set up the orchestration using the Synchronization orchestration method.

  • Synchronization Orchestration - In this orchestration, set UKG Ready as the Source Application and Active Directory as the Target Application. Refer to the Synchronization Orchestration Guide for details on the configurations.

You can set the following configurations in the Synchronization orchestration:

  • Set the Admin Email Options to receive email notifications for success/failures during the orchestration run.
  • Set the email list of the IT admin or manager to receive user creation notification emails and user deactivation notification emails.
  • Enable Group management if Aquera has to manage the groups and group assignment rules based on your group analysis. Set the group names in the restriction list for scoping of the groups which would be managed by Aquera.

Refer to the Synchronization Orchestration Guide for details on Synchronization orchestration configurations.

Orchestration Scenarios

This section details the Orchestration Scenarios.

Creating a user of type NORMAL_ACCOUNT in the Active Directory

To create a user of type NORMAL_ACCOUNT in Active Directory, set the attributes mapping as follows:

Enter "512" as the source attribute expression and map it to the [IAN].userAccountControl attribute of AD. Set the mapping criterion as Map on Create.

Note: 512 is the decimal value corresponding to the NORMAL_ACCOUNT property flag in Active Directory. For all property flags and the corresponding decimal values, refer to the Microsoft documentation.

Escaping Special Characters in OUs and DNs for Active Directory

To escape any special characters in OUs and DNs in the expressions, use a double-backslash before the special character. For example, if the OU name is #Terminated, mention it in the expression as \\#Terminated. 

DEST[IAN].distinguishedName.split(",")[0] + ",OU=\\#Terminated, DC=local"

Attribute Mapping for mandatory password change at first sign-in

To make it mandatory for users to change their password when signing in for the first time, map the Active Directory attribute pwdLastSet to "0" with the mapping option of Map on create only.

If this mapping is not done and users sign into their account for the first time, it will not prompt them to change their password.

To prompt the existing users also for a password reset, set the mapping option as Map on Create and Update.

Requirements to set user password

Active Directory requires a secure LDAPS (LDAP over SSL) connection to set or change user passwords. This is an Active Directory’s security requirement. Random generation of passwords when creating new users does not work without the use of LDAPS. Also, setting the password through attribute mapping requires a secure LDAPS connection.

Note: If the connection is not secured (non-SSL), Active Directory does not allow you to update the user password. 

Active Directory imposes strength requirements on the password and the requirements vary from one instance to another.

Running Orchestration

  • Enter the matching expression to match the source records with the destination records.
  • Run the orchestration in Report Only mode and determine if the correct data is updated in the Active Directory. If the data is not correctly updated, change the matching expression to correctly match users and/or update the data, for example, employeeNumber, in the Active Directory.
  • Increment the Update Limit (maximum number of records to be updated in a single run) for each run in Report Only mode to ensure that the records are updated correctly when the records count is more.
  • Based on your analysis and considerations,
    • Translate the logical maps to the Aquera attribute maps for the fields
    • Translate the logical OU/container locations into tables and expressions
    • Translate the logical group assignment rules into Aquera rules
  • Modify the attributes mapping and repeat the process until the data is correctly updated.
  • Monitor the orchestration logs thoroughly to ensure that no errors are encountered. Modify the orchestration configurations to fix any errors.
  • Once all users are synced, user creations and updates are based on the incremental changes in UKG Ready.

Appendix

This section provides the scripts that may be used by the Customer Support team to analyze the existing users in UKG Ready and Active Directory before setting up the orchestration. It also provides the script to analyze the existing groups of users in Active Directory.

This document explains how analyses can be done using a scheduled script orchestration. In this orchestration, add UKG Ready and Active Directory applications and set the corresponding variable names. In the script, the variable names must be used to refer to the applications.
For details on configuring the script orchestration, refer to the Custom Script Orchestration Guide.

A few sample scripts are provided here for your reference. The scripts may vary based on your use case and the attributes to be retrieved for analysis.

Analyzing the Users in UKG Ready 

Use the below sample script in a scheduled script orchestration for generating a report from UKG Ready for analysis of user status, department, division, and so on.

log.info('UKG ID,UKG USERNAME,UKG FAMILYNAME,UKG GIVENNAME,UKG NAME_FORMATTED,UKG EMAILS_WORK_VALUE,UKG EMAILS_WORK_PRIMARY,UKG ACTIVE,UKG PHONENUMBER_WORK_VALUE,UKG PHONENUMBER_MOBILE_VALUE,UKG EMPLOYEENUMBER,UKG DIVISION,UKG DEPARTMENT,UKG ENT_MANAGER_VALUE,UKG ENT_MANAGER_$REF,UKG ENT_MANAGER_DISPLAYNAME');
let users = ukg.getUsers();
for (let user of users) {
log.info('"'
+ user.id + '","'
+ user.userName + '","'
+ user.name.familyName + '","'
+ user.name.givenName + '","'
+ user.name.formatted  + '","'
+ user.emails[0].value + '","'
+ user.emails[0].primary + '","'
+ user.active + '","'
+ user.phoneNumbers[0].value + '","'
+ user.phoneNumbers[0].type + '","'
+ user.employeeNumber + '","'
+ user.division + '","' 
+ user.department + + '","'
+ user.manager.value + '","'
+ user.manager.$ref + '","'
+ user.manager.displayName + '"');
}

Analyzing the Users in Active Directory

Use the below sample script in a script orchestration for generating a report from Active Directory for analysis of user domain, status, and so on.

log.info('AD ID,AD ACTIVE,AD EMPNO,AD GIVENNAME,AD FAMILYNAME,AD USERNAME,AD SAMACCOUNT,AD DN');

for (let user of AD.getUsers()) {

    log.info('"'

    + user.id + '","'

    + user.active + '","'

    + (user[ENT] && user[ENT].employeeNumber || '') + '","'

    + user.name.givenName + '","'

    + user.name.familyName + '","'

    + user.userName + '","'

    + user[IAN].sAMAccountName + '","'

    + user[IAN].distinguishedName.split(',').slice(1).join(',') + '"');

}

        }
