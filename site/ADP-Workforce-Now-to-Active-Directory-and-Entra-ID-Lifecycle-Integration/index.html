ADP Workforce Now to Active Directory and Entra ID Lifecycle Integration

← Back to index

Overview

The Aquera platform allows your organization to scale complex data management by automating data integration across resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

The integration reads worker and candidate records from ADP Workforce Now. It uses that data to keep user accounts in Active Directory (AD) and group memberships in Entra ID (Azure AD) up to date. It retrieves the selected people from ADP (workers, candidates, or both), then applies configured rules, such as active employees or active full-time employees, to decide which records to process. For each person who meets these rules, the integration looks up an existing Active Directory account using identifiers such as an Employee ID or an email address. 

If no account is found, it creates a new AD user. It adds key fields, including name, login name (sAMAccountName), UPN, email address, department, job title, manager, office location, OU location, and any additional required attributes. If an AD account already exists, the integration compares ADP data with AD data. It updates the AD account when changes are detected, for example, a new title, a different manager, or a move to another department.

When a worker leaves the company or no longer meets the configured rules, the integration disables the AD account or moves it to a “disabled” or other designated OU so that access matches the worker’s employment status.

Entra ID (Azure AD) receives users from AD through directory synchronization. The integration does not create users directly in Entra ID; instead, it manages group memberships for users who already exist there. When Entra ID Group Management is enabled, the integration uses the AD identity and the configuration rules to add or remove the corresponding Entra ID user from the correct Azure AD and Microsoft 365 groups. If Group Assignment for Active Directory is enabled, similar group updates are applied to on-premises AD groups. The optional Email Write Back to ADP feature allows the email address created or updated in AD to be written back into ADP Workforce Now. 

Key Functions

The ADP Workforce Now to Active Directory integration provides the following functions:

  • Create user accounts in Active Directory using ADP data: When a new worker or candidate is added to ADP Workforce Now and meets the configured criteria (for example, active or active full-time), the integration automatically creates a corresponding user account in Active Directory (AD). The new AD account is populated with key details such as name, login name (sAMAccountName), UPN, email address, department, job title, manager, office location, OU location, and any other mapped attributes (including extension attributes).

  • Keep existing AD accounts in sync with ADP: If a worker already has an AD account, the integration compares the worker’s ADP data with AD values on each run. When a difference is found, for example, a change in title, department, manager, or name, the integration updates the existing AD account so that directory information stays aligned with the latest data in ADP Workforce Now.

  • Handle terminations and removals in AD: When a worker is terminated in ADP Workforce Now or no longer meets the configured criteria, the integration treats that person as no longer eligible for active access. Based on the configuration, it either disables the corresponding AD account or moves it to a designated OU (such as a disabled-users OU), ensuring that AD access reflects the worker’s current employment status.

  • Use AD as the source for Entra ID and manage groups: User accounts are created and updated in Active Directory and then synchronized with Entra ID (Azure AD) using directory sync (for example, Azure AD Connect). The integration does not create users directly in Entra ID. Instead, it relies on the AD identity and focuses on managing the user’s Entra ID group memberships after the account becomes available in Azure AD.

  • Manage Entra ID (Azure AD) group memberships: When Entra ID Group Management is enabled, the integration manages Azure AD and Microsoft 365 group memberships for users who have been synchronized from AD into Entra ID. Using the worker’s ADP profile and configured rules, the integration can add or remove these users from Entra ID and Microsoft 365 groups after their AD accounts are created or updated.

  • Manage on-premises AD group memberships: When Group Assignment for Active Directory is enabled, similar group rules are applied directly to on-prem AD groups. Based on ADP data and the configured mappings, the integration adds or removes users from the correct AD security and distribution groups so that group-based access in AD matches the organizational structure and role information stored in ADP.

  • Write email addresses from AD back to ADP: When email writeback to ADP is enabled, the integration can write email addresses generated or updated in Active Directory back to ADP. For example, when a new account is created or a name change results in a new email, update the corresponding record in ADP Workforce Now. This keeps HR records and directory records aligned, with the same current email address stored in both systems.

Prerequisites

The following are the prerequisites:

  • Tenant Account in the Aquera Admin Portal: You must have a tenant account with admin access in the Aquera Admin portal. Visit https://admin.aquera.io to register for a tenant account. Contact your Aquera account manager to get the Aquera tenant access. 
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/
  • Configure Application Connectors in the Aquera Admin Portal: You must configure the following application connectors in the Aquera Admin Portal:

    • ADP Workforce Now: Add and configure the ADP Workforce Now connector in the Aquera Admin portal with the appropriate API credentials. Refer to the ADP Workforce Now Configuration Guide for details on creating the connector and setting up authentication.

    • Active Directory (LDAP): Add and configure the Microsoft Active Directory connector in the Aquera Admin portal. Refer to the Active Directory Configuration Guide for instructions on creating the connector, setting up authentication, setting up a secure LDAP connection, and providing the required service account credentials.

    • Microsoft 365 Entra ID
      Add and configure the Microsoft 365 EntraID connector. This connector is used for managing Entra ID (Azure AD) security groups and Microsoft 365 groups for users synchronized from Active Directory. Refer to the Microsoft 365 Entra ID Configuration Guide for details on app registration, Graph API permissions, and completing tenant consent.

Integration Setup

This section outlines the configuration required to establish integration between ADP Workforce Now and Active Directory, as well as Entra ID, in the Aquera Admin Portal.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin Portal.

  2. In the left navigation panel, select Integrations.

  3. Click Create Integration.

  4. Select ADP Workforce Now to Active Directory and Entra ID as the integration template.

  5. Enter a name and description for the integration.

  6. Select the connector instances that the integration will use:

    • ADP Workforce Now (v28)

    • Active Directory (LDAP)

    • Microsoft 365 Entra ID

  7. Click Continue. The Configuration page is displayed to complete the integration setup.

Configuration

The Configuration tab is used to review and customize the behavior of the ADP Workforce Now to Active Directory and Entra ID Integration.

The following sections are available under the Configuration tab:

Applications

The Applications section lists the application connectors used by the integration. It displays the configured source and target systems and lets you search for, review, and update the selected connector instances.

Parameters

The Parameters section is used to tailor the integration to a specific use case. It contains configurable options that control how data is read from ADP Workforce Now and how user accounts and groups are managed in Active Directory and Entra ID.

Integration Scenarios

Use the Integration Scenarios area to configure the main functional behavior of the integration. Configure the following integration scenarios:

  • Enable User LifeCycle Management: Enables automated creation, update, and deactivation of user accounts in Active Directory based on worker and candidate data from ADP Workforce Now. For example, when a new active worker is added in ADP, a corresponding AD account is created. When a worker is terminated, the related AD account is disabled or moved to a designated OU.

  • Select User Type: Defines which ADP records are processed by the integration, such as workers, candidates, or both. For example, when both Workers and Candidates are selected, the integration can provision AD accounts for employees and also prepare accounts for candidates during the hiring process.

  • Enable EntraID Group Management: Enables management of Entra ID (Azure AD) group memberships for users synchronized from Active Directory, based on ADP-driven rules. For example, if a worker’s department changes in ADP, the integration can remove the user from the old department group in Entra ID and add them to the new department group.

  • Enable Group Assignment for Active Directory: Enables automated assignment and removal of on-premises Active Directory group memberships using rules that reference ADP attributes such as department, location, or job role. For example, when a worker moves into the Sales department, the integration can add the user to the “Sales” AD group and remove the user from the previous department group.

  • Enable Email Writeback to ADP: Ensures that the email address stored in ADP Workforce Now remains consistent with the one in Active Directory by sending email changes from AD to ADP. For example, when a worker’s email address is updated in AD due to a legal name change, the integration updates the corresponding email field in ADP for that worker.

Execution Settings

Configure the following execution-related settings:

  • AD User Attributes to Fetch: Controls which Active Directory attributes are retrieved for each user during sync and provisioning. The list typically includes distinguishedName, objectGUID, objectSid, login attributes (sAMAccountName, userPrincipalName, mail, proxyAddresses), organizational data (company, department, title, manager), and state information (userAccountControl, lastLogonTimestamp), along with any custom attributes. For example, the default list includes extensionAttribute1, allowing the integration to read and write tenant-specific values.

  • Delete AD User (Hard Delete): Determines whether user accounts are permanently removed from Active Directory when deprovisioned, rather than disabled or moved. A hard delete removes the user and its group memberships. For example, when this option is enabled, the integration deletes the corresponding Active Directory user object instead of merely disabling the account.

  • Condition to Deprovision Candidate: Defines the logical condition that must be satisfied before a candidate’s AD account is deprovisioned. The condition is evaluated against ADP candidate fields and is typically used to delete accounts only for candidates who are rejected, withdrawn, or inactive for a specified period. For example, a condition can be configured so that only candidates with a status of “Rejected” for more than 30 days have their AD accounts removed.

Manage Workers

Correlations and Filters

  • Workforce Now Worker Filter: Determines which ADP worker records are included in the integration, based on employment status. Only workers matching the selected filter are considered for create, update, or delete actions. For example, selecting Active Workers ensures only currently employed workers are processed.
  • ADP Correlation Attribute: Specifies the ADP attribute used for identifying the correct worker when matching to an AD account. Common values include Employee Number, Associate ID, or Worker ID.
    For example, choosing Employee Number / Associate ID / Worker ID uses the unique numeric identifier from ADP to match the Active Directory.
  • Active Directory Correlation Attribute: Selects the AD field used to find an existing user account that corresponds to the ADP worker. Typical attributes include Employee Number, sAMAccountName, UPN, or Email. For example, choosing Employee Number matches workers to AD accounts using the employeeNumber attribute.

Update Worker

  • Worker Upsert Condition: Defines when the integration should create or update a worker’s AD account. This setting controls whether all workers, only active workers, or only active full-time workers should be processed. For example, selecting All Workers allows both active and inactive workers to be created or updated.
  • Enable Worker Updates in Active Directory: When enabled, the integration updates AD user records whenever a worker’s profile changes in ADP, such as title, department, or manager.
    For example, when a worker’s department changes in ADP, their AD department attribute is updated automatically.
  • Enable Worker Deletion in Active Directory: Allows the integration to delete AD user accounts when workers meet a configured termination or removal condition in ADP. For example, when a worker’s ADP record meets a termination rule, the integration deletes the corresponding AD account.
  • Map Manager for Worker: Synchronizes the manager listed in ADP to the manager attribute in Active Directory, maintaining accurate reporting relationships. For example, when a worker receives a new manager in ADP, the integration updates the manager's DN in AD.

Manage Candidates

  • Candidate Application Lookback (Days): Defines how many days back the integration looks at the application submission date when selecting candidates from ADP. For example, 30 processes only candidates whose applications were submitted in the last 30 days.

  • Candidate Application Status Filter: Controls which application statuses qualify a candidate for inclusion in the sync. For example, selecting Pre-Hire not started and Background check cleared includes only candidates who are in those hiring stages.

  • Candidate Removal Status Filter: Identifies application statuses indicating a candidate should no longer retain an AD account. For example, selecting Rejected and Marked for Rejection removes or disables AD accounts for candidates who did not move forward in the hiring process.

  • Map Candidate Manager: Copies the hiring manager from ADP into the candidate’s AD record to maintain reporting relationships. For example, when a candidate is assigned to a manager in ADP, the integration writes that manager's name to the manager attribute in Active Directory.

  • ADP Correlation Attribute (Candidate): Specifies which ADP field uniquely matches a candidate to an AD account. For example, using Person OID ensures that each candidate is matched using a stable, system-generated identifier.

  • Active Directory Correlation Attribute (Candidate): Specifies which AD field is used to find the matching AD account for a candidate. For example, selecting Employee ID matches candidates to AD users whose employeeId attribute equals the value stored in ADP.

  • Rejected Candidate Action: Defines what happens to the candidate’s AD account when ADP marks the candidate as rejected. For example, choosing Disable Account keeps the account for audit purposes but prevents the candidate from signing in.

  • Inherit Manager Division and Department: Automatically uses the hiring manager’s division and department when those values are not present on the candidate in ADP. For example, if the candidate’s record has no department, the integration assigns the department from the manager’s profile so the AD account still has a valid organizational placement.

  • Ignore Manually Created Candidates: Skip the candidates that were created manually and not by the integration. For example, if a candidate was added directly in ADP instead of through this integration, the candidate will be skipped.

Active Directory Group Assignment

  • Sync Group Membership on Update: Controls whether the integration should remove users from AD groups that no longer match their current ADP attributes and add them to any newly applicable groups. This setting ensures that group membership always reflects the user’s current role, department, or location. For example, if an employee transfers from Finance to Human Resources, enabling this option removes the user from Finance-Security and assigns the user to HR-Security, keeping their access aligned with their new role.
  • Remove Group Membership on Deactivation or Deletion: Specifies whether all AD group memberships should be removed when a user account is deactivated or deleted. This ensures that users who leave the organization, or are no longer active, cannot retain access to systems protected by group-based permissions. For example, when a worker is terminated and their AD account is disabled, enabling this option ensures the user is removed from all groups.

EntraID Group Management

  • Correlation Attribute from Entra ID: Defines which Entra ID user field is used to match users with Active Directory. For example, selecting User Principal Name (UPN) matches users based on their UPN, such as jane.doe@company.com.
  • Correlation Attribute from Active Directory: Specifies which AD attribute is used to identify the corresponding AD account when syncing Entra ID group membership. For example, choosing SAMAccountName allows matching a user like jd123 between AD and Entra ID.
  • Remove Users from Entra ID Groups on Update: Determines whether the integration should remove users from Entra ID groups that no longer match their updated information. For example, if a user moves from Sales to Finance, this option removes them from all Sales groups and keeps only the appropriate Finance groups.
  • Add Disabled AD Users to Entra ID Groups: Determines whether disabled AD accounts should still be added to the appropriate Entra ID groups. For example, if a contractor’s AD account is disabled but still needs to remain in a compliance group, enabling this option ensures the membership is still applied.

Update User

  • Update AD User Attributes: Keeps existing AD user accounts in sync with ADP by updating mapped fields such as name, job title, department, manager, and email whenever they change.
    For example, if ADP shows that a worker’s title changes from Sales Associate to Sales Manager, this option updates the same user’s title in AD.
    • Update Common Name (CN) and Distinguished Name (DN): Renames the AD object so that both the Common Name (CN) and the full Distinguished Name (DN) reflect the person’s updated name. The DN represents the object’s complete path in Active Directory, including its CN, organizational units (OU), and domain. For example, if Barbara Jane changes her last name to Barbara Miller, the integration updates the AD Distinguished Name (DN) from CN=Barbara Jane,OU=Employees,OU=Users,DC=company,DC=com to CN=Barbara Miller,OU=Employees,OU=Users,DC=company,DC=com.
    • Update User Principal Name (UPN): Updates the user’s sign-in name when the naming rule or the person’s name changes. For example, if the UPN format is first.last@company.com and a worker’s last name changes from Jane to Miller, this option changes the UPN from barbara.jane@company.com to barbara.miller@company.com.
    • Update Primary Email Address (mail): Updates the primary SMTP address in AD to match the current email address defined by the integration. For example, if a user moves to a new regional domain and their email changes from barbara.jane@us.company.com, to barbara.jane@eu.company.com this option updates the mail attribute to the new address.
    • Update sAMAccountName (Legacy Logon Name): Updates the older Windows login name when a user’s name changes and ensures the new login name remains unique in the domain. For example, if your standard is first initial + last name and a worker’s name changes from Barbara Jane to Barbara Miller, this option changes the sAMAccountName from bjane to bmiller.

Candidate Group Assignment Rules

The Candidate Group Assignment Rules tab controls how candidates created from ADP job applications are assigned to Active Directory (AD) and Microsoft Entra ID (Azure AD) groups when their accounts are created. These rules are evaluated only for candidates (not workers) and are applied during initial account provisioning.

AD Candidate Group Rules

The AD Candidate Group Rules section determines which on-premises Active Directory groups a candidate is added to. Each rule checks candidate details from ADP, such as Division or Location, and assigns matching candidates to one or more AD groups.

These rules are evaluated from top to bottom. When a candidate matches a rule, they are added to the AD groups listed in that rule. If no rule matches, a Default rule assigns the candidate to a fallback group, ensuring no candidate is left unassigned.

Consider the following example:

  • Rule 1: If Division = ATL_Robinson, assign the candidate to ATL_Candidates_AD
  • Rule 2: If Division = ARZ_Robinson, assign the candidate to ARZ_Candidates_AD
  • Default Rule: Assign all other candidates to General_Candidates_AD

Result:

  • A candidate with Division ATL_Robinson is added to ATL_Candidates_AD group.
  • A candidate with Division ARZ_Robinson is added to ARZ_Candidates_AD group.
  • Any other candidate is added to General_Candidates_AD group.

Entra Candidate Group Rules

The Entra Candidate Group Rules section controls how candidates are assigned to Microsoft Entra ID (Azure AD) groups. Each rule evaluates candidate attributes from ADP and maps matching candidates to one or more Entra ID groups during account creation. When a candidate meets the rule conditions, they are automatically added to the configured Entra ID groups.

A Default rule can be configured to ensure candidates are still assigned to an Entra ID group when no specific rule matches.

Consider the following example:

  • Rule: If Division = ATL_Robinson, assign the candidate to the ATL_Candidates_Entra group.
  • Default Rule: Assign all other candidates to All_Candidates_Entra group.

Result:

  • A candidate with Division ATL_Robinson is added to ATL_Candidates_Entra group.
  • Any other candidate is added to All_Candidates_Entra group.

Worker Group Assignment Rules

The Worker Group Assignment Rules tab controls how workers (employees) are automatically assigned to Active Directory (AD) and Microsoft Entra ID (Azure AD) groups during ADP worker syncs.

These rules apply when a person is treated as a worker, including:

  • New hires synced as workers.
  • Existing employees whose information changes in ADP.

These rules ensure that workers always have the correct group memberships based on their latest ADP data.

AD Worker Group Rules

The AD Worker Group Rules section determines which on-premises Active Directory groups a worker is added to. Each rule checks worker attributes from ADP, such as Division, Department, or Custom Fields, and assigns matching workers to one or more AD groups.

The rules are evaluated from top to bottom. When a worker matches a rule, they are automatically added to the AD groups listed in that rule. If no rule matches, a Default rule assigns the worker to a fallback AD group, ensuring no worker is left without group access.

These rules are applied during every worker sync, so group memberships can change over time as worker attributes in ADP change. If a person was first created as a candidate and later becomes a worker, the system keeps the same account and updates the group memberships based on the worker rules.

Consider the following example:

  • Rule 1: If Division = ARZ_Robinson and Custom Field = Warranty, assign the worker to ARZ_Warranty_AD group.
  • Default Rule: Assign all other workers to General_Workers_AD group.

Result:

  • A worker with the Division ARZ_Robinson and Warranty field set is added to the ARZ_Warranty_AD group.
  • All other workers are added to General_Workers_AD

Entra Worker Group Rules

The Entra Worker Group Rules section controls how workers are assigned to Microsoft Entra ID (Azure AD) groups during worker syncs. Each rule evaluates worker attributes from ADP and assigns matching workers to one or more Entra ID groups. When a worker meets the rule conditions, they are automatically added to the specified Entra ID groups.

A Default rule can be configured to ensure workers still receive Entra ID group assignments when no specific rule matches.

Consider the following example:

  • Rule: If Custom Field = Warranty, assign the worker to Warranty_Workers_Entra group.
  • Default Rule: Assign all other workers to the All_Workers_Entra group.

Result:

  • Workers with the Warranty field set are added to the Warranty_Workers_Entra group.
  • All other workers are added to the All_Workers_Entra group.

Active Directory OU Rules

The Active Directory OU Rules tab controls how candidates and workers are placed into Organizational Units (OUs) in Active Directory during account creation and updates. Each rule evaluates user attributes from ADP and assigns matching users to a specific Active Directory OU. When a user meets the rule conditions, their account is created in or moved to the specified OU.

A Default rule can be configured to ensure users are still placed into an OU when no specific rule matches.

Consider the following example:

Rule: If Worker Status = Active and Division = ATL_Robinson, assign the user to OU=John,DC=aqueralabs,DC=com.

Default Rule: Assign all other users to OU=Default,DC=aqueralabs,DC=com.

Result:

  • Active users in the ATL_Robinson division are placed in the John OU.
  • All other users are placed in the ****Default OU.

These rules ensure that both new and existing users are always located in the correct Active Directory OU based on their latest ADP data, keeping the directory structure organized and consistent.

Rule Behavior

  • Add Row allows you to add multiple attribute values within the same rule that all map to the same target.
  • Add New Rule allows you to create a separate rule with its conditions and target assignment.
  • Switch to Expression allows you to define advanced logic using expressions when simple attribute matching is not sufficient.
  • Rule Order is important. Rules are evaluated from top to bottom, and the Default rule is applied only when no other rule matches.

Attribute Map

The Attribute Map section shows how ADP Workforce Now attributes are mapped to corresponding attributes in Active Directory (AD) and Microsoft Entra ID (Azure AD). These mappings determine which ADP fields are used to create or update user attributes in each target system.

You can review and modify the source and target attribute mappings to meet your business requirements. Any changes made in this section directly affect how user data flows from ADP to Active Directory and Entra ID during user provisioning and ongoing updates.

Tables 

The Tables tab contains reference tables used by the integration to apply conditional logic during user provisioning and updates. These tables allow you to manage lookup-based decisions without hardcoding values into rules. Each table stores structured data that can be referenced by rules across the integration, such as email domain selection or exclusion lists.

Active Directory User Email Domain Table

This table defines which email domain to use when creating a new Active Directory user, based on company or business entity information from ADP.

For example, users belonging to Company C1 can be created with the email domain @aquera.com, while users from Company C2 can be created with @aqueralabs.com. During user creation, the integration looks up this table and automatically assigns the correct email domain based on the user’s ADP data.

Candidate Emails to Ignore

This table lists candidate email addresses along with a flag indicating whether each candidate should be excluded from integration processing.

If a candidate’s email appears in this table and is marked to be ignored, the integration skips creating or updating that candidate’s account in Active Directory and Entra ID. This is useful for excluding manually created accounts, test users, or specific candidates that should not be processed.
