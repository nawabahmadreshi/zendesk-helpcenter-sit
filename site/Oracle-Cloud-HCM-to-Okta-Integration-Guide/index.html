Oracle Cloud HCM to Okta Integration Guide

← Back to index

Overview

The Aquera platform allows your organization to scale complex data management by automating the integration between various applications. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

The integration helps your organizations keep employee information updated between Oracle Cloud HCM and Okta. It reads worker data, adds manager and location details, and creates new company email addresses when needed. Before updating, it checks both systems to make sure the email is unique and not already in use. This process ensures each employee has the right email and profile in both systems.

This guide explains how the integration reads and updates employee records in Oracle and Okta. It manages when to create or update user emails and provides detailed reports after each run.

Key Functions

The Oracle Cloud HCM to Okta Employee Data Integration performs the following functions:

  • Retrieves employee records from Oracle HCM, either full or incremental, based on recent updates or new hires.
  • Enriches each employee record with related details, such as manager and location information.
  • Generates and validates unique corporate email addresses for employees by checking availability across both Oracle HCM and Okta.
  • Updates or replaces the employee’s primary email in Oracle HCM, ensuring that any previous email records are properly closed.
  • Submits updates to Okta through bulk sessions, creating or updating user accounts as required.
  • Generates detailed reports summarizing the success, failure, or skipped actions for each employee record processed.

Integration Setup

This section outlines how the integration must be configured to synchronize employee data and email updates between Oracle HCM and Okta. The setup ensures that new, updated, or terminated employee records in Oracle HCM are accurately reflected in Okta, maintaining consistency across both systems.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin Portal.
  2. Navigate to Integrations from the left-side panel.
  3. Click Create Integration.
  4. Select Oracle HCM to Okta Employee Data Synchronization.
  5. Enter a name and description for the integration.
  6. Select your Oracle HCM and Okta connector instances.
  7. Click Continue. The Configuration page is displayed to set up the integration details.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section displays the Oracle HCM and Okta connectors linked to the integration. You can change or re-select the application instances if needed.

Parameters

The Parameters section allows you to customize how the integration behaves based on your business requirements. It includes options to define sync modes (full or incremental), control email generation rules, set retry limits, manage rate-limit pacing, and configure reporting preferences.

Integration Features

The Integration Features section allows you to enable or disable key functions that control how data is synchronized between Oracle HCM and Okta.

  • Enable Worker Management: Select this option to sync worker records from Oracle HCM into the Okta .
  • Enable Auto-Creation of Custom Attributes in Okta: Allows the integration to automatically create any missing custom attributes in Okta as part of the sync process.
  • Enable Date Offset: Enables synchronization based on a specific date range or time zone, allowing you to process only records updated after a certain date.
  • Enable Group Management: Activates the ability to manage and update user group assignments in Okta during synchronization.
  • Enable Unique Username/Email Generation: Generates unique usernames or email addresses automatically to prevent duplication across systems.

General Settings

The General Settings section defines the core configuration options that control how the integration runs. Configure the following settings to manage execution mode, logging, notifications, and reporting:

  • Enable Report-Only Mode: Select to generate a report detailing the data changes that would happen, but without applying them.
  • Enable Incremental Mode: Select to run the integration in incremental mode.
  • Enable Debug Mode: Select to generate logs outlining data changes that would occur for debugging.
  • Enable Logging: Select to generate logs outlining data changes.
  • Enable Report Generation:  Select to generate the integration report.
  • Enable Notifications: Select to enable the integration-related notifications.
  • Enable Log Insights: Select to visualize the integration-related metrics.

Worker Settings

The Worker Settings section defines which employee records from Oracle HCM are created, updated, or deleted in Okta. Configure the following settings to control how worker data is synchronized:

  • Users to be Created/Updated in Okta: Select All Active Workers to include every active employee from Oracle HCM whose details should be created or updated in Okta.
  • Users to be Deleted in Okta: Select All Terminated Workers to remove users in Okta whose employment has ended in Oracle HCM.

Manage Policy

The Manage Policy section defines the conditions and rules that control when and how user data is synchronized from Oracle HCM to target applications. Configure the following settings to customize synchronization behavior:

  • Update Trigger from Oracle HCM: Select the condition that determines when user data should sync from Oracle HCM to target applications. For example, you can set the trigger to run whenever an employee’s record is created or updated in Oracle HCM.
  • Enable Custom Update Logic: Turn on this option to apply custom JavaScript rules that control when and how user updates are processed. For example, you can define logic that updates users only if their department or location has changed.
  • Source Application: Choose the Oracle HCM instance that serves as the main source of employee data for synchronization. For example, select Oracle HCM Production if it is your organization’s primary HR system.
  • Pre-Hire Provisioning Days: Set how many days before an employee’s official start date their account should be created in target systems. For example, if you set this to 10, a new hire starting on May 20 will have their Okta account created on May 10.
  • Worker Type Selection: Choose which worker types (for example, full-time, part-time, or contractors) should be included in the synchronization process. For example, select All Workers to include every active employee, or limit it to Full-Time Employees only.
  • Email Format Rule: Define the rule for generating unique and consistent employee email addresses across all target systems. For example, the rule can generate emails in the format firstname.lastname@company.com and, if a duplicate exists, create firstname.lastname2@company.com.
  • Enable Code View: Select this option to view or edit the email generation rule using JavaScript for advanced customization. For example, you can modify the code to include middle initials or custom domain logic.

  • Email Preview: Displays example email addresses generated based on the defined rule.

    For example, preview results may show john.smith@company.com and john.smith2@company.com.

  • Enable Advanced Preview: Turn on this setting to view more sample variations of generated email addresses for testing different name patterns. For example, it can display additional combinations like j.smith@company.com or smith.j@company.com.
  • Target Applications: Select the systems (such as Okta and Oracle) where user accounts will be created, updated, or managed. For example, choose Okta and Oracle, so that both applications stay updated with the latest employee data.

Date Offset Setting

The Date Offset Setting section defines when onboarding and offboarding actions should occur relative to an employee’s hire or termination date. Configure the following settings to manage timing adjustments:

  • Onboarding Date Offset: Select when onboarding should start in relation to the employee’s start date.

    For example, select Offset Before Onboarding Date if you want accounts and access to be created a few days before an employee’s official start date so they’re ready on day one.

  • Days of Offset Before Onboarding: Enter how many days in advance onboarding should start, based on the offset setting. For example, if you set this to 7, an employee starting on May 15 will have their onboarding process triggered on May 8.
  • Offboarding Date Offset: Select when the offboarding process should begin in relation to the employee’s termination date. For example, choose No Offset if you want the offboarding to happen exactly on the termination date, or Offset Before Offboarding Date to disable access a few days before the employee leaves.

Okta Settings

The Okta Settings section defines how Oracle HCM attributes are mapped to Okta, including the creation of custom attributes. Configure the following settings to manage attribute mapping and synchronization:

  • Custom Attributes to be Created in Okta: Select the Oracle HCM attributes that are not available by default in Okta and need to be added as custom attributes. When selected, these attributes are created in Okta during the integration setup. For example, select ManagerPersonNumber and ManagerPersonId to create matching custom attributes in Okta that store each employee’s manager information from Oracle HCM.

Workforce Now Workers Debug Setting

The Workforce Now Workers Debug Setting section allows you to test and troubleshoot the synchronization process using filtered worker data. Configure the following settings to debug specific employee records:

  • Oracle HCM Filter: Select the filter used to retrieve worker data from Oracle HCM, ensuring that only specific records are included for debugging. For example, select Person Number to fetch data for a particular employee or Email Address to search for a worker using their email ID.

  • Person Number: Enter the unique identifier of the worker in Oracle HCM to test or debug their record during synchronization. For example, entering 100177 retrieves the details of the employee with that person number from Oracle HCM for debugging purposes.

    Note: Click the Refresh button to reload the list of available Oracle HCM attributes. For example, if a new field like ManagerDepartment is added in Oracle, clicking Refresh will make it appear in the list for selection.

    • Summary Only: Attribute details are not displayed. Only the count of CRUD operations is shown.
    • Full Details: All attribute details are displayed.
    • Custom: Includes or excludes selected attributes. The following option only appears when the dashboard detail level is set to Custom.
      • Attribute Filter: Controls which event attributes are included or excluded.
        • Included Attributes: Select the attributes to be included in the log.
        • Excluded Attributes: Select the attributes to be excluded from the log.

Reporting Settings

The Reporting Settings section defines how integration results and summaries are generated and shared. Configure the following settings to customize reporting preferences:

  • Select Report Detail Level: Select the desired level of report detail from the dropdown menu.The following are the available options:
    • Summary Only: Attribute details are not displayed. Only the count of CRUD operations is shown.
    • Full Details: All attribute details are displayed.
    • Custom: Includes or excludes selected attributes. The following options only appear when the report detail level is set to Custom.
      • Attribute Filter: Controls which attributes are included or excluded in the report.
        • Included Attributes: Select the attributes to be included in the CSV report.
        • Excluded Attributes: Select the attributes to be excluded from the CSV report.

Notification Settings

The Notification Settings section defines how and when email notifications are sent after each integration run. Configure the following settings to select the detail level and attributes included:

  • Notification Level: Select the level of detail you want to receive in email notifications about the integration’s execution.

    The following options are available:

    • Summary Only: Sends a brief notification showing the number of users created, updated, or deleted.
    • Full Details: Sends a detailed notification that lists all user attributes and changes processed.
    • Custom: Lets you include or exclude specific attributes in the notification.
  • Attribute Filter: Controls which attribute details are included in the notification when Custom is selected.
    • Included Attributes: Sends only the selected attributes in the email notification.
    • Excluded Attributes: Omits the selected attributes from the notification email.
  • Attributes to be Included: Choose the attributes that should appear in the notification when Included Attributes is selected. For example, include First Name, Last Name, and Email Address to see only these details in the notification.
  • Attributes to be Excluded: Choose the attributes that should be hidden in the notification when Excluded Attributes is selected. For example, exclude Employee ID or Last Updated Timestamp if you do not want to include internal system fields.

Note: Click the Refresh button to update the list of available attributes from Oracle HCM before selecting which ones to include or exclude.

  • Notify Admin Via Email: Select to send email notifications to all admins.
    • Admin's Email Addresses: Enter the email addresses of the admins to whom the email notifications must be sent.
    • Admin Email Subject: Enter the content that must be used as the email subject for the email notifications to the admins.
    • Admin Email Notification Template: Enter the content that must be used as the email body for the admin email notifications.
  • Notify Supervisor via Email: Select to send email notifications to all employees.
    • Supervisor Email Addresses: Enter the email addresses of the supervisors to whom the email notifications will be sent.
    • Supervisor Email Subject: Enter the content that must be used as the email subject for the email notifications to the employees
    • Supervisor Email Notification Template: Enter the content that must be used as the email body for the employee email notifications.

Reconciliation Settings

The Reconciliation Settings section defines how user deactivations are managed in Okta, ensuring terminated employees in Oracle HCM are properly deactivated. Configure the following settings to control execution and monitoring:

  • Enable Live Deactivation Mode: Select to perform actual user deactivations in Okta during the integration run. When disabled, the integration runs in Report Only mode and generates a preview report without applying any changes. For example, select this option to automatically deactivate users in Okta when they are terminated in Oracle HCM.
  • Enable User Deactivation Process: Select to activate the reconciliation workflow that deactivates users who have been terminated in Oracle HCM but are still active in Okta. For example, select this option to ensure all inactive employees are properly deactivated across systems.
  • Maximum Deactivations Per Run: Enter the maximum number of users that can be deactivated in a single integration run to control batch size. For example, enter 5 to limit each run to five user deactivations. The recommended range is 5–50 users per run.
  • Test User Filter (Email or Employee ID): Enter a specific user’s email address or employee ID to test the reconciliation process on one user before running it for all. For example, enter maria.smith@company.com to verify the deactivation process for that user only.
  • Send Alerts to Aquera Operations Team: Select to send automatic alerts to the Aquera Operations team for critical events such as integration failures or threshold breaches. For example, select this option to notify the Aquera team if a deactivation process fails or exceeds the configured limit.
  • Email Notification Recipients: Enter the email addresses (comma-separated) of users who should receive reconciliation notifications and status updates. For example, enter it.admin@company.com, hr.support@company.com to notify both IT and HR about each reconciliation run.
  • Okta User Statuses to Process: Select the Okta user statuses that should be included in the reconciliation process. For example, select STAGED, SUSPENDED, and PASSWORD_EXPIRED to evaluate users in these statuses for deactivation.
  • Enable User Exclusion Feature: Select this option to exclude specific users from the deactivation process. For example, select this feature to protect service accounts that must remain active.
  • Excluded Users (Email or Employee ID): Enter the email addresses or employee IDs of users who should be excluded from the deactivation process. For example, enter admin@company.com, test.user@company.com to skip these accounts even if they meet the deactivation criteria.

Monitored Provisioning Settings

The Monitored Provisioning Settings section defines which provisioning operations are tracked. Configure the following settings to monitor user and group changes based on selected actions and criteria:

  • Operation Monitoring:

    Select whether to monitor all provisioning activities or track specific ones only. For example, select All Operations to monitor every action or Customized Monitoring to focus on selected operations.

  • Monitoring Type: Select how you want to customize the monitoring setup. For example, choose Monitor Selected Workers to track specific users or Monitor Selected Operations to monitor certain actions.
  • Workers to Monitor: Select which employees or worker groups you want to monitor. For example, monitor only new hires, contractors, or users from the Finance department.
  • Worker Filter Rule: Define a condition to decide which workers should be monitored. For example, set Department = “IT” to monitor IT employees only.
  • User Delete Monitoring: Select whether to track all user deletions or just specific ones. For example, select All Deletes to monitor every deletion or Select Deletes to track removals based on conditions.
  • Delete Filter Rule: Define a rule to track only certain user deletions. For example, set Employment Type = “Temporary” to monitor deletions for temporary staff.
  • User Create/Update Monitoring: Select whether to track all user creation and updates or just specific ones. For example, choose All Upserts to monitor all adds and updates, or Select Upserts for targeted tracking.
  • Upsert Filter Rule: Define a condition to monitor certain user creation or update events.For example, track updates only when Job Title or Manager ID changes.
  • Fields to Monitor for Changes: Select which user fields should be tracked for changes. For example, choose Email, Department, or Job Title to track updates when these fields change.
  • Group Change Monitoring: Select whether to track all group changes or just specific ones. For example, select All Group Changes to track every addition and removal, or Select Group Changes for key access groups.
  • Group Filter Rule: Define a condition to track specific group changes. For example, monitor group updates only for Admin Access or Security Groups.
  • Group Additions Monitoring: Select whether to track all group additions or only specific ones. For example, choose All Additions to monitor every user added to a group, or Selective Additions for high-privilege groups.
  • Group Removals Monitoring: Select whether to track all group removals or just selected ones. For example, choose All Removes to monitor all removals or Selective Removes to focus on key access groups.

Group Management Settings

The Group Management Settings section defines how users are assigned, updated, or retained in groups. Configure the following settings to automate group management using attribute rules and lifecycle events:

Group Rules

  • Choose Attribute: Select the user attribute that determines group membership. For example, choose Department if group membership is based on the user’s department.
  • Select Target: Select the target group where users matching the chosen attribute will be added. For example, select Finance Team as the target for all users whose department is Finance.
  • Switch to Expression: Switch to expression-based logic for more advanced rule definitions. For example, define a rule like (Department = “IT”) AND (Location = “New York”) to add only New York IT users to the group.
  • Add New Rule: Click to add another group mapping rule. For example, create separate rules for HR, IT, and Finance teams.

Group Retention

  • On Deactivate / Delete: Select how user group memberships should be handled when a user is deactivated or deleted. For example, choose Retain in selected groups to keep users in specific access groups after deactivation.

    The following are the available options:

    • Remove from all assigned groups: Removes the user from every group.
    • Retain in all assigned groups: Keeps the user in all current groups.
    • Retain in selected groups: Keeps the user in specific groups only.

  • On Update: Select how to manage group memberships when a user’s profile is updated. For example, choose Retain in selected groups to keep the user in specific groups even after updates.

    Available options:

    • Remove from all assigned groups – Clears all group memberships on update.
    • Retain in all assigned groups – Keeps all existing memberships.
    • Retain in selected groups – Keeps only selected groups.

  • On Create: Select how group assignments are handled when a new user is created.

    For example, choose Custom Mapping to assign groups based on rules or attributes.

    Available options:

    • Do Not Assign to Any Groups: Skips automatic assignment.
    • Assign to Default Group(s): Adds new users to predefined default groups.
    • Assign Based on Attribute Rules: Uses defined group rules for mapping.
    • Custom Mapping: Uses your defined logic for assigning groups.
    • Exclude Group: Prevents assignment to specific groups.
    • Validate All Groups: Checks group eligibility before assignment.
  • Group Management Restriction List: Enter a comma-separated list of target groups that must be managed through attribute mapping. Users who no longer meet the defined criteria are automatically removed from those groups, while qualifying users are added. For example, enter Finance, HR, IT Security to ensure these groups are strictly managed via mapping rules.

Attribute Mapping

Attribute mapping defines how data fields from Oracle HCM are linked to corresponding fields in Okta. Each mapped attribute ensures that user information, such as names, email addresses, department details, or manager data, is accurately transferred and synchronised during the integration process. You can customize mappings to match business requirements, apply transformations (such as formatting usernames), and include additional attributes to maintain data consistency across connected applications.

It ensures that any changes in Oracle HCM, such as a department transfer or manager update, are automatically reflected in Okta. Administrators can review, adjust, or extend mappings to meet organizational policies and integration needs, making it easier to manage user lifecycles without manual updates.
