HR to Okta Integration Guide

← Back to index

Overview

This guide applies to the 1.0.0 version of the integration.

Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

With this integration, employee records created or updated in the HR system are automatically synchronized with Okta based on the configured schedule. Additionally, when an employee is terminated in the HR system, the corresponding user account in Okta is automatically deactivated (by default), helping ensure that identity data remains accurate and up to date.

This guide explains how to configure the HR to Okta integration and synchronize employee data from your HR application to user accounts in Okta. 

Prerequisites

The following are the prerequisites:

  • Aquera Tenant Account: You must have a tenant account in the Aquera Admin portal. Contact your Aquera account manager to get the Aquera tenant access.
  • Configure Applications in the Aquera Admin Portal: 
    • HR application connector: Ensure that the HR application connector used by the integration is added and configured in the Aquera Admin Portal. 
      Note: This integration is currently tested and certified with ADP Workforce Now (v30) and Dayforce (v29).
    • Okta Connector: Ensure you add and configure the Okta connector (v11) in the Aquera Admin portal. Refer to the Okta Configuration Guide for details.

Integration Setup

Aquera's HR to Okta integration automatically synchronizes employees' data from an HR system to Okta based on a configured schedule.

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel.
  3. Click Create Integration.
  4. Select HR to Okta. The Create New Integration pop-up window is displayed.
  5. Enter a name and a description for the integration.
  6. Click Continue. The Configuration tab of the integration setup page is displayed. The configurations are categorized under various sections within the Configuration tab.
  7. In the Connections section, select the source and target applications.
    • From the Source Application drop-down list, select the HR system from which employees must be retrieved. 
    • Select the Okta connector from the Target Application drop-down list.
  8. Click Save Changes to save the integration configurations.
  9. Set the configurations as per your requirements. Refer to the Integration Configurations Reference Guide for details.

Configuration

The Configuration tab includes a list of configurations required to set up the integration. 

Connections

The Connections section enables you to configure the application connectors that you want to use for the integration.
Note: The Source Application drop-down list shows only the HR system supported by the integration, and the Target Application drop-down list shows all the Okta connector instances configured in your tenant.

Under the Source Match Expression section, you must specify the source attribute and the target attribute that must be used to match employee records between the HR system and Okta. 

Note: Click Switch to Expression Mode to configure a JavaScript expression that must be used to identify the matching records. For example, SOURCE.EmployeeNumber == DEST.EmployeeNumber

For detailed information on all the configuration fields in the Configuration tab, refer to the Integration Configurations Reference Guide.

Link Analysis

The Link Analysis feature identifies duplicate or conflicting resource records across the source and target applications by comparing unique user attributes. It uses smart algorithms to classify resources as linked and unlinked, and supports cleanup between applications.

  1. In the Source Match Expression section, select the source employee attribute and the Okta user attribute that are used to match employee records between the HR system and Okta.
  2. Click Run Link Analysis. 
  3. Once the analysis is completed, click View Report.
    • The Linked tab displays user records for which the values of the defined match attributes (in Source Match Expression) exactly match across the HR and Okta applications.
    • The Unlinked tab displays users for whom no matching HR record was found using the match expression. By default, the tab displays a system-generated Default Name Filter that helps identify potential single-match records across the applications after a successful analysis run. 
  4. Review the list of source user records corresponding to the target records, and if the records match, click Update Correlation Attribute in the Unlinked tab and then click Update to transfer the attribute value from the source application to the target application. The update process runs in the background.
  5. Once the match key is updated in the target application, the corresponding user record is automatically moved to the Linked tab.

For details on the Link Analysis Report and the actions that can be performed on the report, refer to the Integration Analysis Guide.

Mapping and Assignments

The Mapping and Assignments tab consolidates the configuration options for setting the user naming attributes and mapping attributes between the HR and Okta applications. 

User Naming Attributes

The user naming attributes are Username and Email. Aquera's HR to Okta integration enables you to generate unique names for these attributes while creating a user in Okta. The following are the available configurations:

  • Suffix
  • User Naming Configuration

Suffix

This section enables you to configure the domain name for username and email as follows:

  • Username suffix: Enter the domain name to be used as the suffix in the username.
    You can configure a static domain name (for example, aquera.com) as the username suffix or define conditions to assign a specific suffix.
    To define conditions, click Conditional Username Suffix and set conditional logic to assign domain names to users accordingly. 


    This is an AI Prompt field that uses Aquera Intelligence (AI Assistant) to generate domain names during integration runs. For details on how to set up an AI prompt, review and analyze the output generated by the AI Assistant, refer to the Using Aquera AI Assistant for Generating Unique Values in Integrations guide. 
     
  • Email suffix: Enter the domain name to be used as the suffix for the email address. If the email suffix has to match the username suffix, select Email Suffix same as UserName Suffix.
    To assign domain names based on some conditions, click Conditional Email Suffix to define conditional logic and assign a specific work mail suffix. 


    This is an AI Prompt field that uses Aquera Intelligence (AI Assistant) to generate email domain names during integration runs. For details on how to set up an AI prompt, and to review and analyze the output generated by the AI Assistant, refer to the Using Aquera AI Assistant for Generating Unique Values in Integrations guide. 

User Naming Configuration

This section enables you to define the formats to be used for generating values for the Username and Email attributes for a newly created user in Okta.

  • Basic User Naming Configuration: In the Basic mode, select a naming format from the User Naming Format dropdown list to generate a unique value for Username and Email when creating a new user in Okta. The selected format will be used to generate unique names for both the attributes.
    In the basic mode, the Username and Email attributes are mutually dependent and are evaluated together. This means the integration will verify whether the values generated using the selected expression are available for both these attributes before assigning them to the new user. If any value is already in use for an attribute (that means, it is already assigned to a user in Okta), then the integration will try appending a number (starting from 1) to the attribute values and checking for the availability. This process continues for up to 20 attempts or until unique values are generated for both attributes. Once unique values are found for those attributes, those values will be assigned to the new user.
     
  • Advanced User Naming Configuration: Select the Advanced User Naming Configuration checkbox to switch to the Advanced setup mode. In this mode, you can set different formats for Username and Email attributes. 


    When you switch to the Advanced mode, you can select whether email should be mutually dependent with username, as shown below:


    Define the user naming formats in the Prompt to generate the UserName format and Prompt to generate the Email Format fields. For both email and username attributes, the naming format will be applied on user creations and updates.
    These fields are AI Prompt fields that use Aquera Intelligence (AI Assistant) to generate Username and Email values during integration executions. For details on how to set up an AI prompt and to review and analyze the output generated by the AI Assistant, refer to the Using Aquera AI Assistant for Generating Unique Values in Integrations guide.

Attribute Map

The Attribute Map page displays the default mapping from source employee attributes to Okta user attributes. By default, a mapping row is available for the External ID attribute in Okta. 

Perform the following steps to add a new mapping from an employee attribute to an Okta user attribute:

  1. Click Add Row at the end of the attribute mapping list.

  2. Click on the empty field on the source side to include an employee attribute. The list of employee attributes appears. Select an attribute from the dropdown list to add it as the source attribute. 

    You can even write a JavaScript expression in the source field. Click the Enter Expression option at the end of the attributes list to write the expression. The expression is evaluated at runtime and the resulting value is assigned to the corresponding target attribute.
    Note: Click the icon to expand the Expression editor area to write longer expressions.
     

  3. In the Attribute field on the target side, select the Okta attribute name from the drop-down list.
    Note: Refer to Okta schemas for details on the attribute names.
  4. Select the data type of the Okta attribute from the dropdown list.
  5. Select one of the following mapping options:
    • Map on Create and Update: The Okta attribute will be updated during the create and update operations.
    • Map on Create: The Okta attribute will be updated only during the create operation.
    • Map on Update: The Okta attribute will be updated only during the update operation.
    • Don't Map: The Okta attribute will never be updated and the mapping will be ignored during the integration runs.
  6. Click the check mark to add the new mapping to the attribute map.

Note: The values of the default mapping are converted to normalized characters. This means that any characters with diacritics are substituted with the nearest equivalent non-diacritic format of those characters.

Attribute Map Analysis

The Attribute Map Analysis evaluates and compares user attributes across the source and target applications to identify suitable attribute mappings for the integration. It highlights Matched and Unmatched users to guide accurate data correlation.

  1. Click Run Analysis. The following pop-up window is displayed:
  2. Select the source attribute and the target attribute that must be used for correlation. The values in these attributes must be unique for a user and must be the same across the source and target applications for identifying the matching users.
  3. Click Run analysis to run the Attribute Map analysis. The attribute map analysis will analyze all existing attributes between both applications for the eligible users and will generate a report.
  4. After the analysis has been completed, click View Report to view the Attribute Mapper Report.
    You can change the map options and the source attributes, if required.
  5. Click Export Mapped at the top right corner of the page to export and use this Attribute Map in the integration. Only the attributes that are mapped in the report are exported to the attribute map.

For details on the Attribute Map Report and the actions that can be performed, refer to the Integration Analysis Guide.

Preview Employee Data

You can preview the attribute values of an employee record that will be created or updated in Okta based on your mappings.

To preview the attribute values of an employee, perform the following steps:

  1. Click the Enter objects for preview field at the top of the Attribute Map page. A pop-up window is displayed.
  2. Select id and enter the ID of the employee.
  3. Click Preview. The employee data are displayed corresponding to each mapping provided on the Attribute Mapper.

Notes:

  • When the source attributes or expressions provided in the attribute map are invalid or do not return a value, the preview will not appear.
  • When the attributes or expressions evaluate to a value that contains diacritics (such as teléfono), the preview will not accurately show the diacritic signs.

Group Management

The Group Management tab provides options to configure how group memberships must be handled through this integration. You can determine the group assignment, retention, and removal of users based on the update or deactivation of employees in the HR system.

Note: You can also set the manage group memberships via Attribute Map (under Mapping and Assignments) where you can write JavaScript expressions in the mapping, which will return an array of groups.

Select Enable Group Management to set the options for managing user groups. If this checkbox is not selected, the integration does not manage group assignments to users in Okta.

When you select Enable Group Management, the following sections are displayed for further configuration:

  • Group Rules
  • Group Retention

Refer to the following sections for details.

Set Up Group Rules

You can set rules for assigning users to the Okta groups through either the Group Rules section of the Group Management tab or the Attribute Map section of the Mapping and Assignments tab. 
Note: When you use the Attribute Map functionality to set the group rules, you have to write JavaScript conditional expressions to assign the groups. 

In the Group Rules section, you can create conditions that control how users are added to groups. Each rule defines the conditions for assigning users to a set of groups.

Perform the following procedure to create group assignment rules:

  1. Define the conditions for assignment to groups using one of the following ways:
    • Using Condition Builder
    • Using Expressions
  2. If required, click Add Row to add another condition within the same rule. When multiple rows are added within a rule, an OR condition is applied to the rows. That means, for each user, the integration evaluates the conditions in each row sequentially. If the conditions defined in a row are satisfied for a user, the user is assigned to that group and the remaining rows in the rule are not evaluated for that user.
  3. If required, click the + icon at the top-right corner of the rule setting section to add another rule with a set of rows/conditions for group assignments. When multiple rules are set, the integration evaluates every rule for each user.

    Note: You can add a maximum of 10 rules. 

Using Condition Builder

The Condition Builder is the default view for configuring rules. A blank row is displayed by default for configuring the first set of conditions.

  1. Click on Choose Attribute to select an employee attribute that you want to use in the condition.


     
  2. After selecting the attribute, specify the value(s) of the attribute. Any user that matches the specified attribute values (OR operation) will be assigned to the group when the integration runs.

    Note: The attribute values must be entered as chips. However, after you exit the Edit mode, these attribute values appear as comma-separated entries.
  3. If required, add conditions with more attributes and values by clicking the + icon. When multiple attributes are added within the same row, those attribute conditions are evaluated using an AND operation. Only users who match all the specified conditions will be assigned to the group.
     
  4. In the Assign To field, select the groups where users will be placed. 

Using Expressions

Follow this procedure to set the rules using JavaScript expressions. A blank row is displayed by default for configuring the first set of conditions.

  1. Click Switch to Expression. The following options are displayed:
    • For Conditions Only: Select this option to use JavaScript expressions for setting the conditions. You must select the corresponding group from the Assign To field.
    • For Conditions and Assignments: Select this option to use JavaScript expressions for setting the conditions and the group assignments.

You can expand the Expression editor area by clicking the icon and then write the JavaScript expressions.

Note 1: Click Switch to Condition Builder if you want to switch out of the Expression mode.

Note 2: When you switch to the Expression mode or to the Basic Condition mode, all the configured conditions of the rule will be cleared.

Note 3: Expressions can use all the employee and user attributes in the HR and Okta applications. The configured expressions must always be evaluated to true or false.
SOURCE must be used in an expression to represent an attribute in the source application and DEST to represent an attribute in the destination application. For example, SOURCE.EmployeeNumber, SOURCE.HomeOrganization.ShortName, DEST.EmployeeNumber, and so on.  

Set Up Group Retention Policy

The Group Retention section allows you to define how a user's group memberships must be handled when the user record is updated or deactivated.

On Deactivate/Delete

This drop-down list is used to specify what should happen to a user's group memberships in Okta when the user is deactivated. It includes the following options:

  • Remove from all assigned groups: All the group memberships currently assigned to the user will be removed. However, if the deactivated user satisfies some of the group assignment rules and they are not already part of those groups, they will be added to those groups; if they are already part of those groups, they will be retained in those groups.
  • Retain in all assigned groups: All the group memberships currently assigned to the user will be retained, even if the user does not satisfy the rules configured for the group assignments in the Group Rules section.
  • Retain in selected groups: This option has to be used when you want to retain the deactivated users only in specific groups. When you choose this option, the following additional fields are displayed:
    • Groups: In this field, you need to select the specific groups in which a user's membership has to be retained. For example, if a user is part of G1, G2, and G3 groups and if you have specified G3 and G4 in this field, the user's membership in G3 will be retained. Deactivated users will be removed from all other groups.
    • Condition: This is an optional field to define a filter condition to apply the retention rule only to a specific set of users from the deactivated list of users. The group membership will be retained only for the users matching the filter condition. For the remaining set of deactivated users that do not match the filter condition, no action will be taken on their group memberships. Here is a sample filter condition:
      '(SOURCE.WorkAssigmentsPositionDepartmentXRefCode).includes("Admin") && (SOURCE.BusinessTitle).includes("Manager") && (SOURCE.BusinessTitle).includes("HR")'

On Update

This drop-down list is used to specify what should happen to a user's group memberships when the user is updated. It includes the following options:

  • Remove from all assigned groups: All the group memberships currently assigned to the user will be removed. However, if the user satisfies the conditions defined in the Group Rules section, they may be added to new groups based on the configured group rules.
    Note: If the Update Deactivated Users configuration is enabled, a deleted or deactivated user will be added to a new group based on the group rules. However, they will not be added back to an existing group from which they have been removed during this integration run.
  • Retain in all assigned groups: All the group memberships currently assigned to the user will be retained. Even if the user does not satisfy a rule configured for the group assignments in the Group Rules section, they will be retained in those groups.
  • Retain in selected groups: This option has to be used when you want to retain the updated users only in specific groups. When you choose this option, the following additional fields are displayed:
    • Groups: In this field, you need to select the specific groups in which a user's membership has to be retained. For example, if a user is part of G1, G2, and G3 groups and if you have specified G3 and G4 in this field, the user's membership in G3 will be retained on user update.
    • Condition: This is an optional field to define a filter condition to apply the retention rule only to a specific set of user records that were updated. The group membership will be retained only for the users matching the filter condition. For the remaining set of updated users that do not match the filter condition, no action will be taken on their group memberships. Here is a sample filter condition:
      '(SOURCE.WorkAssigmentsPositionDepartmentXRefCode).includes("Admin") && (SOURCE.BusinessTitle).includes("Manager") && (SOURCE.BusinessTitle).includes("HR")'

Group Assignments Analysis

The Group Assignments Analysis feature enables you to resolve group assignment conflicts for users in an application. The analysis identifies specific attributes a user must have to belong to a group and generates a report of assignment rules based on those attributes. The report can then be exported as a group assignment rule and applied to the integration.

  1. Click Run Analysis. The following pop-up window is displayed:
  2. Enter the source attribute and the Okta attribute that must be used for correlation. These attribute values are used for identifying the matching users.
  3. Click Run analysis to run the Group Assignments analysis. It will analyze all the users in the groups and will generate a report that contains the assignment rules based on the pattern of user data in each group.
  4. After the analysis has been completed, click View Report to view the Group Assignments Analysis Report.

    You must enable the mapping for the group assignment rules that you want to include in the integration.
  5. Click Export Mapped at the top right corner of the page to export and use these Group Assignment rules in the integration. Only the mapped rules are exported to the Group Management page.

For details on the Group Assignment Analysis Report and the actions that can be performed, refer to the Integration Analysis Guide.

Known Limitations and Issues

  • You can save the integration without configuring the mandatory fields; however, the integration runs will fail.
  • To view all the Dayforce attributes in the integration, it is required to download a report of all employees from the Data tab of the Dayforce connector configuration page.
  • The Incremental Tuning configuration is designed to function only when the Match expression only option is selected. Hence, the Username option has been removed from this configuration parameter.
  • When you set up group assignment rules, the attribute values must be entered as chips. However, after you exit the Edit mode, these attribute values appear as comma-separated entries.
  • The Import option is not available on the Group Assignment setup page. Hence, group assignment rules cannot be created via a CSV import.
  • The Export option is not available on the Group Assignment setup page. Hence, group assignment rules cannot be exported as a CSV file.
  • You can save a group management rule even if some rows in the rule have groups with empty conditions. This issue will be fixed in a future release.
  • During the Group Assignments Analysis, the source and target correlation attributes must be manually entered. No drop-down list is available to select an attribute from a list of attributes.
  • In Attribute Map, when you search for a source attribute using the attribute name, duplicate attribute names may be returned in the search result. This issue will be fixed in a future release.
