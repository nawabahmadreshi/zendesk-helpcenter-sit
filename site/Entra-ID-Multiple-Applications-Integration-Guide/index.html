Entra ID / Multiple Applications Integration Guide

← Back to index

Overview

The Aquera platform enables your organization to scale complex data management by automating data integration across various resource types. Aquera's integrations facilitate automatic provisioning, ensuring consistency across systems. It enables organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately enhancing efficiency in managing complex IT infrastructures and applications.

This integration provides a SCIM-based provisioning system that synchronizes user data between Microsoft Entra ID (formerly Azure AD) and target applications.

By default, the integration uses the User Principal Name (UPN) from Entra ID and matches it with the Username in the target application to identify user records, unless custom correlation attributes are configured in the Applications section.

Key Functions

This integration automates identity lifecycle management with the following primary capabilities:

  • Synchronizes user data from Entra ID to multiple target applications.
  • Manages group assignments.
  • Generates password for new users in the target applications.
  • Sends notifications about provisioning events via email and Microsoft Teams.
  • Generates detailed logs and reports.

Prerequisites

The following are the prerequisites:

  • Tenant Account in the Aquera Admin Portal: You must have a tenant account with admin access in the Aquera Admin portal. Visit https://admin.aquera.io and register to get a tenant account. Contact your Aquera account manager to get the Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/.
  • User Account in Entra ID: You must have a user account in Entra ID with permission to configure provisioning (for example, Application Administrator, Cloud Application Administrator, Application Owner, or Global Administrator).
  • Enterprise Application in Entra ID: If your use case requires an enterprise application in Entra ID, refer to the Creating an Enterprise Application in Entra ID section to learn how to create an enterprise application.
  • Microsoft 365 Application Connector in the Aquera Admin Portal: You need to configure the Microsoft 365 application connector in the Aquera Admin Portal. Refer to the Microsoft 365 Configuration Guide for configuration details.
  • Target Application Connector in the Aquera Admin Portal: You need to configure a connector in the Aquera Admin Portal for your target application. For example, if Box is your target application, then you must have a Box application connector in the Aquera Admin Portal. Refer to the configuration guide for your application connector for configuration details.
    Note: If your use case requires multiple target applications, then you must have a connector for each target application.

Creating Enterprise Application in Entra ID

You must have an enterprise application in Entra ID for each target application included in the integration. This enterprise application will serve as a source for user records and will be used to provision users to the target applications.

Perform the following steps to create an enterprise application:

  1. Log in to your Microsoft Azure portal.
  2. Search and select Enterprise Applications.
  3. Click New application.
  4. Search for the application for which you want to create an enterprise application, and select it from the search results. For example:
  5. Enter a name for the application, and click Create.

    The enterprise application is created and available on the Enterprise applications page.

Note: To learn how to assign users and groups to an enterprise application, refer to this Microsoft Documentation.

Integration Setup

This section details the configuration and mapping process for integrating Entra ID with target applications. It outlines the steps to create the integration, set up integration preferences, and define how data is mapped.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Search for Entra ID / Multiple Applications in the search box next to the Explore Templates section, and then select it to create a new integration.
  5. Enter a name and a description for the integration.
  6. Click Continue.

Once created, the following tabs are displayed:

  • Overview
  • Source
  • Applications
  • Rules
  • Operation
  • Tables
  • Schedule

Overview

Source

Navigate to the Source tab to identify the Microsoft Entra ID application connector used for the integration and define the required source settings.

The following sections are displayed in this tab for configuration:

Source Application

This section allows you to identify the application used for the integration. The following field is displayed:

  • Entra ID application: Select the Entra ID application connector that you have configured in the Aquera Admin portal. If you do not find your application connector in the list, click the refresh icon available next to the field to reload the available connectors.

Source Configuration

Configure the following source settings as per your requirement:

  • Enable custom eligibility rule: By default, all active users are eligible for provisioning. Enable this to define a custom eligibility rule that identifies users eligible for provisioning.
    When enabled, the following fields are displayed for configuration:
    • Custom eligibility rule builder: Enter your prompt to define a rule that identifies users eligible for provisioning. This field uses Aquera Intelligence (AI Assistant). For details on how to set up an AI prompt, review and analyze the output generated by the AI Assistant, refer to the Using Aquera AI Assistant for Generating Unique Values in Integrations guide.
    • Action for users not eligible for provisioning: Select the action to be performed on users who do not meet the provisioning eligibility criteria.
      The following are the available options:
      • Skip: Select this to ignore users who do not meet the provisioning eligibility criteria. No provisioning or deprovisioning action is performed for the user, and the user is excluded from processing during the integration execution.
      • Terminate: Select this to automatically terminate users who do not meet the provisioning eligibility criteria.

  • Customize how users are matched between Entra ID and the target application: By default, user records are matched using the User Principal Name (UPN) in Entra ID and the Username in the target application.     • Entra ID attribute for matching users: Select the correlation attribute in Entra ID that uniquely identifies a user record in your target application. For example, select employeeId if you want to use it to match user records in the target application.
    • Target application attribute for matching users: Select an attribute in the target application that matches the selected Entra ID attribute value to identify common users. For example, select ENT.employeeNumber if you want to use it to match user records in Entra ID.
  • Generate password for new users: By default, a password is not generated for new users during account provisioning in the target application. Turn on this toggle button to generate a random password automatically.
    When enabled, the following fields are displayed for further configuration:
    • Generate password using policy: Turn on this toggle button if you want passwords to be generated based on a defined password policy.
    • Policy for password generation: Select the password policy to be used for this integration. All password policies created in the Password Policies section will be available for selection.
  • Enable safeguard limits: Turn on this toggle button to define limits that restrict the number or percentage of provisioning operations. When enabled, the following fields are displayed for further configuration:
    • Safeguard limit type: Select whether the safeguard limits are defined as a number or a percentage.
    • Safeguard limit setup: Specify whether a single safeguard limit is applied across all operations or separate limits are configured per operation type. The following are the available options:
      • Single limit for all operations: Select this if you want to define a limit for all operations. When selected, the following field is displayed for configuration:
        • Safeguard limit for all operations: Enter a value to define the safeguard limit for all operations. After the limit is reached, no create, update, or terminate operations will be executed.
      • Separate limits per operation type: Select this if you want to define a separate limit for all operations. When selected, the following fields are displayed for configuration:
        • Select operations to apply safeguard limit: Select the operations for which you want to enable safeguard limit. The available options are:
          • Creation: Select this option to enable safeguard limit for user creation. When enabled, the following field is displayed for further configuration:
            • Create limit: Enter a value to define the safeguard limit for creation. After the limit is reached, create operations will no longer be executed.
          • Update: Select this option to enable safeguard limit for user update. When enabled, the following field is displayed for further configuration:
            • Update limit: Enter a value to define the safeguard limit for update. After the limit is reached, update operations will no longer be executed.
          • Termination: Select this option to enable safeguard limit for user termination. When enabled, the following field is displayed for further configuration:

    Source attributes are listed on the left side and the target attributes are listed on the right side of the page. You can modify only the source attributes as needed, to align with your specific use case.
    JavaScript expressions can be used to transform source attribute values and construct the values that are updated in the target attributes. You can enter the expression directly in the source attribute field or click the expand icon to write a longer expression.

Access Bundles

An access bundle is a logical group created within the integration system to make group assignments easier. It works as an intermediate layer between the source system and the target application groups.

In this approach, users are first assigned to access bundles based on specific attributes configured in the Assignments section. These access bundle are then mapped to the appropriate groups in the target applications.

This section allows you to define access bundles and map them to the target application groups.

In this section, the following fields are displayed:

  • Name: Provide a name for the bundle.
  • Description: Provide a short description for the bundle.
  • Target applications managed by this access bundle for group assignments: Select the target applications for which you want to manage group assignments through this access bundle.
    Note: All target applications configured in the Applications section will be available for selection. If you do not find your application in the dropdown list, click the refresh icon available adjacent to the field to refresh the list.
  • Target application groups for user assignment through this access bundle: Select the group names available in your target applications to which users will be assigned through this access bundle.
    Note: All groups from the applications selected in the Select Applications field will be available for selection. If you do not find your group's name in the dropdown list, click the refresh icon available adjacent to the field to refresh the list.

Assignments

Use this section to define rules for assigning access bundles to users based on Microsoft Entra ID attributes, Entra ID groups, or enterprise applications. The integration evaluates these rules during execution and assigns the configured access bundles in the Access Bundles section to matching users.

The following rule types are available:

  • Attribute Based
  • Entra ID Group Based
  • Enterprise App Based

Attribute Based

Use this tab to assign access bundles based on Microsoft Entra ID user attribute values.

For example, you can assign access bundles to users whose Department attribute is set to Engineering.

The following columns are available by default:

  • Attribute column: Select the Microsoft Entra ID user attribute to evaluate. For example, Department.
  • Access Bundles: Select one or more access bundles to assign when the attribute condition matches.

Refer to the Modifying Assignment Rules to learn how to edit rule assignments.

Entra ID Group Based

Use this tab to assign access bundles based on users' Microsoft Entra ID group memberships.

The following columns are available by default:

  • Entra Groups: Select the Microsoft Entra ID groups to be evaluated.
  • Attribute column: Select an additional user attribute to further filter matching users, if required. For example, Department.
  • Access Bundles: Select one or more access bundles to assign to users who match the configured conditions.

Refer to the Modifying Assignment Rules to learn how to edit rule assignments.

Enterprise App Based

Use this tab to assign access bundles to users associated with specific enterprise applications in Microsoft Entra ID.

The following columns are available by default:

  • Enterprise Applications: Select the enterprise applications to be evaluated.
  • Attribute column: Select an additional user attribute to further filter matching users, if required. For example, Department.
  • Access Bundles: Select one or more access bundles to assign to users who match the configured conditions.

Refer to the Modifying Assignment Rules to learn how to edit rule assignments.

Modifying Assignment Rules

This section explains how to edit access rules for assigning access bundles to users.

Perform the following steps to edit access rules:

  1. Click the vertical ellipsis next to any column, and then click Add Attribute to view the list of available attributes you can use to set the assignment rule.

    If you want to remove the existing column, click Delete Column.
  2. Search and select the attribute you want to use to define a condition. For example, select Country.
  3. In the next row, double-click and select the attribute value to define the rule. If multiple values are selected, OR condition is applied among them. For example, in the following sample condition, a user's department must be either QA or Development to satisfy this assignment rule.
  4. In the Access Bundles column, select the target access bundle names to which users who satisfy the specified condition will be assigned.
  5. If required to add another condition and assignment for the same selected attribute, click Add Row.
    Note: You can add any number of rows within a rule.
  6. If required to include another attribute to the assignment rule, click the vertical ellipsis next to a column, and then click Add Attribute.
    If multiple columns are added for a rule, AND condition is applied between different columns. For example, in the following sample condition, a user's department must be QA or Development, and the country must be USA to satisfy this assignment rule.

    You can use multiple attributes and their corresponding values to define a assignment rule.
  7. If needed, click the + icon at the top right corner to add another rule, as explained above.
  8. Click Save Changes at the top-right corner of the page to save the configuration.
  9. To see other action items, such as View Info, Insert Row, Copy Row, Disable Rule, and Delete Row, click the horizontal ellipsis next to the row.

      • Select Channel: Select the channel in the selected team where chat notifications will be sent.
  • Notification body template: Enter the content to be used as the notification body. You can use available customization features, such as variables and Improve Description to configure the email body.

Execution

This section enables you to customize the execution settings for the integration. It has the following sub-sections:

  • Advanced Settings
  • Source Filter
  • Synchronization Setting

Advanced Settings

It has the following parameter for configuration:

  • Enable report-only mode: Turn on this toggle button to generate a report detailing the data changes that would happen, but without applying them.
  • Enable debug mode: Turn on this toggle button to generate detailed logs for debugging purposes.

Source Filter

This section allows you to configure filters to limit which records are processed by the integration. Only records that match the defined filter criteria are included.

It has the following parameters for configuration:

  • Filter attribute: Select the Entra ID attribute that should be used for filtering records.
  • Attribute value: Enter the value for the selected attribute to determine which records are included in the integration.

Synchronization Setting

This section allows you to configure the synchronization settings for the integration. It has the following parameters for configuration:

  • Enable full synchronization: Turn on this toggle button to allow the integration to run in full synchronization mode, in which all eligible user records are retrieved from Entra ID instead of retrieving only the records changed since the last synchronization run.
    When disabled, the integration always runs in incremental mode, with the first execution automatically performing a full synchronization.
    When you enable this, the following fields are displayed for further configuration:

    • Full sync frequency: Select how often full synchronization runs. For example, selecting Every 6 hours from the drop-down menu will execute the integration every 6 hours in full synchronization mode.

      Note:

      The integration execution is triggered based on the schedule configured in the Schedule section. The Full sync frequency setting does not change how often the integration executes. Instead, it determines which scheduled executions should run in full synchronization mode.
      For example, if the schedule is configured as Continuous and the Full sync frequency is configured as Every 6 hours, the integration executes continuously based on the configured schedule settings. However, only one execution every 6 hours runs in full synchronization mode. All other executions between those full synchronization are in incremental mode.
      For example, if a full synchronization execution happens at 12 AM, the next full synchronization execution happens at 6 AM. All executions between 12 AM and 6 AM execute in incremental synchronization mode.

    • Disable incremental sync: Turn on this toggle button to run the integration only in full synchronization mode and skip incremental synchronization runs. When disabled, the integration runs in both full and incremental synchronization modes based on the configured synchronization settings.
  • Reset last full sync timestamp: Turn on this toggle button to reset the last full synchronization timestamp used by the integration. This affects how incremental synchronization identifies changes in future runs. After the reset is applied, incremental synchronization considers changes starting from the configured reset date and time.
    When enabled, the following fields are displayed for further configuration:
    • Reset effective date: Specify the date that should be used as the last full sync timestamp.
    • Reset time: Specify the time that should be used as the last full sync timestamp.

Tables

In the Tables tab, you can add tables to the integration. Tables are used to define custom source attributes and synchronize them with the target application. If the required data is not available in the source application, you can define it in a table and use it to update the target application. Once defined, the table can be referenced in attribute mapping expressions to set or update values automatically.

For example, to set the timezone attribute of users in the target application to Europe/Paris, create a table during integration setup with timezone as an attribute name and Europe/Paris as a value. Once defined, reference the table in the source attribute of the attribute mapper using the following format:
TABLE.<table_name>.<key_name> or TABLE.<table_name>[<key_name>]

For example: TABLE.test.timezone

Note: You can configure expressions with nested tables in the attributes mapper, if required.
For example: TABLE.UserHireRecord[TABLE.UserRecord["empDate"]]

Refer to the Using Tables in Integrations guide for more details.
