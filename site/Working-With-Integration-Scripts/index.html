Working With Integration Scripts

← Back to index

Overview

Integrations can be set up using scripts. The script is based on JavaScript, and the script supports application SCIM functions, utility functions, log functions, OpenAPI functions, and external libraries. It is possible to access the user attributes of an application in the script. If tables are configured in the integration, you can access the tables in the script.

When an event trigger is set, the script supports accessing the events of the application. It is possible to set a response in the script to inform the success or failure of an operation. 

The script editor provides a tooltip option that describes the supported functions. 

This guide explains how to work with the integration scripts.

For details on how to set up integrations using scripts, refer to the Integration Using Scripts guide.

Functions

The following table lists all the functions that can be used in the integration scripts:


Application Functions

This section details the application functions that can be used with the connectors in the integration scripts.

The Application functions can be invoked as <AppVariableName>.<functionName> where, <AppVariableName> is the name of the variable that you have configured in the integration to refer to the provisioned connector, and the <functionName> is the name of the function to be invoked.

Notes:

  • The <AppVariableName> should be unique. It can include a combination of letters and numbers. It should always start with a letter. It should not have only numbers. It should not be a JavaScript keyword. It should not include any spaces or special characters.  
  • The group functions can be used for a connector in the scripts only if the connector supports groups.

The following application functions are supported in a script:

getAqueraTenantAttributes

This function retrieves the values of all the tenant attributes configured for the application.

 

get{ResourceType}s

This function retrieves all the records of the specific resource type from the application. The resource type can be users, groups, and so on based on the resource types supported by the application.

Note:

For retrieving all users from an application, use the getUsers function in the integration scripts. The importAllUsers function has been deprecated and must no longer be used. If you have used the importAllUsers function in V5 or an earlier version of the integration, a warning message is displayed. You can replace the importAllUsers function with the getUsers function. Starting with V6 version, importAllUsers is not supported; you must use the getUsers function instead. If you are upgrading your integration from V5 to V6, make sure that you update the function name and the parameters accordingly.


get{ResourceType}

This function retrieves the specified data record of the specific resource type if it is found in the application. Returns null if the specified data record is not found. The resource type can be users, groups, and so on based on the resource types supported by the application.


importUpdated{ResourceType}Since

This function retrieves all the records of the specific resource type that are modified after a specified date. The resource type can be users, groups, and so on based on the resource types supported by the application.


post{ResourceType}

This function creates a new data record of the specific resource type in the application. It returns the object that was created including the ID. The resource type can be users, groups, and so on based on the resource types supported by the application.


put{ResourceType}

This function updates an existing data record of the specific resource type in the application. The resource type can be users, groups, and so on based on the resource types supported by the application.

You can pass the parameters in one of the following ways:

  • The ID of the data record and the object that includes the attribute values to be updated.
  • The object that includes the ID of the data record and the attribute values to be updated.


patch{ResourceType}

This function updates an existing data record of the specific resource type in the application. The resource type can be users, groups, and so on based on the resource types supported by the application.


delete{ResourceType}

This function deletes a data record from the application. The resource type can be users, groups, and so on based on the resource types supported by the application.


new{ResourceType}

This function returns an object template of a specific resource type to view the attributes supported by the application. The resource type can be users, groups, and so on based on the resource types supported by the application.

In the empty object returned by this function, you can set values for the attributes and pass this object in the functions that have the object as the parameter. For example, this object can be passed in functions such as create{ResourceType}, update{ResourceType}, delete{ResourceType}, and so on.


importAllUsersFromGroup

This function retrieves all the user records assigned to a specified group in the application.


importAllGroupNames

This function retrieves all the group display names in the application.


findUserByUsername

This function retrieves a specific user if it is found in the application. Returns null if no user is found.


activateUser

This function activates a user in the application.


deactivateUser

This function deactivates a user in the application.


addUserToGroup

This function adds a user to a specific group.


removeUserFromGroup

This function removes a user from a specific group.


setUserPassword

This function sets the password for a user in the application.


isSubordinate

This function verifies if a user is a subordinate of a specific manager. 


Utility Functions

This section details the utility functions that do not require an application prefix. The utility functions will be called in the script as utils.<functionName>. 

The following are the supported utility functions:
 

sendEmail

This function is used to send an email to a specified list of recipients with a subject, a body, and any attachments, if required.  

Format utils.sendEmail(string[] to, string subject, string body, string type, string from, string[] cc, string[] bcc, object[] attachments)
Parameters

(mandatory)to: The recipient's email ID.

(mandatory)subject: The email subject.

(mandatory)body: The email content.

type: The valid value can be "text" or "html". The text parameter directs the body to be processed as plain text. The HTML parameter directs the body to be processed as HTML. If not specified, the body will be processed as plain text.

from: This parameter enables you to specify the sender email ID on behalf of which emails will be sent to users. To set up the sender email ID in Aquera's service, you must first send a request to support@aquera.com. When the email ID is set up, a verification email will be sent to that email ID. The user of that email ID must click the verification link included in the email to allow Aquera to send emails on behalf of that user.

cc: This is the CC (Carbon Copy) component in an email. This can be arrays of strings to include multiple email addresses. 

bcc: This is the BCC (Blind Carbon Copy) component in an email. This can be arrays of strings to include multiple email addresses.

attachments: This is an array of email attachment objects, where each object contains the following properties:

  • (mandatory)content {type: string}: The file content. For example, This is a sample file
  • (mandatory)fileName {type: string}: The name of the CSV/text/xlsx file. For example, users.csv, users.xlsx
  • contentEncoding {type: string}: The encoding of the file content. For example, ascii, utf8, base64
  • compress {type: boolean}: Indicates if the file is a zip file. For example, true
  • compressedFileName {type: string}: The name of the compressed file, if compress is true. For example, files.zip
  • contentType {type: string}: The content type of the file: "text/csv" for a CSV file, "text/plain" for a text file, "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet" for an Excel file.

Note: Currently, only CSV, text, and Excel files are supported as attachments. The attachments feature is now supported only in the Integration V5 version.

Returns This function does not return anything.

generatePassword

This function generates a random password of a specified length.

The password includes a combination of letters (lowercase and uppercase), numbers, and special characters. Any of these special characters !$#% can be included in the generated password.


pause

This function pauses the execution of the integration for the specified time. 


daysSince

This function calculates the days between the specified past date and the current date. 


setPersistentVariableValue

This function sets a value to a persistent variable. A persistent variable is a variable that can be used across multiple integrations in your tenant.

The value of a persistent variable will be retained even after the integration is executed and can be used and modified in successive executions of the same integration and other integrations.


getPersistentVariableValue

This function retrieves the current value from a persistent variable. You can use a persistent variable that is defined in another integration.


removePersistentVariableValue

This function removes a persistent variable. After a persistent variable is removed and if it is used in any other integration, then that integration will fail.


now

This function generates the current timestamp in the ISO date format.

For example, the date format returned is 2023-02-27T11:57:17.994Z.


today

This function generates the current date in the format YYYY-MM-DD.

For example, the date format returned is 2023-02-27.


setResponse

This function sets the status code and the body for the response.

 

toNormalizedDiacritics

This function replaces all the characters that have diacritics with the closest non-diacritic format of those characters. 

A diacritic is any mark, shape, stroke, or sign added or attached to a letter. They are used to indicate which syllable of a word should be stressed, if vowels should be pronounced in specific ways, or to separate one word from another that has an identical spelling and/or pronunciation. Examples: Renée, François, Michèle, Chloë, Siân, Sørina.


writeIntegrationReport

This function generates a CSV report for integration.

The CSV report generated following the integration run can be downloaded by navigating to the Integrations module page, selecting the History column of your integration to display the Execution Report panel, and then clicking on Download Report (CSV).


updateStatus

This function can be used for the following two scenarios:

  • Keep a running script alive: You can use this function in scripts when the script takes longer to compute or process. This function will prevent the script from going into an incomplete state.
  • Monitor the progress of long-running integrations: You can write custom statuses to be displayed in the RESPONSE DATA column of Integration Logs. This is required to monitor the progress of the current user record processing when there are sufficiently large numbers of users to be processed. 
Format utils.updateStatus(String status)
Parameter

(mandatory) status: The status message to be displayed in the Response Data column of Integration Logs.

Returns This function does not return anything.
Example utils.updateStatus("Processing User: " + totalUsers)

evaluateExpression

This function is used to execute an expression that has been defined in the expression builder and run it against a data object. It takes a data object as input, applies the expression logic, and then returns the result.

Note: When you set up a script-based integration, you have the option to create a parameter of type Expression Builder. This parameter enables you to configure a source schema that defines the input attributes and a response schema that defines the output structure. Refer to the Integration Using Scripts guide for details on Expression Builder.

 

How it Works:

  1. You define an expression in the Expression Builder parameter of your integration. The expression uses a source schema (input attributes) and produces results according to a response schema (output structure).
  2. When the evaluateExpression function is called, it runs the configured expression against the provided object and returns the outcome.
Format utils.evaluateExpression(expression_builder, {Object})
Parameters

(mandatory)attributeMap: The variable name of the expression builder that must be used to validate the object. Refer to the Integration Using Scripts Guide for details.

(mandatory)object: The object containing the attributes and their values.

Returns This function returns the status (success: true or false) and the execution result in case of success or the error message in case of failure.
Example utils.evaluateExpression(expressionBuilderName, {person__legalName__givenName:"Tom",person__legalName__familyName:"Bill"});
Sample Script

//A portion of a sample script

const result = utils.evaluateExpression(expression_builder,{person__legalName__givenName:"Tom",person__legalName__familyName:"Bill"});

log.info(`result: ${JSON.stringify(result)}`);

evaluateRules

This function is used to evaluate the rules configured in the rule builder against a source object and return the names of the rules that evaluated to true.

Note: When you set up a script-based integration, you have the option to create a parameter of type Rule Builder. This parameter enables you to define fields for setting up configurable rules in an integration. Refer to the Integration Using Scripts guide for details on Rule Builder.

 

How it Works:

  1. You define the fields in the Rule Builder parameter for setting up rules in an integration.
  2. In the rule configuration section of the integration setup page, the end user sets up rules, for example, group assignment rules.
  3. When the evaluateRules function is called, it runs the configured rules against the provided object (for example, the user object that is passed as a parameter) and returns the names of the rules that evaluated to true. For example, the function returns the group assignment rules that are satisfied for a user object.
Format utils.evaluateRules(Object ruleBuilder, Object source)
Parameters

(mandatory)ruleBuilder: The variable name of the Rule Builder parameter that must be used to evaluate the source object. Refer to the Integration Using Scripts Guide for details.

(mandatory)source: The source object containing the attributes and their values.

Returns This function returns an array of strings representing the names of the rules that are evaluated to true.
Example utils.evaluateRules(ruleBuilderName, user);

Instructions for Writing Scripts

This section provides general instructions and guidelines to be followed while writing scripts to ensure that the script is executed without any errors.

Accessing Tables

This section details how to access tables in integration scripts. Tables define a key-name and key-value pair. Tables are added in integrations to create custom attributes and sync them to applications. 

A table can be accessed using the following formats:

  • To specify the key name defined in a table, enclose it within double quotes:
    <table_variable_name>["<key_name>"]
  • To specify an application's schema attribute as the key name, do not enclose it in double quotes:
    <table_variable_name>[<key_name>]

For details on how to add tables, refer to the Integration Using Scripts guide.

Examples:

  1. Consider the following Employee table (the table variable name is Emp) having key names 5135 and Marissa.

    Use the following format to access a key value using the key name defined in the table:
    • Emp["5135"]
    • Emp["Marissa"]
       

  2. Consider the Employee5135 table having the following key names:

    Use the following format to access a key value using the application's schema attribute name as the key name in the table:

    let usrObject = <AppVariableName>.findUserById("00u622hn3hEksuO1N5d7");
Emp5135[usrObject.userName] 
Emp5135[usrObject.name.givenName]
Emp5135[usrObject.addresses.find(address => address.type == 'work').locality]
Emp5135[usrObject[EnterpriseUser].department]
Emp5135[usrObject[IANUser].status]

    For the scenario Emp5135[usrObject.userName], the value of the userName marrissa@example.com will be searched across all the key names in the Employee5135 table and value1 will be returned.
     

Notes:

  • The table name and variable name should be unique. The variable name can include a combination of letters and numbers. It should always start with a letter. It should not have only numbers. It should not contain any spaces or special characters and is case-sensitive.
  • The key name should be unique. The key name can include a combination of letters and numbers.
  • The key name supports all the special characters except the following:
    • / (forward slash) 
    • \ (backslash)
    • " " (double quotes)
  • When a new table is added to an integration or a table is modified, save the changes and perform a page reload to access the table in the script. 

Accessing User Attributes

This section explains how to access the user attributes in the integration scripts.

  • The core attributes are accessed directly at the top level of the object using the following format:
    <userObjectVariable>.<coreAttributeName>
    Examples: userObject1.id, userObject1.userName, userObject1.name.givenName.

     
  • The enterprise attributes are accessed by using the built-in variable EnterpriseUser using the following format:
    <userObjectVariable>[EnterpriseUser].<enterpriseAttributeName>
    Examples: userObject1[EnterpriseUser].manager.value, userObject1[EnterpriseUser].organization, userObject1[EnterpriseUser].department.

     
  • The IAN (abbreviated for Integrated Application Network) attributes are accessed by using the built-in variable IANUser using the following format:
    <userObjectVariable>[IANUser].<IANattributeName>
    Examples: userObject1[IANUser].terminationDate, userObject1[IANUser].lastLoginDate, userObject1[IANUser].hireDate.

Accessing Complex Multivalued Attributes

This section explains how to access complex multivalued attributes in the integration scripts. Complex multivalued user attributes are emails, phone numbers, and addresses.

To retrieve a complex multivalued attribute like emails, use the following format:
let workEmail = userObject1.emails.find(email => email.type == 'work').value;
This will retrieve the email address of type work and then you can use workEmail in your expression.

To add or modify a complex multivalued attribute like emails, use the following format:
userObj1.emails=[ 
             { 
                  type:"work",
                  value:"bjensen@example.com”
            }]

Note: The type work must be supported in the schema of the connector.

Accessing Events

This section explains how to access events when an event trigger is set up in the integration.

An event will be triggered when there is a change in the data in an application. The data change in an application can be a new user creation, existing user modification, user activation, user addition to a group, user removal from a group, and so on. You need to subscribe to the events in the application to receive notifications for specific data changes.

Set an event trigger for the application in the integration. Refer to Integration using Scripts for details on setting an event trigger. 

You can access events in the script using the format: request.<requestAttribute>

Note: The request attributes depend on the application for which the event trigger is set.

The following is an example of logging the request attributes of an event received from the Okta application:

log.info(JSON.stringify(request.eventType));

log.info(JSON.stringify(request.data.events[0].uuid));

A sample request object of an Okta application is attached for your reference. You can view the request object of an application by using:

log.info(JSON.stringify(request));

Note: If you have not added any applications in the integration, then the request object printed in the logs will be empty.

Setting Response

This section explains how to set a response in the script.

You may be required to set a response for the integration runs to inform the success or failure of an operation. Then, you can set a status code and a status message for the response.

The response will contain the following properties:

  • statusCode
  • headers
  • body

You can set the response using the format:

response.<property>

Example:

response.statusCode = 401;
response.body = "{errorMessage: Unauthorized}";
 

Handling Exceptions

Exceptions occur because of any mistakes in the script, an erroneous server response, and for many other reasons. You can handle exceptions in the script using a try-catch block. 

The code in the try block is executed, and if it throws an exception, the code in the catch block will be executed.

The following is the format of a try-catch block:

try{  

<JavaScript expression>; 

} 

catch(error){  

<JavaScript expression>; 

}

The following is a sample to handle an exception during the update of a user record:

var users = app.importAllUsers();

for (var user of users) {

      try{  

            if (user[IANUser].employeeType == "Emp") {

                 app.updateUser(user);

             }

           } 

           catch(error){  

                    log.info("Transaction failed for user: " + user.userName)

                    log.error("Error is: " + error.message); 

            } 

}

Adding Comments

JavaScript comments can be used to explain the JavaScript code and to make it more readable. The JavaScript comments can also be used to prevent an execution when testing alternative code. You can add single-line or multi-line comments in the script:

  • Single-line comments start with //. Any text between // and the end of the line will be ignored and not executed by the integration.
    Example:
    // This is a single-line comment
    Shortcuts to add single-line comments:
    Mac: Select the code and give Cmd + / (slash)
    Windows: Select the code and give Ctrl + / (slash)

  • Multi-line comments start with /* and end with */. Any text between /* and */ will be ignored and not executed by the integration.
    Example:
    /*
    This is a
    multi-line
    comment
    */
    Shortcuts to add multi-line comments:
    Mac: Select the code and give Shift + Option + A
    Windows: Select the code and give Shift + Alt + A

Troubleshooting

How can you prevent an integration script execution from entering an incomplete state?

When running an integration script, if the script takes a long time to compute or process, the integration script execution may enter an incomplete state. To prevent this, the integration script should regularly send a message to the container to indicate that the script is still alive. You can use the following function to keep the integration script running:

Format <AppVariableName>.updateStatus(String status)
Parameter

(mandatory) status: The status message to be displayed in the Response Data column of Integration Logs.

Returns This function does not return anything.
Example <AppVariableName>.updateStatus("Processing User: " + totalUsers)
Sample Script 1
var allUsers = okta.getUsers();
let i=0;
for (var singleUser of allUsers) {
      i++;
      okta.updateStatus(`Processing User: ${i}`);
      log.info(i + " " +singleUser.userName);
      utils.pause(2000); 
}
Sample Script 2
const activeJobs = [];
do {
const hcmjobs = hcm.invokeOperation('getRestApiResourcesJobs', parameters);
hcm.updateStatus(`More HCM Jobs: ${hcmJobs.hasMore}`);
      for(const i = 0; i < hcmJobs.length; i++){
           hcm.updateStatus(`Checking for Active HCM jobs`);
           if(hcmJobs[i].status == "Active"){
              activeJobs.push(hcmJobs[i]);
            }
        }
} while (hcmjobs.hasMore)

Note: This function can be used inside a do-while, for loop, and nested loops (for example, for loop inside a do-while loop).
