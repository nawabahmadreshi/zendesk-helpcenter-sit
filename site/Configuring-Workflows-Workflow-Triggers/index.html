Overview

This guide details how to run workflows based on a schedule or an event (for example, user data changes) in an application. There are two types of workflow triggers — Schedule Triggers and Event Triggers. Click the Event Trigger / Schedule Trigger element to view the Trigger configuration panel with options to configure a trigger for workflow execution. 

For more details on all the capabilities of workflows, refer to the Workflows user guide.

Schedule Trigger

You can set a schedule to trigger the workflow at specific intervals. Select a Basic schedule or an Advanced schedule option depending on the frequency.

Basic Schedule

You can set a basic schedule for the following time intervals:

  • 15 minutes
  • 30 minutes
  • 1 hour
  • 4 hours
  • 6 hours
  • 12 hours
  • 24 hours

Advanced Schedule

You can set an advanced schedule based on the following options:

  • Select Continuous to run the workflow continuously.
    Note: There can be a delay of a maximum 5 minutes to start the next instance of the workflow after the previous instance ends.

  • Select Minutes to schedule the workflow to run every X minutes, where you should select X minutes from the Repeat Every dropdown list. The available options are from 15 to 60 minutes.

  • Select Hourly to schedule the workflow to run every X hours, where you should select X hours from the Repeat Every dropdown list. The available options are from 1 to 24 hours.

  • Select Daily to schedule the workflow to run every X days at a specific time, where you should select the day frequency from the Repeat Every dropdown list and select the time from the Starting at(Time) dropdown list. You can set up a maximum of 8 daily schedules. The available options for the days are from 1 to 6. The Starting at(Time) that you select here is the UTC time.

  • Select Weekly to schedule the workflow to run on specific days of the week at a specific time, where you should select the days in the Repeat Every field and select the time from the Starting at dropdown list. The Starting at time that you select here is the UTC time.

Note: If you select Never, the workflow never runs automatically. You can click the Execute button at the top-right corner of the workflow page to manually run the workflow. 

Event Trigger

Event trigger initiates the workflow when specific events occur in an application. An event trigger is created in cases where there is a data change in an application such as a new user creation, existing user modification, or user deactivation. The application or the orchestration sends the event trigger, which in turn triggers the workflow.

Configure Event Trigger

This section details how to configure an event trigger:

Source Type: Select the type of source (Orchestration / Application) from which an event will be sent. 

Source Name: Select the name of the source that sends the events. The supported source can be an orchestration or a connector based on the selected source type.

Event Name: Select the name of the event from the dropdown list.

  1. Orchestration Events:

    • User Created by Sync Orchestration: The workflow triggers after a user creation event in the target application configured in a synchronization orchestration.

      The workflow triggers for each target user creation and receives an event request containing the created user object. 

      To receive the event notification, configure the New User Orchestration Hook in the sync orchestration. For more details, refer to the       Synchronization orchestration guide.
    • User Updated by Sync Orchestration: The workflow triggers after a user update event in the target application configured in a synchronization orchestration.

      The workflow triggers for each target user update and receives an event request containing the updated user object. 

      To receive the event notification, configure the Update User Orchestration Hook in the sync orchestration. For more details, refer to the Synchronization orchestration guide.

    • User Deactivated by Sync Orchestration: The workflow triggers after a user deactivation event in the target application configured in a synchronization orchestration.

      The workflow triggers for each target user deactivation and receives an event request containing the deactivated user object.

      To receive the event notification, configure the Deprovision User Orchestration Hook in the sync orchestration. For more details, refer to the Synchronization orchestration guide.
  2. Application Events: The workflow triggers when there is a data change in an application. A list of application events is shown in the Event Name field if the connector selected in the Source Name field supports the Events resource type. For example, ADP Enterprise HR is one of the connectors that supports Events resource type.

    When the workflow is triggered, it receives an event request containing the attributes of the event schema.

    Note: You may have to configure the events for which the workflow needs to be triggered in the respective application's UI. Refer to the respective connector configuration guides for more details on the Events resource type.
  3. Users Modified Since Event: The event checks for modified users since the last workflow run and executes the workflow for each modified user. The event works only if the configured connector supports filtering users by the meta.lastModified attribute. The workflow triggers for each user modified after the meta.lastModified date and receives an event request containing the user object of the modified user.

For details on accessing the event request, refer to the Configuring Workflows: Accessing Event Request and Filtering Data guide.

Schedule Event Execution: Set a schedule for the execution of the event trigger. The available options are the same as Schedule Trigger. However, you get an additional option to specify the schedule start and end dates in Minutes, Hourly, Daily, and Weekly schedules. For more details, refer to the Schedule Trigger section.
