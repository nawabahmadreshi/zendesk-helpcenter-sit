Integration Expressions - Keywords, Methods, and Functions

← Back to index

Overview

Expressions are necessary for complex data transformations. Aquera provides extensions to JavaScript expressions to support full JavaScript syntax, and the extensions are designed to make it easier to work with SCIM attributes. 

All the JavaScript operators and functions are supported in the expressions of integrations. JavaScript expressions are used in some configuration fields, attribute mappings, tables, and groups when you set up integrations. 

  • In User Integration, Data Integration, and Bulk Integration, you can write JavaScript expressions in configurations such as Match Expression, Create Condition, and Update Condition based on your requirements.
  • In the Attribute Map tab of the integrations, you can write JavaScript expressions to perform operations on the values received from the source applications. When the integration runs, the methods, and functions in the expressions are executed on the source attribute's values, and the resulting values are updated in the target attributes. For attribute mapping details, refer to the respective integration guide.
  • In the Group Rules tab of User Integration, you can write JavaScript expressions to set the rules for updating group-related data. For details on how to manage groups, refer to the Managing Groups in User Integrations guide.

Note: You must use SCIM Filter Expressions in the Source SCIM Filter configuration of integrations since it follows the SCIM RFC specification. Refer to the SCIM Filter Expressions in Integrations guide for details on SCIM filter expressions.

This guide details Aquera's keywords, inbuilt methods, and inbuilt functions that can be used in integration expressions to perform the respective actions.

Keywords Used in Expressions

The following are the keywords that can be used in the expressions of integrations:

SOURCE

References the source entry object.
Examples: SOURCE.userName, SOURCE[ENT].department=="Accounting"

DEST

References the destination entry object
Examples: DEST.userName, DEST[IAN].hireDate

ENT

References the SCIM enterprise schema.
Example: SOURCE[ENT].employeeNumber refers to the SOURCE employeeNumber field that resides in the enterprise schema.

IAN

References Aquera's SCIM extension schema. IAN stands for Integrated Aquera Network.
Example: SOURCE[IAN].employmentStatus

TABLE

References a defined table.
Example: TABLE.hireDate where hireDate is the name of the table.

UTILS

Keyword prefix for utility functions.

Inbuilt Methods Used in Expressions

The following are the inbuilt methods that can be used with string objects in the expressions of integrations:

toNormalizedDiacritics()

This method replaces all the characters that have diacritics in a string with the closest non-diacritic format of those characters.

For example, if the given name is "Jénsen", using the method toNormalizedDiacritics in the expression "SOURCE.name.givenName.toNormalizedDiacritics()" will return "Jensen".  

Sample Expressions:

  • (SOURCE.emails[home].value ? SOURCE.emails[home].value.trim().toNormalizedDiacritics() : undefined)
  • (SOURCE.name.familyName.substring(0, 1).toUpperCase() + SOURCE.name.familyName.toLowerCase().substring(1)).toNormalizedDiacritics()
  • (SOURCE.name.givenName.substring(0, 1).toUpperCase() + SOURCE.name.givenName.toLowerCase().substring(1)).toNormalizedDiacritics() 

toCapitalized()

This method converts the first character of each word in a string to uppercase and all the remaining characters in the string to lowercase. Along with the case conversion, it also does the following operations:

  • Trim spaces at the beginning and end of the string
  • Replaces instances of two or more spaces in the string with a single space character
  • Removes all non-printable characters from the string

For example, if the display name is "barbara  jénsen", using the method toCapitalized in the expression "SOURCE.displayName.toCapitalized()" will return "Barbara Jnsen".  

toSAMAccountName()

This method converts the string to a valid SAM Account Name for Active Directory by truncating the string to 20 characters or less and by removing the following invalid characters from the string:

" [ ] : ; | = + * ? < > / \

This method also removes all non-printable characters from the string.

For example, if the work email is "barbarajénsen@example.com", using the method toSAMAccountName in the expression "SOURCE.emails[work].value.toSAMAccountName()" will return "barbarajnsen@example".  

Note: The period character (.) is allowed in the SAM Account Name, but it cannot be the final character.

toEmail()

This method converts a string to a valid email address. It includes removing invalid characters and replacing diacritics with valid email-compatible characters.

For example, if the work email is "barbarajénsen@example.com", using the method toEmail in the expression "SOURCE.emails[work].value.toEmail()" will return "barbarajensen@example.com". 

toPrintable()

This method removes all non-printable characters (including tab spaces and new lines) from a string.

For example, if the work email is "barbarajénsen@example.com", using the method toPrintable in the expression "SOURCE.emails[work].value.toPrintable()" will return "barbarajnsen@example.com". 

Inbuilt Function Used in Expressions

The following is the inbuilt function that can be used in the expressions of integrations:

UTILS.isSubordinate(String userid, String managerId)

This function returns true if the user's manager or any manager of the user's manager is equal to the managerid that is passed in the function. This function takes the user ID and the manager's user ID and looks up the management hierarchy to see if the user reports directly or indirectly to the manager user. This function is useful in creating groups of users that fall within the reporting structure of a higher-level organization supervisor.

Keywords Used in Attribute Mapping Expressions

The following are the keywords that can be used in the Attribute Mapping expressions:

%ATTEMPT

%ATTEMPT is useful to resolve duplicate attribute values when mapping to attributes that must have unique values.

%ATTEMPT can be used to configure and prioritize a sequence of values for a group of attributes. Initially, the value 1 will be used for all attributes containing the keyword, and if they all result in a defined and unique value, then the process is complete; otherwise, %ATTEMPT moves on to use 2 up to 20.

The set of defined attributes will be sent in an initial POST and when a 409 POST failure is encountered due to a conflict, it will increment %ATTEMPT in all fields containing the sequence and try up to 20 post retries. PUT retries are also possible.

Example: 

userName Mapping:
if(%ATTEMPT == 1){
SOURCE.name.givenName + SOURCE.name.familyName
}
else {
SOURCE.name.givenName + SOURCE.name.familyName + (%ATTEMPT-1)
}

email Mapping:
if(%ATTEMPT == 1){
SOURCE.name.givenName + SOURCE.name.familyName + "@office.com"
}
else {
SOURCE.name.givenName + SOURCE.name.familyName + (%ATTEMPT-1)+ "@office.com"
}

In this example, the value of %ATTEMPT will be the same for both attributes.

Note: There can be situations where the displayName attribute generates the same value across all %ATTEMPT retries (1–20), preventing the system from resolving duplicate conflicts. As a result, the integration repeatedly retried and logged the warning: “unable to determine consistent set of %ATTEMPT attributes.” Additionally, cached value validation details are not included in the report, which is why only this warning message was visible in the logs. 

Hence, the displayName expression is updated to include %ATTEMPT so that a unique value is generated on each retry and duplicate conflicts can be resolved correctly.

Use the expression as follows:

displayName = <baseDisplayName> + "_" + %ATTEMPT

Example: 

If your current expression is:

user.firstName + " " + user.lastName

Then change it to:

displayName = user.firstName + " " + user.lastName + "_" + %ATTEMPT

That ensures each retry produces a different display name.

%UATTEMPT

%UATTEMPT uses the same logic as %ATTEMPT; however, %UATTEMPT operates on each attribute containing the keyword independently rather than as a set.

%UATTEMPT can be used to achieve the uniqueness of all the attributes in the same integration. This keyword operates like %ATTEMPT but settles on a unique value without regard for any other attributes.

%ATTEMPT enforces the same attempt value for multiple attributes. However, %UATTEMPT may use different attempt values for multiple attributes.

This keyword is available from User Integration V4 onwards.

userName Mapping:
if(%UATTEMPT == 1){
SOURCE.name.givenName + SOURCE.name.familyName
}
else {
SOURCE.name.givenName + SOURCE.name.familyName + (%UATTEMPT-1)
}

email Mapping:
if(%UATTEMPT == 1){
SOURCE.name.givenName + SOURCE.name.familyName + "@office.com"
}
else {
SOURCE.name.givenName + SOURCE.name.familyName + (%UATTEMPT-1) + "@office.com"
}

In this example, the value of %UATTEMPT might be different for both attributes.

 

%USEQ

%USEQ is similar to %UATTEMPT, but rather than increasing from 1 to 20, %USEQ is replaced in the string with a unique sequence number.

%USEQ tries to keep the existing value when the value already matches the pattern (that is, removing or replacing %USEQ with a number results in a string satisfying the mapping).

All fields containing a %ATTEMPT or %USEQ will be incremented up to 20 times, should user creation fail as duplicate.

For full runs, Integration guarantees that a map with %USEQ in it results in a unique value for the mapped attribute for each user. 

Example:

userName Mapping:
SOURCE.name.givenName + SOURCE.name.familyName + "%USEQ"

email Mapping:
SOURCE.name.givenName + SOURCE.name.familyName + "%USEQ" + "@office.com"

In this example, the value of %USEQ might be different for both attributes.

Automatic Reference Navigation

SCIM supports a special type of attribute that is a reference to another object.  The manager of an object is an example of this type of reference.  In the Group Expressions and Attribute Map expressions, this can be used to access the attribute value of a user's manager.  

For example, SOURCE[ENT].manager.ref[ENT].employeeID is used to access the employee ID of the user's manager.

Escaping Special Characters in Expressions

To escape any special characters in the Javascript expressions, use a double-backslash before the special character.

For example, if the OU name in an Active Directory is #Terminated, mention it in the expression as \\#Terminated. 

Sample Expression:

DEST[IAN].distinguishedName.split(",")[0] + ",OU=\\#Terminated, DC=local"

Checking for Null Characters

To check for the null value of an attribute, for example, the jobFunctionName attribute in the Source application, use the expression as follows:

SOURCE[IAN].jobFunctionName != undefined

This expression will return true or false based on whether the attribute has a null value.
