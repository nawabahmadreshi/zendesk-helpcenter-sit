Overview

This guide applies to the latest public version of the integration (version 2.0.2).

Aquera provides an innovative platform that allows your organization to scale complex data management by automating data synchronization from ADP Vantage employees to Active Directory users.

Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

In this integration, every employee newly created or modified in ADP Vantage is updated in the Active Directory according to the schedule.

This guide covers details on how to set up the Active Directory / ADP Vantage integration to synchronize employees data from ADP Vantage to the Active Directory users. 

Prerequisites

The following are the prerequisites:

  • Aquera Tenant Account: You must have a tenant account in the Aquera Admin portal. Visit https://admin.aquera.io and register to get a tenant account. Contact your Aquera account manager to get the Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/
     
  • Configure Applications in the Aquera Admin Portal: 
     
    • ADP Vantage Connector: Ensure you add and configure the ADP Vantage connector in the Aquera Admin portal. It is recommended to configure the version 29 of the ADP Vantage connector. For details, refer to the ADP Vantage Configuration Guide.
      • To import a role with the minimum required permissions for this integration into your ADP Vantage service account, refer to the Importing a Role with Minimum Permissions section.
    • Active Directory Connector: Ensure you add and configure the Active Directory connector version 8 in the Aquera Admin portal. Refer to the Active Directory Version 8 Configuration Guide for details.

Note:

In the connector configuration page, the tenant value must be the hostname or IP address of a specific Active Directory domain controller. You must not configure the hostname of a load balancer that may be used in front of domain controllers for high availability.

Integration Setup

Aquera's Active Directory / ADP Vantage integration automatically synchronizes employees data from ADP Vantage to Active Directory based on a configured schedule.

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel.
  3. Click Create Integration.
  4. Select Active Directory / ADP Vantage. The Create New Integration pop-up window is displayed.
  5. Enter a name and a description for the integration.
  6. Select ADP Vantage connector version 29 and Active Directory connector version 8 from the corresponding drop-down lists.
  7. Click Continue. The integration is created and the Configuration tab of the integration setup page is displayed. The configurations are categorized under various sections within the Configuration tab.
  8. Set the configurations as per your requirements. Refer to the Integration Configurations Reference Guide for details.
  9. Click Save Changes to save the Active Directory / ADP Vantage integration configurations.

Link Analysis

The Link Analysis feature identifies duplicate or conflicting resource records across the source and target applications by comparing unique user attributes. It uses smart algorithms to classify resources as linked and unlinked, and supports cleanup between applications.

  1. In the Source Match Expression section, select the ADP Vantage employee attribute and the Active Directory user attribute that are used to match employee records between ADP Vantage and Active Directory.
  2. Click Run Link Analysis. 
  3. Once the analysis is completed, click View Report.
    • The Linked tab displays user records for which the values of the defined match attributes (in Source Match Expression) exactly match across the ADP Vantage and Active Directory applications.
    • The Unlinked tab displays users for whom no matching ADP Vantage record was found using the match expression. By default, the tab displays a system-generated Default Name Filter that helps identify potential single-match records across the applications after a successful analysis run. 
  4. Review the list of source user records corresponding to the target records, and if the records match, click Update Correlation Attribute in the Unlinked tab and then click Update to transfer the attribute value from the source application to the target application. The update process runs in the background.
  5. Once the match key is updated in the target application, the corresponding user record is automatically moved to the Linked tab.

For details on the Link Analysis Report and the actions that can be performed on the report, refer to the Integration Analysis Guide.

Mapping and Assignments

The Mapping and Assignments tab consolidates the configuration options for setting the user naming attributes, specifying organizational unit (OU) assignment rules, and mapping attributes between ADP Vantage and Active Directory. 

User Naming Attributes

In Active Directory, the user naming attributes are User Principal Name (UPN), SAM Account Name (SAM), Common Name (CN), and Work Mail. These attributes are used to identify user objects in AD.

  • UPN: The User Principal Name (UPN) attribute is the user logon name (ULN) that is used to log on to a domain. The UPN must be unique in the forest. By convention, UPN should map to the user's email address. A UPN consists of a prefix (the user account name) and a suffix (a DNS domain name). The prefix is joined with the suffix using the @ symbol. The format of UPN is username@DomainName. For example, barbarajensen@aquera.com
     
  • SAM: The SAM Account Name (SAM) attribute is the logon name used to support clients and servers running earlier versions of Windows. This name must be less than 20 characters. The SAM must be unique within the domain. The format of the SAM is DomainName\username. For example, Aquera\barbarajensen
     
  • CN: The Common Name (CN) attribute is the name of an object in the directory. It is a key component in the Distinguished Name (DN), which uniquely identifies a user in the directory. The Common Name must be unique in the container or the organizational unit. For instance, you can specify the first name, followed by a space, and the last name as the common name of a user. For example, Barbara Jensen 
     
  • Mail: The Mail attribute is the user's email address, often set to match the UPN. The mail should be unique within the same domain. The mail attribute typically follows standard email formatting. For example, barbarajensen@sample.com

AD requires unique values for UPN/ULN, SAM, CN, and Mail attributes. Aquera's Active Directory / ADP Vantage integration enables you to generate unique names for these attributes while creating a user in Active Directory. The following are the available configurations:

  • Suffix
  • User Naming Configuration

User Naming Configuration

This section enables you to define the formats to be used for generating values for the UPN/ULN, SAM, CN, and Work Mail attributes for a newly created user in Active Directory.

  • Basic User Naming Configuration: The Basic mode is enabled by default. In this mode, select a naming format from the ULN/ Mail/ SAM/ CN Format dropdown list to generate values for UPN, SAM, CN, and Work Mail when creating a new user in Active Directory. The selected format will be used to generate unique names for all four user naming attributes.
    In the basic mode, the attributes UPN/ULN, SAM, CN, and Work Mail are mutually dependent and are evaluated together. This means the integration will verify whether the values generated using the selected expression are available for all these attributes before assigning them to the new user. If any value is already in use for an attribute (that means, it is already assigned to a user in Active Directory), then the integration will try appending a number (starting from 1) to the attribute values and checking for the availability. This process continues for up to 20 attempts or until unique values are generated for all four attributes. Once unique values are found for those attributes, those values will be assigned to the new user.
     
  • Advanced User Naming Configuration: Select the Advanced User Naming Configuration checkbox to switch to the Advanced setup mode. In this mode, you can set different formats for each of the user naming attributes. 



    When you switch to the Advanced mode, you can select the user naming attributes that should be mutually dependent with the UPN/ULN, as shown below:


    For each of the user naming attributes, you can set when the naming format has to be applied:
    • On user creations
    • On user updates
    • On both user creations and updates


      Define the user naming formats in the Prompt to generate the UPN/ ULN format, Prompt to generate the Common Name format, Prompt to generate the Mail format, and Prompt to generate the SAM Account Name fields.
      These fields are AI Prompt fields that use Aquera Intelligence (AI Assistant) to generate UPN, Common Name, SAM Account Name, and Work Mail values during integration executions. OU Assignment

In Active Directory (AD), an Organizational Unit (OU) is a container object that organizes and manages resources such as users, groups, computers, and other OUs within a domain. Each OU has a distinguished name that specifies its location in the Active Directory hierarchy. An organization can create OUs based on departments, geographic locations, or roles. It is highly recommended to place user objects in OUs to help with better management, policy application, and administrative delegation.

Aquera's Active Directory / ADP Vantage integration allows you to define rules for assigning users to Active Directory OUs. The following are the available configurations:

  • Show OUs having users only: Select this checkbox to list only the OUs containing users, in the OU drop-down lists. OUs that do not have any users are not listed in the OU fields in this tab.
  • Default Assignment OU Condition: From this drop-down list, select whether only new users (Create option) or both new and rehired users (Create & Reactivate option) must be considered for the assignment to the default OU. 
  • Default Assignment OU: Select the default OU where new and rehired users will be placed if no conditional OU assignment rules exist or the user does not meet the conditional OU assignment rules configured. 
    • New users and rehires will be moved to the default OU if they do not meet any conditional OU assignments or no conditional OU assignments are defined.
    • Updated users or deactivated users will never be moved to the default OU.
    • If new or rehired users satisfy any conditional OU assignments, they will be added to the corresponding OUs.
  • Default Deactivated OU: Switch the toggle ON to select the default OU where deactivated users will be placed if no conditional OU assignments exist or the deactivated user does not meet the conditional OU assignment rules.
    • Updated users will never be moved to the default deactivated OU.
    • If an active user is deactivated and the user satisfies any conditional OU assignments, they will be added to the corresponding OU; they will not be added to the default deactivated OU.
    • If any update is made to a deactivated user and the user is already part of an OU, they will be retained in the same OU.
  • Conditional OU Assignment: Switch the toggle ON to add OU assignment rules. Refer to the following section for setting up conditional OU assignment rules.

Configure OU Assignment Rules

OU assignment rules determine the set of users that must be placed in the OUs in Active Directory. 

  1. Click the Assign On drop-down list and select one of the following options:
    • Create: To assign a user to an OU during the user creation operation. The OU and the DN attributes of the AD user will be updated during the user creation operation.
    • Update: To assign a user to an OU during the user update operation. The OU and the DN attributes of the AD user will be updated during the user update operation.
    • Create & Update: To assign a user to an OU during the user creation and update operations. The OU and the DN attributes of the AD user will be updated during the user creation and update operations.
  2. Define the conditions for assignment to OUs using one of the following ways:
    • Using Condition Builder
    • Using Expressions
  3. If required, click Add Row to add another condition within the same rule. When multiple rows are added within a rule, an OR condition is applied to the rows. That means, for each user, the integration evaluates the conditions in each row sequentially. If the conditions defined in a row are satisfied for a user, the user is assigned to that OU and the remaining rows in the rule are not evaluated for that user.
  4. If required, click the + icon at the top-right corner of the rule setting section to add another rule with a set of rows/conditions for OU assignments. When multiple rules are set, the integration evaluates every rule for each user.

    Note: You can add a maximum of 10 rules. 

Using Condition Builder

The Condition Builder is the default view for configuring rules. A blank row is displayed by default for configuring the first set of conditions.

  1. Click on Choose Attribute to select a ADP Vantage user attribute that you want to use in the condition.


     
  2. After selecting the attribute, specify the value(s) of the attribute as comma-separated values. Any ADP Vantage user that matches the specified attribute values (OR operation) will be assigned to the OU when the integration runs.
  3. If required, add conditions with more attributes and values by clicking the + icon. When multiple attributes are added within the same row, those attribute conditions are evaluated using an AND operation. Only users who match all the specified conditions will be assigned to the OU.
  4. In the Assign To field, select the organizational unit where users will be placed. To ensure all the required OUs are listed in this OU list, ensure you have configured all the base DNs in the Active Directory connector's tenant attribute, User Base Distinguished Names.

Using Expressions

Follow this procedure to set the rules using JavaScript expressions. A blank row is displayed by default for configuring the first set of conditions.

  1. Click Switch to Expression. The following options are displayed:
    • For Conditions Only: Select this option to use JavaScript expressions for setting the conditions. You must select the corresponding OU from the Assign To field.
    • For Conditions and Assignments: Select this option to use JavaScript expressions for setting the conditions and the OU assignments.

You can expand the expression editor area by clicking the icon and then write the JavaScript expressions.

Note: Click Switch to Condition Builder if you want to switch out of the Expression mode.

Refresh OU List

When the configuration of the User Base Distinguished Names is updated in the Active Directory connector or when the OU names are updated in the Active Directory application, it is required to refresh the OU list displayed on this page. Click Refresh OUs to retrieve the most recent OU list.

In Active Directory, each object has a Distinguished Name (DN), which serves as a unique identifier within the directory. A DN describes the fully qualified path to an entry within the Active Directory hierarchy. A DN is made up of attribute=value pairs, separated by commas. Example of a user DN:
CN=Barbara Jensen,OU=Sales,DC=example,DC=com

A user DN is formed from the combination of the CN and the OU assigned to the user. For example, if the CN for the user is Barbara Jensen3 and the OU where the user will be placed is OU=Admins,DC=example,DC=com, then the DN for the user will be:
CN=Barbara Jensen3,OU=Admins,DC=example,DC=com

Attribute Map

The Attribute Map page displays the default mapping from ADP Vantage Employee attributes to Active Directory User attributes. 

Perform the following steps to add a new mapping from a ADP Vantage attribute to an AD attribute:

  1. Click Add Row at the end of the attribute mapping list.

  2. Click on the empty field on the source side to include a ADP Vantage attribute. The list of ADP Vantage attributes appears, categorized into Suggested Attributes and Other Attributes. The Suggested Attributes category contains the attributes that most closely match the selected target attribute. Depending on the target attribute, the list of Suggested Attributes changes. Select an attribute from the dropdown list to add it as the source attribute. 

    You can even write a JavaScript expression in the source field. Click the Enter Expression option at the end of the ADP Vantage attributes list to write the expression. The expression is evaluated at runtime and the resulting value is assigned to the corresponding target attribute.
    Note: Click the icon to expand the Expression editor area to write longer expressions.
     

  3. In the Attribute field on the target side, select the Active Directory attribute name from the drop-down list.
    Note: Refer to Active Directory schemas for details on the attribute names.
  4. Select the data type of the AD attribute from the dropdown list.
  5. Select one of the following mapping options:
    • Map on Create and Update: The AD attribute will be updated during the create and update operations.
    • Map on Create: The AD attribute will be updated only during the create operation.
    • Map on Update: The AD attribute will be updated only during the update operation.
    • Don't Map: The AD attribute will never be updated and the mapping will be ignored during the integration runs.
  6. Finally, click the check mark to add the new mapping to the attribute map.

Note 1: The values of the default mapping are converted to normalized characters. This means that any characters with diacritics are substituted with the nearest equivalent non-diacritic format of those characters.

Set Up Group Retention Policy

The Group Retention section allows you to define how a user's group memberships must be handled when the ADP Vantage user is updated or deactivated.

On Deactivate/Deletion

This drop-down list is used to specify what should happen to a user's group memberships when the ADP Vantage user is deactivated. It includes the following options:

  • Remove from all assigned groups: All the group memberships currently assigned to the user will be removed. However, if the deactivated user satisfies some of the group assignment rules and they are not already part of those groups, they will be added to those groups; if they are already part of those groups, they will be retained in those groups.
  • Retain in all assigned groups: All the group memberships currently assigned to the user will be retained, even if the user does not satisfy the rules configured for the group assignments in the Group Rules section.
  • Retain in selected groups: This option has to be used when you want to retain the deactivated users only in specific groups. When you choose this option, the following additional fields are displayed:
    • Groups: In this field, you need to select the specific groups in which a user's membership has to be retained. For example, if a user is part of G1, G2, and G3 groups and if you have specified G3 and G4 in this field, the user's membership in G3 will be retained. Deactivated users will be removed from all other groups.
    • Condition: This is an optional field to define a filter condition to apply the retention rule only to a specific set of users from the deactivated list of users. The group membership will be retained only for the users matching the filter condition. For the remaining set of deactivated users that do not match the filter condition, no action will be taken on their group memberships. Here is a sample filter condition:
      '(SOURCE.WorkAssigmentsPositionDepartmentXRefCode).includes("Admin") && (SOURCE.BusinessTitle).includes("Manager") && (SOURCE.BusinessTitle).includes("HR")'

On Update

This drop-down list is used to specify what should happen to a user's group memberships when the user is updated. It includes the following options:

  • Remove from all assigned groups: All the group memberships currently assigned to the user will be removed. However, if the user satisfies the conditions defined in the Group Rules section, they may be added to new groups based on the configured group rules.
    Note: If the Update Deactivated Users configuration is enabled, a deleted or deactivated user will be added to a new group based on the group rules. However, they will not be added back to an existing group from which they have been removed during this integration run.
  • Retain in all assigned groups: All the group memberships currently assigned to the user will be retained. Even if the user does not satisfy a rule configured for the group assignments in the Group Rules section, they will be retained in those groups.
  • Retain in selected groups: This option has to be used when you want to retain the updated users only in specific groups. When you choose this option, the following additional fields are displayed:
    • Groups: In this field, you need to select the specific groups in which a user's membership has to be retained. For example, if a user is part of G1, G2, and G3 groups and if you have specified G3 and G4 in this field, the user's membership in G3 will be retained on user update.
    • Condition: This is an optional field to define a filter condition to apply the retention rule only to a specific set of user records that were updated. The group membership will be retained only for the users matching the filter condition. For the remaining set of updated users that do not match the filter condition, no action will be taken on their group memberships. Here is a sample filter condition:
      '(SOURCE.WorkAssigmentsPositionDepartmentXRefCode).includes("Admin") && (SOURCE.BusinessTitle).includes("Manager") && (SOURCE.BusinessTitle).includes("HR")'

Known Limitations and Issues

  • You can save the integration without configuring the mandatory fields; however, the integration runs will fail.
  • To view all the ADP Vantage attributes in the integration, it is required to download a report of all employees from the Data tab of the ADP Vantage connector configuration page.
  • The Incremental Tuning configuration is designed to function only when the Match expression only option is selected. Hence, the Username option has been removed from this configuration parameter.
  • When you set up group assignment rules or OU assignment rules, the attribute values must be entered as chips. However, after you exit the Edit mode, these attribute values appear as comma-separated entries.
  • The Import option is not available on the OU Assignment and the Group Assignment setup pages. Hence, OU assignment rules and group assignment rules cannot be created via a CSV import.
  • The Export option is not available on the OU Assignment and the Group Assignment setup pages. Hence, OU assignment rules and group assignment rules cannot be exported as a CSV file.
  • You can save a group management rule even if some rows in the rule have groups with empty conditions. This issue will be fixed in a future release.
  • During the OU Assignments Analysis and the Group Assignments Analysis, the source and target correlation attributes must be manually entered. No drop-down list is available to select an attribute from a list of attributes.
  • In Attribute Map, when you search for a source attribute using the attribute name, duplicate attribute names may be returned in the search result. This issue will be fixed in a future release.
