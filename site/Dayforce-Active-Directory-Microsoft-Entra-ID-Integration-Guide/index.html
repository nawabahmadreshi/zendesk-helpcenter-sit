Overview

The Aquera platform enables your organization to scale complex data management by automating data integration across various resource types. Aquera's integrations facilitate automatic provisioning, ensuring consistency across systems. It enables organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately enhancing efficiency in managing complex IT infrastructures and applications.

This integration automates provisioning across Dayforce, Active Directory, and Microsoft Entra ID. It synchronizes employee and candidate data, manages group assignments, and handles onboarding, updates, and terminations. It also supports automated access assignments and provides detailed notifications, logs, and reports.

This guide covers the end-to-end tasks in setting up the integration to synchronize employee and candidate data from Dayforce to the directory.

Key Functions

This integration automates identity lifecycle management with the following primary capabilities:

  • Synchronizes employee and candidate data from Dayforce to Active Directory and Microsoft Entra ID. Refer to the Supported Provisioning Methods section for the various provisioning methods supported by the integration.
  • Generates passwords for new users in Active Directory and Microsoft Entra ID.
  • Manages group assignments.
  • Sends notifications about provisioning events.
  • Generates detailed logs and reports.

Supported Provisioning Methods

This section explains how employee and candidate records from Dayforce are created, updated, and deactivated across different directory environments based on the selected provisioning method.

The integration supports the following provisioning methods:

  • Active Directory Provisioning: In this provisioning method, employee and candidate records from Dayforce are created, updated, and deactivated in Active Directory. The integration synchronizes data from Dayforce to Active Directory and applies the required changes, including account creation, attribute updates, password management, Organizational Unit (OU) assignments, and group assignments. You can use this method if your use case requires automated provisioning from Dayforce to Active Directory.
  • Entra ID Directory Provisioning: In this provisioning method, employee and candidate records from Dayforce are created, updated, and deactivated in Microsoft Entra ID. The integration manages the user lifecycle by synchronizing user attributes and handling access assignments in Entra ID.
  • Hybrid Directory Provisioning: This provisioning method combines Active Directory and Microsoft Entra ID by provisioning users in Active Directory, and then synchronizing the records to Entra ID via Microsoft Entra Connect. Employee and candidate records from Dayforce are initially created and managed in Active Directory.
    Note: Microsoft Entra Connect (formerly Azure AD Connect) is a Microsoft tool that integrates your on-premises Active Directory with Microsoft Entra ID. It syncs users and access information from Active Directory to Entra ID.
  • Dual Directory Provisioning: This provisioning method enables independent provisioning to both Active Directory and Microsoft Entra ID. In this method, employee and candidate records from Dayforce are created, updated, and deactivated separately in both directories.

Note: The provisioning method is selected and configured in the Directory section.

Prerequisites

The following are the prerequisites:

  • Tenant Account in the Aquera Admin Portal: You must have a tenant account with admin access in the Aquera Admin portal. Visit https://admin.aquera.io and register to get a tenant account. Contact your Aquera account manager to get the Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/.
  • User Account in Entra ID: You must have a user account in Entra ID with the permission to configure provisioning (for example, Application Administrator, Cloud Application Administrator, Application Owner, or Global Administrator).
    Notes:
    • The Exchange Administrator role is required for Exchange-related features, such as mailbox conversion.
    • The SharePoint Administrator role is required for SharePoint-related features, such as providing OneDrive access to the user's manager.
  • Admin Access to Dayforce: You must have admin access to your Dayforce instance.
  • Dayforce Field-Level Access: The specific field-level permissions must be enabled for the service account to manage user information in Dayforce. For more details, refer to
  • Application Connectors in the Aquera Admin Portal: You need to configure the following application connectors in the Aquera Admin Portal:
    • Dayforce: Refer to the Dayforce Configuration Guide for configuration details.
    • Active Directory: Refer to the Active Directory (AD) Configuration Guide for configuration details.
    • Microsoft 365: Refer to the Microsoft 365 Configuration Guide for configuration details.

Integration Setup

This section details the configuration and mapping process for integrating Dayforce with Active Directory and Microsoft Entra ID. It outlines the steps to create the integration, set up integration preferences, and define how data is mapped.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Search for Dayforce / Active Directory / Microsoft Entra ID in the search box next to the Explore Templates section, and then select it to create a new integration.
  5. Enter a name and a description for the integration.
  6. Click Continue.

Once created, the following tabs are displayed:

  • Overview
  • Source
  • Directory
  • Policy
  • Execution
  • Notification Center
  • Tables
  • Schedule

Employee Source Configuration

The following fields are displayed:

  • Source Application: Select the Dayforce application connector that you have configured in the Aquera Admin portal for employee management. If you do not find your application connector in the list, click the refresh icon available next to the field to reload the available connectors.
  • Employee onboarding/offboarding criteria: Select one of the following options to define how employee accounts from Dayforce are evaluated for creation and termination in the directory:
    • Based on status: Select this option to evaluate account creations and terminations based on the employee’s status in Dayforce. When selected, the integration uses the configured statuses to determine which users are eligible for account creation or termination in the directory. The following parameters are available for further configuration:
      • Status(es) to be considered for account activation in the target application: Select the statuses that should be considered active. Employees with these statuses in Dayforce are considered for account creation in the directory.
      • Status(es) to be considered for account deactivation in the target application: Select the statuses that should be considered inactive. Employees with these statuses in Dayforce are considered for account termination in the directory.
    • Based on date: Select this option to evaluate account creations and terminations using a date attribute in Dayforce. When selected, the integration uses the configured date attribute to determine when employee accounts are considered for creation or termination in the directory. The following parameters are available for further configuration:
      • Date to be considered for account activation in the target application: Select a date attribute in Dayforce to determine when an employee becomes eligible for account creation in the directory. When this date is reached, the employee record will be created in the directory.
      • Date to be considered for account deactivation in the target application: Select a date attribute in Dayforce to determine when an employee becomes eligible for account termination in the directory. When this date is reached, the employee record will be terminated.

Synchronization Settings

This section enables you to configure how data is synchronized between Dayforce, Active Directory, and Microsoft Entra ID.

It has the following settings:

  • Enable full synchronization: Enable this option to run the integration in full synchronization mode. When disabled, only incremental changes are synchronized.
    • Full Sync Frequency: Select the frequency of full sync from Dayforce to Active Directory and Entra ID. The following options are available:
      • Every 6 hours: Select this to execute a full sync after every 6 hours.
      • Every 12 hours: Select this to execute a full sync after every 12 hours.
      • Every 24 hours: Select this to execute a full sync after every 24 hours.
    • Disable Incremental Sync: Select this option to stop incremental synchronization. When disabled, incremental changes are synchronized as they occur.
      Note: If Full Sync Frequency is set to every 6 hours and Disable Incremental Sync is enabled, the integration runs only Full synchronization at 12 AM, 8 AM, and 4 PM. The runs at 4 AM, 12 PM, and 8 PM are ignored.

Notes:

  • The first synchronization always runs as a full sync, regardless of this setting. After the initial sync, subsequent syncs run incrementally based on the schedule configured in the Schedule tab.
  • If Full Sync Frequency is set to every 6 hours and the Schedule frequency is set to every 4 hours, the integration runs every 4 hours. During these runs, the integration alternates between Full and Incremental synchronization. For example, Full synchronization runs at 12 AM, 8 AM, and 4 PM, while Incremental synchronization runs at 4 AM, 12 PM, and 8 PM.

Target Directory Assignment Settings

This section enables you to configure how user accounts are assigned within the target directories.

It has the following settings:

  • Enable custom eligibility rules for the target directory: Enable this option to define a custom eligibility rule that identifies employees and candidates eligible for directory assignment. By default, all active employees and candidates with the Offer Accepted status are eligible for directory assignment.
    • Custom eligibility rule builder prompt: Enter your prompt to define a rule to identify employees or candidates who are eligible for directory assignment using Aquera Intelligence. Refer to the Using Aquera AI Assistant for Generating Unique Values in Integrations guide for more details.
  • Enable custom assignment rules for the target directory: Enable this option to apply custom rules for assigning directory accounts. When disabled, directory accounts are assigned to all employees and candidates based on the default synchronization configuration.
    • Directory Assignment Rule: Configure rules to determine how directory accounts are assigned to employees and candidates. 
      Perform the following steps to create new Employee rules:
      1. Click the vertical ellipsis next to Choose Attribute, and then click Select Attribute to view the list of available attributes you can use to set the eligibility rule.
      2. Search and select the attribute you want to use to define a condition. For example, select Application_Status_Group_Reference_Code.
      3. In the next row, double-click and select the attribute value to define the rule. If multiple values are selected, an OR condition is applied among them.
      4. In the Directory Column, double-click and select the provisioning method to define the rule. Refer to the Supported Provisioning Methods section for more details.
      5. Click Add Row to add additional conditions for the selected attribute. Multiple rows can be added within a rule.
      6. Click the ellipsis and select Delete Row to delete a row as needed.
      7. Click + to add another rule, as explained above. 
      8. Click Save Changes at the top-right corner of the page to save the configuration.

Note: Follow the above steps to add rules for Candidates.

Sync the user's manager info to the target: Enable this option to sync the user's manager information to the target directory during user provisioning. Business Groups

A business group is a logical group created within the integration system to make group assignments easier. It works as an intermediate layer between the source system and the target application groups.

In this approach, users are first assigned to business groups based on specific attributes configured in the Access Rules section. These business groups are then mapped to the appropriate groups in the target applications.

The Business Groups section allows you to define business groups and map them to the target application groups.

In this section, if the List View toggle button is disabled, the following parameters are displayed:

  • Name: Provide a name for the group.
  • Description: Provide a short description for the group.
  • Owner: Enter the email addresses of the group owners.
  • Is Sub group: Enable this parameter to include other business groups as a subgroup to this group. Users assigned to the selected subgroups will also be part of this group. When enabled, the following parameter is displayed for further configuration:
    • Sub Business Groups: Select one or more business groups you want to include in this group.
  • Business Groups with Application Groups: Select this checkbox to map the business group to the target application groups.
    When enabled, the following fields are displayed for further configuration:
    • Select Application: Select the target applications for which you want to manage group assignments through this business group.
    • Entitlement Type: Select an entitlement type available in your target applications to which users will be assigned through this business group. For example, select Licenses.
    • Entitlements: Select an entitlement from the dropdown menu to which users will be assigned through this business group. For example, select Licenses.

Perform the following steps to define access rules for the employees:

  1. Click the vertical ellipsis next to Choose Attribute, and then click Select Attribute to view the list of available attributes you can use to set the assignment rule.
  2. Search and select the attribute you want to use to define a condition. For example, select Employment Status Reason.
  3. In the below row, double-click and select the attribute value to define the rule. If multiple values are selected, OR condition is applied among them. For example, in the following sample condition, a user's employment status reason must be either New Hire or Rehire to satisfy this assignment rule.
  4. If required to add another condition and assignment for the same selected attribute, click Add Row. You can add any number of rows within a rule.
  5. If required to include another attribute to the group assignment rule, click the vertical ellipsis in the first column, and then click Add Attribute.
    If multiple columns are added for a rule, AND condition is applied between different columns. For example, in the following sample condition, employee's Employment Status Reason must be either New Hire or Rehire, and Primary Work Assignment Department must be Engineering to satisfy this assignment rule. You can use multiple attributes and their corresponding values to define a group assignment rule.
  6. Under the Organisational Unit (OU) field, double-click and select the OU to which users who satisfy the specified condition will be assigned.
  7. If needed, click the + icon at the top right corner to add another rule, as explained above.
  8. Click Save Changes at the top-right corner of the page to save the configuration.

Note: If required to delete a row, click the horizontal ellipsis next to the row and select Delete Row.

Tables

In the Tables tab, you can add tables to the integration. Tables are used to define custom source attributes and synchronize them with the target application. If the required data is not available in the source application, you can define it in a table and use it to update the target application. Once defined, the table can be referenced in attribute mapping expressions to set or update values automatically.

For example, to set the timezone attribute of users in the target application to Europe/Paris, create a table during integration setup with timezone as an attribute name and Europe/Paris as a value. Once defined, reference the table in the source attribute of the attribute mapper using the following format:
TABLE.<table_name>.<key_name> or TABLE.<table_name>[<key_name>]

For example: TABLE.test.timezone

Note: You can configure expressions with nested tables in the attributes mapper, if required.
For example: TABLE.UserHireRecord[TABLE.UserRecord["empDate"]]

Refer to the Using Tables in Integrations guide for more details.
