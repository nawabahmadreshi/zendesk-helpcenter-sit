Overview 

Aquera transforms how organizations manage complex data by automating integrations across multiple applications, eliminating manual processes, and reducing errors. Aquera Log Insights provides centralized visibility into the execution of integration workflows across connected applications. As organizations scale automated integrations, consistent monitoring is required to ensure data accuracy, operational reliability, and timely issue resolution.

Aquera Log Insights delivers this visibility through capturing structured logs generated during integration execution, including workflows for user provisioning, data transformation, and directory synchronization. The platform enriches raw log data with essential metadata such as user information(if selected), timestamps, integration names, and execution status, enabling you to search, analyze, and act on integration data in real time. 

Key Concepts and Terminology

Log Insights collects and stores logs generated by integration scripts, including key details such as time, user, and status. These logs are organized into Index patterns, which can be thought of as folders grouped by date:

  •  The index patterns enable simultaneous searching across multiple related log groups, providing a comprehensive view of integration workflows.  For example, instead of searching for user-provisioning-january, user-provisioning-february, and user-provisioning-march separately, you can use the pattern user-provisioning-* to search all user-provisioning logs across all time-periods. 
  •  A @timestamp field is used within index patterns to retrieve data for specific dates or time ranges, which can then be visualized through the dashboard interface.  For example, the index pattern application-*-2024-08-14 retrieves all application logs from August 14, 2024, enabling users to analyze daily activity patterns and identify trends or issues within that specific timeframe. 
  • In Log Insights, each log entry includes a @timestamp field, making it easy to filter by time, chart trends, and quickly find when events occurred.

Log Insights Capabilities

  • Centralized integration monitoring: Capture structured logs from provisioning, transformation, and directory sync workflows with enriched metadata (timestamps, run IDs, execution status).
  • Pattern-based search and filtering: Search across related logs using index patterns and filter results by time range.
  • Dashboard visualization: View preconfigured dashboards and widgets for monitoring key metrics.
  • Audit and troubleshooting: Track execution using run identifiers, developer context, and detailed audit trails.
  • Query and time controls: Save queries, view query history, and apply flexible time ranges using presets or custom ranges.

  • Enable Log Insights: The Log Insights option must be enabled in Parameters > General Settings to capture integration metrics and display them in the Log Insights dashboard. This setting is required to generate and view Log Insights dashboards.

How Log Insight Works 

The log insights workflow can be broken down into the following step-by-step actions, depending on the integration runtime and Log Insights UI behavior.

The following is the step-by-step procedure:

  1. Integration Trigger: The script-based integration runs either manually or on a schedule 
  2. Logging Begins: The script logs execution stages using the log function and writes structured records using a write function. Each object typically represents a user, transaction, or system response.
  3. Log Enrichment: As each log entry is recorded, it's enriched with the following metadata:
    Integration name
    • Run identifier (run ID)
    • Timestamps
    • User references/IDs
    • Execution status
    • Custom developer context (if provided)
  4. Transmission to Indexing Layer: These logs are automatically pushed to a backend indexing service. Each integration writes to a named index for isolation and efficient retrieval.
  5. UI Availability: Logs are made instantly available in the Log Insights UI under the selected integration. Users can view logs in a tabular layout, expand entries to inspect full JSON payloads, and apply dynamic filters.
  6. Dashboard Update: Dashboards that reflect metrics such as success/failure rates, daily volume, and top errors update in near real-time as logs arrive.

Default dashboards

When Log Insights is enabled for an integration, the following dashboards may be created:

  • Integrations Summary: Displays high-level insights by integration category (e.g., ITSM, ITO, IRIS, HR, and JML).
  • Application Transactions: Created when the integration calls SCIM methods, showing SCIM-related activity and outcomes.

Reporting Tab

The Reporting tab in Log Insights provides a comprehensive interface for exporting log data and visualizations into downloadable report files. It is designed for users who need to capture log information at a specific point in time, generate recurring data extracts, or maintain records for auditing, monitoring, and analysis purposes. Reporting helps bridge the gap between live log exploration and offline use by allowing users to save, schedule, and retrieve report outputs without manually rerunning queries.

The Reporting tab is organized into two primary sections:

  • Reports: displays generated report outputs
  • Report Definitions: manages report configuration templates

Reports

The Reports display the execution history of generated reports. Each row represents a single completed report output generated from a report definition.

Reports are immutable snapshots of data captured at the moment of generation. Once created, a report reflects the exact data available at that time and does not change, even if the underlying logs or saved searches are updated later.

The following are the key components available under the Reports section: 

  1. Reports Header:  Displays the total number of generated reports available. This count includes reports across all states, including created, pending, shared, error, and archived.
  2. Search Reports: Helps users quickly find reports by name. Searching filters the report list in real time and is useful when working with numerous generated reports.
  3. Reports List: Lists all generated reports and provides detailed information for each report instance. The table includes the following columns:
    • Name: The name of the report, inherited from the report definition.
    • Source: Indicates whether the report was generated from a saved search, dashboard, or visualization.
    • Type: Shows how the report was generated, either Schedule or On demand.
    • Creation time: The date and time when the report was generated.
    • State: Displays the report's current lifecycle state.
    • Actions: Provides access to download the report output when available.
  4. Filters: The Reports screen provides filter controls to narrow down the list of reports:
    • Type filter: Allows filtering reports by Schedule or On demand.
    • State filter: Allows filtering reports by lifecycle state, including Created, Pending, Shared, Error, and Archived.
      Note: Filters help users focus on specific report types or troubleshoot failed or pending reports.
  5. Report Table: The table shows how many reports are currently displayed and the total number of available reports. It updates dynamically as filters or search terms are applied.
  6. Rows per Page: Let users choose how many reports to display per page. Increasing the number of rows per page can reduce the need for pagination when reviewing many reports.
  7. Pagination Controls: Allow users to navigate between report pages. These controls are useful when the total number of reports exceeds the selected rows-per-page limit.

When Reports Are Created

Reports appear in the Reports list when they are generated either automatically or manually.  A report is created each time a scheduled report runs according to its configured schedule, or when a user manually triggers report generation using an on-demand report definition.

Notes:

  • Whenever a report is generated, a new report entry is created in the Reports list, even if the same report definition is used multiple times.

  • Changes made to a report definition, including edits or deletions, do not affect reports that have already been generated.

  • Previously generated reports remain available for viewing, downloading, or management until they are explicitly deleted.

Report Definitions

The Report Definitions section contains the configuration layer for reporting. A report definition acts as a reusable template that controls how reports are generated.

A report definition specifies:

  • The data source used for reporting
  • The method of generation (manual or scheduled)
  • The recurrence pattern, if scheduling is enabled

Report definitions do not contain data themselves. Instead, they describe how data should be collected and exported when a report is generated.

The following are the key components available under the Report Definitions section:

  1. Report Definitions Header: Displays the total number of report definitions available. This count includes both active and disabled definitions.
  2. Search Bar: Allows you to quickly find a specific report definition by typing its name or related keywords. The list updates dynamically as you type.
  3. Refresh: Reloads the list of report definitions to reflect the latest changes, such as newly created, updated, enabled, or disabled definitions.
  4. Create: Opens the Create Report Definition screen, where you can configure a new report by selecting the data source, time range, file format, and trigger type.
  5. Report Definitions Table: The central table that lists all configured report definitions. Each row represents a single report definition and displays:
    1. Name: The name of the report definition. Selecting the name opens the definition for review or editing.
    2. Source: Indicates whether the report is based on a saved search or a dashboard.
    3. Type: Shows whether the report definition is On demand or Schedule.
    4. Schedule details: Displays scheduling information for scheduled reports. A dash indicates that no schedule is configured.
    5. Last Updated: Shows when the report definition was last modified.
    6. Status: Indicates whether the report definition is Active or Disabled.
  6. Items per Page Dropdown: Allows you to select how many report definitions are displayed per page, making it easier to navigate large lists.
  7. Pagination Controls: Enable navigation between pages when the number of report definitions exceeds the selected items-per-page value

Report Definition

The Report Definition page enables you to configure how reports are generated in Log Insights. It also allows you to define the data source, time range, output format, and execution behaviour for a report before it is generated.

A report definition created here can be used to generate reports on demand or automatically on a schedule, depending on the selected options.

The Report settings section contains the following core configuration fields that define the report.

  • Name: Specifies the report definition's name. This name identifies the report definition and is applied to all reports generated from it.
    Note: Only alphanumeric characters, spaces, underscores (_), and hyphens (-) are allowed. Using clear and descriptive names makes it easier to identify reports later.
  • Description (optional):  Used to provide additional context about the purpose of the report. This is useful for explaining what the report contains or why it was created, especially when multiple users manage reports.
  • Report Source: Defines the source of the report data. You can choose one of the following sources:
    • Dashboard: When Dashboard is selected, the report is generated from an existing dashboard. This option is typically used to export aggregated views, charts, or visual summaries.
      After selecting this option, you must choose a dashboard from the Select dashboard dropdown.
    • Visualization: When Visualization is selected, the report is generated from a single visualization. This is useful when you want to export a specific chart or visual representation of log data. After selecting this option, you must choose a visualization from the Select visualization dropdown.
    • Saved Search: When Saved search is selected, the report is generated from the results of a saved search query. This option is commonly used for exporting raw or filtered log records. After selecting this option, you must choose a saved search from the Select saved search dropdown.
      • Record Limit (Saved Search Only): When using Saved search as the report source, the Record limit field becomes available. This field controls the maximum number of log records included in the report output.
        Note: Setting an appropriate record limit helps manage report size and performance, especially for large datasets. Reports generated from saved searches are limited to 10,000 records per execution.

  • Time Range: The Time range setting defines the period of data included in the report.
    Time ranges can be relative (for example, last 30 minutes) or set to specific dates using the Show dates option. The selected time range is evaluated relative to the report execution time.

    For scheduled reports, the time range is recalculated at each run, ensuring that each report captures data relevant to that execution window.

  • File Format: The File format option determines how the report output is generated.
    • PDF is typically used for dashboards and visualizations. PDF format is not supported for the Saved Search option. 
    • CSV is typically used for the Saved Searches data exports. 
      Note: The available file format options depend on the selected report source.
  • Report Trigger: The Report trigger setting controls how and when reports are generated from this definition.
    • On demand: When On demand is selected, reports are generated only when a user manually triggers them. This option is suitable for one-time exports or adhoc analysis.
    • Schedule: When Schedule is selected, reports are generated automatically according to the defined schedule. Scheduled reports create new report entries at each execution without manual intervention.
      • When Schedule is selected as the report trigger, additional scheduling options become available. These settings control how often and when reports are generated automatically.
        • Request Type: Defines how the scheduled report is executed.
        • Recurring: Generates reports repeatedly based on the selected frequency. This option is used for ongoing, periodic reporting.
        • Frequency: Specifies how often the report is generated. For example, selecting the Daily schedule causes the report to run once a day.
          Note: Available frequency options may vary depending on configuration, but commonly include daily or other recurring intervals.
        • Request Time: Specifies the exact time at which the scheduled report runs. The report is generated at this time according to the selected frequency.
          Note: Using a defined request time ensures reports are created consistently, such as at the start or end of a business day. 
          Once scheduled, the report definition will automatically generate new reports according to the configured frequency and request time, as long as the definition remains active.

Relationship Between Reports and Report Definitions

Report definitions and reports follow a one-to-many relationship. A single report definition can generate multiple reports over time, with each report representing a snapshot of data captured when the report is executed. For example, a scheduled report definition that runs every five minutes will generate a new report every five minutes, and each generated report will appear as a separate entry in the Reports section.

Scheduling Behavior

Scheduling enables reports to be generated automatically without requiring manual user intervention. When scheduling is enabled, reports are created at regular intervals as defined in the report definition, and each scheduled execution produces a separate report entry. Scheduling continues as long as the report definition remains active.

Time Range Selector in Log Insights

The Time Range Selector controls the time window for the data displayed across all visualizations and tables on the dashboard. It allows you to analyze data for a specific period or continuously track the most recent activity.

The time range selector supports three modes: Absolute, Relative, and Now.

  1. Absolute: The Absolute mode selects a fixed start date and time. This option is useful when you want to analyze data for a specific historical period.

    When an absolute time range is selected:

    • The start date and time remain fixed
    • The displayed data does not change unless the time range is manually updated
    • Refreshing the dashboard reloads data only for the selected fixed window
    • This mode is commonly used for audits, investigations, or the review of events that occurred within a known time frame.

  2. Relative: The Relative mode defines a time range relative to the current time, such as the last 15 minutes, last 24 hours, or last 3 months.

    When a relative time range is selected:

    • The start time automatically adjusts based on the current time.
    • Each refresh recalculates the time window.
    • Optional rounding (e.g., to the month) can be applied to improve time boundaries.
    • Relative time ranges are ideal for ongoing monitoring and trend analysis.

  3. Now: The Now mode dynamically sets the start time to the moment of each refresh.

    When Now is selected:

    • The start date and time are automatically set to the current time on every refresh.
    • The time range always reflects the most up-to-date data available.
    • This option is best suited for real-time monitoring scenarios that require the latest data.
    • Changes to the time range apply immediately to all visualizations and data tables on the dashboard.
