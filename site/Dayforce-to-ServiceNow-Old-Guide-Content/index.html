Overview

The Aquera platform allows your organization to efficiently scale complex data management by automating the integration of diverse resource data across different applications and systems.

The Dayforce to ServiceNow Data Synchronization integration automates the process of synchronizing user, department, and location data from Dayforce to ServiceNow. It also creates tickets in ServiceNow for the following events in Dayforce:

  • Employee onboarding
  • Employee role changes
  • Employee offboarding

The integration creates, updates, and deletes records in ServiceNow based on the current state in Dayforce. For example:

  • If a department exists in Dayforce but not in ServiceNow, it is created.
  • If a department exists in both but with different details, it is updated.
  • If a department exists in ServiceNow but not in Dayforce, it is deleted (if enabled)

This integration is triggered based on the schedule that you set under the Trigger tab.

The Dayforce to ServiceNow Data Synchronization integration performs the following tasks:

  • Retrieves the details of the employees who have been hired, whose roles have been changed, and who have been terminated from Dayforce and synchronizes the data with ServiceNow. Incremental retrieval is performed during every integration run within a period of 24 hours.
  • Creates tickets in ServiceNow for employee onboarding, employee role changes, and employee offboarding events.
    • Creates a ticket if no ticket is already available for the employee event
    • Updates the ticket if a ticket is already available and there are some changes to the event from the last integration run, for example, the effective hire date has been modified, the termination date has been modified, and so on.
  • Retrieves the data of users, locations, and departments from Dayforce and synchronizes the data with ServiceNow. The number of creations, updates, and deletions in ServiceNow for each type of data can be controlled by configuring the parameters available in the integration setup.
  • Creates a CSV report of the changes in ServiceNow and sends the report via email to the email recipients, based on the parameters configured.

Integration Setup

This section details the configuration process for integrating Dayforce with ServiceNow. It covers creating the integration, setting parameters, and establishing automated synchronization of employee records, departments, locations, and tickets from Dayforce into ServiceNow.

This integration syncs the Dayforce data of all employees, locations, departments, and creates tickets for any events related to employee onboarding, role changes, and offboarding in ServiceNow.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Select Dayforce to ServiceNow.
  5. Enter a name and a description for the integration.
  6. Select your Dayforce and ServiceNow connector instances.
  7. Click Continue. The Configuration page is displayed to configure the integration.

Parameters

The Parameters section enables you to customize the integration settings according to your specific use case. It has the following configuration parameters:

Integration Scenarios 

  • Enable Ticket Management: Select this checkbox to create and update onboarding and offboarding tickets in ServiceNow based on Dayforce events.
  • Enable Department Management: Select this checkbox to synchronize department records between Dayforce and ServiceNow.
  • Enable Location Management: Select this checkbox to keep location records in ServiceNow aligned with Dayforce organizational data.
  • Enable User Management: Select this checkbox to create, update, or deactivate ServiceNow user accounts from Dayforce employee records.

Execution Features

Configure the following execution related settings:

  • Enable Report-Only Mode: Select this option to generate a report detailing the data changes that would occur, but without applying them.
  • Enable Logging: Select to generate logs outlining data changes.
  • Enable Report Generation: Select to generate the integration report.
  • Enable Email Notifications: Select this checkbox to enable the integration-related notifications.

  • Enable Log Insights: Select to visualize the integration-related metrics.

    Debug Settings: Select to generate logs outlining data changes that would occur for debugging.

Onboarding Tickets Generation

  • Enable Onboarding Ticket Generation: Select to enable onboarding ticket creation in ServiceNow for new hires from Dayforce. For example, when an employee’s status changes to Active in Dayforce, an onboarding ticket is automatically created in ServiceNow.
  • Statuses to fetch Dayforce Employees: Select the Dayforce employment statuses to include for onboarding ticket creation. For example, choose statuses like Active or Hired to identify employees starting soon.
  • Select Onboarding Date Type: Select the employee date field (for example, Hire Date or Start Date) used to determine when onboarding tickets should be created.
  • Enable Ticket Pre-Processing: Select to create onboarding tickets a few days before the onboarding date. For example, create tickets 5 days before an employee’s hire date to prepare access or assets in advance. Refer to the Sample Ticket Description Template.
  • Enable Employee Number Filter: Select to limit onboarding ticket creation to a specific employee number for testing. For example, use this to test ticket creation for employee number 100245 before enabling it for all employees. 
    • Create ticket X-days before provided date: Enter the number of days before the onboarding date to create the ticket. For example, if you choose Hire Date and set the workflow to create a ticket 5 days before the date, the ticket will be created 5 days before the employee’s hire date.
      Note: The selected Onboarding Date Type determines the provided date.
  • ServiceNow Trigger: Select the trigger type used to initiate ticket creation in ServiceNow. For example, select Service Catalog Item if onboarding tickets should be generated through ServiceNow’s catalog requests. 
    The following options are available in the dropdown:
    • Ticket Type: Select the ServiceNow ticket record type used for onboarding (for example, Incident, Change Request, Problem, HR Case, Catalog Request, or Custom Ticket Type). For example, select HR Case to generate tickets under HR case management.
    • Service Catalog Category: Select the ServiceNow catalog category (such as Software, Hardware, or Facilities) that contains the catalog item you want to use. For example, if you select Software, the Service Catalog Service Request field will automatically populate with the related items from that category.
      • Service Catalog Service Request: Select the specific ServiceNow catalog item from the chosen category that should trigger this workflow. The list of available items is automatically filtered based on the selected Service Catalog Category.
    • Flow Designer API Endpoint: Enter the REST API endpoint URL generated by the ServiceNow Flow Designer. This endpoint allows ServiceNow to initiate the workflow when the flow is executed.
    • Order Guides: Select one or more ServiceNow order guides that should trigger this workflow. An order guide groups multiple related catalog items (such as hardware, software, and access requests) under a single request. The workflow will start whenever a selected order guide is submitted in ServiceNow.

Offboarding Tickets Generation

  • Enable Offboarding Ticket Generation: Select to enable automatic offboarding ticket creation in ServiceNow for employees marked as terminated in Dayforce. For example, when an employee’s termination date is reached, an offboarding ticket is automatically generated in ServiceNow.
  • Statuses to fetch Dayforce employees: Select the Dayforce employment statuses to include for offboarding ticket creation. For example, choose statuses like Terminated or Inactive to capture employees leaving the organization.
  • Select Offboarding Date Type: Select the employee date field (for example, Termination Date or Last Working Day) used to determine when offboarding tickets should be created.
  • Enable Ticket Pre-Processing: Select to create offboarding tickets a few days before the onboarding date. For example, create tickets 3 days before the termination date to ensure system access removal in time. Refer to the Sample Ticket Description Template.
    • Create ticket X-days before provided date: Enter the offboarding date type that should be used to calculate when the ticket will be created. For example, if you choose Termination Date and set the workflow to create a ticket 3 days before the date, the ticket will be created 3 days before the employee’s termination date.
      Note: The selected Offboarding Date Type determines the provided date.
  • Enable Employee Number Filter: Select to limit offboarding ticket creation to a specific employee number for testing. For example, use this to verify the process for employee number 102198 before running for all terminations.
  • ServiceNow Trigger: Select the trigger type used to initiate ticket creation in ServiceNow. For example, select Flow Designer Trigger if offboarding workflows are automated using ServiceNow Flow Designer. 
    The following are the options: 
    • Ticket Type: Select the ServiceNow ticket record type used for offboarding (for example, Change Request, HR Case, Problem, Catalog Request, or Custom Ticket Type). For example, select Change Request to create tickets that track deprovisioning tasks.
    • Service Catalog Category: Select the ServiceNow catalog category (such as Software, Hardware, or Facilities) that contains the catalog item you want to use. For example, if you select Software, the Service Catalog Service Request field will automatically populate with the related items from that category.
      • Service Catalog Service Request: Select the specific ServiceNow catalog item from the chosen category that should trigger this workflow. The list of available items is automatically filtered based on the selected Service Catalog Category.
    • Flow Designer API Endpoint: Enter the REST API endpoint URL generated by the ServiceNow Flow Designer. This endpoint allows ServiceNow to initiate the workflow when the flow is executed.
    • Order Guides: Select one or more ServiceNow order guides that should trigger this workflow. An order guide groups multiple related catalog items (such as hardware, software, and access requests) under a single request. The workflow will start whenever a selected order guide is submitted in ServiceNow.
  • Set Offboarding Employee Number Value: Enter the employee number to test the offboarding workflow.

Departments Sync

  • Record Matching Field: Select the field used to match department records between Dayforce and ServiceNow. For example, select Name if both systems use the department name as the unique identifier, or ID if both use a numeric department ID.
  • Enable Department Creation: Select to allow new departments from Dayforce to be created in ServiceNow. For example, when a new department Finance Operations is added in Dayforce, it will automatically appear in ServiceNow.
  • Enable Creation Filter Condition: Select to apply a condition that filters which new departments should be created. For example, create only departments belonging to Corporate Division.
  • Enable Department Update: Select to allow existing department records in ServiceNow to update automatically when details change in Dayforce. For example, if Support is renamed to Customer Support, the change will sync to ServiceNow.
  • Enable Update Filter Condition: Select to apply a condition that filters which departments should be updated. For example, update only departments located in North America.
  • Enable Department Removal: Select to allow departments removed from Dayforce to be deleted in ServiceNow. For example, if Legacy IT is deleted in Dayforce, it will also be removed from ServiceNow.
  • Enable Deletion Filter Condition: Select to apply a condition that filters which departments can be deleted. For example, delete only departments marked as Inactive in Dayforce.
  • Enable Sync Record Limits: Select to limit the number of department records that can be created, updated, or deleted in each sync run. For example, restrict updates to 100 records per run to prevent accidental bulk changes.
  • Set Creation Record Limit: Specify the maximum number of department records to create per sync run. For example, set 100 to limit bulk insertions.
  • Set Update Record Limit: Specify the maximum number of department records to update per sync run. For example, set 100 to control how many updates occur in one run.
  • Set Deletion Record Limit: Specify the maximum number of department records to delete per sync run. For example, set 10 to prevent unintended mass deletions.

Locations Sync

  • Record Matching Field: Select the field used to match location records between Dayforce and ServiceNow. For example, select Name if both systems use the same location name, or Primary location if both reference a common location ID.
  • Enable Location Creation: Select to allow new locations from Dayforce to be created in ServiceNow. For example, when a new office Berlin HQ is added in Dayforce, it will automatically appear in ServiceNow.
  • Enable Filter Condition for Insertions: Select to apply a filter condition that controls which new locations are created in ServiceNow. For example, create only locations where the country equals United States.
  • Select attribute to filter insertion records in ServiceNow’s Locations: Select the attribute used to apply the insertion filter. For example, select country to create locations based on a specific country.
  • Select match condition to filter insertion records in ServiceNow’s Locations: Select how the system matches the chosen attribute to its value. For example, select Equals To to match locations where the country equals “USA.”
  • Provide value to filter insertion records in ServiceNow’s Locations: Select or enter the value used for the match condition. For example, enter USA to create only U.S. locations.
  • Enable Location Updates: Select to allow existing location records in ServiceNow to update automatically when details change in Dayforce. For example, if New York Office is renamed to NYC Office, the change will sync automatically.
  • Enable Filter Condition for Updates: Select to apply a condition that limits which locations are updated in ServiceNow. For example, update only locations where the region equals Europe.
  • Attribute to filter updation records in ServiceNow’s Locations: Select the attribute used to apply the update filter. For example, select a region to update records by geographic region.
  • Match condition to filter updation records in ServiceNow’s Locations: Select how to compare the chosen attribute for updates. For example, select Contains to update records with region names containing “APAC.”
  • Value for filtering updation records in ServiceNow’s Locations: Select or enter the value that determines which locations should be updated. For example, enter Europe to update only European offices.
  • Enable Location Deletion: Select to allow locations removed from Dayforce to be deleted in ServiceNow. For example, if the Toronto Branch is deleted in Dayforce, it will also be removed in ServiceNow.
  • Enable Filter Condition for Deletion: Select to apply a condition that controls which locations are deleted from ServiceNow. For example, delete only locations where the city equals Closed.
  • Attribute to filter for deletion records in ServiceNow’s Locations: Select the attribute used to apply the deletion filter. For example, select the city if deletions should apply to locations in a specific city.
  • Match condition to filter deletion records in ServiceNow’s Locations: Select the match condition used for deletion filtering. For example, select Equals To if you want to delete only locations that exactly match the chosen value.
  • Value to filter deletion records in ServiceNow’s Locations: Select or enter the value that defines which locations should be deleted. For example, enter Closed to remove all locations marked as closed in Dayforce.
  • Organization Levels to Sync: Select the organizational levels from Dayforce to include in the location sync. For example, include Corporate, Region, and Site to map locations at multiple hierarchy levels.
  • Limit Data Changes: Select to limit how many location records are created, updated, or deleted per sync run. For example, set limits to avoid bulk changes during testing or initial syncs.
  • Set Creation Limit: Select to specify the maximum number of location records to create per sync run. For example, set 100 to limit the new locations created in one execution.
  • Set Update Limit: Select to specify the maximum number of location records to update per sync run. For example, set 100 to control update volume.
  • Set Deletion Limit: Select to specify the maximum number of location records to delete per sync run. For example, set 10 to prevent unintentional bulk deletions.

Users Sync

  • Record Matching Field: Select the field used to match user records between Dayforce and ServiceNow. For example, select the Employee number if both systems use a unique employee ID to identify the same user.
  • Enable User Insertion: Select to allow new users from Dayforce to be created in ServiceNow. For example, when a new hire, John Smith, is added in Dayforce, their ServiceNow user record will be created automatically.
  • Enable Filter Condition for Insertion: Select to apply a filter condition that controls which new users are created in ServiceNow. For example, create only users where the city equals New York.
  • Attribute to filter for records to insert in ServiceNow’s Users: Select the attribute used to apply the insertion filter. For example, select city to create users based on a specific location.
  • Match condition to filter insertion records in ServiceNow’s Users: Select how the chosen attribute is matched to its value. For example, select Equals To to create users only when the city exactly matches “New York.”
  • Value to filter insertion records in ServiceNow’s Users: Select or enter the value to compare against the chosen attribute. For example, enter New York to create only users based in that city.
  • Enable User Updates: Select to allow existing user records in ServiceNow to update automatically when details change in Dayforce. For example, if a user’s job title changes from Analyst to Senior Analyst in Dayforce, it will be updated in ServiceNow.
  • Enable Filter Condition for Updation: Select to apply a filter condition that controls which users are updated in ServiceNow. For example, update only users belonging to the Engineering department.
  • Select attribute to filter updation records in ServiceNow’s Users: Select the attribute used to apply the update filter. For example, select department to update users by their assigned department.
  • Match condition to filter updation records in ServiceNow’s Users: Select how to compare the chosen attribute for updates. For example, select Contains to update all users where the department name contains “Tech.”
  • Value to filter updation records in ServiceNow’s Users: Select or enter the value used for matching during updates. For example, enter Engineering to update only users in that department.
  • Enable User Deletion: Select to allow users removed or terminated in Dayforce to be deleted from ServiceNow. For example, when an employee leaves the company, their ServiceNow account will be removed automatically.
  • Enable Filter Condition for Deletion: Select to apply a condition that limits which user records are deleted from ServiceNow. For example, delete only users where the status equals Inactive.
  • Select attribute to filter for deletion records in ServiceNow’s Users: Select the attribute used to apply the deletion filter. For example, select status if deletions should apply only to inactive user accounts.
  • Provide match condition to filter deletion records in ServiceNow’s Users: Select the match condition used for deletion filtering. For example, select Equals To to delete only users whose status exactly matches “Inactive.”
  • Provide value to filter deletion records in ServiceNow’s Users: Select or enter the value that defines which users should be deleted. For example, enter Inactive to remove all users whose status is inactive in Dayforce.
  • Limit Data Changes: Select to restrict how many user records can be created, updated, or deleted per sync run. For example, limit updates to 100 records per run to prevent large-scale unintended changes.
  • Set Creation Limit: Select to specify the maximum number of user records created per sync run. For example, set 100 to control how many new users are created in each run.
  • Set Update Limit: Select to specify the maximum number of user records updated per sync run. For example, set 100 to limit bulk updates.
  • Set Deletion Limit: Select to specify the maximum number of user records deleted per sync run. For example, set 10 to prevent accidental mass deletions.
  • Enable Incremental User Sync: Select to process and sync only those user records that were created, updated, or deleted since the last successful integration run. For example, when enabled, only new or changed records since the last sync will be processed, improving performance.

Sample Ticket Description Template

You can customize the offboarding, onboarding, and role change ticket descriptions using dynamic placeholders to insert employee details when the ticket is created. 

The following is the ticket description template:

[ticketType] Employee ticketType process required for employeeName (employeeNumber) | Effective: date 

Note: During ticket creation, these placeholders will be replaced with the actual ticket type (Onboarding, Offboarding, Role Change) and the employee’s details, as follows: 

  • [Onboarding] Employee Offboarding process required for John Doe (Emp.No. 441) | Effective: June 19, 2025.
  • [Offboarding] Employee Offboarding process required for John Doe (Emp.No. 441) | Effective: June 19, 2025.
  • [Role Change] Employee Role Change process required for John Doe (Emp.No. 441) | Effective: June 19, 2025.

Attribute Mapping

The Attribute Mapping tab allows you to define how each field in Dayforce corresponds to a specific field in ServiceNow. This configuration ensures that, during every integration cycle, data is accurately translated and aligned between the two systems, maintaining consistency and data integrity across all entities. You can refer to the Attribute Mapping tab while setting up your integration to visualize and verify the field-level mappings.
