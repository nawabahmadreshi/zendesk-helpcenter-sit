Overview

The Aquera platform allows your organization to scale complex data management by automating data integration across resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

The Active Directory to Jira Service Management (Jira Assets) integration synchronizes user data from Active Directory to Jira Assets. It ensures that identity information in Jira remains accurate and up-to-date. The integration handles new, updated, or inactive users in Active Directory and ensures these changes are reflected in Jira Assets by creating, updating, or managing user records as needed.

The integration monitors changes in Active Directory, such as the creation of new users, attribute updates (for example, department, title, and manager), and de-provisioned accounts. It maps the AD attributes to corresponding fields in Jira Assets. The integration also establishes manager relationships using asset references, enabling hierarchical visibility within Jira Assets.

This guide provides details on the configurations available in the Aquera Admin Portal for setting up and managing the Active Directory to Jira Service Management (Assets) integration.

Key Functions

The Active Directory to Jira Service Management (Jira Assets) integration performs the following functions:

  • Retrieves Active Directory users across one or more domains and supports both full sync (all users) and incremental sync (only users changed since the last run).

  • Builds a safe AD user filter that includes only user accounts and excludes computer accounts. In Active Directory, computer accounts are identities created for devices (like laptops, desktops, and servers)

  • Enriches users with manager details by collecting and resolving manager Distinguished Names (DNs) to ensure that manager attributes can be mapped correctly.

  • Checks Jira Assets once to identify the right workspace and record type (like User). Then it loads the existing records and compares them with Active Directory users. 
    Note: In Jira Assets, a workspace is where data is stored, and a record type (like User or Employee) is how that data is grouped.

  • Uses a configured correlation attribute (for example, employee ID/email / username) to match AD users to Jira Assets and then categorizes users into Create, Update, or Terminate/Deactivate buckets.

  • The integration treats inactive users differently, depending on whether they already exist in Jira Assets.

    • If a user is inactive in Active Directory and does NOT exist in Jira Assets

      • The integration does nothing.

      • It does not create a new user record in Jira Assets for inactive users.

    • If a user is inactive in Active Directory and already exists in Jira Assets
      The integration looks at your configuration:

      • If Delete Inactive Users is enabled, the user record is deleted from Jira Assets.

      • If Delete Inactive Users is disabled, the user record is updated (for example, it is not marked inactive or modified based on mappings).

  • Applies configurable conditions (filters) and execution limits to control which users are processed and how many records are handled per run.

  • Maps AD attributes to Jira Assets attributes using the Attribute Map, including optional manager mapping using Jira Assets object references (for hierarchy/reporting).

Prerequisites

The following are the prerequisites:

  • Aquera Tenant Account: You must have a tenant account in the Aquera Admin portal. Visit https://admin.aquera.io to register for a tenant account. Contact your Aquera account manager to get Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/

  • Configure Application Connectors in the Aquera Admin Portal: You must configure the required application connectors in the Aquera Admin Portal based on your environment:

    • Active Directory: Configure the Active Directory connector so Aquera can read user details (and manager information if required).

    • Jira Service Management (Jira Assets): Configure the Jira connector with access to Assets so that Aquera can create and update records in the selected workspace and object type.

Integration Setup

This section describes the configuration required for the Active Directory to Jira Service Management (Jira Assets) integration. It covers selecting the applications involved and configuring the integration settings used to create and update Jira Asset records for Active Directory users.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin Portal.

  2. Navigate to Integrations from the left-side panel.

  3. Click Create Integration.

  4. Select Active Directory to Jira Service Management (Jira Assets).

  5. Enter a name and description for the integration.

  6. Click Continue. The Configuration page is displayed.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section shows the application connectors required for this integration. This section allows you to select:

  • The Active Directory connector retrieves user details.

  • The Jira Service Management (Jira Assets) connector is used to create or update user records.

Parameters

The Parameters section lets you customize the integration settings for your use case. It includes settings to control how users are matched, which records are processed, and how Jira Assets records are created or updated.

Integration Features

  • Enable User Lifecycle Management: Select to enable user provisioning and perform incremental updates.

General Settings

This section enables you to configure the following integration features:

  • Enable Report-Only Mode: Select to generate a report detailing the updates that would happen, but without applying them.
  • Enable Full Sync: Select to process all Active Directory users during the run.
  • Enable Logging: Select to generate logs outlining data changes.
  • Enable Report Generation: Select this checkbox to generate the integration report.
  • Enable Email Notifications: Select this checkbox to enable the integration-related notifications.
  • Enable Log Insights: Select this checkbox to visualize the integration-related metrics.

Manage Users 

  • AD Application List: Select one or more Active Directory instances to pull user data for synchronization. For example, selecting Active Directory–Corporate and Active Directory–Regional syncs users from both directories.

Active Directory Settings

  • Select AD Domain: Specifies the Active Directory domain to search for users. For example, selecting dc=aqueratest,dc=in restricts the sync to users in that domain.

  • AD Filter Name: Choose the Active Directory attribute that determines which users are included in the sync. For example, selecting employeeID limits the sync to users identified by their employee ID.

  • Filter Criteria Value: Enter the value used to match users for the selected filter. For example, entering five retrieves users with employeeID = 5.

JIRA Settings

  • Target Schema (Jira Assets): Determines where user records are stored in Jira Assets. For example, selecting test ensures all users are created or updated within that schema.

  • Target Object Type (Jira Assets): Defines the Jira Assets object type used to store users. For example, selecting Employees stores each Active Directory user as an Employee object.

  • Enable Manager Mapping: Links each user to their manager in Jira Assets based on Active Directory manager data. For example, if John reports to Sarah in AD, Sarah is set as John’s manager in Jira Assets.

  • Delete Inactive Users: Controls how inactive users are handled. For example, when enabled, users who leave the organization are permanently removed from Jira Assets.

  • Manager Attribute Name (Jira Assets): Specifies the Jira Assets attribute that stores the manager reference. For example, entering the Manager Name links users to their manager using this field.

Provisioning Threshold Settings

  • Import Limit: Enter the maximum number of users to be processed in a single run. For example, entering 10 processes only includes the first ten users.

  • Create Limit: Enter the maximum number of new users that can be created in one run. For example, entering 1 creates only one new user, which is useful for testing.

  • Update Limit: Enter the maximum number of existing users to update in a single run. For example, entering one updates only one user at a time.

  • Delete Limit: Enter the maximum number of users that can be deleted in one run. For example, entering one deletes only one inactive user during the run.

Correlation Settings
  • Source Correlation Attribute: Select the Active Directory attribute that uniquely identifies each user. For example, selecting User Principal Name matches users using their email-style login.
  • Target Correlation Attribute: Select the Jira Assets attribute used to identify user records uniquely. For example, selecting Employee ID ensures users are matched using the same employee identifier in Jira Assets.
  • Condition to Create Employees: Specify which Active Directory users are allowed to be created in Jira Assets. For example, selecting All Active Users creates records only for active employees.
  • Condition to Update Employees: Specify which users are allowed to be updated in Jira Assets. For example, selecting All Active Users updates only the details of active employees.
  • Condition to Terminate an Employee: Specify which users can be removed or deactivated in Jira Assets. For example, selecting All Inactive Users handles users who are no longer active.

Logging Settings 

Configure the following log settings:

  • Select the Log Detail Level: Select the desired log detail level from the dropdown menu. The following are the available options:
    • Summary Only: Attribute details are not displayed. Only the count of CRUD operations is shown.
    • Full Details: All attribute details are displayed.
    • Custom: Includes or excludes selected attribute details. Selecting the Custom option will display the Attribute Filter drop-down menu with the following options:
      • Included Attributes: Select the attributes to be included in the log.
      • Excluded Attributes: Select the attributes to be excluded from the log.

  • Store Logs: Enable storing log data for later review and reporting.

Log Insights

Configure the following settings to customize log insights:

  • Dashboard Name: Enter the name of the dashboard where log insights will be created or updated. For example, entering AD to Jira Assets – Log Insights creates a dashboard with this name to display integration analytics.

  • Select the Log Detail Level: Select how much information should be captured in log insights. For example, selecting Full Details records detailed information for each processed user and action.

    • Summary Only: Captures only high-level counts and does not include attribute-level details. For example, the dashboard shows the number of records created, updated, or deleted.

    • Full Details: Captures all available attribute-level details for each processed record. For example, user attributes such as name, email, department, and manager are included.

    • Custom: Allows you to include or exclude specific attributes in log insights. Selecting this option displays the Attribute Filter settings.

      • Included Attributes: Select the attributes that should appear in the log insights. For example, include employeeID, displayName, and department.

      • Excluded Attributes: Select the attributes that should be excluded from the log insights. For example, exclude phoneNumber or managerDN.

  • Store Log Insights: Enable saving log insight data for reporting and analysis.

 

Report Settings

Configure the following report settings:

  • Generate Integration Report: Select to generate a CSV report for each integration run. For example, when enabled, a CSV file is created with details of users processed and the actions performed in Jira Assets (create, update, or delete).

  • Select the Report Detail Level: Select how much information should be included in the report. For example, selecting Full Details includes all mapped user attributes in the CSV.

    • Summary Only: Includes only overall counts and does not include attribute-level details. For example, the report shows how many records were created, updated, or deleted.

    • Full Details: Includes complete attribute-level details for each processed record. For example, the report includes fields such as name, email, department, title, and manager.

    • Custom: Includes or excludes specific attributes in the report. For example, you can generate a report that excludes sensitive fields such as mobile or homeAddress.

      • Included Attributes: Select the attributes that must appear in the report. For example, include employeeID, displayName, and department.

      • Excluded Attributes: Select the attributes that should not appear in the report. For example, exclude phoneNumber or managerDN.

Notification Settings

  • Admin's Email Addresses: Enter the email addresses of the admins to whom the email notifications will be sent

  • Notification Domain: Specify the domain context for the notification emails.

  • Admin Email Subject: Enter the content that must be used as the email subject for sending email notifications to the admins.

  • Admin Email Notification Template: Enter the content that must be used as the email body for all admin email notifications

  • Send Integration Report via Email: Select to receive email notification with a report for all integration runs.

  • Notify Aquera Support on Failure: Enable notifying Aquera Support when the integration fails.

Email Report Settings

  • Email Report Detail Level: Select the level of detail to include in email reports.
     Custom lets you choose exactly which attributes appear in the report.

    • Summary Only: Includes only overall counts and does not include attribute-level details. For example, the email report shows how many records were created, updated, or deleted.

    • Full Details: Includes complete attribute-level details for each processed record. For example, the email report includes fields such as name, email, department, title, and manager.

    • Custom: Includes or excludes specific attributes in the email report. For example, you can generate an email report that excludes sensitive fields such as mobile or homeAddress.

      • Included Attributes: Select the attributes that must appear in the email report. For example, include employeeID, displayName, and department.

      • Excluded Attributes: Select the attributes that should not appear in the email report. For example, exclude phoneNumber or managerDN.

  • .Include Attribute Values in Email Report: Enable including actual attribute values in the email report. For example, when enabled, the report shows real user data instead of only summary counts.

Attribute Map

The Attribute Map lets you choose which Active Directory user attributes should be mapped to Jira Assets fields. The integration uses these mappings to populate the selected Jira Assets attributes when creating or updating user records.
