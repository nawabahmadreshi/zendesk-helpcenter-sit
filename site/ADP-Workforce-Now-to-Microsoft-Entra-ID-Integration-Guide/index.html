Overview

The Aquera platform enables organizations to scale complex identity and data management by automating integrations across enterprise systems. Aquera integrations facilitate automated provisioning and data synchronization, ensuring consistency across applications and infrastructure. By orchestrating identity lifecycle processes and automating repetitive tasks, organizations can streamline operations, dynamically scale resources, and improve the efficiency of managing complex IT environments.

The ADP Workforce Now to Microsoft Entra ID Integration automates the synchronization of worker data from ADP Workforce Now to Microsoft Entra ID. This integration ensures that user accounts and contacts in Entra ID are created, updated, and deactivated automatically based on authoritative HR data from ADP.

The integration retrieves worker records from ADP Workforce Now, applies configured eligibility rules, and matches each worker to an existing Entra ID user or contact using a defined correlation attribute. If no matching record is found, a new account or contact is created. If a record already exists, the integration updates it when changes are detected, such as updates to job title, department, manager, or other profile attributes. When a worker becomes inactive or no longer meets the configured eligibility conditions, the integration applies lifecycle rules to update the corresponding record status in Entra ID.

The integration also supports incremental synchronization and configurable lifecycle offsets to control when provisioning, updates, or deactivation actions occur relative to worker lifecycle events.

Key Features

The ADP Workforce Now to Microsoft Entra ID Integration provides the following capabilities to automate identity lifecycle management and maintain accurate user and contact records in Microsoft Entra ID:

  • Creates Microsoft Entra ID user accounts for eligible workers from ADP Workforce Now, including automatic generation of a temporary password.
  • Updates existing Microsoft Entra ID user accounts by applying attribute changes detected in ADP Workforce Now.
  • Deactivates Microsoft Entra ID user accounts based on configurable lifecycle rules, with deactivation prioritized when both update and deactivation conditions apply.
  • Creates Microsoft Entra ID contacts for workers based on configured contact eligibility rules.
  • Updates existing Microsoft Entra ID contacts when corresponding worker attributes change in ADP Workforce Now.
  • Removes Microsoft Entra ID contacts when workers no longer meet the configured contact eligibility conditions.
  • Supports flexible user and contact correlation using configurable attributes such as Associate ID, Position ID, or Work Email.
  • Applies configurable eligibility rules to determine which workers are created, updated, or deactivated, including support for custom filtering and date-based conditions.
  • Supports lifecycle date offsets for onboarding and offboarding, allowing provisioning or deactivation to occur relative to worker lifecycle events.
  • Manages Microsoft Entra ID group memberships based on defined assignment rules.
  • Processes only worker records that have changed since the last successful run using stored last-modified timestamps.

Integration Setup

This section outlines the configuration required to establish the integration between ADP Workforce Now and Microsoft Entra ID using the Microsoft 365 connector in the Aquera Admin Portal.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin Portal.
  2. In the left navigation panel, select Integrations.
  3. Click Create Integration.
  4. Select ADP Workforce Now / Entra ID as the integration template.
  5. Enter a Name and Description for the integration.
  6. Select the connector instances that the integration will use:
    • ADP Workforce Now
    • Microsoft 365 (Entra ID)
  7. Click Continue.

The Configuration page is displayed to complete the integration setup.

Configuration

The Configuration tab allows you to review and customize the behavior of the ADP Workforce Now to Microsoft Entra ID Integration.

The following sections are available under the Configuration tab:

Applications

The Applications section displays the configured source and target connectors used by the integration.

This section allows you to review, search, and update the selected connector instances associated with the integration.

Parameters

The Parameters section allows you to configure how ADP Workforce Now worker records are processed and synchronized with Microsoft Entra ID.

This section includes settings that control worker eligibility rules, account creation and deactivation behavior, incremental synchronization, optional onboarding and offboarding date offsets, contact management behavior, and Microsoft Entra ID group assignment logic.

These settings determine which workers are created, updated, or deactivated in Entra ID and how user accounts, contacts, and group memberships are managed based on ADP Workforce Now data.

Integration Features

  • Enable user management: Enables synchronization of ADP worker records to Microsoft Entra ID user accounts. For example, when a new employee is added in ADP Workforce Now and meets the eligibility rules, the integration creates the corresponding user account in Microsoft Entra ID.
  • Enable user date offset: Enables processing of users based on configured hire or termination date offsets. For example, if a hire date offset of -2 days is configured, the integration can create the user account two days before the employee’s official hire date.
  • Enable contact management: Enables synchronization of ADP worker records as contacts in Microsoft Entra ID. For example, a contractor or external worker in ADP Workforce Now can be created as a contact in Entra ID instead of a user account.
  • Enable contact date offset: Enables processing of contacts based on configured lifecycle date offsets. For example, if a contact offset of +1 day is configured for termination, the integration removes or deactivates the contact one day after the worker’s termination date.
  • Enable password policy: Enables automatic generation of temporary passwords for newly created Microsoft Entra ID user accounts. For example, when a new employee account is created in Entra ID, the integration generates a temporary password based on the configured password policy.
  • Enable incremental flow: Enables incremental synchronization so that only workers updated since the previous integration run are processed. For example, if a worker’s department or title changes in ADP Workforce Now, only that updated record is processed during the next run.
  • Enable Entra group management: Enables automatic assignment and removal of users from Microsoft Entra ID groups based on configured group assignment rules. For example, if a worker belongs to the Sales department in ADP, the integration can automatically add the user to the Sales-Entra group.

General Settings

  • Enable logging: Select to generate logs outlining data changes.
  • Enable notifications: Select to enable the integration-related notifications.

Worker Management

This section allows you to configure how worker records from ADP Workforce Now are created, updated, and deactivated in Microsoft Entra ID.

Note: The Worker Management section appears when the Enable user management checkbox is selected in the Integration Features section of the Parameters tab.

The Worker Management section includes the following settings:

  • Workers eligible for provisioning in Entra: Select which ADP workers should be created or updated in Microsoft Entra ID.
    The following options are available:
    • All workers: Provisions and updates all workers from ADP Workforce Now, regardless of employment status. For example, both active and inactive workers are synchronized to Entra ID.
    • All active workers: Provisions and updates only workers whose employment status is active. For example, terminated or retired workers are excluded.
    • All active full-time workers: Provisions and updates only workers who are both active and classified as full-time in ADP Workforce Now. For example, part-time or contract workers are excluded.
    • Custom: Allows you to define a rule to determine which workers should be provisioned.
      When Custom is selected, the following field is displayed:

      • Custom upsert condition: Provide a JavaScript expression to identify which workers should be created or updated in Microsoft Entra ID.
        Note: This field is available only when Custom is selected for the Workers eligible for provisioning parameter in Entra.

        Examples:

        • Provision workers only when a work email address is available

          !!person__emailAddresses?.work

        • Provision workers only when their employment status is Active

          workerStatus__statusCode?.codeValue==="Active"

        • Provision workers only when they belong to the Sales department

          workerOrganization__departmentName==="Sales"

        • Provision workers only when their original hire date is today or in the future

          newDate(workerDates?.originalHireDate)>=newDate()
  • Deactivate condition type: Select how the integration determines which workers should be deactivated in Microsoft Entra ID.
    The following options are available:
    • Basic: Uses predefined worker status rules to determine which users should be deactivated.
      When Basic is selected, the following field is displayed:
      • Workers eligible for deactivation in Entra: Select which worker statuses trigger account deactivation in Microsoft Entra ID. For example, selecting All Terminated Workers ensures that user accounts are deactivated when employees leave the organization.
        The following options are available:
        • All terminated workers: Deactivates user accounts for workers whose employment status is marked as terminated in ADP Workforce Now.
        • All deceased workers: Deactivates user accounts for workers whose status in ADP Workforce Now is marked as deceased.
        • All retired workers: Deactivates user accounts for workers whose status in ADP Workforce Now is marked as retired.
    • Custom: Allows you to define a custom rule to determine which users should be deactivated.
      When Custom is selected, the following field is displayed:

      • Custom deactivate condition: Enter a JavaScript rule to determine which workers should be deactivated in Microsoft Entra ID.

        Examples:

        • Deactivate users when the worker status is Terminated

          workerStatus__statusCode?.codeValue==="Terminated"

        • Deactivate users when the worker status is Terminated and the termination date has passed

          workerStatus__statusCode?.codeValue==="Terminated"&&
newDate(workerDates?.terminationDate)<=newDate()

        • Deactivate users when the worker status is Retired

          workerStatus__statusCode?.codeValue==="Retired"

        • Deactivate users when the worker is no longer Active

          workerStatus__statusCode?.codeValue!=="Active"
    • None: Disables automatic deactivation logic. When this option is selected, the integration does not automatically deactivate any users.

User Correlation Settings

The User Correlation Settings section defines how worker records from ADP Workforce Now are matched with existing user accounts in Microsoft Entra ID. Proper correlation ensures that existing users are updated instead of creating duplicate accounts.

The user correlation section includes the following settings: 

  • Workforce Now correlation attribute: Select the ADP Workforce Now field that will be used to search for an existing user in Microsoft Entra ID.
    The following options are available:
    • Associate ID: Uses the worker’s Associate ID from ADP Workforce Now as the lookup value. For example, if the Associate ID is A10234, the integration searches for a user in Entra ID who has the same value for the selected Entra correlation attribute.
    • Position ID: Uses the worker’s Position ID from ADP Workforce Now as the lookup value.
    • Work Mail: Uses the worker’s primary work email address from ADP Workforce Now as the lookup value. For example, barbara.jensen@example.com is used to search for a matching user in Entra ID.

  • Entra correlation attribute: Select the Microsoft Entra ID attribute that will be compared with the selected ADP field during user lookup.
    The following options are available:

    • Employee ID: Uses the Employee ID attribute in Entra ID for comparison.
    • UPN: Uses the User Principal Name attribute in Entra ID for comparison.
    • Mail: Uses the Mail attribute in Entra ID for comparison.

    For example, if Work Mail is selected in ADP and UPN is selected in Entra ID, the integration searches for a user whose UPN matches the worker’s Work Mail value.

User Provisioning Schedule

This section allows you to configure when user accounts are created or deactivated in Microsoft Entra ID relative to worker lifecycle events in ADP Workforce Now.

Note: The User Provisioning Schedule section appears when the Enable user date offset checkbox is selected in the Integration Features section of the Parameters tab.
The following settings are available.

User Onboarding Date and Time

Configure the following settings to control when user accounts are created relative to a worker’s onboarding date:

  • User account creation timing: Select when the user account should be created relative to the worker’s onboarding date in ADP Workforce Now.
    The following options are available:

    • On onboarding date: Creates the user account on the worker’s onboarding date. For example, if the onboarding date is July 10, the account is created on July 10.

    • Before onboarding date*: Creates the user account a specified number of days before the onboarding date.
      When Before onboarding date is selected, the following fields are displayed:

      • Days before onboarding date*: Enter the number of days before the onboarding date when the account should be created. For example, if the onboarding date is July 10 and the value is 5, the account is created on July 5.

      • Timezone priority*: Select the time zone priority used to determine which time zone is applied when calculating the onboarding schedule. The integration evaluates the selected options in order and uses the first available time zone to determine the execution time.
        The following options are available:

        • Employee Timezone: Uses the worker’s configured time zone from ADP Workforce Now to determine when the onboarding action should occur. For example, if the execution time is 08:00 and the worker’s time zone is Asia/Kolkata, the account creation process runs at 8:00 AM in that time zone.
          The following options are available:

          • UTC: Uses Coordinated Universal Time (UTC) as the reference time zone for executing the onboarding action. This option ensures that the account creation process runs at the same universal time regardless of the worker’s geographic location.

          • PST: Uses Pacific Standard Time (PST) as the reference time zone for executing the onboarding action. For example, if the execution time is 08:00, the account creation process runs at 8:00 AM PST.

      • Execution time*: Select the time when the onboarding action should occur. For example, selecting 08:00 ensures that the user account is created at 8:00 AM based on the configured time zone priority.

  • After onboarding date*: Creates the user account a specified number of days after the onboarding date.
    When After onboarding date is selected, the following fields are displayed:

    • Days after onboarding date*: Enter the number of days after the onboarding date when the account should be created. For example, if the onboarding date is July 10 and the value is 2, the account is created on July 12.

    • Timezone priority*: Select the time zone priority used to determine which time zone is applied when calculating the onboarding schedule.
      The following options are available:

      • Employee Timezone: Uses the worker’s configured time zone from ADP Workforce Now to determine when the onboarding action should occur.

      • UTC: Uses Coordinated Universal Time (UTC) as the reference time zone for executing the onboarding action.

      • PST: Uses Pacific Standard Time (PST) as the reference time zone for executing the onboarding action.

  • Execution time: Select the time when the onboarding action should occur. For example, selecting 08:00 ensures that the user account is created at 8:00 AM based on the configured time zone priority.

Log Settings

The Log Settings section allows you to configure how integration activity is recorded, including the level of detail captured and how sensitive information is handled in logs.

Note: The Log Settings section appears when the Enable Logging checkbox is selected in the General Settings section of the Parameters tab.

  • Log Type: Select the type of data that should appear in the logs.
    The following options are available:
    • Minimum: Logs only the operations performed during the integration run. Operations are recorded without GUIDs or field details. For example, the log may display User Created, User Updated, or User Deleted without listing any attributes or identifiers.
    • Metadata: Logs the operations performed along with the names of the fields that were changed, but does not include the GUID. For example, the log may show Updated fields: Department and Title without displaying the associated user identifier.
    • Non-PII Data: Logs the operations performed, field names, and selected non-PII values. The GUID is not included. For example, the log may display Department updated to Sales while masking sensitive information such as date of birth.
    • GUID with Non-PII Data: Logs the GUID together with the operations performed, field names, and selected non-PII values. This option provides detailed traceability while still preventing exposure of sensitive information.
  • Log Level: Select the desired log detail level from the dropdown menu.
    The following options are available:
    • Full: Displays all attribute details recorded during the integration run.
    • Include List: Displays only the attributes specified in the Custom Inclusion List field.
      If Include List is selected, the following field is displayed:
      • Custom Inclusion List: Enter the attributes that should be displayed in the logs. Only the attributes specified in this list will appear in the log entries.
    • Exclude List: Displays all attribute details except those specified in the Custom Exclusion List field.
      If Exclude List is selected, the following field is displayed:
      • Custom Exclusion List: Enter the attributes that should be excluded from the logs display.
  • PII Fields: Enter the attributes that contain Personally Identifiable Information (PII) so they can be masked or excluded in logs. For example, adding dateOfBirth ensures that this sensitive value does not appear in integration logs.
  • GUID Fields: Enter the fields that store Globally Unique Identifiers (GUIDs) so they are handled correctly in logs. For example, adding managerId ensures the integration treats this value as a system-generated identifier.
  • Employee Identifier Fields: Enter the attributes that represent employee identifiers so they appear consistently in logs. For example, adding associateOID ensures the employee identifier is logged correctly for each operation.

Attribute Map

The Attribute Map section defines how worker attributes from ADP Workforce Now are mapped to corresponding attributes in Microsoft Entra ID. These mappings control how employee data from ADP is used when creating or updating identities in Entra ID.

The following mapping configurations are available:

  • Entra User Map: This mapping is used when provisioning user accounts in Microsoft Entra ID. Each row defines how an ADP worker attribute or expression is mapped to a specific Entra ID user attribute. For example, the worker’s Work Email can be mapped to the User Principal Name (UPN) and Mail attributes, while Legal First Name and Legal Last Name can be combined to populate the Display Name.
  • Entra Contact Map: This mapping is used when provisioning contacts instead of user accounts in Microsoft Entra ID. Similar to the user map, each row defines how attributes from ADP Workforce Now are mapped to contact attributes in Entra ID.

Mappings may include expressions that transform source values before they are sent to Entra ID. For example, the Mail Nickname can be generated by extracting the portion of the worker’s Work Email before the @ symbol.

Administrators can review and modify these mappings to align with their organization’s identity provisioning requirements. Any changes made here determine how worker data from ADP Workforce Now flows into Microsoft Entra ID during provisioning and subsequent updates.

ADP Workforce Now Worker Group Assignment Rules

The Group Rules section determines which Microsoft Entra ID groups a worker is assigned to based on attribute values from ADP Workforce Now.

Administrators define rules that evaluate specific worker attributes from ADP and automatically assign users to the appropriate Entra ID groups during user provisioning and updates.

Remove Groups on Deactivation

When Remove Groups On Deactivation is enabled, users are automatically removed from the groups managed by this integration upon account deactivation in ADP Workforce Now.

Group Assignment Rules

Each rule evaluates a selected ADP Workforce Now attribute and assigns workers to one or more groups when the rule conditions are met.

Common attributes that can be used include:

  • AssociateOID
  • Associate ID
  • Birth Date
  • Gender Code Value
  • Custom employment attributes such as Employment > Custom Fields > Sales Territory

The selected attribute becomes the condition used to evaluate each rule.

Each rule includes the following components:

  • A value from the selected ADP Workforce Now attribute
  • A worker status condition (for example, Active)
  • One or more Group Assignments in Microsoft Entra ID

Rules are evaluated from top to bottom. When a worker matches a rule, the worker is assigned to the configured groups. A worker can match multiple rules and therefore be added to multiple groups.

If no rule matches, the worker is not assigned to additional groups unless a fallback rule is configured.

These rules are applied during initial user creation and subsequent updates.

Example

Attribute used for rule evaluation:

Employment > Custom Fields > Sales Territory

  • Rule 1: If Sales Territory = Boston North and Status = Active, assign the worker to Apps_Sales.
  • Rule 2: If Sales Territory = Connecticut and Status = Active, assign the worker to Sales-District.
  • Rule 3: If Sales Territory = New England and Status = Active, assign the worker to Sales-Distribution.

Evaluation results:

  • A worker with Sales Territory = Boston North and Active status is assigned to Apps_Sales.
  • A worker with Sales Territory = Connecticut and Active status is assigned to Sales-District.
  • A worker with Sales Territory = New England and Active status is assigned to Sales-Distribution.
  • Workers who do not match any rule are not assigned to additional groups.

Using a Different Attribute

If another attribute, such as Associate ID, Gender Code Value, or Birth Date, is selected, the same rule logic applies. The selected attribute is used as the value evaluated in each rule, and workers who match the rule conditions are assigned to the configured groups.

Expression Mode

You can switch the rule builder to Expression Mode by selecting Switch to Expression.

Two expression options are available:

  • For Conditions Only: Use expressions to define rule conditions while keeping group assignments configured in the UI.
  • For Conditions and Assignments: Use expressions to define both rule conditions and group assignments.

Expression mode allows administrators to create more advanced and flexible rule logic when the standard rule builder does not meet specific requirements.

Update the Integration to the Latest Version

You can upgrade an integration built from a template to the latest version to access new features and enhancements. 

Perform the following steps to upgrade an integration:

  1. Navigate to the Integrations module on the Aquera Admin portal.
  2. Click the vertical ellipsis icon next to an integration and click View Details. The Details panel is displayed with the following sections:
    • INTEGRATION DETAILS: Displays the integration name, description, creation date, last update date, and the user who created or last updated it.
    • TEMPLATE DETAILS: Displays the current version number and its features under CURRENT VERSION. If a newer version is available, the version number and the changes are shown under NEW VERSION AVAILABLE with an Upgrade button.
  3. Click Upgrade. The Upgrade Integration pop-up window is displayed.
  4. Provide the following details for the new integration to be created after the upgrade:
    • New Integration Name: Enter the name of the new integration.
    • Description: Provide a description for the new integration.
      The current version number and the upgraded version number are displayed.
  5. Click Upgrade. A new integration is created with the same configuration details as the existing one, but with features or enhancements from the upgraded version.
