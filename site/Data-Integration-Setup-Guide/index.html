Overview

Aquera's Data Integration lets you synchronize data of any resource type between a source application and a target application.

With Data Integration, you can set up integrations for all resource types that the connector supports. For example, you can configure a data integration that synchronizes the employees between the source and target applications (connectors), as long as both the source and target applications support the Employees resource type.

You can also synchronize different resource types from the source application to the target application. For example, you can synchronize the Candidates resource type from the source application to the Employees resource type in the target application. 

This guide explains how to set up Data Integration in the Aquera Admin portal.

Notes:

  • Data Integration is recommended for synchronizing resource types other than Users and Groups. Aquera recommends User Integration for synchronizing Users and Groups resource types.
  • If Data Integration is unavailable in your tenant, contact the Aquera Support team.

Prerequisite

Ensure that the source and target applications (connectors) are added and configured in the Aquera Admin portal before creating the integration. Refer to the corresponding configuration guides for details on configuring the connectors.

Creating a Data Integration

You can create a data integration using one of the following options:

  • Create your own Data Integration
  • Create Data Integration using a Template

Create your own Data Integration

Create a new data integration where you need to fill in every configuration field manually.

Perform the following steps to create a new data integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel which displays the list of integrations created by the user. 
    Note: The number of integration records displayed on this page is based on the display size of the user's screen.
  3. Click Create Integration.
  4. Select Data Integration.
  5. Enter a name and a description for the integration.
  6. Click Continue. A new page is displayed with the fields and tabs to configure the integration.
  7. Enter the details on the Configuration tab. For further details, refer to the Configuration section.
  8. Click Create to create the data integration.

Create Data Integration using a Template

Templates are pre-configured integrations created by Aquera for specific use cases. You can use the templates as-is or modify them to match your organization's needs. You can view the templates if Aquera has enabled templates in your tenant. 

Perform the following steps to create a data integration using a template:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel which displays the list of integrations created by the user. 
    Note: The number of integration records displayed on this page is based on the display size of the user's screen.
  3. Click Create Integration.
  4. Select a pre-built data integration template based on your requirements.
  5. Enter a name and a description for the integration.
  6. Select a source connector in the Source Application dropdown field.
  7. Select a target connector in the Target Application dropdown field.
  8. Click Continue. A new page is displayed with the fields and tabs to configure the integration.
  9. You can update the configurations if required. For further details, refer to the Configuration section.
  10. Click Create to create the data integration.

Basic Details

This section explains the fields to configure the basic details. 

Field

Description

Name

Enter the name of the integration.

Source Application

Select the source application from where the data of the resource type must be retrieved. The data of created and modified attributes of the resource type is retrieved from this application as part of the integration.

Source Resource Type

Select the resource type of the source application. Example: Employees, Roles.

Target Application

Select the target application in which the data of created and modified resources must be updated as part of the integration.

Target Resource Type

Select the resource type of the target application. Example: Employees, Roles.

Report Only

Select this checkbox to generate a report that shows the data that would be changed, without actually making those changes. This can be selected for the first instance of integration to identify any issues. You can clear this selection after verifying the report and confirming the expected result.

Update Limit

The maximum number of records that are allowed to be updated in the target application during a single run of the integration. Set this field to unlimited if you do not want to limit the number of records.
Note: This configuration works only if the selected resource in the target connector supports the respective create / update operation.

Admin Email List

The administrative email addresses to which notifications on this integration have to be sent. Use a comma to separate multiple email addresses.
Example: admin1@example.com, admin2@example.com 

Source Match Expression

The source attribute or JavaScript expression used to identify the target record that matches a source record during resource imports. 

Note: This configuration works only if the selected resource in the target connector supports the GET operation.

If the integration does not find a match, the source record will be considered as a new resource and the resource will be created in the target. If that source record is not a new resource, then the expression needs to be modified. You must run the integration in report mode to verify this.
Example: SOURCE[ENT].employeeNumber

Dest Match Field

Select the target attribute used to match the expression in the Source Match Expression field from the dropdown list. 
Example: DEST[ENT].employeeNumber

Incremental Mode

When the Incremental Mode checkbox is selected, integrations run in incremental import.

In Incremental import, the source records that have been created or modified in the source application since the last integration run will be updated in the target application.

Note: The Incremental Mode configuration works only if the selected resource supports incremental import. 

Import Limit

The maximum number of records that must be retrieved from the source application during a single run of the integration. Set this field to unlimited if you do not want to limit the number of imported records. The default value is unlimited, which retrieves all the records of the selected resource from the source application.

The limit can be set accordingly to verify and debug the source-to-target mapping. You can set the import limit to 1 to verify that the record is imported correctly.

Source Filter

The expression that can be used for filtering the selected resource during the source import, if required. This field is primarily used to verify the data by filtering records from the source application.

Note: The Source Filter configuration works only if the selected resource in the source connector supports the SCIM filter expression provided. Refer to the respective Connector Configuration Guides for more details on the supported filters.

Create Condition

The expression that is used to determine whether a resource record has to be created in the target application.

The following are the valid values:

  • true: Creates resource records in the target application based on the Match Expression provided.
  • false: Does not create any records in the target application.

Update Condition

The expression that is used to determine whether a resource record has to be updated in the target system.

The following are the valid values:

  • true: Updates resource records in the target application based on the Match Expression provided.
  • false: Does not update any records in the target application.

Report Mode

The logging mode of integration details. This setting controls the level of details to be captured in the integration logs. 

The following are the valid values listed in the order of the level of logging: 

  • Minimal
  • Normal
  • Debug
  • Verbose

The default value is normal. 

For more information on the log-level details for each report mode, refer to the Integration Logging Modes guide.

New Integration Hook

If you want to perform a set of tasks when a new resource record is created in a target application, configure the New Integration Hook.

In the New Integration Hook, select a Script or Workflow integration that has to be executed when a new resource record is created in the target application. The selected integration will be triggered during the resource record creation.

Notes:

  • New Integration Hook supports Script integration version 3 and Workflow integration version 3.
  • Logs will be generated for each resource record creation. For example, if there are 10 resource records created in the target application, then there will be 10 integration logs generated for the selected integration.

Update Integration Hook

If you want to perform a set of tasks when a resource record is updated in a target application, configure the Update Integration Hook. 

In the Update Integration Hook, select a Script or Workflow integration that has to be executed when a resource record is updated in the target application. The selected integration will be triggered during the update of the resource record.

Notes: 

  • Update Integration Hook supports Script integration version 3 and Workflow integration version 3.
  • Logs will be generated for each resource record update. For example, if there are 10 resource records updated in the target application, then there will be 10 integration logs generated for the selected integration.

Fire Hooks on Error

Select this checkbox to trigger the integration specified in the New Integration Hook and Update Integration Hook even if there is a failure in the creations and updates of the resource record.

However, if you do not select this checkbox, the integration specified in the respective Integration Hooks will be triggered only for successful creations and updates of the resource records, and not for failures during the creations and updates.

Admin Email Options

This section explains the fields to configure the integration email report. For more information on the contents of the email report, refer to the Integration Email Report section.

Field

 Description

Admin Email Subject

Specify the subject of the email that will be sent to the Admin Email List.

The default template of the email subject is: {{name}} Integration Report

The following keywords can be used in the Admin Email Subject:

  • {{name}} This is replaced with the name of the integration
  • {{status}} This is replaced with the status of the integration

Examples:

  • {{name}} Integration Report and Status is {{status}}
  • [{{name}}] - [{{status}}] Integration Report

Send Always

Sends an email report to the Admin Email List when the integration run is a success, failure, or error. The report is not sent when the status is canceled.

Safeguard Email List

 Enter a comma-separated list of email IDs to which an email notification must be sent whenever the Destination Modification Safeguard is triggered for the integration.

Send on Report Only

Sends an email report to the Admin Email List when the integration is run in Report Only mode.

Send on Update

Sends an email report to the Admin Email List when at least one resource record is updated in the target application.

Send on Failure

Sends an email report to the Admin Email List when the integration run is a failure.

Send on Error

Sends an email report to the Admin Email List when errors occur during the integration run.

Send on Safeguard Trigger

Sends an email report to the email IDs specified in the Safeguard Email List whenever the Destination Modification Safeguard is triggered.

Note: Email IDs configured in the Admin Email List will also receive this report.

Send Details

Sends the integration log report in a zip format as an email attachment. This email attachment will be sent only if at least one of the following options is selected:

  • Send Always
  • Send on Report Only
  • Send on Update
  • Send on Failure
  • Send on Error

Send CSV Report

Sends the integration log report as a zipped CSV file to the email recipients. This is sent as an email attachment only if at least one of the following options is selected:

  • Send Always
  • Send on Report Only
  • Send on Update
  • Send on Failure
  • Send on Error

The CSV file will have the following columns:

  • Source: The source attribute values. The attribute values will vary based on the Report Mode set. 
  • Destination: The target attribute values. The attribute values will vary based on the Report Mode set. 
    Note: For create operations, the Destination will be empty.
  • Attribute: The name of the attribute.
  • Old Value: The previous value of the attribute if any.
    Note: For create operations, the Old Value will be empty.
  • New Value: The new/updated value of the attribute.
  • Action: The operation performed such as Create / Update.
  • Status: The status of the operation. The following are valid statuses:
    • Success: This is shown when the creation, update, or removal is successful.
    • Failure: This is shown when the creation, update, or removal fails.
    • Pending: This is shown when Report Only mode is enabled or Update Limit has been reached.
    • Ignored: This is shown when records were not processed in the target application. Example: matches multiple DEST records.
  • Reasons for Errors or Failures: The reason for the Failure, Pending, or Ignored status. Examples:
    • updateLimit count reached: creation aborted
    • {errorCode:E0000001,errorSummary:Api validation failed: login,errorLink:E0000001,errorId:oaesHm95FOHT9GDa3zSlAwPRg,errorCauses:[{errorSummary:login: An object with this field already exists in the current organization}]} 
    • create condition not satisfied

Notes: 

  • The CSV file will be empty if there are no creations/updates to the records in the target application.
  • When the create / update condition is not satisfied, those records will not be shown in the CSV file.
  • Ensure to set the Report Mode to debug / verbose for the CSV file to show the data correctly.
  • For complex attributes, the sub-attribute names will be shown in the Old Value/New Value columns. 
  • If you have selected both Send Details and Send CSV Report, the recipients will receive an email with a single ZIP file containing text and CSV files.
  • The maximum size of the ZIP file supported in the email report is 40 MB.

Note: Ensure to click Save Changes to save the integration after making the required configuration changes.

Integration Email Report

This section explains the details that are included in the integration email report. The integration email report will have the following details: 

  • For a successful integration run, the content of the email report will contain the following messages:
    • For each attribute creation: <attribute_name> : created
    • For each attribute update: <attribute_name> : updated   
  • For failed integration  runs, the email report will have the same content as the integration logs.
  • When Report Only mode is enabled, the email report will have the same content as the integration logs and is based on the selected Report Mode. For information on the details that are captured in the logs for each of the report modes, refer to the Integration Logging Modes
    guide.
  • All the integration email report will contain the integration parameters and their values. For details on the integration parameters, refer to the Integration Logs guide.

Advanced Details

This section explains the fields to configure the advanced details. 

Field Description

Source Caching

Select an option from the dropdown list to enable, disable, or refresh the caching of source records. The following are the available options:

  • disabled: Select this option if you do not want the source records to be cached.
  • enabled: Select this option to cache the source records so the synchronization runs can be faster. The source records will be securely stored in an encrypted format using a hash of the configured synchronization ID.
    The source records that are retrieved during the first synchronization run will be cached for 7 days or until disabled. The subsequent synchronization runs will skip the source records loading stage as the same set of cached source records are reused which leads to faster synchronization runs.
  • refresh: Select this option to refresh the cache with the latest source records in the next synchronization run. 

Target Caching

Select an option from the dropdown list to enable, disable, or refresh the caching of target records. The following are the available options:

  • disabled: Select this option if you do not want the target records to be cached. The target records will be updated only when the target caching is disabled.
  • enabled: Select this option to cache the target records so the synchronization runs can be faster. The target records will be securely stored in an encrypted format using a hash of the configured synchronization ID. 
    The target records that are retrieved during the first synchronization run will be cached for 7 days or until disabled. The subsequent synchronization runs will skip the target records loading stage as the same set of cached target records are reused which leads to faster synchronization runs.
  • refresh: Select this option to refresh the cache with the latest target records in the next synchronization run.

Incremental Mode Timestamp

The timestamp that must be considered when you run the synchronization in the Incremental Mode. The source records that were added or modified after this timestamp will be considered for the incremental import. You have to set this timestamp manually before each incremental run.

You can do an incremental import at any time by setting a timestamp in the Incremental Mode Timestamp and selecting the Incremental Mode checkbox. 

If you have not selected the Incremental Mode checkbox and have set a timestamp in the Incremental Mode Timestamp, the synchronization will run in the full import mode.

Note: Do not enclose the timestamp in single or double quotes. Provide the timestamp as a valid date. For example, 2023-04-19T14:15:16.148Z, 2023-04-19T14:15Z, 2023-04-19T14:15-08:00, 2023-04-19T14:15+01:00, 3/15/2024 09:00:00 GMT, 3/15/2024 09:00:00 PDT, 3-15-2024 09:00:00 PDT.

Skip Create and Update Failures

When this checkbox is selected, during incremental execution of the synchronization, if there are failures in creating or updating destination records, those transactions will be skipped in the next incremental run. The last modified timestamp will be updated for the source records including the source records that encountered failures in the previous incremental run. Hence, in the next incremental run, only the changes from the last modified timestamp will be considered.

When this checkbox is not selected, those failed transactions will be attempted on the next incremental execution. The last modified timestamps will remain the same as that of the previous incremental run so that the synchronization will re-read all updates that were part of the previous incremental run. This action may result in longer delays while the failures continue through repeated executions of the synchronization.

The Email From Address enables you to specify the sender email address associated with all emails sent by the synchronization. The default email address is noreply@aquera.com.

To modify the , you must first send a request to support@aquera.com. A verification email will be sent to that email address. You must follow the instructions in the email to grant Aquera permission to send emails on behalf of your email account. 
Refer to Verifying Email Address and Configuring DKIM Settings for more details.

Destination Modification Safeguard Enabled

Select this checkbox to trigger the safeguard which will abort the integration run if it would result in the number of modifications exceeding the Max Destination Modification Percentage. 

The modifications include the creation, update, and deletion of records in the target application.

Note: The safeguard calculation ignores Update Limit. The use of Import Limit will reduce the number of modifications in the calculation.

Max Destination Modification Percentage

When Destination Modification Safeguard Enabled checkbox is selected, the integration run will be aborted if the number of modifications exceeds the configured percentage. The default percentage set is 25%. 

When Destination Modification Safeguard is triggered, the integration run will be aborted and there will be no creations or updates in the target applications.

The Destination Modification Safeguard will be triggered based on the following calculation:   

For full imports:

((target records that will be created + target records that will be updated) / (target records that will be created + source records matched)) > MaxDestinationModificationPercentage

For incremental imports:

((target records that will be created + target records that will be updated) / (target records that will be created + total matched records)) > MaxDestinationModificationPercentage 

Here, the total matched records include all of the matches seen in the previous integration runs for the incremental import.

During incremental runs, when a safeguard is triggered, the Incremental Mode Timestamp will not be incremented and hence no records will be skipped.

Note: If the Report Mode is selected as debug or verbose, the integration logs will include the details of the modification safeguard calculation and detail if the safeguard was triggered or not.

Attribute Mapping

You can map the source attributes to the target attributes by navigating to the Attribute Map tab. This is required to be performed to synchronize the data from the source application to the target application. You can also write expressions to modify the value of the source attribute before updating the value in the target attribute.

Perform the following steps to map attributes:

Navigate to your integration and click the Attribute Map tab to perform attribute mapping. The default mapping of attributes from the source to the target application is displayed on the page. The source attributes are listed on the left side and the target attributes are listed on the right side of the page.

The attributes from the source application are specified as SOURCE.<attributeName>.

After the source-to-target attribute mapping is completed, click Save Changes at the top-right corner of the page to save the integration.

You can perform the following actions as per your requirements:

Preview Data

You can preview the data values from the source and target resource by providing the resource ID in the Enter objects for preview search box.

Filter Attributes by an Attribute Name

To filter the list of attributes by the attribute's name, enter the attribute name in the Search Attributes box.

Select a Mapping Option for an Attribute

Set the appropriate mapping option from the source attribute to the target attribute. Specify if the mapping of an attribute is to be done for create and update operations, create operation, or update operation. You can also choose not to map from the source to the target attribute.

Click the dropdown list present between source and target attributes and select any of the following options for an attribute:

  • Map on Create and Update: The target attribute will be updated during the create and update operations
  • Map on Create: The target attribute will be updated only during the create operation 
  • Map on Update: The target attribute will be updated only during the update operation 
  • Don't Map: The target attribute will never be updated

For example, if you select the Map on Create option for the attribute SOURCE.firstName, this attribute value will only be updated in the target application during the create operation.

Note: All the target attributes with mutability as "Immutable" support Map on Create option. It does not support Map on Create and Update or Map on Update options.

Filter Attributes by the Mapping Option

You can filter the list of attributes using the Map filter present between source and target attributes.

For example, if you select the Map on Create and Update option, it will list all the target attributes that will be updated during the create and update operations.

Write Expressions in Attribute Mapping

You can write expressions to modify the value of the source attribute and construct a value to be updated in the target attribute. Write the expression directly in the source attribute field or click the Expand icon in the source attribute field to write a long expression.

On clicking the Expand icon, the Expression Editor window appears, where you can write an expression to perform operations on the attribute values. These expressions can contain variables, attribute names, methods, and functions. When the integration runs, the expressions are evaluated and the resulting values are updated in the target application.

For more details on Aquera's keywords, inbuilt methods, and functions, refer to the Integration Expressions - Keywords, Methods, and Functions guide.

selectUniqueValue Function in Data Integration

Data Integration supports selectUniqueValue function that lets the integration select the first unique value. You can use the selectUniqueValue function in the Source Attribute/Expression section of the Attribute Map page.

The syntax of the function is as follows:

selectUniqueValue(SOURCE.attribute1, SOURCE.attribute2)

For example, consider the function selectUniqueValue(SOURCE.firstName, SOURCE.lastName) is mapped to the FirstName target attribute. If the value of SOURCE.firstName is not the same as the value of the FirstName target attribute across all target resources, the integration considers it as the unique value and maps it to the FirstName target attribute.

If the value of SOURCE.FirstName is the same as the FirstName target attribute, the integration checks the second parameter, which is SOURCE.lastName. If the value of SOURCE.lastName is not the same as the value of the FirstName target attribute across all target resources, the integration considers it as the unique value and maps it to the  FirstName target attribute.

If the integration doesn't find a unique value across all parameters passed in the selectUniqueValue function, it processes all the other records and includes the failed record in the report.

Notes:

  • selectUniqueValue is currently available only on Data Integration.
  • selectUniqueValue is applicable only when the mapping option is set to Map on Create. For more details on mapping options, refer to the Select a Mapping Option for an Attribute section.

Delete an Attribute Mapping

To delete an attribute mapping, navigate to the target attribute field and click the Delete icon.

Array Mapping

You can map target attributes of the type array of objects in the attribute mapping. The array of objects is denoted by the symbol [{}] in the attribute map. To map a source attribute to the target attribute which is an array of objects, click the expand icon in the SOURCE ATTRIBUTE / EXPRESSION field and provide values for the sub-attributes of the selected object array. For more details, refer to the Array Mapping in Data Integration guide.

Tables in Integration

Tables define a name-value pair to create custom attributes. Tables are added in integrations to create custom source attributes and sync them to the target application. Tables are used when an application attribute is empty or particular data is not sent from an application, and you want to update a value for that attribute in another application.

For details on how to add tables and use them in integrations, refer to the Building Integrations - Tables guide.

Scheduling the Integration Run

To schedule an integration, navigate to the Schedule tab of the integration. You can set a time frequency to run the integrations automatically. For more details, refer to the Integration Schedule Setup Guide. 

User Actions on an Integration

You can perform actions on an integration such as execute, upgrade, copy, modify, delete, export, import, and view logs. For further details, refer to the User Actions on Integrations guide.
