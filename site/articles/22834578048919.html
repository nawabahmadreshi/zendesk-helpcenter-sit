Overview

ADP Lyric HCM is ADP's latest HCM platform that bundles HR, payroll, talent, workforce management, benefits management, and analytics solutions.

This guide helps users create an ADP Lyric HCM connector on the Aquera platform.

For more details on the ADP Lyric HCM application, visit the ADP Lyric HCM website.

Supported Features

The ADP Lyric HCM connector supports the following features:

The ADP Lyric HCM connector also supports various features as per the OpenAPI specification. Refer to the OpenAPI Specification section to know all supported OpenAPI operations.

Prerequisites

The following are the prerequisites:

  • Administrator access to the ADP Lyric HCM instance: You must have the admin credentials of your ADP Lyric HCM instance.
  • Bearer Token: You must have a bearer token for authentication. Refer to the Authentication section for more details.

Connector Configuration

Perform the following steps to configure the ADP Lyric HCM connector:

    1. Log in to the Aquera Admin portal.
    2. Navigate to Applications and click Add Application.
    3. Search for ADP Lyric HCM and click the ADP Lyric HCM icon.
    4. Enter the values for the following fields under the Configuration tab:
      • Basic Details:
        • Display Name: Enter a display name for your application.
        • Description: Enter a description for your application.
        • Source Owner: Enter valid email IDs of the source owners.
        • Subscribe to error notification: Enable the toggle button to send notification emails to source owners for errors in the user data transactions. Click Edit Preference to select the error types for which you want to receive notifications. After you configure your preference, you will receive error notifications only for the selected error types. By default, all error types are selected.
        • Pick your protocol: Select the protocol from the dropdown list based on your requirement. SCIM is the default value.
        • Copy this URL: This field displays the base URL of the connector API endpoint. The URL is populated based on your selected protocol. Copy and paste this URL while provisioning to the identity service provider.
      • Tenant Attributes: Refer to the Aquera Tenant Attributes section for details.
    5. Click Create to create the ADP Lyric HCM connector.

Refer to the Getting Started with Connectors guide to learn more about connector setup including different tabs of the connector configuration page.

Aquera Tenant Attributes

The following are the Aquera tenant attributes:

  • Make active X days before hire date: Enter the number of days before the hire date when the user account of a new hire has to be marked as active. For example, if you set this attribute to 5 and the hire date of a new hire is 06-Dec-2021, the user account of the new hire will be marked as active on 01-Dec-2021, which is 5 days before their hire date. This setting applies only to the users whose hiring process is completed. This attribute considers the original hire date that is present in the SCIM response.
  • Activation time for new employee: Enter the time when the user account of an employee must be activated on the hire date. The time must be entered only in the UTC offset format. It is the difference in hours and minutes from the Coordinated Universal Time (UTC) and local time of a particular place and date. The format is HH:MM:SS +/- HH:MM. For example, 12:30:00-08:00, which falls in the PST time zone.
    Note: If the checkbox Use activation time/deactivation time based on employee's timezone is selected, the mentioned difference in hours and minutes from the Coordinated Universal Time (UTC) is ignored and the mentioned local time along with the employee's timezone according to the Use activation time/deactivation time based on employee's timezone scenarios is considered.
  • Keep active for X days after termination: Enter the number of days after the termination date for which the account of a terminated user has to be kept active. For example, if you set this attribute to 2 and the termination date of a user is 06-Dec-2021, the user account will be marked as active till 08-Dec-2021, which is 2 days after their termination date. 
  • Deactivation time for terminating an employee: Enter the time when the user account of an employee must be deactivated on the day of termination of the employee. The time must be entered only in the UTC offset format. It is the difference in hours and minutes from the Coordinated Universal Time (UTC) and local time of a particular place and date. The format is HH:MM:SS +/- HH:MM. For example, 12:30:00-08:00, which falls in the PST time zone.
    Note: If the checkbox Use activation time/deactivation time based on employee's timezone is selected, the mentioned difference in hours and minutes from the Coordinated Universal Time (UTC) is ignored and the mentioned local time along with the employee's timezone according to the Use activation time/deactivation time based on employee's timezone scenarios is considered.
  • Make active X days before rehire date: Enter the number of days before the rehire date when the account of a rehired user has to be marked as active. For example, if you set this attribute to 2 and the rehire date of a user is 06-Dec-2021, the user account will be marked as active on 04-Dec-2021, which is 2 days before their rehire date.
  • Use activation time/deactivation time based on employee's timezone: If this checkbox is selected, the activation or deactivation time is considered based on the employee's timezone. If the employee's timezone is not available then the activation or deactivation time is considered based on the employee's work address. In work address, employee's city is checked first and if the employee's city is not available then employee's work country is considered. If none of these details are available, then the employee will be activated based on the time configured in the Activation time for new employee and deactivated based on the time configured in the Deactivation time for terminating an employee.
    If this checkbox is not selected, the employee will be activated based on the time configured in the Activation time for new employee and deactivated based on the time configured in the Deactivation time for terminating an employee.
  • Deactivate employees without work email: Select this checkbox to ignore the employee accounts with which no work email is associated. The employee accounts without work email will be deactivated. By default, the checkbox is not selected.
  • Import employees having domains matching with the below list only (Rest of them will be marked as inactive): The comma-separated list of domains from which users need to be retrieved. For example, @adp.com,@adp1.com. Users from all other domains will be deactivated. Please note that the inactive users from the provided domains list will be still marked as inactive only.
    Note: The Deactivate employees without work email, Import employees having domains matching with the below list only, and Make users inactive based on ADP Lyric HCM status settings override the following tenant attribute settings:
    • Make active X days before hire date
    • Activation time for new employee
    • Keep active for X days after termination
    • Deactivation time for terminating an employee
    • Make active X days before rehire date
    • Use activation time/deactivation time based on employee's timezone
  • Make users inactive based on ADP Lyric HCM status: Enter the additional status of employees in ADP Lyric HCM that must be considered as inactive. You must enter only Leave as the status based on your requirement. The other values, such as Retired, Terminated, and Deceased are considered inactive by default.
  • Attributes to be excluded in the response: Enter the comma-separated list of attributes that must be excluded from the response. For example, title, name.formatted,originalHiredate, jobCode, ethnicity. All other attributes will be included in the response.
    • The exact names of the attributes must be entered in this field. You can expand the Users list under the Schema on the Aquera Admin portal to see the list of user attributes and find their exact names.
    • ADP Custom attributes can also be removed from the response. You must enter the exact names of the custom attributes as configured in the ADP UI. For example, assignment_stringFields_{codeFields.nameCode.codeValue}, person_dateFields_{dateFields.nameCode.codeValue}, stringFields_{stringFields.nameCode.codeValue}
    • To exclude complex attributes of array type, such as emails, phoneNumbers, and addresses, the type value must also be entered. The format is <attributename>[type]. For example, phoneNumbers[home], emails[work].
      Notes:
        • The Id, userName, and active attributes cannot be removed from the response.
        • To exclude the manager attributes, the entire manager object has to be excluded by entering the value as manager in the field.
        • This tenant attribute is not supported by the resource type 'employee'.
  • Attributes to be included in the response: Enter the comma-separated list of attributes that must be included in the response. Any other user attributes will be excluded from the response.
    • The exact names of the attributes must be entered in this field. You can expand the Users list under the Schema on the Aquera Admin portal to see the list of user attributes and find their exact names.
    • ADP Custom attributes can also be included in the response. You must enter the exact names of the custom attributes as configured in the ADP UI. For example, assignment_stringFields_{codeFields.nameCode.codeValue}, person_dateFields_{dateFields.nameCode.codeValue}, stringFields_{stringFields.nameCode.codeValue}
    • To include complex attributes of array type, such as emails, phoneNumbers, and addresses, the type value must also be entered. The format is <attributename>[type]. For example, phoneNumbers[home], emails[work].
      Notes: 
        • The Id, userName, and active attributes will always be included in the response, even if they are not set in this tenant attribute.
        • To include the manager attributes, the entire manager object has to be included by entering the value as manager in the field.
        • This tenant attribute is not supported by the resource type 'employee'.
  • Get Earnings Summary: Select this checkbox to include earnings summary in the EmployeePayrollRun response.
  • Get Statutory Deductions Summary: Select this checkbox to include statutory deductions summary in the EmployeePayrollRun response.
  • Get Non-Statutory Deductions Summary: Select this checkbox to include non-statutory deductions summary in the EmployeePayrollRun response.
  • Get Reportable Earnings and Benefits Summary: Select this checkbox to include reportable earnings and benefits summary in the EmployeePayrollRun response.
  • Get Allocated Earnings: Select this checkbox to include allocated earnings in the EmployeePayrollRun response.
  • Get Allocated Statutory Deductions: Select this checkbox to include allocated statutory deductions in the EmployeePayrollRun response.
  • Get Allocated Non-Statutory Deductions: Select this checkbox to include allocated non-statutory deductions in the EmployeePayrollRun response.
  • Get Allocated Reportable Earnings and Benefits: Select this checkbox to include allocated reportable earnings and benefits in the EmployeePayrollRun response.

Error Code

Description

Resolution
400

Invalid request

Verify the payload of the input request.

OpenAPI Specification

OpenAPI Specification is an API description format to define the structure and syntax of REST APIs. You can view the OpenAPI specification document in an API tool which defines the elements of an API. The OpenAPI operations can be used in Aquera's workflows and integrations using scripts.

Perform the following steps to view the OpenAPI specification in an API tool:

  1. Copy the OpenAPI URL from Copy this URL field on the connector configuration page of the Aquera Admin portal.
  2. Add /document at the end of the copied URL.
  3. Run the URL in an API tool, such as Postman using the GET method with the correct authentication credentials. The tool will display the OpenAPI specification document with all the supported OpenAPI operations. You can view the parameters and request body in the specification document.

You can also view the OpenAPI specification document by clicking the View OpenAPI Document button.

Performing OpenAPI Operation

Using an API Tool

You can perform OpenAPI operations using an OpenAPI tool as follows:

  1. Copy the OpenAPI URL from Copy this URL field on the connector configuration page. 
  2. Add /invoke at the end of the copied URL.
  3. Paste the URL in an API tool.
  4. Provide the required credentials for authentication.
  5. Pass the operation ID, parameters, and properties in the request body.
    Note: The parameters/properties vary depending on the type of operation.
  6. Run the URL using the POST method.

The following is the format to be passed in the request body:

{
"operationId": <<Operation ID>>,
"parameters": {<<parameter name>>: <<parameter value>>},
"requestBody": {<<property name>>: <<property value>>}
}

Using Workflows

You can perform OpenAPI operations using Workflows. Contact the Aquera support team for configuring Workflows.

Using Scripts

You can perform OpenAPI operations using Scripts. Refer to the Integration Using Scripts guide for details on how to set up an integration using scripts.

OpenAPI operations can be invoked using the following format:

<AppVariableName>.<OpenAPI_MethodName>({parameterName: parameterValue})

Here, <AppVariableName> is the name of the variable that you have configured in the script to refer to the provisioned connector, and the <OpenAPI_MethodName> is the name of the OpenAPI method to invoke the corresponding OpenAPI operation. 

Provide a comma-separated list of parameter names with parameter values enclosed in curly braces.

For further details on how to invoke OpenAPI methods using scripts, refer to the Working With Integration Scripts guide.
