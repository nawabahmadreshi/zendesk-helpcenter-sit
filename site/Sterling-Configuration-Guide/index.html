What is Orchestration

Orchestration is the automated configuration and coordination amongst applications. In a SCIM-to-SCIM Orchestration, the SCIM Gateway manages synchronous connections between the Source and the Target.

The SCIM Gateway translates identity management platform SCIM requests into REST-based API calls, SOAP/web services-based APIs calls, SQL commands, LDAP operations, robotic automation, SDK library calls, and queue messages across your entire IT infrastructure covering all the places where your users have accounts. The SCIM gateway creates, updates, deletes, or deactivates accounts from any application, database, directory, or device. Additionally, the gateway synchronizes passwords into your applications that use secure web authentication (SWA) or into your client's applications. 

Orchestration Expressions

NAME : Provide a Name to the Orchestration

ORCHESTRATION TYPE : Choose the Orchestration Type, for Example, SCIMtoSCIM, Scheduled

SOURCE APPLICATION : References the source entry object

TARGET APPLICATION : References the dest entry object

ENT : The expression used to determine which target record matches a given source record.

REPORT ONLY : When selected, a report is generated which shows what would be changed without actually making the changes.
UPDATE LIMIT : The maximum number of records to update in a single run.
ADMIN EMAIL LIST: Distribution list specifying administrative email recipients.
MATCH EXPRESSION : Expression used to determine which target record matches a given source record.
INCREMENTAL MODE : When selected, lastModified is maintained for the source application, and only source records newer than the timestamp will be processed. This operation should be considered when there the number of source records is very large.
IMPORT LIMIT: The maximum number of source records to import. This can be useful while determining/debugging mapping conditions.

SOURCE SCIM FILTER: Optional SCIM filter applied to source import.

CREATE CONDITION: Optional expression used to determine if a given user should be created.

UPDATE CONDITION: Optional expression used to determine if a given user should be updated.

MAP MANAGER: Select to map manager to target.

Orchestration Parameters

Source SCIM

This parameter to the orchestration defines the SOURCE system

This source must be configured as ADP in the applications list for your tenant.  

Target SCIM

This parameter to the orchestration defines the DESTINATION system

This destination must be configured as Active Directory in the applications list for your tenant.  

Report Only

This parameter when true directs the orchestration to run in read-only mode.

A report is generated that can be examined to ensure that the users will be created/updated/deleted in the Active Directory.

Mapper

This parameter is used to create a map of the user in Active Directory from expressions from ADP.  These expressions allow you to specify how each attribute will be populated.

The Mapper is Client Dependent and varies from Connector to Connector. 

Group Management

if the group expression is undefined then the group list will not be modified during the PUT or POST.  

If you would like to remove a user from all of their groups then the expression must evaluate to the empty list [].

The managed group list provides a list of groups that the Orchestration will manage.  

Only these groups will be modified by the orchestration.  

This allows the target to have groups that are managed in the target, for example, Active Directory security groups managed by Active Directory and groups managed by the orchestration.

The list of groups will be the displayName of each group.

The groupDisplays must be unique in the target directory.  

If the group name is redundant in the target then that group cannot be managed as the orchestration will not know which group of the two to add the user.

Email Templates

The email templates allow for the customization of email messages that are sent to the manager and/or administrator.  

These templates allow for substitution of attribute values of the corresponding user in the email message itself by utilizing {{attribute_name}}.

Group Rules

Users can define their own rules for the groups with regard to updating the data.

The Group Rules Expression can utilize any valid Javascript syntax for determining if a user is in a group.  

Schedule

Once the orchestration is completed, updates reflecting the changes made at the source in the target can be automated.

Function

UTILS.isSubordinate(User ID, Manager ID) - This function is used to determine if a user is within the reporting hierarchy of a manager.  

For example, if the user is within the organization in the role of a Developer, then this function can be used to determine the Role.  This function takes the user.id and the manager user.id and will look up the management hierarchy to see if the user's immediate manager is the manager.id, if it is it returns true.

If the user has a manager and it is not the one being searched for it will do the same operation on the user's manager.  This continues until the manager is found or it reaches a user that does not have a manager.  If the user does not have a manager, then it will return false.

Custom Scripts

The Custom Script Orchestration is a scheduled orchestration that can access any number of applications.  The script is based on javascript so all functions and language constructs are available in the script.

Each application has a name that can be used to access the actions on that application.  The actions available are as follows:

 

Actions

application_name.importAllUsers() -- This will return all of the Users in the application.  This will be an array of user objects.

application_name.importAllUsersFromGroup(string group | object group) -- This will return an array of all users that are assigned to this group in the application.

application_name.importUpdatedUsers(integer count, string units) -- This will return an array of all users that have been updated in the last count units.  The units can be year, month, day, hour, minute, second.

application_name.addUserToGroup(string group | object group, string id | string userName | object user) -- This action will add the specified user to the specified group.

application_name.removeUserFromGroup(string group | object group, string id | string userName | object user) -- This action will remove the specified user from the specified group.

application_name.findUser(string id | string userName | object user) -- This will return a user if it can be found in the application.  Or if no user is found then it will return undefined.

application_name.createUser(object user) -- this will create a new user in the application.  It will return the user object that was created including the id.

application_name.updateUser(string id, object user || object user) -- update an existing user by specifying and id and user object or just a user object with an id attribute.

application_name.deleteUser(string id | string userName | object user) -- delete a user in the application.

application_name.setUserPassword(string id | string userName | object user, string password) -- set the password of the user in the application.

application_name.activateUser(string id | string userName | object user) -- set the active = true to activate the user in the application.

application_name.deactivateUser(string id | string userName | object user) -- set the active = false to deactivate the user in the application.

Functions

sendEmail(string to, string subject, string body) -- will send an email to the specified to with a subject and body.

generatePassword( empty | int length) -- creates a random password of specified length.

findType(array object | string type) -- this will look through the array of objects that have an attribute named type and will return the first object that has that specified type.

log(string details) -- write log information in the resulting orchestration run.

SCIM Objects

User

The user object contains 3 sections core, enterprise, and extended.

The core attributes are accessed directly at the top level of the object as in user.id, user.userName, or user.name.givenName.

The enterprise object can be accessed by using the built in variable ENT (abbreviated for enterprise).  Examples of this are user[ENT].manager.value, user[ENT].organization, user[ENT].department.

The extended object can be accessed by using the built in variable IAN (abbreviated for integrated application network).  For example user[IAN].terminationDate, user[IAN].lastLoginDate.

HR Onboarding Bridge

The HR Onboarding Bridge Orchestration provides a one way synchronization from any Human Resources Management System to a directory or application.  The Orchestration can be scheduled to run on a frequency specified in the schedule section of the configuration.

 

Orchestration Parameters

source_scim

This parameter to the orchestration defines the SOURCE system that the bridge will read information from.  This source must be configured as an application in the applications list for your tenant.  The application must have it's credential configured so that the orchestration can communicate to the source HR system to retrieve all of the records to be synchronized to the destination.

target_scim

This parameter to the orchestration defines the DESTINATION system that the bridge will write users to.  This destination must be configured as an application in the applications list for your tenant.  The application must have it's credential configured so that the orchestration can communicate to the target directory or application to create/update/delete users.

report_only

This parameter when true directs the orchestration to run in read only mode to generate a report that can be examined to ensure that the users will be created/updated/deleted in the destination directory or application correctly.

map

This parameter is used to create a map of the user in the destination directory or application from expressions from the source system.  These expressions allow you to specify how each attribute will populated.

active : this attribute determines the state of the user in the destination application or directory.  The most basic form will be an exact copy active = SOURCE.active.  But you can support other use cases:

Employee Moving to Contractor:  When the title field in the Source HR is set to CONSULTANT then the user in the destitination will not be deactivated even after they are terminated.  active = SOURCE.title == 'CONSULTANT' || SOURCE.active will make this possible.

 

Group Management

if the group expression is undefined then the group list will not be modified during the PUT or POST.  If you would like to remove a user from all of their groups then the expression must evaluate to the empty list [].

The managed group list provides a list of groups that the Orchestration will manage.  Only these groups will be modified by the orchestration.  This allows the target to have groups that are managed in the target, for example AD security groups managed by AD and groups managed by the orchestration.

The list of groups will be the displayName of each group.

The groupDisplays must be unique in the target directory.  If the group name is redundant in the target then that group cannot be managed as the orchestration will not know which group of the two to add the user to.

 

Email Templates

The email templates allow for customization of the onboarding and off boarding email messages that are sent to the manager and/or administrator.  These templates allow for substitution of attribute values of the corresponding user in the email message itself by utilizing {{attribute_name}}.

For example to include an IAN attribute as part of the response you will need to include {{[IAN].Location}} for example.
