Active Directory / Dayforce Integration Guide

Overview

Aquera provides an innovative platform that allows your organization to scale complex data management by automating data synchronization from Dayforce employees to Active Directory users.

Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

In this integration, every employee newly created or modified in Dayforce is updated in the Active Directory according to the schedule.

This guide covers details on how to set up the Active Directory/Dayforce integration to synchronize employees data from Dayforce to the Active Directory users. 

Note:

This guide is applicable only for the latest version (1.2.9) of the Active Directory / Dayforce integration.

    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/

    • Active Directory Connector: Ensure you add and configure the Active Directory connector version 8 in the Aquera Admin portal. Refer to the Active Directory Version 8 Configuration Guide for details.

Integration Setup

Aquera's Active Directory / Dayforce integration automatically synchronizes employees data from Dayforce to Active Directory based on a configured schedule.

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel.
  3. Click Create Integration.
  4. Select Active Directory / Dayforce. The Create New Integration pop-up window is displayed.
  5. Enter a name and a description for the integration.
  6. Select Dayforce connector version 29 and Active Directory connector version 8 from the corresponding drop-down lists.
  7. Click Continue. The integration is created and the Configuration tab of the integration setup page is displayed. The configurations are categorized under various sections within the Configuration tab.
  8. Set the configurations as per your requirements. Refer to the Integration Configurations Reference Guide for details.
  9. Click Save Changes to save the Active Directory / Dayforce integration configurations.

User Naming Attributes

In Active Directory, the user naming attributes are User Principal Name (UPN), SAM Account Name (SAM), Common Name (CN), and Work Mail. These attributes are used to identify user objects in AD.

  • UPN: The User Principal Name (UPN) attribute is the user logon name (ULN) that is used to log on to a domain. The UPN must be unique in the forest. By convention, UPN should map to the user's email address. A UPN consists of a prefix (the user account name) and a suffix (a DNS domain name). The prefix is joined with the suffix using the @ symbol. The format of UPN is username@DomainName. For example, barbarajensen@aquera.com
     
  • SAM: The SAM Account Name (SAM) attribute is the logon name used to support clients and servers running earlier versions of Windows. This name must be less than 20 characters. The SAM must be unique within the domain. The format of the SAM is DomainName\username. For example, Aquera\barbarajensen
     
  • CN: The Common Name (CN) attribute is the name of an object in the directory. It is a key component in the Distinguished Name (DN), which uniquely identifies a user in the directory. The Common Name must be unique in the container or the organizational unit. For instance, you can specify the first name, followed by a space, and the last name as the common name of a user. For example, Barbara Jensen 
     
  • Mail: The Mail attribute is the user's email address, often set to match the UPN. The mail should be unique within the same domain. The mail attribute typically follows standard email formatting. For example, barbarajensen@sample.com

AD requires unique values for UPN/ULN, SAM, CN, and Mail attributes. Aquera's Active Directory / Dayforce integration enables you to generate unique names for these attributes while creating a user in Active Directory. The following are the available configurations:

  • Suffix: Configure the domain names for UPN/ULN and Work Mail. For example, aquera.com 

    • UPN/ULN suffix: Enter the domain name to be used as the suffix in the UPN/ULN.
      You can configure a static domain name (for example, aquera.com) as the UPN/ULN suffix or define conditions to assign a specific UPN suffix.
      To define conditions, click Conditional UPN/ULN Suffix and set conditional logic to assign domain names to users accordingly. 

      This is an AI Prompt field that uses Aquera Intelligence (AI Assistant) to generate domain names during integration runs. For details on how to set up an AI prompt, review and analyze the output generated by the AI Assistant, refer to the Using Aquera AI Assistant for Generating Unique Values in Integrations guide. 
       

    • Work Mail suffix: Enter the domain name to be used as the suffix for the work email address. If the work email suffix has to match the UPN suffix, select Work Mail Suffix same as UPN/ULN Suffix.
      To assign domain names based on some conditions, click Conditional Work Mail Suffix to define conditional logic and assign a specific work mail suffix. 


      This is an AI Prompt field that uses Aquera Intelligence (AI Assistant) to generate email domain names during integration runs. For details on how to set up an AI prompt, review and analyze the output generated by the AI Assistant, refer to the Using Aquera AI Assistant for Generating Unique Values in Integrations guide. 


      Note: Leave the Work Mail Suffix field blank if you want the integration to create new users without a work email address in Active Directory.

       

  • User Naming Configuration: Define the formats to be used for generating values for the UPN/ULN, SAM, CN, and Work Mail attributes for a newly created user in Active Directory.
    • Basic Mode: The Basic mode is enabled by default. In this mode, select a naming format from the ULN/ Mail/ SAM/ CN Format dropdown list to generate values for UPN, SAM, CN, and Work Mail when creating a new user in Active Directory. The selected format will be used to generate unique names for all four user naming attributes.
      In the basic mode, the attributes UPN/ULN, SAM, CN, and Work Mail are mutually dependent and are evaluated together. This means the integration will verify whether the values generated using the selected expression are available for all these attributes before assigning them to the new user. If any value is already in use for an attribute (that means, it is already assigned to a user in Active Directory), then the integration will try appending a number (starting from 1) to the attribute values and checking for the availability. This process continues for up to 20 attempts or until unique values are generated for all four attributes. Once unique values are found for those attributes, those values will be assigned to the new user.

      Note: The values of the expressions in the basic mode drop-down list are converted into normalized characters. This means that any characters with diacritics are substituted with the nearest equivalent non-diacritic format of those characters.
       

    • Advanced Mode: Select the Advanced User Naming Configuration checkbox to switch to the Advanced setup mode. In this mode, you can set different formats for each of the user naming attributes. 



      When you switch to the Advanced mode, you can select the user naming attributes that should be mutually dependent with the UPN/ULN. 

      The integration will check for a unique value in the first format for all selected attributes. If any value is unavailable (already assigned to users), it will proceed to the next format in sequence. The process continues until it finds available values for all attributes to assign to a user. Additionally,  Advanced mode allows you to configure separate naming formats for user creation and update operations.

      When you add a format, you need to select whether the format needs to be followed for user creations (On Create), user updates (On Update), or for both operations (On Create & Update). You can write JavaScript expressions for UPN, SAM, CN, and Work Mail attributes to generate unique names for AD users. To access Dayforce attributes, you must specify SOURCE.<Attribute Name> in the expressions. You can click Add Format to add multiple formats. You can include a maximum of 20 formats.

      The integration looks for unique values starting with format 1 for the selected attributes (mutually dependent attributes). If it does not find a unique value for at least one of the mutually dependent attributes, it will proceed to the next format in sequence. This process continues for all the formats you have added until it identifies unique values for all the selected attributes in the same format. If the integration cannot find unique values across the configured formats, the integration will throw a failure.

      The following is a scenario explaining how the unique names are generated:

      Consider that you have selected UPN, SAM, and Work Mail as mutually dependent. This means the values evaluated based on the first format or expression for UPN, SAM, and Mail must be available for creating or updating a user. If the values are already used for existing users, the integration checks for the next format or expression. This continues until it finds a format where the unique values are available for all these attributes.

      In this scenario, CN is an independent attribute. This means it is not dependent on other attributes (UPN, SAM, and Mail). Hence, the first available format for CN will be assigned to the user.

      Suppose you have given the following JavaScript expressions for UPN, SAM, Mail, and CN. When a new user with the name Barbara Jane Jensen is to be created in AD, the integration evaluates the JavaScript expressions to assign values for the UPN, SAM, Mail, and CN attributes of the user.

      UPN/ULN:

      Format 1: SOURCE.FirstName + "." + SOURCE.LastName
      Format 2: SOURCE.FirstName + "." + SOURCE.LastName + 1
      Format 3: SOURCE.FirstName + "." + SOURCE.LastName + 2

      SAM:
      Format 1: SOURCE.FirstName + SOURCE.MiddleName + "." + SOURCE.LastName
      Format 2: SOURCE.FirstName + SOURCE.MiddleName + "." + SOURCE.LastName + 1
      Format 3: SOURCE.FirstName + SOURCE.MiddleName + "." + SOURCE.LastName + 2

      CN:
      Format 1: SOURCE.FirstName + " " + SOURCE.LastName
      Format 2: SOURCE.FirstName + " " + SOURCE.LastName + 1
      Format 3: SOURCE.FirstName + " " + SOURCE.LastName + 2

      Work Mail:
      Format 1: SOURCE.FirstName + "." + SOURCE.LastName
      Format 2: SOURCE.FirstName + "." + SOURCE.LastName + 1
      Format 3: SOURCE.FirstName + "." + SOURCE.LastName + 2
       

      The following are the values generated from the defined expressions for the new user, Barbara Jane Jensen:

      UPN/ULN:
      Format 1: Barbara.Jensen
      Format 2: Barbara.Jensen1
      Format 3: Barbara.Jensen2

       

      SAM:
      Format 1: BarbaraJane.Jensen
      Format 2: BarbaraJane.Jensen1
      Format 3: BarbaraJane.Jensen2

      CN:

      Format 1: Barbara Jensen
      Format 2: Barbara Jensen1
      Format 3: Barbara Jensen2

      WorkMail:

      Format 1: Barbara.Jensen
      Format 2: Barbara.Jensen1
      Format 3: Barbara.Jensen2


      The integration first verifies the attributes that are selected as mutually dependent. It evaluates each configured format in sequence, starting from format 1 and continuing through subsequent formats up to the last defined format (maximum of 20 formats) till it finds unique values for all the attributes. If a format configured for a mutually dependent attribute evaluates to a value that is already in use for another user, the integration will move to the next format to find unique values for all the mutually dependent attributes.

      Case 1: For format 1, if the UPN, SAM, and Work Mail values are not assigned to any user, then this format will be used for the new user. In this case, the UPN value of Barbara.Jensen, the SAM value of BarbaraJane.Jensen, and the Work Mail value of Barbara.Jensen will be assigned to the new user Barbara Jane Jensen.

      Case 2: For format 1, if the UPN and Work Mail values are already assigned to an existing user, the integration checks for the next format (which is format 2) for all the 3 mutually dependent attributes, even if one of those attribute values is available for format 1. This means UPN, Work Mail, and SAM values for a defined format must be available to be assigned for a new user creation. For format 2, if the UPN, Work Mail, and SAM values are not already assigned to any user, then this format will be used for the new user creation. In this case, the UPN value of Barbara.Jensen1, the SAM value of BarbaraJane.Jensen1, and the Work Mail value of Barbara.Jensen1 will be assigned to the new user Barbara Jane Jensen.

      The integration then evaluates the formats for the CN. Since CN is an independent attribute, the first format that generates a unique CN value for the user will be considered. Suppose the CN values of format 1 and format 2 are already used for existing users, and format 3 is not used by any user, then the integration assigns the format 3 value, which is Barbara Jensen2, for the CN of the new user Barbara Jane Jensen.

 

OU Assignment

In Active Directory (AD), an Organizational Unit (OU) is a container object that organizes and manages resources such as users, groups, computers, and other OUs within a domain. Each OU has a distinguished name that specifies its location in the Active Directory hierarchy. An organization can create OUs based on departments, geographic locations, or roles. It is highly recommended to place user objects in OUs to help with better management, policy application, and administrative delegation.

Aquera's Active Directory / Dayforce integration allows you to define rules for assigning users to Active Directory OUs. The following are the available configurations:

  • Show OUs having users only: Select this checkbox to list only the OUs containing users, in the OU drop-down lists. OUs that do not have any users are not listed in the OU fields in this tab.
  • Default Assignment OU Condition: From this drop-down list, select whether only new users (Create option) or both new and rehired users (Create & Reactivate option) must be considered for the assignment to the default OU. 
  • Default Assignment OU: Select the default OU where new and rehired users will be placed if no conditional OU assignment rules exist or the user does not meet the conditional OU assignment rules configured. 
    • New users and rehires will be moved to the default OU if they do not meet any conditional OU assignments or no conditional OU assignments are defined.
    • Updated users or deactivated users will never be moved to the default OU.
    • If new or rehired users satisfy any conditional OU assignments, they will be added to the corresponding OUs.
  • Default Deactivated OU: Switch the toggle ON to select the default OU where deactivated users will be placed if no conditional OU assignments exist or the deactivated user does not meet the conditional OU assignment rules.
    • Updated users will never be moved to the default deactivated OU.
    • If an active user is deactivated and the user satisfies any conditional OU assignments, they will be added to the corresponding OU; they will not be added to the default deactivated OU.
    • If any update is made to a deactivated user and the user is already part of an OU, they will be retained in the same OU.
  • Conditional OU Assignment: Switch the toggle ON to add OU assignment rules. Refer to the following section for setting up conditional OU assignment rules.

Configure OU Assignment Rules

OU assignment rules determine the set of users that must be placed in the OUs in Active Directory. 

  1. Click the Assign On drop-down list and select one of the following options:
    • Create: To assign a user to an OU during the user creation operation. The OU and the DN attributes of the AD user will be updated during the user creation operation.
    • Update: To assign a user to an OU during the user update operation. The OU and the DN attributes of the AD user will be updated during the user update operation.
    • Create & Update: To assign a user to an OU during the user creation and update operations. The OU and the DN attributes of the AD user will be updated during the user creation and update operations.
  2. Define the conditions for assignment to OUs using one of the following ways:
    • Using Condition Builder
    • Using Expressions
  3. If required, click Add Row to add another condition within the same rule. When multiple rows are added within a rule, an OR condition is applied to the rows. That means, for each user, the integration evaluates the conditions in each row sequentially. If the conditions defined in a row are satisfied for a user, the user is assigned to that OU and the remaining rows in the rule are not evaluated for that user.
  4. If required, click the + icon at the top-right corner of the rule setting section to add another rule with a set of rows/conditions for OU assignments. When multiple rules are set, the integration evaluates every rule for each user.

    Note: You can add a maximum of 10 rules. 

Using Condition Builder

The Condition Builder is the default view for configuring rules. A blank row is displayed by default for configuring the first set of conditions.

  1. Click on Choose Attribute to select a Dayforce user attribute that you want to use in the condition.


     
  2. After selecting the attribute, specify the value(s) of the attribute as comma-separated values. Any Dayforce user that matches the specified attribute values (OR operation) will be assigned to the OU when the integration runs.
  3. If required, add conditions with more attributes and values by clicking the + icon. When multiple attributes are added within the same row, those attribute conditions are evaluated using an AND operation. Only users who match all the specified conditions will be assigned to the OU.
  4. In the Assign To field, select the organizational unit where users will be placed. To ensure all the required OUs are listed in this OU list, ensure you have configured all the base DNs in the Active Directory connector's tenant attribute, User Base Distinguished Names.

Using Expressions

Follow this procedure to set the rules using JavaScript expressions. A blank row is displayed by default for configuring the first set of conditions.

  1. Click Switch to Expression. The following options are displayed:
    • For Conditions Only: Select this option to use JavaScript expressions for setting the conditions. You must select the corresponding OU from the Assign To field.
    • For Conditions and Assignments: Select this option to use JavaScript expressions for setting the conditions and the OU assignments.

You can expand the Custom Expression editor area by clicking the icon and then write the Javascript expressions.

Note: Click Switch to Condition Builder if you want to switch out of the Expression mode.

Refresh OU List

When the configuration of the User Base Distinguished Names is updated in the Active Directory connector or when the OU names are updated in the Active Directory application, it is required to refresh the OU list displayed on this page. Click Refresh OUs to retrieve the most recent OU list.

In Active Directory, each object has a Distinguished Name (DN), which serves as a unique identifier within the directory. A DN describes the fully qualified path to an entry within the Active Directory hierarchy. A DN is made up of attribute=value pairs, separated by commas. Example of a user DN:
CN=Barbara Jensen,OU=Sales,DC=example,DC=com

A user DN is formed from the combination of the CN and the OU assigned to the user. For example, if the CN for the user is Barbara Jensen3 and the OU where the user will be placed is OU=Admins,DC=example,DC=com, then the DN for the user will be:
CN=Barbara Jensen3,OU=Admins,DC=example,DC=com

Group Management

The Group Management tab provides options to configure how group memberships must be handled through this integration. You can determine the group assignment, retention, and removal of users based on the update or deactivation of users in Dayforce.

Note: You can also set the manage group memberships via Attribute Map (under Mapping and Assignments) where you can write JavaScript expressions in the mapping, which will return an array of groups.

Select Enable Group Management to set the options for managing user groups. If this checkbox is not selected, the integration does not manage group assignments to users in the Active Directory.

When you select Enable Group Management, the Group Rules section will be displayed on this tab. Refer to the following section for details.

Set Up Group Rules

You can set rules for assigning users to the Active Directory groups through either the Group Rules section of the Group Management tab or the Attribute Map section of the Mapping and Assignments tab. 
Note: When you use the Attribute Map functionality to set the group rules, you have to write JavaScript conditional expressions to assign the groups. 

In the Group Rules section, you can create conditions that control how users are added to groups. Each rule defines the conditions for assigning users to a set of groups.

Perform the following procedure to create group assignment rules:

  1. Click Add New Rule to create a new group assignment rule. The following fields are displayed.
    • Choose Attribute: Click on this label to view the list of Dayforce employee attributes.
      • Search for the attribute using which you want to set a condition. 
      • Select the required attribute from the list and configure the attribute value to form the rule condition. Only those users who meet this condition will satisfy this group rule. If multiple values are provided, an OR condition is applied among those values. For example, in the following sample condition, a user's Pay Policy XRefCode must be either USA or India for them to satisfy this group assignment rule.
    • Assign To: Select the group(s) to which the users who satisfy the specified condition must be assigned. The list shows all the groups from the target AD application.
  2. Click Add Row to add another condition and group assignment for the same Dayforce attribute. 

    Note 1: You can add any number of rows within a group rule.

    Note 2: You can delete a row by using the delete icon at the right end of the row.
  3. [Optional] If you want to write a JavaScript expression to set the condition, click Switch to Expression.
    Note: When you switch to the Expression mode or to the Basic Condition mode, all the configured conditions of the rule will be cleared.
    • Click Continue to configure the expression.
    • Expressions can use all the employee and user attributes in the source and destination applications. The configured expressions must always be evaluated to true or false.
      SOURCE must be used in an expression to represent an attribute in the source application and DEST to represent an attribute in the destination application. For example, SOURCE.EmployeeNumber, SOURCE.HomeOrganization.ShortName, DEST.EmployeeNumber, and so on.  
  4. Click Add New Rule to add another group assignment rule, as explained above.
    Note: You can add any number of group rules; however, you can see only 5 rules on a single page. If you add more than 5 rules, navigation options will be available at the bottom of the page to navigate to other pages for configuring or viewing the remaining rules.

Set Up Group Retention Policy

The Group Retention section allows you to define how a user's group memberships must be handled when the Dayforce user is updated or deactivated.

On Deactivate

This drop-down list is used to specify what should happen to a user's group memberships when the Dayforce user is deactivated. It includes the following options:

  • Remove from all assigned groups: All the group memberships currently assigned to the user will be removed. However, if the deactivated user satisfies some of the group assignment rules and they are not already part of those groups, they will be added to those groups; if they are already part of those groups, they will be retained in those groups.
  • Retain in all assigned groups: All the group memberships currently assigned to the user will be retained, even if the user does not satisfy the rules configured for the group assignments in the Group Rules section.
  • Retain in selected groups: This option has to be used when you want to retain the deactivated users only in specific groups. When you choose this option, the following additional fields are displayed:
    • Groups: In this field, you need to select the specific groups in which a user's membership has to be retained. For example, if a user is part of G1, G2, and G3 groups and if you have specified G3 and G4 in this field, the user's membership in G3 will be retained. Deactivated users will be removed from all other groups.
    • Condition: This is an optional field to define a filter condition to apply the retention rule only to a specific set of users from the deactivated list of users. The group membership will be retained only for the users matching the filter condition. For the remaining set of deactivated users that do not match the filter condition, no action will be taken on their group memberships. Here is a sample filter condition:
      '(SOURCE.WorkAssigmentsPositionDepartmentXRefCode).includes("Admin") && (SOURCE.BusinessTitle).includes("Manager") && (SOURCE.BusinessTitle).includes("HR")'

On Update

This drop-down list is used to specify what should happen to a user's group memberships when the user is updated. It includes the following options:

  • Remove from all assigned groups: All the group memberships currently assigned to the user will be removed. However, if the user satisfies the conditions defined in the Group Rules section, they may be added to new groups based on the configured group rules.
    Note: If the Update Deactivated Users configuration is enabled, a deleted or deactivated user will be added to a new group based on the group rules. However, they will not be added back to an existing group from which they have been removed during this integration run.
  • Retain in all assigned groups: All the group memberships currently assigned to the user will be retained. Even if the user does not satisfy a rule configured for the group assignments in the Group Rules section, they will be retained in those groups.
  • Retain in selected groups: This option has to be used when you want to retain the updated users only in specific groups. When you choose this option, the following additional fields are displayed:
    • Groups: In this field, you need to select the specific groups in which a user's membership has to be retained. For example, if a user is part of G1, G2, and G3 groups and if you have specified G3 and G4 in this field, the user's membership in G3 will be retained on user update.
    • Condition: This is an optional field to define a filter condition to apply the retention rule only to a specific set of user records that were updated. The group membership will be retained only for the users matching the filter condition. For the remaining set of updated users that do not match the filter condition, no action will be taken on their group memberships. Here is a sample filter condition:
      '(SOURCE.WorkAssigmentsPositionDepartmentXRefCode).includes("Admin") && (SOURCE.BusinessTitle).includes("Manager") && (SOURCE.BusinessTitle).includes("HR")'

Limitations

  • You can save the integration even without mandatory fields configured, but the integration runs will fail.
  • It is required to download a report of all employees in the Dayforce connector (through the Data tab) to view all the Dayforce attributes in this integration.
  • The Incremental Tuning configuration is designed to function only when the option match expression only is selected, as this integration does not support incremental tuning with userName.
  • Enabling the Default Deactivated OU toggle without selecting an OU from the dropdown will cause errors during the integration run.
  • Enabling the Conditional OU Assignment toggle without adding a rule (condition or expression) will cause errors during the integration run.
  • Debug logging for a full run with a large number of source records can exceed back-end limits, resulting in reports that are too big to save. This will be fixed in a future release.
