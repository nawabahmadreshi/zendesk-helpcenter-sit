User Integration Setup Guide - UKG Pro to Active Directory

← Back to index

Overview

This guide covers the end-to-end tasks involved in setting up integration to sync users in UKG Pro with Active Directory.

The following tasks are involved in setting up the integration from UKG Pro to Active Directory:

  • Initial Preparation
  • Configuring Applications in Aquera Admin
  • Integration Setup
  • Running Integration

Initial Preparation

Aquera Admin Console

  1. Your IT admin must register as an Aquera Admin.
  2. Once registered successfully, Aquera creates a tenant for the IT Admin and provides access to it.

Active Directory

Determine the architecture to be followed for setting up communication between Aquera and the Windows Active Directory Domain Controller. Refer to this Aquera guide for detailed information on the different architectural approaches.

Based on the selected approach, the following procedures may have to be performed:

  • A Windows Server must be ready to deploy Aquera Agent onto the server where Domain Controller is present. For instructions on installing Aquera Agent, refer to the Installation and Configuration Guide.
  • A firewall must be set up to allow communication with Aquera.
  • The static IP address of Aquera needs to be whitelisted. Contact the Aquera Support team.

To allow Aquera to make updates to Active Directory (for creating, updating, and deleting users), LDAPS communication to Active Directory needs to be set up, which requires a TLS connection. Active Directory Certificate Services must be enabled for this. To install Active Directory Certificate Services, see docs.microsoft.com.

Configuring Applications in Aquera Admin

The following applications must be added and configured in the Aquera Admin console:

  • UKG Pro. Refer to the UKG Pro Configuration Guide for details.
  • Active Directory. Refer to the Active Directory Configuration Guide for details.

Integration Setup

This section details how the integration must be configured to allow data push from UKG Pro to Active Directory. 

It also describes the various analyses to be performed and the important points to be considered before setting up the integration.

Considerations

  • The naming convention for the User Principal Name (UPN) in the Active Directory
  • The naming convention for the SAM Account Name in the Active Directory
  • Fields currently populated in the Active Directory
  • The criteria for matching a user record from Oracle HCM Cloud with that in the Active Directory.

General Analysis Steps

Before setting up the integration, the following analyses need to be done:

  • Fill out the attached mapping template for mapping the UKG Pro user attributes to Active Directory user attributes.

  • Expression language can be used in the mapping to create UPNs and SAM Account Names without duplicates or errors. In the expressions, you can use inbuilt methods such as toNormalizedDiacritics() and toSAMAccountName() for this purpose. You can also use the keywords such as %ATTEMPT, %USEQ, SEQUENCENUMBER to avoid duplicates. 

    For details, refer to the Integrations Expressions Guide.

  • Define the logical OU mapping rules for user containers.

For detailed analyses of the UKG Pro users and the users/groups in Active Directory using scripts, refer to the Appendix.

You can also analyze the user's data using the Reports option in the Data tab of the Applications page in the Aquera Admin console. Refer to the User Reports Guide for details.

Building Integration

This document describes how to set up the integration using the User Integration method.

  • User Integration - In this integration, set UKG Pro as the Source Application and Active Directory as the Target Application. Refer to the User Integration Setup Guide for details on the configurations.

You can set the following configurations in the User Integration:

  • Set the Admin Email Options to receive email notifications for success/failures during the integration run.
  • Set the email list of the IT admin or manager to receive user creation notification emails and user deactivation notification emails.
  • Use tables for creating lookup tables based on location and department values from the source.
  • Enable Group management if Aquera has to manage the groups and group assignment rules based on your group analysis. Set the group names in the restriction list for scoping of the groups which would be managed by Aquera.

Refer to the User Integration Setup Guide for details on User Integration configurations.

Integration Scenarios

This section details the integration scenarios.

Creating a user of type NORMAL_ACCOUNT in the Active Directory

To create a user of type NORMAL_ACCOUNT in Active Directory, set the attributes mapping as follows:

Enter "512" as the source attribute expression and map it to the [IAN].userAccountControl attribute of AD. Set the mapping criterion as Map on Create.

Note: 512 is the decimal value corresponding to the NORMAL_ACCOUNT property flag in Active Directory. For all property flags and the corresponding decimal values, refer to the Microsoft documentation.

Escaping Special Characters in OUs and DNs for Active Directory

To escape any special characters in OUs and DNs in the expressions, use a double-backslash before the special character. For example, if the OU name is #Terminated, mention it in the expression as \\#Terminated. 

DEST[IAN].distinguishedName.split(",")[0] + ",OU=\\#Terminated, DC=local"

Attribute Mapping for mandatory password change at first sign-in

To make it mandatory for users to change their password when signing in for the first time, map the Active Directory attribute pwdLastSet to "0" with the mapping option of Map on create only.


 

If this mapping is not done and users sign into their account for the first time, it will not prompt them to change their password.

Requirements to set user password

Active Directory requires a secure LDAPS (LDAP over SSL) connection to set or change user passwords. This is an Active Directory’s security requirement. Random generation of passwords when creating new users does not work without the use of LDAPS. Also, setting the password through attribute mapping requires a secure LDAPS connection.

Note: If the connection is not secured (non-SSL), Active Directory does not allow you to update the user password. 

Active Directory imposes strength requirements on the password and the requirements vary from one instance to another.

Running Integration

  • Enter the matching expression to match the source records with the destination records.
  • Run the integration in Report Only mode and determine if the correct data is updated in the Active Directory. If the data is not correctly updated, change the matching expression to correctly match users and/or update the data, for example, employeeNumber, in the Active Directory.
  • Increment the Update Limit (maximum number of records to be updated in a single run) for each run in Report Only mode to ensure that the records are updated correctly when the records count is more.
  • Based on your analysis and considerations,
    • Translate the logical maps to the Aquera attribute maps for the fields
    • Translate the logical OU/container locations into tables and expressions
    • Translate the logical group assignment rules into Aquera rules
  • Modify the attributes mapping and repeat the process until the data is correctly updated.
  • Monitor the integration logs thoroughly to ensure that no errors are encountered. Modify the integration configurations to fix any errors.
  • Once all users are synced, user creations and updates are based on the incremental changes in Oracle HCM Cloud.

Appendix

This section provides the scripts that may be used by the Customer Support team to analyze the existing users in UKG Pro and Active Directory before setting up the user integration. It also provides the script to analyze the existing groups of users in Active Directory.

This document explains how analyses can be done using a script integration. In this integration, add UKG Pro and Active Directory applications and set the corresponding variable names. In the script, the variable names must be used to refer to the applications.
For details on configuring the script integration, refer to the Integration Setup using Scripts Guide.

A few sample scripts are provided here for your reference. The scripts may vary based on your use case  and the attributes to be retrieved for analysis.

Analyzing the Users in UKG Pro

Use the below sample script in a script integration for generating a report from UKG Pro for analysis of user status, department, and usertype, and so on. 


let ukg_users = ukg_user.getUsers();
 
      let i = 0;
      log.info('UKG ID,UKG STATUS,UKG EMPNO,UKG GIVENNAME,UKG FAMILYNAME,UKG WORK EMAIL,UKG NICKNAME,UKG USERNAME,UKG MANAGER');
      for (i = 0; i < ukg_users.length; i++)   {
      let ukg_id = ukg_users[i].id;
      let ukg_status = ukg_users[i][IAN].status;
      let ukg_emp = ukg_users[i][ENT].employeeNumber;
      let ukg_given = ukg_users[i].name.givenName;
      let ukg_family = ukg_users[i].name.familyName;
          let ukg_work = ukg_users[i].emails && ukg_users[i].emails.find(x => x.type == 'work') && ukg_users[i].emails.find(x => x.type == 'work').value || '';
      let ukg_username = ukg_users[i].userName;
      let ukg_manager = ukg_users[i].manager;
      log.info('"' 
      + ukg_id + '","' 
      + ukg_status + '","'
       + ukg_emp + '","'
       + ukg_given + '","' 
       + ukg_family + '","'
       + ukg_work + '","'
      + ukg_username + '","'
       + ukg_manager + '"');
      }

Analyzing the Users in Active Directory

Use the below sample script below in a script integration for generating a report from Active Directory for analysis of user domain, status, and so on.

log.info('AD ID,AD ACTIVE,AD EMPNO,AD GIVENNAME,AD FAMILYNAME,AD USERNAME,AD SAMACCOUNT,AD DN');
let users = AD.getUsers();
for (let user of users) {
log.info('"'
+ user.id + '","'
+ user.active + '","'
+ (user[ENT] && user[ENT].employeeNumber || '') + '","'
+ user.name.givenName + '","'
+ user.name.familyName + '","'
+ user.userName + '","'
+ user[IAN].sAMAccountName + '","'
+ user[IAN].distinguishedName.split(',').slice(1).join(',') + '"');
}
