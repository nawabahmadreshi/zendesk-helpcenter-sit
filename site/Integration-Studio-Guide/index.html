Overview

You can build an integration template using the Integration Studio. In an integration template, you can add various parameters to allow end users to configure their integration preferences. As an integration developer, you need to write JavaScript to perform tasks on the data across applications. Depending on the resources supported by the application, the type of data that the integration handles can vary - it can be users, groups, tickets, and so on. You can perform various tasks on the data, including reading, creating, updating, deleting, and performing general tasks such as sending emails and creating service requests. You can set up the integration to run either on a specific schedule or based on the data events in an application. When the integration runs, the data will be synchronized across various applications and systems.

This user guide explains how you can build an integration using Integration Studio in the Aquera Admin portal.

The script supports SCIM functions and open API methods supported by an application, utility functions, and log functions. It is possible to access the schema attributes of an application in the script. If tables are configured in the integration, you can access tables in the script. For further details on the functions supported in the script and how to work with the scripts, refer to the Working With Integration Scripts guide.

Prerequisites

Configure the required connectors correctly in the Aquera Admin portal before creating an integration. Refer to the corresponding configuration guides for details on configuring the required connectors. 

Build an Integration Template

Perform the following steps to build an integration template:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel, which displays the list of integrations created by the user. 
  3. Click Create Integration.
  4. Select Integration.
  5. Enter a name and a description for the integration.
  6. Click Continue. A new page is displayed to configure the integration.
  7. Click Create to start building the integration template.

Integration Panels

An integration page displays the following panels:

  • Configuration
  • Script
  • Flow Chart
  • Documentation
  • Assistant
  • Summary

To view a panel, click the icon at the top right portion of the integration page and select the panel that you would like to view on the integration page.

You can rearrange the panels by dragging and dropping them on the integration page. Click the icon next to the name of a panel to drag and drop the panel into the desired position on the integration page.

Each panel includes an Expand icon and a Close icon at its top right corner.

 

  • When you click the Expand icon on a panel, the panel expands and hides all the other panels on the page. In the Expanded mode, you can click the Collapse icon to return to the multi-panel view.
  • When you click the Close icon, the panel will be closed. However, you cannot close all the panels; at least one panel will always remain open.

Note: You can view the Script, Flow Chart, Documentation, Assistant, and Summary panels on the integration page only if you are an Aquera employee and if those features are enabled in your tenant. You can switch to the customer view by clicking the Hide Code option from the ellipsis icon at the top-right corner of the page. To switch back to the Aquera employee view (with the script panel), click Show Code.

Unsaved Changes Confirmation

When users make changes on the integration page and attempt to leave without saving, an Unsaved Changes confirmation dialog is displayed. This helps prevent data loss.

The dialog appears when users try to navigate away from the integration after making unsaved changes, such as updating the script, changing configuration fields, adding or editing parameters, modifying groups or subgroups, updating triggers, changing rule builder settings, or modifying attribute mappings.

The dialog provides the following options:

  • Discard Changes: Leaves the integration page without saving the changes. Any unsaved updates are lost.

  • Save Changes: Saves the pending changes before leaving the page.

  • Close: Closes the dialog and keeps the user on the current integration page.

For example, if a user updates the integration script and attempts to navigate away without saving by using the browser back button, a swipe-back gesture, or the UI back button, the Unsaved Changes dialog is displayed. The user can then choose to save the changes, discard them, or remain on the page.

Configuration Panel

The Configuration panel allows you to add the necessary applications, add configuration parameters, create attribute maps between different applications, set up a trigger, and add tables to be used in the script.

Applications

The Applications section allows you to add applications to perform the operations required for the integration. You can add multiple applications as needed. Ensure to add at least one valid application for successful integration. 

Notes: 

  • If no application is configured in the integration, the Applications sub-group will not be displayed in the configuration panel for end users.
  •  You can change the default name, Applications, to another name, if required, by navigating to the Sub Group section in the Configuration panel.

Add Application

Perform the following steps to add an application:

  1. Click the + icon next to the Applications to display the Add Application pop-up window.
  2. Select the required connector instance from the Application drop-down.
  3. Enter a unique variable name for the connector in the Variable field. This name will be used to refer to the connector in the script.
    Note: The variable name must be unique within the script configured for a specific integration. It can contain letters and numbers but must start with a letter, cannot be a JavaScript keyword, and should not include spaces or special characters.
  4. Enter a display name for the connector in the Display Name field.
  5. Select whether you want to use the SCIM methods, Open API methods, or both in this integration from the Protocol field, based on your requirements. Based on the selection, the methods are available for use in the integration script.
    Note: The Protocol field is displayed only if the selected application connector supports both SCIM and Open API protocols.
  6. Click the Add button to add the connector to be used in the integration.

Refresh Application

After adding applications to the integration, if any changes have been made to the application schema, you must use the Refresh icon adjacent to the Applications section to retrieve the most recent schema for all applications.

Additionally, you have the option to click the ellipsis icon next to each application and select the refresh icon to retrieve the latest schema for individual applications.

Parameters

The Parameters section lets you define settings for the integration by adding configurable fields. These fields store values as variables, which you can then use in your integration script to customize how the integration behaves.

It also provides an option to organize these parameters as tabs (groups), subgroups within tabs, or sections. You can even create parameters without any grouping.

Note 1: For end users, the Parameters subgroup will be displayed under the Configuration tab only if at least one parameter is added in this subgroup. Else, the Parameters subgroup will not be displayed.

Note 2: You can change the default name, Parameters, to another name, if required, by navigating to the Sub Group section in the Configuration panel.

This section explains how to add a configuration parameter to an integration. If you want to add the new parameter to a specific tab (group), to a subgroup within a tab, or to a section, you may create those groups, subgroups, or sections first and then proceed with adding a parameter.

  1. Click the + icon in the Parameters section. The Add Parameter pop-up window is displayed.

      • Multiple Text Inputs: This option allows you to create a field where you can enter values as chips (by typing each value and pressing Enter). If required, you can set default values using the Value field.

Enable or Disable Rules

End users can enable or disable rules at both the rule tab level and the individual row level within the Rule Builder.

  • Rule tab level: End users can use the Disable Rules or Enable Rules option from the rule tab menu to disable or enable all rows in that rule tab. When the rule tab is disabled, all rows in that tab are displayed with a light grey background to indicate that the rules are inactive. The rows remain editable even when they are disabled.

  • Row level: End users can use the Disable Rule or Enable Rule option from a row’s action menu to disable or enable an individual row. When a row is disabled, it is displayed with a light grey background to indicate that the rule is inactive. Other rows in the same rule tab are not affected.

Note: 

  • Disabled rules are not evaluated when the integration runs. If a rule tab is disabled, all disabled rows in that tab are skipped during execution. If an individual row is disabled, only that row is skipped, while the remaining enabled rows continue to be evaluated.

  • If an individual row is disabled and the entire rule is then disabled, the row-level disabled state is retained. When the rule is re-enabled, it becomes active, but the previously disabled row remains disabled until it is enabled separately.

  • If an entire rule tab is disabled and the end user enables a specific row, the rule tab-level disable is removed. The selected row becomes active, while the remaining rows remain individually disabled. This allows the end user to activate only the required row while preserving the disabled state of the other rows.

Script or JSON Mapping on Refresh

When an Attribute Map is refreshed based on selected parameters, the mappings can be generated or updated using Script or JSON. This allows the mapper to update source-to-target mappings dynamically based on the selected parameter values.

The Script or JSON output can define the following:

  • The source attribute or source expression.
  • The target attribute.
  • The mapping option that controls when the target attribute should be updated.

Use the mapWhen property in the Script or JSON output to define the mapping option for each target attribute.

The following are the supported mapWhen values:

"mapWhen": "create"
"mapWhen": "update"
"mapWhen": "both"
"mapWhen": "none"

These values work as follows:

  • create: Updates the target attribute only during create operations.
  • update: Updates the target attribute only during update operations.
  • both: Updates the target attribute during both create and update operations.
  • none: Does not update the target attribute.

For example, if a mapping contains "mapWhen": "update", the target attribute is updated only when the integration performs an update operation.

Import Integration Access Control

Integration import, Re-upload JSON, and core integration options are controlled using tenant-level feature flags. These controls ensure that new customer tenants create integrations from built-in templates by default.
The following feature flags are available:
  • enableImportIntegration: Controls access to Import Integration and Re-upload JSON.
  • enableCoreIntegrations: Controls access to Core Integration and User Integration.
For new customer tenants, both feature flags are disabled by default.
  • When enableImportIntegration is disabled, Import Integration is hidden, and Re-upload JSON is not displayed in the integration action menu.
  • When enableCoreIntegrations is disabled, Core Integration and User Integration options are hidden.
  • Enable these feature flags only when the tenant requires import access, custom scripting, or core integration access.
Note: Existing tenants continue to have the same access as before. To retain existing behavior, the required feature flags must be added to the existing tenant records.
