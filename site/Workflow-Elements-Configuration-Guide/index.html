Overview

To set up a sequence of tasks in a Workflow, you need to add elements and configure the elements in the Workflow. An element is an individual task in a Workflow. To add an element to the Workflow, drag the required item from the Components section and drop the item on the + icon of the Workflow. When you click an element in a Workflow, the element configuration panel opens where you can configure the element.

This guide details how to configure elements in Workflows. For further details on Workflows, refer to the Workflow User Guide. 

Operators

Operators control the flow of tasks in the Workflow. This section explains how to configure Operators.

If-else

You can use If-else to perform a set of tasks when the condition is true and perform a different set of tasks when the condition is false. The Workflow evaluates the condition to either True or False and executes the tasks in the corresponding True or False flow.

Input

You can provide a condition using one of the following ways:

  • Configure the Attribute, Operator, and Value fields.

    Value

  • Click the Switch to Expression icon and then click the expand icon to write a condition using JavaScript expressions. 

Cases

You can use Cases to perform a different set of tasks based on different values of an attribute or expression evaluation. The Workflow evaluates the attribute or expression, compares it with each case value, and if there is a match, executes the tasks of the corresponding case flow.

Persistent Data

You can use Persistent Data to store data that can be used across multiple Workflows in the same tenant. Persistent Data once created can be updated any number of times by any Workflow within a tenant. 

Persistent Data will be retained even after a Workflow execution is completed and it can be used or modified in successive executions of the same Workflow and other Workflow in the same tenant.

The following are sample use cases to explain the use of Persistent Data:

  • Assume you have configured multiple Workflows and you want these Workflows to run in such a way that only the incremental data of users after the previous execution date/time of the Workflow need to be processed in another Workflow.
    In this scenario, you can create Persistent Data to store the date and time of execution of that Workflow. The same Persistent Data can be used in a different Workflow to use the run date/time of the other Workflow, for obtaining the incremental data. The execution time in the Persistent Data keeps changing for every run of these individual Workflows and that value will be retained for use and modification by successive Workflows.

  • Assume you have configured multiple Workflows performing specific tasks and want to generate a report containing the results of the tasks that are processed.
    In this scenario, when each task is completed, you can add the result of each task processing in Persistent Data. Finally, when all the Workflow executions are completed, you can read the Persistent Data to generate a report.

Field Description

Wait for a Response*

When you set this option to True, a synchronous calling occurs, pausing the current Workflow and triggering the selected Workflow. The current Workflow waits for a response from the selected Workflow and once the response is received, it resumes the current Workflow.

Output

The output displays a system-defined variable name that stores the array of the imported data records. You can also modify the name of the output variable and confirm the change.

By default, the variable name is created in the format, <operationName>_<sequence_number>. The sequence number is incremented for the same operation for each application added to the Workflow. For example, if the operation is importAllUsers, the first output variable name will be importAllUsers, the next incremental output variable name will be importAllUsers_1, and so on.

Note: Based on your requirements, you may need to add a for-each loop after an Import All operation since it returns an array of data objects that you need to iterate over to retrieve a single data object. For example, if the Workflow includes an application with Import All Users, you must add a for-each loop after this task to iterate over the variable importAllUsers. Each user object will be stored in the output variable singleElement.

Appendix

Configure Log Levels for Elements

You can configure the log level and log message for individual elements in a Workflow in the Logs section of each element. The Logs section of an element has the following fields:

  • Log Level: Select a log level from the drop-down list.
  • Log Message: Provide a message to be logged. You can include attributes or variables in this field if required. You can also click the expand icon to open the Expressions Editor to write JavaScript code.

The logs are printed in the Workflow logs based on the global Log Level and the element Log Level settings in the Workflow:

Global Log Level Element Log Level Description
Default Default

Prints the following in the Workflow logs:

  • The messages of the Print Log elements.
  • The messages of the Logs section of each element with the log level set as Default.
Info Default / Info

Prints the following in the Workflow logs:

  • The messages of the Print Log elements.
  • Each step of the Workflow execution.
  • The messages of the Logs section of each element with the log level set as Default or Info.
Verbose Default / Info / Verbose

Prints the following in the Workflow logs:

  • The messages of the Print Log elements.
  • Each step of the Workflow execution.
  • The input and output values of each element.
  • The messages of the Logs section of each element with the log level set as Default, Info, or Verbose.

Note: The global Log Level setting in the Workflow can be configured by navigating to the vertical ellipsis at the top right corner of the Workflow page and clicking on Settings. 

Filtering Data

You can write expressions in the expression mode to filter the attributes.

You must use SCIM filter expressions to filter the attributes of the Application's Import All operation. SCIM filter expressions follow the SCIM RFC specification. 

However, you must use JavaScript expressions in the configuration of Workflow elements such as If-else and While. All the JavaScript operators are supported in the JavaScript expressions.

This section provides information on the SCIM filter expressions to filter the attributes using attribute operators.

Note: The SCIM filter expression will work only if the connector added in the Workflows supports filtering by the specified operators and attribute names. Refer to the Supported Filters section of the respective connector configuration guides for further details.

The following table lists the attribute operators that can be used in a SCIM filter expression:

Operator Description SCIM Filter Expressions
equal

The attribute evaluation and the value must be identical to meet the filter condition.

Core: userName eq "bjensen"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department eq "Sales"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID eq "Q7DETWGV"
not equal

The attribute evaluation and the value must not be identical to meet the filter condition.

Core: userName ne "bjensen"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department ne "Sales"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID ne "Q7DETWGV"
greater than

If the attribute evaluation is greater than the value, the filter condition is met. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTime types, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:employeeNumber gt 100

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID gt 100
less than

If the attribute evaluation is less than the value, the filter condition is met. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTime types, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:employeeNumber lt 100

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID lt 100
greater than or equal to

If the attribute evaluation is greater than or equal to the value, the filter condition is met. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTime types, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:employeeNumber ge 100

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID ge 100
less than or equal to

If the attribute evaluation is less than or equal to the value, the filter condition is met. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTime types, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:employeeNumber le 100

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID le 100
contains

The value must be a substring of the attribute evaluation to meet the filter condition.

Core: userName co "@domain.com"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department co "Sale"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID co "Q7"
starts with

The value must be a substring matching at the beginning of the attribute evaluation to meet the filter condition.

Core: userName sw "b"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department sw "S"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID sw "Q"
ends with

The value must be a substring matching at the end of the attribute evaluation to meet the filter condition.

Core: userName ew "@domain.com"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department ew "s"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID ew "V"
is defined

If the attribute evaluation has a non-empty or non-null value, or if it contains a non-empty node for complex attributes, the filter condition is met.

Core: userName pr

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department pr

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID pr

Status Variables

The following are Aquera's status variables that you can use in the Print Log element or the Logs section of any element:

  • currentStatus: Shows the status of the most recent element execution in the Workflow, such as success or failure.
  • statusCode: Shows the response code of the most recent element execution in the Workflow, for example, 200, 404, 500, and so on.
  • statusMessage: Shows the detailed status message (such as error summary, error code, ..)  when there is a failure in the most recent element execution in the Workflow.

Note: The status variables will be printed in the Workflow logs only if the Execution Mode is set to Normal. The execution mode setting in the Workflow can be configured by navigating to the vertical ellipsis at the top right corner of the Workflow page and clicking on Settings. 

Constants

The following are Aquera's Constants that you can use in the input fields of the Workflow based on your use case:

  • now: Returns the current date and time in ISO format, for example, 2023-07-27T12:49:40.923Z.

  • today: Returns the current date in ISO format, for example, 2023-07-27.

Keywords

The following are Aquera's keywords that you can view in the schema and attribute mapper of objects and use in the JavaScript expressions:

  • ENT (Enterprise): References the SCIM enterprise schema of the users' resource type.
    Example: <objectVariableName>.[ENT].employeeNumber
  • IAN (Integrated Aquera Network): References Aquera's SCIM extension to the custom schema of the users' resource type.
    Example: <objectVariableName>.[IAN].hireDate
  • GIAN (Group Integrated Aquera Network): References Aquera's SCIM extension to the custom schema of groups' resource type.
    Example: <objectVariableName>.[GIAN].groupType

Auto Suggestions and Tooltips in Expressions Editor

The Expressions Editor provides auto-suggestions of variables and functions. 

When you click the i icon next to a function name, you can view the tooltip that provides details of the parameters and description of the function.  

Using Data Values in Configuration Fields

You can include data values in the element configuration fields. The following table shows how values of different data types can be used in the configuration fields.

Type of Data Usage of Value in Configuration Fields
String Provide the value within double quotes. For example, "Barbara".
Number Provide the value without quotes. For example, 100.
Date Time Provide value within double quotes. For example, "2011-05-13T04:42:34Z".
Boolean Provide true or false without any quotes. For example, true.
Object Provide the object reference without any quotes. For example, userObject.
