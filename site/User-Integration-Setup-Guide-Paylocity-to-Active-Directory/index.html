User Integration Setup Guide - Paylocity to Active Directory

Overview

This guide covers the end-to-end tasks involved in setting up a synchronization to sync users in Paylocity with Active Directory.

The following tasks are involved in setting up the synchronization from Paylocity to Active Directory:

  • Configuring Applications in Aquera Admin
  • Synchronization Setup
  • Running the Synchronization

Configuring Applications in Aquera Admin

The following applications must be added and configured in the Aquera Admin console:

  • Paylocity. Refer to the Paylocity Configuration Guide for details.
  • Active Directory. Refer to the Active Directory Configuration Guide for details.

Synchronization Setup

This section details how the user synchronization must be configured to allow data push from Paylocity to Active Directory. 

It also describes the various analyses to be performed and the important points to be considered before setting up the user synchronization.

Considerations

  • A naming convention for the User Principal Name (UPN) in the Active Directory
  • A naming convention for the SAM Account Name in the Active Directory
  • Fields currently populated in the Active Directory
  • The username can be retrieved from the Paylocity system
  • The criteria for matching a user record from Paylocity with that in the Active Directory.

General Analysis Steps

Before setting up the synchronization, the following analyses need to be done:

  • Fill out the attached mapping template for mapping Paylocity attributes to Active Directory attributes.
  • Expression language can be used in the mapping to create UPNs and SAM Account Names without duplicates or errors. In the expressions, you can use inbuilt methods such as toNormalizedDiacritics() and toSAMAccountName() for this purpose. You can also use the keywords such as %ATTEMPT, %USEQ, SEQUENCENUMBER to avoid duplicates. For details, refer to the Synchronization Expressions - Keywords, Methods, and Functions Guide.
  • Define the logical OU mapping rules for user containers.

For detailed analyses of Paylocity users and the users/groups in Active Directory using scripts, refer to the Appendix.

You can also analyze the user's data using the Reports option in the Data tab of the Applications page in the Aquera Admin console. Refer to the User Reports Guide for details.

Building a Synchronization

This section describes how to set up synchronization using the User Synchronization method.

  • User Synchronization - In this synchronization, set Paylocity as the Source Application and Active Directory as the Target Application. Refer to the User Synchronization Setup Guide for details on the synchronization configurations.

You can set the following configurations in the user synchronization:

  • Set the Admin Email Options to receive email notifications for success/failures during the synchronization run.
  • Set the email list of the IT admin or manager to receive user creation notification emails and user deactivation notification emails.
  • Use tables for creating lookup tables based on location and department values from the source.
  • Enable Group management if Aquera has to manage the groups and group assignment rules based on your group analysis. Set the group names in the restriction list for scoping of the groups which would be managed by Aquera.

Refer to the User Synchronization Setup Guide for details on the synchronization configurations.

Note: For the custom attributes from the source system, the custom fields are not displayed automatically for the Active Directory in the Attribute Mapping page. You must manually add the Active Directory custom fields to the Attribute Mapping page and complete the mapping.

Synchronization Scenarios

This section details the synchronization scenarios.

Creating a user of type NORMAL_ACCOUNT in the Active Directory

To create a user of type NORMAL_ACCOUNT in Active Directory, set the attributes mapping as follows:

Enter "512" as the source attribute expression and map it to the [IAN].userAccountControl attribute of AD. Set the mapping criterion as Map on Create.

Note: 512 is the decimal value corresponding to the NORMAL_ACCOUNT property flag in Active Directory. For all property flags and the corresponding decimal values, refer to the Microsoft documentation.

Escaping Special Characters in OUs and DNs for Active Directory

To escape any special characters in OUs and DNs in the expressions, use a double-backslash before the special character. For example, if the OU name is #Terminated, mention it in the expression as \\#Terminated. 

DEST[IAN].distinguishedName.split(",")[0] + ",OU=\\#Terminated, DC=local"

Prompting for Password Change at Login

To make it mandatory for the new users in Active Directory to change their password on their first login, map the Active Directory attribute pwdLastSet to "0" with the mapping option of Map on create only. If this mapping is not done and users sign into their account for the first time, it will not prompt them to change their password.

To prompt the existing users also for a password reset, set the mapping option as Map on Create and Update.

Requirements to Set User Password

Active Directory requires a secure LDAPS (LDAP over SSL) connection to set or change user passwords. This is an Active Directory’s security requirement. Random generation of passwords when creating new users does not work without the use of LDAPS. Also, setting the password through attribute mapping requires a secure LDAPS connection.

Note: If the connection is not secured (non-SSL), Active Directory does not allow you to update the user password. 

Active Directory imposes strength requirements on the password and the requirements vary from one instance to another.

Running the Synchronization

  • Enter the matching expression to match the source records with the destination records.
  • Run the synchronization in Report Only mode and determine if the correct data is updated in the Active Directory. If the data is not correctly updated, change the matching expression to correctly match users and/or update the data, for example, employeeNumber, in the Active Directory.
  • Increment the Update Limit (maximum number of records to be updated in a single run) for each run in Report Only mode to ensure that the records are updated correctly when the records count is more.
  • Based on your analysis and considerations,
    • Translate the logical maps to the Aquera attribute maps for the fields
    • Translate the logical OU/container locations into tables and expressions
    • Translate the logical group assignment rules into Aquera rules
  • Modify the attributes mapping and repeat the process until the data is correctly updated.
  • Monitor the synchronization logs thoroughly to ensure that no errors are encountered. Modify the synchronization configurations to fix any errors.
  • Once all users are synced, user creations and updates are based on the incremental changes in Paylocity.

Appendix

This section provides the scripts that may be used by the Customer Support team to analyze the existing users in Paylocity and Active Directory before setting up the synchronization. It also provides the script to analyze the existing groups of users in Active Directory.

This document explains how analyses can be done using a script synchronization. In script synchronization, add Paylocity and Active Directory applications and set the corresponding variable names. In the script, the variable names must be used to refer to the applications.
For details on configuring the script synchronization, refer to the Synchronization Setup using Scripts Guide.

A few sample scripts are provided here for your reference. The scripts may vary based on your use case and the attributes to be retrieved for analysis.

Analyzing the Users in Paylocity 

Use the below sample script in a script synchronization for generating a report from Paylocity for analysis of username, user title, work location, and department. The work location and department are mapped to IAN attributes.

const ENT = "urn:ietf:params:scim:schemas:extension:enterprise:2.0:User";
let employees = payloc.getUsers();
for (let employee of employees) 
{
    log.info(employee.userName + ',' + employee.active + ',' + 
    employee.title + ',' + employee[IAN].worklocation + ',' + 
    employee[IAN].costCenter1);
}

Analyzing the Users in Active Directory

Use the below sample script in a script synchronization for generating a report from Active Directory for analysis of user domain, status, and so on.

log.info('AD ID,AD ACTIVE,AD EMPNO,AD GIVENNAME,AD FAMILYNAME,AD USERNAME,AD SAMACCOUNT,AD DN');

for (let user of AD.getUsers()) {

    log.info('"'

    + user.id + '","'

    + user.active + '","'

    + (user[ENT] && user[ENT].employeeNumber || '') + '","'

    + user.name.givenName + '","'

    + user.name.familyName + '","'

    + user.userName + '","'

    + user[IAN].sAMAccountName + '","'

    + user[IAN].distinguishedName.split(',').slice(1).join(',') + '"');

}

Analyzing the Groups in Active Directory

Use the below script to analyze the groups of existing users in the Active Directory.

let ad_users = AD.getUsers();
let users_length = 0;
let ad_users_length = ad_users.length;
let i = 0;
let total = users_length > ad_users_length ? users_length : ad_users_length;
log.info('AD ID,GROUP NAME');
for (i = 0; i < total; i++) {
    let groups = ad_users[i].groups;
    if (groups) {
        for (let group of groups) {
            log.info(ad_users[i].userName + ',' + group.display);
        }
    }
}
