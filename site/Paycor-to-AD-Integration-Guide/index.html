Overview

The Aquera platform allows your organization to scale complex data management by automating data integration across resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

In this integration, every employee newly created or modified in Paycor is updated in the Active Directory according to the schedule.

This guide covers details on how to set up the HR to Active Directory integration to synchronize employee data from Paycor to Active Directory. 

Prerequisites

The following are the prerequisites:

  • Aquera Tenant Account: You must have a tenant account in the Aquera Admin portal. Visit https://admin.aquera.io to register for a tenant account. Contact your Aquera account manager to get Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/
  • Configure Application Connectors in the Aquera Admin Portal: You need to configure the following application connectors in the Aquera Admin Portal:
    • Paycor Connector: Refer to the Paycor Configuration Guide for details.
    • Active Directory Connector: Refer to the Active Directory Version 8 Configuration Guide for details.

Integration Setup

This section outlines how the integration must be configured to push employee data from Paycor to Active Directory.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Select Paycor to Active Directory Script Integration.
  5. Enter a name and a description for the integration.
  6. Select your Paycor and Active Directory connector instances.
  7. Click Continue. The Configuration page is displayed to configure the integration.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Parameters

The Parameters section enables you to customize the integration settings according to your specific use case. It has the following settings:

  • UPN/ULN Suffix: Configure the domain name that must be used as the suffix for UPN/ULN. You can configure a static domain name as the UPN/ULN suffix ( for example, aquera.com) or define conditions to assign a specific UPN/ULN suffix.
  • Same as UPN/ULN Suffix: Select this checkbox if Work Mail Suffix is the same as UPN/ULN suffix.
  • Work Mail Suffix: Configure the domain name that must be used as the suffix in the Work Mail. For example, aquera1.com. If you want the same UPN suffix to be used as the Work Mail suffix, select Same as UPN/ULN Suffix.

Onboarding

The Onboarding section enables you to customize the integration settings according to your specific use case. It has the following settings:

  • Enable Onboarding: Select this checkbox to onboard users. This is a mandatory setting. 
    Note: It is mandatory to enable this checkbox to customize the user's onboarding.
  • Create Condition
    • Switch to Expression Mode: If you prefer to write a detailed prompt for creating a user, click Switch to Expression Mode.
    • Create Condition: Select the desired create condition from the dropdown menu.
    • Generate Password: Select this checkbox to generate a random password while creating a new user.
      • Password Length: Enter the length of the password while creating a new user.
      • Special Character Set for Password: Defines the set of special characters available for inclusion in generated passwords. Passwords will also include alphanumeric characters. The first character will always be alphabetic.
  • Send Email on Create
    • Admin's Email Addresses: Enter the email addresses of the admins to whom the email notifications will be sent
    •  Recipient's Email Addresses: Enter the email addresses of the recipients to whom the email notifications will be sent
    •  Admin Email Subject: Enter the content that must be used as the email subject for sending email notifications to the admins
    •  Admin Email Notification Template: Enter the content that must be used as the email body for all admin email notifications
    • Send as HTML: Select this checkbox to send the e-mail in HTML format. You can use the standard HTML tags in the Creation Email Body field to format the email body.
  • New User Integration Hook: If you want to perform a set of tasks when a new user is created in a target application, configure the New User Integration Hook. For example, when an employee joins an organization, if you want to create a ticket in an ITSM that provides access to the relevant applications or systems for the new user, you can select a Script or a Workflow Integration in the New User Integration Hook that performs this task for each new user.

Profile Update

The Profile Update section enables you to customize the integration settings according to your specific use case. It has the following settings:

  • Enable Profile Update: Select this checkbox to update the user's profile. This is a mandatory setting. 
    Note: It is mandatory to enable this checkbox to customize the user's profile.
  • Update Condition
    • Switch to Expression Mode: If you prefer to write a detailed prompt for creating a user, click Switch to Expression Mode.
    • Update Condition: Select the desired update condition from the dropdown menu.
  • Update Deactivated Users: Select this checkbox to update the deactivated source user records in the target system, provided the records satisfy the Update Condition configured for the integration. If this checkbox is not selected, the data of users who are deactivated in the source system are not updated in the target system.
  • Update User Integration Hook: If you want to perform a set of tasks when a user is updated in a target application, configure the Update User Integration Hook. For example, when an existing employee moves into a new role in an organization, if you want to perform a task that provides access or removes access for the relevant applications or systems for the updated user based on the new role, you can select a Script Integration or a Workflow Integration in the Update User Integration Hook that performs this task for each user updated.

Offboarding

The Offboarding section enables you to customize the integration settings according to your specific use case. It has the following settings:

  • Enable Offboarding: Select this checkbox to offboard users. This is a mandatory setting. 
    Note: It is mandatory to enable this checkbox to customize the user's offboarding.
  • Termination Delay: The number of days to wait after a user record in the source application is deactivated, before deleting the user record in the target application. Once the delay period is over, the user is deleted in the next integration run. The user record will be deleted only if the target connector supports the user delete operation. If the target connector does not support the delete operation, the user will only be deactivated in the target application.
  • Deactivate Users Missing in the Source Application: Select this checkbox to deactivate all the target user records for which the associated source records are not present in the source application.
  • Send Email on Deactivation
    • Admin's Email Addresses: Enter the email addresses of the admins to whom the email notifications will be sent
    •  Recipient's Email Addresses: Enter the email addresses of the recipients to whom the email notifications will be sent
    •  Admin Email Subject: Enter the content that must be used as the email subject for sending email notifications to the admins
    •  Admin Email Notification Template: Enter the content that must be used as the email body for all admin email notifications
    • Send as HTML: Select this checkbox to send the e-mail in HTML format. You can use the standard HTML tags in the Creation Email Body field to format the email body.
  •  Deprovision User Integration Hook: If you want to perform a set of tasks when a user is deactivated in a target application, configure the Deprovision User Integration Hook. For example, when an employee leaves an organization, if you want to perform a task that removes user's access from the relevant applications or systems for the deactivated user, you can select a Script or a Workflow integration in the Deprovision User Integration Hook that performs this task for each user deactivated.

Additional Settings

The Additional Settings section enables you to customize the integration settings according to your specific use case. It has the following settings:

  • Source Match Expression
    • Switch to Expression Mode: If you prefer to write a detailed prompt for creating a user, click Switch to Expression Mode.
    • Source Attribute: The source attribute values. The attribute values will vary based on the Report Mode set. 
    • Target Attribute: The target attribute values. The attribute values will vary based on the Report Mode set. 
  • Source Filter Condition
    • Switch to Expression Mode: If you prefer to write a detailed prompt for creating a user, click Switch to Expression Mode.
    • Employee Filter Criteria: The Employee Filter Criteria fields are displayed for you to set the conditions for filtering the employees.
    • Operator: The Operator
    • Value: The value
  • Map Manager: Select this checkbox to map users' managers from source to target. The enterprise manager attributes in the user objects are used for mapping.
  • Phone Number Formatting: The standard format to which users' phone numbers must be converted during the integration. The valid values are:
    • disabled: By default, the conversion of phone numbers to a specific format is disabled.
    • automatic: Convert the phone number to automatic format.
    • E164: Convert the phone number to E.164 international telephone number format, which is represented as
      + <country code> <subscriber number including area code>.
    • INTERNATIONAL: Convert the phone number to international format.
    • NATIONAL,country=US: Convert the phone number to the format followed inthe  United States.
  • Timezone Offset: The UTC timezone offset value to be used for generating reports. For example, a value of -8 in this field indicates PST, 0 indicates GMT, and so on.
  • Fire Hooks on Error: Select this checkbox to trigger the integration specified in the New User Integration Hook and Update User Integration Hook even if there is a failure in user creations and updates.

  • Skip Create and Update Failure: When this checkbox is selected, during incremental execution of the integration, if there are failures in creating or updating destination users, those transactions will be skipped in the next incremental run. The last modified timestamp will be updated for the source records, including the source records that encountered failures in the previous incremental run. Hence, in the next incremental run, only the changes from the last modified timestamp will be considered.

    When this checkbox is not selected, then those failed transactions will be attempted on the next incremental execution. The last modified timestamps will remain the same as those of the previous incremental run, so that the integration will re-read all updates that were part of the previous incremental run. This action may result in longer delays while the failures continue through repeated executions of the integration.

  • Destination Modification Safeguard Enabled: Select this checkbox to trigger the safeguard, which will abort the integration run if it would result in the number of modifications exceeding the Max Destination Modification Percentage. 
  • Incremental Tuning: Select an algorithm used to identify matching target records for incremental imports:
    • match expression and userName: When this option is selected, the integration tries to find a matching target using the configured Match Expression. If a matching record is not found, only then does the integration try to find a matching target using the configured userName mapping. This is the default option.
    • match expression only: When this option is selected, the integration attempts to locate a matching target by searching based on the configured Match Expression. The match expression must be reasonably straightforward, and the filtering of users by the attribute specified in the Match Expression field must be supported by the target application.
    • userName only: When this option is selected, the integration attempts to locate a matching target by searching based on the configured userName mapping. A match is determined by a userName filter, which returns a target that satisfies the match expression. The match expression may be as complicated as necessary with this option. 
  • Enable Incremental Cutoff: Select this checkbox to enable the incremental cutoff. This configuration works when you run the integration in the Incremental Mode.

Group Management

The Group Management tab provides options to configure how group memberships must be handled through this integration. You can determine the group assignment, retention, and removal of users based on the update or deactivation of users in Paycor.

Note: You can also set the manage group memberships via Attribute Map (under Mapping and Assignments) where you can write JavaScript expressions in the mapping, which will return an array of groups.

Select Enable Group Management to set the options for managing user groups. If this checkbox is not selected, the integration does not manage group assignments to users in the Active Directory.

When you select Enable Group Management, the Group Rules section will be displayed on this tab. Refer to the following section for details.

Set Up Group Rules

You can set rules for assigning users to the Active Directory groups through either the Group Rules section of the Group Management tab or the Attribute Map section of the Mapping and Assignments tab. 
Note: When you use the Attribute Map functionality to set the group rules, you have to write JavaScript conditional expressions to assign the groups. 

In the Group Rules section, you can create conditions that control how users are added to groups. Each rule defines the conditions for assigning users to a set of groups.

Perform the following procedure to create group assignment rules:

  1. Click Add New Rule to create a new group assignment rule. The following fields are displayed.
    • Choose Attribute: Click on this label to view the list of Paycor employee attributes.
      • Search for the attribute using which you want to set a condition. 
      • Select the required attribute from the list and configure the attribute value to form the rule condition. Only those users who meet this condition will satisfy this group rule. If multiple values are provided, an OR condition is applied among those values. For example, in the following sample condition, a user's work location must be either USA or India for them to satisfy this group assignment rule.
    • Assign To: Select the group(s) to which the users who satisfy the specified condition must be assigned. The list shows all the groups from the target AD application.
  2. Click Add Row to add another condition and group assignment for the same Paycor attribute. 

    Note 1: At a time, only two rows will be displayed on the page. If you add multiple rows of conditions, use the navigation buttons to navigate to other rows.

    Note 2: You can delete a row by using the delete icon at the right end of the row.
  3. [Optional] If you want to write a JavaScript expression to set the condition, click Switch to Expression.
    Note: When you switch to the Expression mode or to the Basic Condition mode, all the configured conditions of the rule will be cleared.
    • Click Continue to configure the expression.
    • Expressions can use all the employee and user attributes in the source and destination applications. The configured expressions must always be evaluated to true or false.
      SOURCE must be used in an expression to represent an attribute in the source application and DEST to represent an attribute in the destination application. For example, SOURCE.activeEmploymentDetail__employeeNumber, DEST.EmployeeNumber, and so on.  
  4. Click Add New Rule to add another group assignment rule, as explained above.

Set Up Group Retention Policy

The Group Retention section allows you to define how a user's group memberships must be handled when the Paycor employee is updated or deactivated.

On Deactivate

This drop-down list is used to specify what should happen to a user's group memberships when the Paycor user is deactivated. It includes the following options:

  • Remove from all assigned groups: All the group memberships currently assigned to the user will be removed. However, if the deactivated user satisfies some of the group assignment rules and they are not already part of those groups, they will be added to those groups; if they are already part of those groups, they will be retained in those groups.
  • Retain in all assigned groups: All the group memberships currently assigned to the user will be retained, even if the user does not satisfy the rules configured for the group assignments in the Group Rules section.
  • Retain in selected groups: This option has to be used when you want to retain the deactivated users only in specific groups. When you choose this option, the following additional fields are displayed:
    • Groups: In this field, you need to select the specific groups in which a user's membership has to be retained. For example, if a user is part of G1, G2, and G3 groups and if you have specified G3 and G4 in this field, the user's membership in G3 will be retained. Deactivated users will be removed from all other groups.
    • Condition: This is an optional field to define a filter condition to apply the retention rule only to a specific set of users from the deactivated list of users. The group membership will be retained only for the users matching the filter condition. For the remaining set of deactivated users that do not match the filter condition, no action will be taken on their group memberships. Here is a sample filter condition:
      '((SOURCE.activeEmploymentDetail.department).includes("Admin") && (SOURCE.activeEmploymentDetail__positionData__jobTitle).includes("Manager")'

On Update

This drop-down list is used to specify what should happen to a user's group memberships when the user is updated. It includes the following options:

  • Remove from all assigned groups: All the group memberships currently assigned to the user will be removed. However, if the user satisfies the conditions defined in the Group Rules section, they may be added to new groups based on the configured group rules.
    Note: If the Update Deactivated Users configuration is enabled, a deleted or deactivated user will be added to a new group based on the group rules. However, they will not be added back to an existing group from which they have been removed during this integration run.
  • Retain in all assigned groups: All the group memberships currently assigned to the user will be retained. Even if the user does not satisfy a rule configured for the group assignments in the Group Rules section, they will be retained in those groups.
  • Retain in selected groups: This option has to be used when you want to retain the updated users only in specific groups. When you choose this option, the following additional fields are displayed:
    • Groups: In this field, you need to select the specific groups in which a user's membership has to be retained. For example, if a user is part of G1, G2, and G3 groups and if you have specified G3 and G4 in this field, the user's membership in G3 will be retained on user update.
    • Condition: This is an optional field to define a filter condition to apply the retention rule only to a specific set of users that were updated. The group membership will be retained only for the users matching the filter condition. For the remaining set of updated users that do not match the filter condition, no action will be taken on their group memberships. Here is a sample filter condition:
      '(SOURCE.activeEmploymentDetail.department).includes("Admin") && (SOURCE.activeEmploymentDetail__positionData__jobTitle).includes("Manager")'

Attribute Map

The Attribute Map tab defines how data is mapped from the source to the target application. The default mapping of attributes is displayed on the page. The source attributes are listed on the left side and the target attributes are listed on the right side of the page.

You can modify only the source attributes as needed, to align with your specific use case.

You can write JavaScript expressions to modify the value of the source attribute and construct a value to be updated in the target attribute. Write the expression directly in the source attribute field or click the Expand icon in the source attribute field to write a long-expression.

All attributes marked with a red asterisk (*) indicates required fields for successful provisioning.

After the source-to-target attribute mapping is completed, click Save Changes at the top-right corner of the page to save the integration.
