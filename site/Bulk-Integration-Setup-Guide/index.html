Bulk Integration Setup Guide

← Back to index

Overview

In Bulk Integration, all modified and new user records are updated as a batch to the target application, as per the defined integration schedule. The user data are sent to the target application as an array of user records. The Bulk Integration can be configured only if the target application supports bulk or batch import of data. 

The following are some of the applications that support batch import: TigerConnect, SumTotal, and Flaik.

This guide explains how you can set up a Bulk Integration in the Aquera Admin portal.

Prerequisites

Ensure that the required source and target connectors are added and configured correctly in the Aquera Admin portal, before creating the Bulk Integration. Refer to the corresponding configuration guides for details on configuring your required connectors.

Creating a Bulk Integration

You can create a bulk user integration using one of the following options:

  • Create your own Bulk User Integration
  • Create Bulk User Integration using a Template

Create your own Bulk User Integration

Create a new bulk user integration where you need to fill in every configuration field manually.

Perform the following steps to create a new bulk user integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel which displays the list of integrations created by the user. 
    Note: The number of integration records displayed on this page is based on the display size of the user's screen.
  3. Click Create Integration.
  4. Select Bulk Integration.
  5. Enter a name and a description for the integration.
  6. Click Continue. A new page is displayed with the fields and tabs to configure the integration.
  7. Enter the details on the Configuration tab.
  8. Click Create to create the bulk integration.

Create Bulk User Integration using a Template

Templates are pre-built integrations created by Aquera for specific use cases. You can use the templates as-is or modify them to match your organization's needs. You can view the templates if Aquera has enabled templates in your tenant. 

Perform the following steps to create a bulk user integration using a template:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel which displays the list of integrations created by the user. 
    Note: The number of integration records displayed on this page is based on the display size of the user's screen.
  3. Click Create Integration. 
  4. Select a pre-built bulk user integration template based on your requirements.
  5. Enter a name and a description for the integration.
  6. Select a source connector in the Source Application dropdown field.
  7. Select a target connector in the Target Application dropdown field.
  8. Click Continue. A new page is displayed with the fields and tabs to configure the integration.
  9. You can update the configurations if required.
  10. Click Create to create the bulk integration.

Basic Details

This section explains the fields to configure the basic details. 

Field Description

Source Application

Select the source application from where the data of users must be retrieved. The added and modified user data are retrieved from this application as part of the integration.

Target Application

Select the target application to which the bulk (batch) data of modified users are updated as part of the integration. The target application must support the bulk import of data.

Report Only

 Select this checkbox to generate a report that shows the data that would be changed, without actually making those changes. This can be selected for the first instance of integration to identify any issues. You can clear this selection after verifying the report and confirming that the result is as expected.

Admin Email List

The administrative email addresses to which notifications on this integration have to be sent. Multiple email addresses must be separated by a comma.
Example: admin1@example.com, admin2@example.com 

Incremental Mode

Select this checkbox to run the integration in incremental import (incremental run). During incremental import, the source records that have been created or modified in the source application since the last integration run will be updated in the target application. Integrations run in the incremental import only if the source connector supports filtering users by the meta.lastModified attribute.

When the Incremental Mode checkbox is not selected, integrations run in full import (full run). During full import, all the source records that have been created or modified in the source application will be updated in the target application.

Incremental Mode Initial Timestamp

The timestamp that must be considered when you run the integration in incremental mode for the first time. The source records that were added or modified after this timestamp will only be considered for the first incremental run of integration.

The timestamp has to be entered with the time zone designator (TZD) in the format, 
YYYY-MM-DDThh:mm:ss.sTZD

Example: 2020-06-18T08:30z, 2020-06-18T01:30-0700

Import Limit

The maximum number of user records that are imported from the source application during a single run of the integration. Set this field to unlimited if you do not want to limit the number of imported records.

The limit can be set accordingly to verify and debug the source-to-target mapping. You can set the import limit to 1 to verify that the user is imported correctly.

Source SCIM Filter

The expression that can be used for filtering the users during source import, if required. This field is primarily used to verify the user data by filtering users from the source application.

Example: CON:userName eq "bjensen@example.com"

Notes:
1. When you use Source SCIM Filter, ensure that Incremental Mode is not enabled.

2. If an expression starts with CON:, the filter is executed in the connector and the resulting user records are retrieved in the integration. Otherwise, all the user records are retrieved in the connector and then the filter is applied by the integration.

3. CON: can be used in the expression only if the source connector supports the corresponding filter operation. Refer to the respective Connector Configuration Guides for further details on the supported filters.

4. For enterprise and IAN attributes, the external namespace also must be included with the attribute name. For example, CON:urn:ietf:params:scim:schemas:ian:2.0:User:employeeID gt 100

For details on the operators and the syntax that can be used for filtering the user data, refer to the SCIM Filter Expressions in Integrations guide.

Create Condition

The JavaScript expression that is used to determine the source set of users that must be created in the target application during batch integration.

The following are sample values:

  • SOURCE.active or SOURCE.active==true: Creates users in the target application whose active attribute is set to true in the source application.
  • true: Creates users in the target application whose active attribute is set to true or false in the source application.
  • false: Does not create any users in the target application.
  • SOURCE.active==false: Creates users in the target application whose active attribute is set to false in the source application.
  • SOURCE[IAN].hireDate > new Date(): Creates users in the target application whose hire date is greater than the current date.

Update Condition

The JavaScript expression that is used to determine the source set of users that must be updated in the target application during the batch integration.

The following are sample values:

  • SOURCE.active: Updates users in the target application whose active attribute is set to true in the source application. 
  • true: Updates users in the target application whose active attribute is set to true or false in the source application.
  • false: Does not update any users in the target application.
  • SOURCE.active == false: Deactivates users in the target application whose active attribute is set to false in the source application.
  • SOURCE[IAN].startDate > new Date(): Updates users in the target application whose start date is greater than the current date.

Notes:

  • If the target application supports reading users' data from it (GET operation), then both Create Condition and Update Condition apply. The Create Condition determines the users that will be created (users not already existing in target) and the Update condition determines the users that will be updated (users already exist) in the bulk operation.
  • If the target application does not support GET operation, then the Update Condition will be ignored. In that case, the Create Condition is used to determine the user records that will be included in the bulk update.

Map Manager

 Select this checkbox to map users' managers from the source to the target application. The manager attributes in the user objects are used for mapping.

Update Limit

The maximum number of user records that are allowed to be updated in the target application during a single run of the integration. Set this field to unlimited if you do not want to limit the number of records.

Note: The Update Limit must be set to unlimited when the integration runs in the incremental mode. Otherwise, a few user updates may be skipped during the incremental runs.

Match Expression

The expression that is used to identify the target record that matches a source record. This expression can be given only if the target connector supports GET operation.

The default match expression in this field is SOURCE[ENT].employeeNumber == DEST[ENT].employeeNumber, which means matching the source and target records with the users' employee number attribute value.

Creation Email Details

This section explains the fields to configure the creation email details. 

Field Description

Send New User's Manager Email

 Select this checkbox to send an e-mail notification to the manager when a new user is created in the target application as part of the integration.

Creation Distribution List

 The email addresses to which an e-mail notification has to be sent when a new user is created in the target application as part of the integration. Multiple e-mail addresses must be separated by a comma.

Creation Email Subject

 The subject line to be used in the e-mail notification that will be sent when a user is created in the target application. For example, New Employee.

Creation Email Body

The template that has to be used for the body of the notification e-mail, which will be sent when a new user is created in the target application. The target attributes can be included in the template to add any details of the created user in the e-mail body.

For example:
name : {{name.givenName}} {{name.familyName}}

email : {{emails[work].value}}

username : {{userName}}

password : {{password}}

manager: {(ENT.manager.displayName}}

department: {{IAN.userDepartment}}

Advanced Details

This section explains the fields to configure the advanced details. 

Field Description

Phone Number Formatting

The standard format to which the phone numbers of users must be converted during the integration process. The valid values are:

  • disabled: By default, the conversion of phone numbers to a specific format is disabled.
  • automatic: 
  • E164: The E.164 international telephone numbering format is + <country code> <subscriber number including area code>.
  • INTERNATIONAL: International format
  • NATIONAL,country=US: The format that is being followed in a nation. You must replace US with the appropriate country code.

Note: The automatic formatting does not work when you write to ADP. To set the international phone number format of +1-xxx-xxxxx during the integration, set the field value as INTERNATIONAL,template=+.-...-.......

Report Mode

The logging mode of integration details. This setting controls the level of details to be captured in the integration logs.

The following are the valid values listed in the order starting with the least to the most detailed level of logging: 

  • minimal
  • normal
  • debug
  • verbose

You can use the minimal or normal mode for the basic log details. For a detailed analysis of the integration, you can use the debug or verbose mode.

For further information on the log-level details for each report mode, refer to the Integration Logging Modes guide.

The Email From Address enables you to specify the sender email address associated with all emails sent by the integration. The default email address is noreply@aquera.com.

To modify the , you must first send a request to support@aquera.com. A verification email will be sent to that email address. You must follow the instructions in the email to grant Aquera permission to send emails on behalf of your email account. 

Refer to Verifying Email Address and Configuring DKIM Settings for more details.

Note: This setting is available from Bulk Integration Version 3 onwards.

Update Deactivated Users

Select this checkbox to update the deactivated source user records in the target system provided the records satisfy the Update Condition configured for the integration. If this checkbox is not selected, the data of users that are deactivated in the source system are not updated in the target system.

This setting works only when the integration runs in full import mode. 

Note: This setting is available from Bulk Integration Version 3 onwards.

Select this checkbox to support the creation of inactive users in the target application. You must set a relevant expression in the Create Condition configuration to create inactive users in the target application.

To create inactive users, the target application must support the creation of inactive users.

Note: This setting is available from Bulk Integration Version 3 onwards.

Click Save Changes to save the integration after making the required configuration changes.

Mapping of Attributes

You can map the source SCIM attributes to the target SCIM attributes by navigating to the Attribute Map tab. This is required to be performed to synchronize the data from the source application to the target application. You can also write expressions to modify the value of the source attribute before updating the value in the target attribute.

For further details, refer to the Attribute Mapping in Integrations guide.

Adding and Using Tables

Tables define a name-value pair to create custom attributes. Tables are added in integrations to create custom source attributes, and POST (sync) them to the target application. Tables are used when an application attribute is empty or particular data is not sent from an application, and you want to update a value for that attribute in another application.

For details on how to add tables and use them in integrations, refer to the Using Tables in Integrations guide.

Scheduling an Integration Run

To schedule an integration, navigate to the Schedule tab of the integration. You can set a time frequency to run the integrations automatically. Refer to the Integration Schedule Setup Guide for further details.

User Actions on an integration

You can perform actions on an integration such as execute, upgrade, copy, modify, delete, export, import, and view logs. For further details, refer to the User Actions on Integrations guide.
