Dayforce to Active Directory (Multidomain)

← Back to index

Overview

The Aquera platform allows your organization to scale complex data management by automating data integration across resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

This integration connects Dayforce with Active Directory in a multi-domain environment and keeps user accounts synchronized with employee data. It uses Dayforce name attributes and work location (Site) to construct each user’s Common Name (CN) and Distinguished Name (DN). Based on this information, the integration determines the correct Active Directory domain and Organizational Unit (OU) and automatically creates, updates, moves, or recreates user accounts when names, locations, or domain assignments change.

The integration ensures that every user has a unique, valid identity in Active Directory and is always assigned to the correct domain and OU. It continuously aligns CN, DN, and login attributes such as UPN and sAMAccountName with the latest Dayforce data, keeping Active Directory accurate and up to date.

The integration supports both full synchronization and incremental runs. A full sync processes all eligible Dayforce employees, while incremental runs process only those users whose data has changed since the previous execution.

This guide explains how to configure and set up the integration to synchronize employee data from Dayforce to Active Directory across multiple domains.

Key Features

The Dayforce to Active Directory (Multidomain) integration provides the following functions:

  • When a new user is added in Dayforce, the integration creates an Active Directory account, builds the Common Name (CN) from the user’s Dayforce name fields, and sets the correct Distinguished Name (DN).

  • If a user’s Site (work location) changes in Dayforce, the integration automatically moves the user account to the correct Organizational Unit (OU). If the new Site belongs to a different Active Directory domain, the integration creates the user account in the proper domain.

  • If a user’s name changes in Dayforce, the integration updates the user’s Common Name (CN) and Distinguished Name (DN). It automatically renames the Active Directory account to match the updated name.

  • For every new account, the integration first checks whether the common name (CN), login name (UPN), Windows logon name, or email address is already in use by someone else in any Active Directory domain. If it finds a duplicate, it adds a sequence number to the CN, tries again (up to 10 times), and then updates the DN to use the final CN value that is not in use.
    Note: If no unique value is found after 10 tries, the user cannot be created.

  • As user data changes over time, the integration keeps the user’s name, login details, and AD domain aligned with the latest Dayforce data.

  • The integration supports full synchronization, incremental runs, and report-only mode. In report-only mode, selected Active Directory domains generate reports without applying changes, while other domains continue to synchronize user data as usual.

  • Synchronizes group memberships and manager relationships for users. Groups and managers are assigned only when the integration can identify a single, correct match. If more than one matching group or manager exists in different Active Directory domains, the integration does not apply the assignment to prevent incorrect access or reporting relationships.

  • Manages user account status and termination using Dayforce data. User accounts are automatically enabled, disabled, or terminated as required, including upon location changes, based on attributes such as Site and LegalEntityAddress.

  • Supports automatic password generation when creating user accounts, using either a default or a custom password policy. It also provides email notifications and execution reports for user creation, updates, terminations, and integration run results.

Integration Flow

This integration connects Dayforce with Active Directory in a multi-domain environment. It retrieves employee information from Dayforce, including FirstName, PreferredLastName, LastName, and work location (Site) from the employee’s OrgUnit structure, which represents the employee’s actual workplace, such as a hotel, office, or property.

The integration uses this information to build two key identity components in Active Directory:

  1. Common Name (CN):
    The CN is created using the configured Dayforce name fields. For example, FirstName + PreferredLastName results in a value such as Barbara Jane.

  2. Distinguished Name (DN):
    The employee’s Site is matched against the Location-to-Domain mapping table, which defines the correct Organizational Unit (OU) and Active Directory domain for that Site. The integration then combines the CN with the mapped OU and domain to construct the full DN, for example:
    CN=Barbara Jane,OU=Operations,DC=Aquera1,DC=com

Once the CN and DN are generated, the integration uses them during Active Directory account creation. Before creating or updating a user, the integration verifies that key identity attributes are unique across all configured Active Directory domains. These attributes include Common Name (CN), User Principal Name (UPN), sAMAccountName (Windows logon name), and email address.

  • If any of these values already exist, the integration automatically modifies the value by appending a numeric suffix and retries until a unique value is found. For example, if barbara.jane already exists, the integration attempts barbara.jane1, then barbara.jane2, and continues until a unique value is identified.

  • If the CN changes, only the CN portion of the DN is updated. The OU and domain remain unchanged, as the employee’s Site in Dayforce determines them.

  • After an account is created, the integration continuously keeps the user information synchronized with Dayforce. During each run, the integration rebuilds the CN and DN using the latest Dayforce data and compares them with the current values in Active Directory.

  • If the user’s name changes, the CN and DN are updated, and the Active Directory account is renamed.

  • If the user’s Site changes, the DN is recalculated, and the account is moved to the appropriate OU.

  • If the new Site maps to a different Active Directory domain, the integration disables the existing account in the current domain and creates a new account in the correct target domain.

This ensures that each user’s name, login details, domain placement, and AD directory locations are always aligned with the most current information in Dayforce.

Integration Setup

Aquera’s Dayforce to Active Directory (multidomain) integration keeps employee data in sync by  transferring information from Dayforce to Active Directory.

Follow the steps below to set up the integration:

  1. Log in to the Aquera Admin Portal.

  2. Go to the Integrations section in the left panel.

  3. Click Create Integration.

  4. Choose Dayforce to AD. A Create New Integration pop-up window will appear.

  5. Enter a name and description for the integration.

  6. Click Continue. You will be taken to the Configuration tab, which contains multiple configuration sections.

  7. Under Connection, select Dayforce connector version 26 (or higher) as the Source Application and Active Directory connector version 8 as the Target Application.

  8. Click Create to add the Dayforce to AD integration.

  9. Review and configure the settings.

Integration Features

Configure the following integration settings:

  • Enable Manager Assignment: Sync the employee’s manager information from Dayforce to Active Directory.
  • Enable Group Assignments: Enable synchronization of group memberships for users during the integration run. For example, if a user is mapped to the FrontDesk group in Dayforce, the user is automatically added to the FrontDesk group in Active Directory.

Attribute Map

The Attribute Map defines how employee data from Dayforce is mapped into Active Directory during integration runs. These mappings control which AD attributes are updated and what values are written into those fields.

This integration includes two attribute maps.

Employee Attribute Mapping

This attribute map updates standard Active Directory user profile fields using employee data from Dayforce. It ensures the user’s AD identity fields remain accurate, such as name, department, email, and employee identifiers.

The key mappings include:

  • ad.UserData.ACCOUNTDISABLE: Controls whether the AD account is disabled. For example, mapping false keeps the account enabled.

  • ad.UserData.cn: Sets the user’s common name (CN). For example, the integration uses the employee’s Common Name when available, otherwise it builds the value using the employee’s first name.

  • ad.UserData.company: Updates the AD company field. For example, this can be set from Dayforce or left blank if the value is not available.

  • ad.UserData.department: Updates the AD department field. For example, this is mapped from the employee’s Primary Work Assignment Department Short Name.

Location Based Termination Attribute Mapping

This attribute map applies termination updates in Active Directory when an employee is terminated in Dayforce. It ensures the AD account is disabled, moved into the correct OU, and updated with termination-specific values.

The key mappings include:

  • ad.UserData.ACCOUNTDISABLE: Disables the user’s AD account during termination. For example, mapping true disables the account.

  • ad.UserData.MOVE_TO_OU: Moves the user to a specific OU during termination. For example, mapping OU=Disabled moves the user into the Disabled OU.

  • ad.UserData.description: Updates the AD description during termination. For example, when the user is terminated, this text is written to the user’s AD profile.

  • ad.UserData.manager: Updates the AD manager field during termination. For example, the integration uses the employee’s ActiveManagerXRefCode when available, otherwise it falls back to a default manager such as 1616.

  • ad.UserData.memberOf: Assigns the user to a default set of AD groups during termination. For example, the integration adds the user to AqueraTestDomainG1, LabsInGroup1, and LabsInGroup2.

Tables

Tables let you supply additional values that may not exist in Dayforce but are required to create or update users correctly in Active Directory. The integration uses these tables during runs to populate missing values, such as the correct AD domain, OU, or security groups, based on the employee’s location or other attributes.

This integration includes the following tables:

Location-to-Domain Mapping

Defines which AD domain and OU each Dayforce Site (location) belongs to, and helps build the full DN (Distinguished Name) during account creation and updates.

For example:

  • Administration > OU=Barbara,DC=hotel1,DC=com
  • Location  > OU=JoeSmith,DC=hotel2,DC=com

This ensures each user is created in the right domain and OU based on their Dayforce location.

Location-Based Group Assignment

This table maps employee location values to the Active Directory groups that should be assigned to the user. For example, users in USA can be assigned to one set of groups, while users in CAN can be assigned to a different set. 

For example:

  • CAN: Assigns the user to TEST_001.

  • USA: Assigns the user to TEST_Labs_01 and TEST_002.
