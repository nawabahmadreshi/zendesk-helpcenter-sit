Overview

The Aquera platform allows your organization to efficiently scale complex data management by automating the integration of diverse resource data across different applications and systems.

The integration synchronizes companies, departments, locations, managers, and positions from Oracle Cloud HCM to ServiceNow. It detects changes in Oracle Cloud HCM and applies them in ServiceNow, ensuring both systems always reflect the latest organizational structure.

For each data entity:

  • Create: If a record exists in Oracle Cloud HCM but not in ServiceNow, it is created in ServiceNow.
  • Update: If a record exists in both systems but has been updated in Oracle Cloud HCM, the ServiceNow record is updated accordingly.
  • Delete: If a record exists in ServiceNow but not in Oracle Cloud HCM, it can be deleted from ServiceNow (if deletion is enabled).

The integration runs on a configurable schedule, fetching only records changed since the last successful sync to optimize performance and minimize data transfer. After each run, the integration generates a detailed CSV report of all changes (inserts, updates, deletes) and emails it to designated recipients. This integration is triggered based on the schedule that you set under the trigger tab.

Key Functions

The Oracle Cloud HCM / ServiceNow Integration provides the following functions:

  • Retrieves company, department, location, manager, and position data from Oracle Cloud HCM.
  • Retrieves corresponding records from ServiceNow for each data type.
  • Compares Oracle Cloud HCM and ServiceNow records using attribute mapping to identify:
    • Records to insert (new in Oracle Cloud HCM, missing in ServiceNow)
    • Records to update (exist in both, but details differ)
    • Records to delete (exist in ServiceNow, missing in Oracle Cloud HCM, if deletion is enabled)
  • Executes insert, update, and delete operations in ServiceNow in configurable batches.
  • Generates detailed CSV reports of all changes (inserts, updates, deletes) in each run.
  • Sends summary emails (with optional CSV attachment) to configured recipients after each sync.
  • Provides configurable controls for:
    • Enabling/disabling insert, update, and delete.
    • Limiting the number of operations per run.

Integration Setup

This section details the configuration process for integrating Oracle Cloud HCM with ServiceNow. It covers creating the integration, setting parameters, and establishing automated synchronization of companies, departments, locations, managers, and positions from Oracle Cloud HCM to ServiceNow.

This integration syncs the Oracle Cloud HCM data of all companies, departments, locations, managers, and positions. It creates tickets for any events related to employee onboarding, role changes, and offboarding in ServiceNow.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Select Oracle Cloud HCM to ServiceNow.
  5. Enter a name and a description for the integration.
  6. Select your Oracle Cloud HCM and ServiceNow connector instances.
  7. Click Continue. The Configuration page is displayed to configure the integration.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Parameters

The Parameters section enables you to customize the integration settings according to your specific use case. It has the following settings:

Global Settings

Configure the following global settings:

  • Report Only: Generates a simulation report of all intended changes without modifying any data in ServiceNow. Useful for testing and validation.
  • Enable Email Notifications: Activates email alerts to keep stakeholders informed about each integration run.
    Note: The following fields are enabled only when Enable Email Notifications is selected.  
    • Recipients' Email Addresses: Specify one or more email addresses to receive integration notifications and reports.
    • Email Body: Customize the email body content sent to recipients, enabling personalized messaging.
  • Enable Report Generation: Generate a detailed integration report after every sync.
  • Logging Level: Control the amount of detail recorded in logs for monitoring, troubleshooting, and audit purposes.
  • Enable Analytics: Allow integration results to be visualized and analyzed through a dashboard.

Company Data Sync

Configure the following settings to control how company records are created, updated, deleted, and filtered during the integration:

  • Create Company Data: Adds new company records into ServiceNow when they exist in Oracle Cloud HCM but are missing in ServiceNow. For example, Oracle Cloud HCM has a company called Riverstone Corp, which is not found in ServiceNow. The sync will create this record in the target table.
    • Enable Filter Condition for Create: Ensures only certain company records are created by applying a predefined filter. For example, when the filter allows creation only if the Company Code starts with US, then records like US001 are created, while others are skipped.
      • Create Filter Attribute: The ServiceNow field is checked during filtering to determine whether a record should be created. For example, selecting Company Code enables filtering records based on regional identifiers, such as APAC001 or EMEA_CORP.
      • Create Filter Condition: The comparison logic used to match the attribute with a value, such as Equals or Starts With. For example, if the condition is Starts With and the value is US, then records like US_FINANCE and US001 are included for creation.
      • Create Filter Value: The specific value used to determine whether the record passes the filter condition. For example, If the value is EMEA and the condition is Contains, then companies like Global_EMEA and EMEA_SALES will be created.
  • Update Company Data: Applies changes to existing company records in ServiceNow when they’ve been updated in Oracle Cloud HCM. The company NovaTech Inc. changed its name and address in Oracle Cloud HCM. These changes will be updated in ServiceNow.
    • Enable Filter Condition for Update: Applies updates only to records that meet specific criteria defined by a filter. For example, if the filter is set to update companies with names containing Global, then GlobalEdge and Acme Global will be updated.
      • Update Filter Attribute: The ServiceNow field used to evaluate whether a company should be updated. For example, choosing Company Name as the filter field allows targeting names like Corp or Tech during updates.
      • Update Filter Condition: Defines how to compare the attribute value, such as Equals, Contains, or Starts With. For example, a condition of Contains and value Corp will update records like HealthCorp or TechCorp.
      • Update Filter Value: The value that a field must match to qualify for update. For example, if the value is Finance and the condition is Ends With, only companies like Global Finance will be updated.
  • Delete Company Data in ServiceNow: Removes company records from ServiceNow that no longer exist in Oracle Cloud HCM. For example, the record for OldBridge Ltd. has been deleted in Oracle Cloud HCM. It will be removed from ServiceNow during the sync.
    • Enable Filter Condition for Delete: Deletes only the records that match the filtering condition. For example, if deletion is allowed only for records with codes that start with TEMP, then TEMP001 will be deleted, while HR_TEMP will not.
      • Delete Filter Attribute: The ServiceNow field used to decide which records qualify for deletion. For example, using Company Code allows filtering out codes like OLD_CORP or ARCHIVE_002.
      • Delete Filter Condition: Sets the comparison logic for deletion filters, such as Equals To or Ends With. For example, using Equals To with the value DELETE_ME deletes only the record exactly matching that code.
      • Delete Filter Value: The value used to compare against the selected field for deletion.
        For example: If the value is _OLD and the condition is Ends With, records like HR_OLD and DATA_OLD will be deleted.
  • Limit Company Data Executions: Controls the number of create, update, and delete operations that can happen in each sync run. For example, during testing, you might limit the sync to 10 creates, 15 updates, and 5 deletions to avoid processing a large volume.
  • Set Creation Limit: Sets how many new company records can be created in a single sync run. For example, if 20 records qualify but the limit is set to 10, only the first 10 will be created.
  • Set Update Limit: Sets how many company records can be updated in one sync execution. For example, if 50 records are eligible but the update limit is 25, only 25 will be processed.
  • Set Deletion Limit: Sets how many records can be deleted during each sync. For example, if 12 records match the filter but the limit is 4, only 4 of them will be deleted.
  • Company Data Table Name: The ServiceNow table where company records are stored and synced. For example, this might be the standard core_company table or a custom table, such as u_companies_staging.
  • Company Data Primary Key: The field in ServiceNow used to uniquely match and identify company records. For example, the key x_1641040_admin_u_company_code ensures records align correctly between Oracle Cloud HCM and ServiceNow.

Department Sync

Configure the following settings to manage the creation, update, deletion, and filtering of department records during the integration:

  • Create Department Data: Adds new department records into ServiceNow when they exist in Oracle Cloud HCM but are missing in ServiceNow. For example, Oracle Cloud HCM includes a department called Corporate Strategy, which doesn’t exist in ServiceNow. The sync will create this department record.
    • Enable Filter Condition for Create: Ensures only certain department records are created by applying a predefined filter. For example, if the filter is set to create departments whose names start with HR, only records like HR_Talent and HR_Admin are included.
      • Create Filter Attribute: The ServiceNow field is checked during filtering to determine whether a department record should be created. For example, selecting Department Name allows filtering by values like Finance, HR, or Marketing.
      • Create Filter Condition: The comparison logic used to match the selected field with a value, such as Equals, Contains, or Starts With. For example, if the condition is Starts With and the value is IT, then records like IT_Support and IT_Development will be considered for creation.
      • Create Filter Value: The specific value used to determine whether a department passes the filter condition. For example, if the value is Admin and the condition is Ends With, departments like Office Admin and HR_Admin will be created.
  • Update Department Data in ServiceNow: Applies changes to existing department records in ServiceNow when they’ve been updated in Oracle Cloud HCM. For example, the department Finance Operations changed its name in Oracle Cloud HCM to Finance and Accounting. This change is updated in ServiceNow.
    • Enable Filter Condition for Update: Applies updates only to department records that meet specific criteria defined by a filter, for example, updating only departments whose names contain Operations, such as IT Operations and Finance Operations.
      • Update Filter Attribute: The ServiceNow field used to determine if a department record should be updated. For example, selecting Department ID or Name allows you to filter updates based on specific values like DPT1002 or Sales Team.
      • Update Filter Condition: Defines how the selected field is compared to the provided value, such as Equals, Contains, or Ends With. For example, if the condition is Contains and value is Support, it updates records like Customer Support and Tech Support.
      • Update Filter Value: The value that a department field must match to qualify for update. For example, using the value Global with the condition Contains allows updates to departments like Global Finance or Global HR.
  • Delete Department Data in ServiceNow: Removes department records from ServiceNow that no longer exist in Oracle Cloud HCM. For example, if Legal Affairs has been removed in Oracle Cloud HCM, it will also be deleted from ServiceNow.
    • Enable Filter Condition for Delete: Deletes only the department records that match the filtering condition. For example, if the deletion filter matches codes starting with OLD_, then departments like OLD_IT or OLD_MKTG are deleted.
      • Delete Filter Attribute: The ServiceNow field used to decide which department records qualify for deletion. For example, selecting Department Code allows filtering values like DPT9999 or ARCHIVED_01.
      • Delete Filter Condition: The comparison logic used to decide if a department should be deleted, based on the field and value. For example, if the condition is Equals To and the value is REMOVE_THIS, only that exact match is deleted.
      • Delete Filter Value: The value used in the condition to determine which records should be deleted. For example, if the value is Test and the condition is Contains, then departments like Test_Dept or QA_Test will be deleted.
  • Limit Department Data Executions: Controls how many create, update, and delete operations are performed per sync run. For example, you can set the sync to perform no more than 5 creations, 10 updates, and 3 deletions during a single run.
  • Set Creation Limit: Sets how many new department records can be created per sync run. For example, if 12 departments are eligible but the limit is 5, only the first 5 will be created.
  • Set Update Limit: Sets how many department records can be updated in one run. For example, if 25 are eligible for update but the limit is 10, only 10 updates are performed.
  • Set Deletion Limit: Sets how many department records can be deleted during the sync. For example, if 8 departments are flagged for deletion and the limit is 3, only 3 will be removed.
  • Department Data Table Name: The ServiceNow table where department records are stored and synced. For example, this might be cmn_department or a custom table like u_dept_sync.
  • Department Data Primary Key: The field used in ServiceNow to uniquely identify each department during sync. For example, id or department_code can be used to align records with Oracle Cloud HCM data.

Location Sync

Configure the following settings to manage how location records are created, updated, deleted, and filtered during the integration:

  • Create Location Data: Adds new location records into ServiceNow when they exist in Oracle Cloud HCM but are missing in ServiceNow. For example, Oracle Cloud HCM includes a location called Boston HQ, which isn’t found in ServiceNow. The sync will add this record.
    • Enable Filter Condition for Create: Ensures only specific location records are created by applying a filter. For example, if the filter is set to allow only locations that start with NY_, then NY_001 and NY_HR will be created, but CA_WEST will not.
      • Create Filter Attribute: The ServiceNow field used to evaluate whether the location qualifies for creation. For example, selecting Location Code allows filtering values like LOC1001 or US_EAST.
      • Create Filter Condition: The logic used to compare the field value to the specified filter value, such as Equals, Contains, or Starts With. For example, if the condition is Starts With and value is US, records like US001 or US_WEST are considered for creation.
      • Create Filter Value: The value to be compared against the selected field to determine eligibility. For example, if the value is EUROPE and the condition is Contains, records like EUROPE_WEST and UK_EUROPE will be created.
  • Update Location Data: Applies updates to existing location records in ServiceNow when details have changed in Oracle Cloud HCM. For example, the location Chicago Branch updated its address in Oracle Cloud HCM. The change is synced to ServiceNow.
    • Enable Filter Condition for Update: Applies updates only to location records that match the defined filter condition. For example, update only records whose location names include Warehouse, such as NY Warehouse or West Warehouse.
      • Update Filter Attribute: The ServiceNow field used to apply the update filter. For example, selecting Location Name allows filtering updates by partial matches like Regional or Support.
      • Update Filter Condition: The rule used to match the selected field against a value. For example, using Contains with the value Lab will update records like Innovation Lab and Testing Lab.
      • Update Filter Value: The value used for update filtering, based on the selected condition. For example, if the value is Asia and the condition is Ends With, records like Office_Asia or Support_Asia are updated.
  • Delete Location Data in ServiceNow: Deletes location records in ServiceNow that no longer exist in Oracle Cloud HCM. For example, if Toronto Office has been removed in Oracle Cloud HCM, it will be deleted from ServiceNow.
    • Enable Filter Condition for Delete: Ensures only matching location records are deleted during sync. For example, delete only those locations whose code starts with TEMP_, like TEMP_NY or TEMP_LON.
      • Delete Filter Attribute: The ServiceNow field used to determine which records qualify for deletion. For example, selecting Location Code allows targeting values like CLOSED_001 or OLD_OFFICE.
      • Delete Filter Condition: Defines how the attribute should match the given value before deletion occurs. For example, if the condition is Equals To and the value is DELETE_THIS, only that exact record is deleted.
      • Delete Filter Value: The value to compare against the selected field to determine deletion eligibility. For example, if the value is PHASED and the condition is Contains, records like PHASED_OUT_EU and NA_PHASED will be deleted.
  • Limit Location Data Executions: Controls how many create, update, and delete operations are performed per sync run. For example, limit the sync to 8 creations, 12 updates, and 3 deletions to reduce load.
  • Set Creation Limit: Sets the maximum number of location records that can be created per run. For example, if 10 records qualify but the limit is 4, only 4 will be created.
  • Set Update Limit: Sets the maximum number of location records that can be updated per run. For example, if 20 are eligible but the limit is 5, only 5 will be processed.
  • Set Deletion Limit: Sets the maximum number of location records that can be deleted per run. For example, if 9 match the delete criteria but the limit is 2, only 2 will be removed.
  • Location Data Table Name: The ServiceNow table where location records are stored and synchronized. For example, this could be cmn_location or a custom table like u_location_master.
  • Location Data Primary Key: The field used to uniquely identify each location record in ServiceNow. For example, x_1641040_admin_u_location_code is a unique field used to match Oracle Cloud HCM location records during sync.

Position Sync

Configure the following settings to manage how position records are created, updated, deleted, and filtered during the integration:

  • Create Position Data in ServiceNow: Adds new position records into ServiceNow when they exist in Oracle Cloud HCM but are missing in ServiceNow. For example, Oracle Cloud HCM has a position called Senior Data Analyst, which is not yet listed in ServiceNow. The sync will add this new position.
    • Enable Filter Condition for Create: Ensures only specific position records are created based on a defined filter. For example, if the filter allows creation only for positions that start with ENG_, then positions like ENG_DEV01 and ENG_QA02 are created, while HR_EXEC01 is skipped.
      • Create Filter Attribute: The ServiceNow field used to evaluate if the position qualifies for creation. For example, selecting Position Code allows filtering on codes such as POS001, ENG_LEAD, or TEMP123.
      • Create Filter Condition: The logic used to compare the attribute to a filter value, like Equals or Starts With. For example, if the condition is Contains and the value is Lead, it includes records such as Team Lead, Finance Lead, or TechLead01 for creation.
      • Create Filter Value: The value to match against the selected field to allow record creation. For example, using Manager with condition Ends With includes records like Project Manager, IT Manager, or HR_Manager.
  • Update Position Data: Applies changes to position records in ServiceNow when updates occur in Oracle Cloud HCM. For example, the position Marketing Analyst now has an updated title or description in Oracle Cloud HCM. That change will be reflected in ServiceNow.
    • Enable Filter Condition for Update: Applies updates only to position records that meet a filter condition. For example, only update records where the position name contains Engineer, such as Senior Engineer or DevOps Engineer.
      • Update Filter Attribute: The ServiceNow field used to determine whether a position record should be updated. For example, selecting Position Name allows filtering for updates on records like Analyst, Consultant, or Specialist.
      • Update Filter Condition: The comparison rule used to determine if the field value matches the update condition. For example, using Equals To with the value Sales Lead ensures only that exact match is updated.
      • Update Filter Value: The value used to determine eligibility for updating the record. For example, with value Tech and condition Contains, updates apply to records like Tech Support and TechOps Lead.
  • Delete Position Data in ServiceNow: Deletes position records from ServiceNow that no longer exist in Oracle Cloud HCM. For example, if Contract Recruiter has been removed from Oracle Cloud HCM, it will be deleted from ServiceNow during the next sync.
    • Enable Filter Condition for Delete: Ensures only position records that meet a filter condition are deleted. For example, delete only positions with codes starting with TEMP_, such as TEMP001 or TEMP_ADMIN.
      • Delete Filter Attribute: The ServiceNow field used to decide whether the position qualifies for deletion. For example, choosing Position Code allows filtering based on structured codes, such as OLD_POS01 or ARCHIVED_ENG.
      • Delete Filter Condition: The logic used to compare the field with the specified value for deletion. For example, using Contains with value ARCHIVE deletes records such as ARCHIVE_QA, ARCHIVE2023, or HR_ARCHIVE.
      • Delete Filter Value: The specific value that determines which records should be deleted based on the filter. For example, using the value Contract with Ends With will remove records such as HR_Contract or Consulting_Contract.
  • Limit Position Data Executions: Limits the number of position record operations per sync run. For example, limit the sync to perform only 10 creations, 20 updates, and 5 deletions to reduce data volume during testing or rollout.
  • Set Creation Limit: Specifies the maximum number of new position records that can be created during a sync run. For example, if 15 positions qualify for creation and the limit is 5, only the first 5 will be processed.
  • Set Update Limit: Specifies the maximum number of position records that can be updated during a single sync. For example, if 50 are eligible but the limit is 25, only 25 will be updated during the run.
  • Set Deletion Limit: Sets the maximum number of deletions allowed during a sync run. For example, if 10 records qualify for deletion and the limit is 3, only 3 will be removed.
  • Position Data Table Name: The ServiceNow table where position records are stored and synchronized. For example, this might be x_1641040_admin_u_admin_positions or another custom position table used for integration.
  • Position Data Primary Key: The field used to identify position records in ServiceNow during sync uniquely. For example, position_code ensures alignment between Oracle Cloud HCM and ServiceNow for each record.

Manager Sync

Configure the following settings to manage how manager records are updated and filtered during the integration: 

  • Update Manager Data in ServiceNow: Applies changes to manager records in ServiceNow when updates are detected in Oracle Cloud HCM. For example, if the email or name of John Smith, a manager in Oracle Cloud HCM, has changed, those updates are applied to his record in ServiceNow.
    • Enable Filter Condition for Update: Ensures that only matching manager records are updated based on a defined filter. For example, only update managers whose usernames start with mgr_, such as mgr_jane or mgr_steve.
      • Update Filter Attribute: The ServiceNow field used to determine if a manager record qualifies for update. For example, selecting user_name allows filtering on values, such as  john.doe or mgr_jane.
      • Update Filter Condition: Defines how to compare the selected field with the filter value. For example, using Contains with value admin updates records, such as  admin_tom, it_admin, or hr_admin.
      • Update Filter Value: The specific value that must match (as per the chosen condition) for the update to occur. For example, using value team with condition Ends With will update records, such as  sales_team or hr_team.
  • Limit Manager Data Executions: Controls the number of manager updates that can be made in a single sync run. For example, limit the sync to 10 updates to prevent large-scale changes during initial testing.
  • Set Update Limit: Specifies the maximum number of manager records that can be updated during a single run. For example, if 20 managers qualify for updates and the limit is 5, only the first 5 are processed.
  • Manager Data Table Name: The ServiceNow table where manager data is stored and updated. For example, this could be sys_user or a custom user table, such as u_manager_profiles.
  • Manager Data Primary Key: The unique field in ServiceNow used to match manager records during sync. For example, using user_name ensures that the correct ServiceNow user aligns with Oracle Cloud HCM manager data.
  • Manager Data Fields to Expand: Lists nested fields from Oracle Cloud HCM to retrieve during the sync. For example, expanding fields like workRelationships, assignments, managers, or emails ensures detailed manager data is fetched and processed.
  • Fetch Selected Manager Data Fields: Restricts the sync to fetch only the selected fields from Oracle Cloud HCM. For example, selecting fields such as PersonId and emails.EmailAddress, or assignments.PrimaryFlag ensures only relevant data is fetched and improves sync performance.

 

Attribute Mapping

The Attribute Map tab allows users to configure how data from Oracle Cloud HCM is transferred into ServiceNow by defining attribute-level mappings.

The target attributes from ServiceNow are fixed and pre-defined, ensuring consistency with the destination schema. However, you have full control over the source application attributes.

You can enter custom expressions or field paths from Oracle Cloud HCM that determine what data will be mapped. Each row in the mapping table represents a one-way data flow from a user-defined Oracle Cloud HCM expression to a specific ServiceNow attribute. Refer to the Attribute tab for complete mapping. 

You can toggle the view to JSON for advanced users who prefer editing mappings in code format.
