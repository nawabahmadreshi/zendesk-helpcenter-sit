Overview

The Aquera platform enables your organization to scale complex data management by automating data integration across various resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It enables organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately enhancing efficiency in managing complex IT infrastructures and applications.

This guide covers the end-to-end tasks involved in setting up the ADP Workforce Now / HaloPSA integration to create onboarding and offboarding tickets in HaloPSA based on employee onboarding and offboarding in ADP Workforce Now.

This integration is triggered when the Aquera User Integration retrieves a newly onboarded / offboarded employee from ADP Workforce Now and then successfully creates/deletes that user in a target system (for example, Active Directory). The ADP Workforce Now / HaloPSA integration must be selected in the New User Integration Hook and the Deprovision User Integration Hook configuration fields. When the user integration runs, the ADP Workforce Now / HaloPSA integration is triggered for every successful creation or deletion of a user in the target system configured in the user integration.

Key Functions

The ADP Workforce Now / HaloPSA integration performs the following functions:

  • Syncs worker and candidate (pre-hire) records from ADP Workforce Now into the Okta custom identity application.
  • Updates the email ID of workers from Okta to the ADP Workforce Now application (Writeback).
  • Creates new custom attributes in Okta.
  • Allows records to be synchronized based on the specified date offset and time zone.
  • Deletes the user records upon deactivation in Okta.
  • Automatically generates unique usernames and email IDs in Okta.

Integration Setup

This section details the configuration and mapping process for integrating ADP Workforce Now with Okta. It outlines the steps to create the integration, set up integration preferences, and define how worker data is mapped from ADP to Okta.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Select ADP Workforce Now / Okta.
  5. Enter a name and a description for the integration.
  6. Select your ADP Workforce Now and Okta connector instances.
  7. Click Continue. The Configuration page is displayed to configure the integration.

Configuration

On the Configuration page, the following sections are displayed:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Parameters

The Parameters section enables you to customize the integration features and general settings according to your specific use case.

Integration Features

Enable the following features based on your requirements:

  • Enable Worker Management: Select this checkbox to sync worker records from ADP into the Okta custom identity application. When enabled, the Worker Management section is displayed to configure integration settings for worker management.
  • Enable Candidate Management: Select this checkbox to sync candidate records from ADP into the Okta custom identity application. When enabled, the Candidate Management section is displayed to configure integration settings for candidate management.
  • Link Manager in Okta: Select this checkbox to link the manager custom object in Okta.
  • Enable Okta Email Writeback: Select this checkbox to enable updating the email ID of workers from Okta to the ADP Workforce Now application. When enabled, the Okta Writeback section is displayed for further configuration.
  • Enable Auto-Creation of Custom Attributes in Okta: Select this checkbox to authorize the creation of new custom attributes in Okta. When enabled, the Okta Attributes section is displayed to configure the custom attributes to be created in Okta.
  • Enable Monitored Provisioning: Select this checkbox to enable the monitored provisioning. When enabled, the Monitored Provision section is displayed for further configuration.
  • Enable Date Offset: Select this checkbox to enable date offset, allowing records to be synchronized based on the specified date offset and time zone. When enabled, the Date Offset section is displayed for configuration.
  • Enable Deletion of User when Deactivated in Okta: Select this checkbox to delete the user record upon deactivation in Okta. When enabled, the Okta Deletions section is displayed to configure the deletion policy.
  • Generate Unique Username/Email: Select this checkbox to automatically generate unique usernames and email IDs in Okta. When enabled, the Unique Identity section is displayed for further configuration.
  • Enable Okta Group Management: Select this checkbox to manage group assignments in Okta. When enabled, the Group Rules tab is displayed for further configuration.

General Settings

Configure the following settings:

  • Enable Report-Only Mode: Select this checkbox to execute the integration in the report-only mode. The report-only mode generates a report to show the data that would be changed without actually making those changes.
  • Enable Debug Mode: Select this checkbox to generate logs outlining data changes that would occur, for debugging purposes. When enabled, the Debug Mode section is displayed for further configuration.
  • Enable Logging: Select this checkbox to generate logs outlining data changes. When enabled, the Log Settings section is displayed for further configuration.
  • Enable Report Generation: Select this checkbox to generate the integration report. When enabled, the Report Settings section is displayed for further configuration.
  • Enable Notifications: Select this checkbox to enable integration-related notifications. When enabled, the Notification Settings section is displayed for further configuration.
  • Enable Log Insights: Select this to visualize the integration-related metrics. When enabled, the Log Insights Settings section is displayed for further configuration.

Worker Management

The Worker Management section enables you to customize the integration settings for worker management according to your specific use case.
Note: This section appears only if you have selected the Enable Worker Management setting in the Parameters section.

It has the following settings:

  • User to be Created/Updated in Okta*: Select the ADP Workforce Now workers whose records will be created or updated in Okta. The following are the available options:
    • All Workers: Select this to consider all workers from ADP to be created or updated in Okta.
    • All Active Workers: Select this to consider all active workers to be created or updated in Okta.
    • All Active Full Time Workers: Select this to consider all active full-time workers to be created or updated in Okta.
    • Custom: Select this to specify a custom condition for identifying the workers from ADP to be created or updated in Okta.

  • Custom Upsert Condition*: Provide a JavaScript expression to identify the workers to be created or updated in Okta.
    Note: This parameter is available only if you select Custom in the User to be Created/Updated in Okta parameter.
    Examples:

    • Upsert workers whose first name is "Bob"
    • person__legalName__givenName === "Bob"
    • Upsert workers whose first name starts with "A"
    • person__legalName__givenName__startsWith("A")
    • Upsert workers whose original hire date is today or in the future
    • new Date(workerDates?__originalHireDate) >= new Date()

Monitored Provision

This section allows you to configure the settings for monitored provisioning. It has the following settings:
Note: This section appears only if you have selected the Enable Monitored Provisioning in the Parameters section.

  • Monitor Email Addresses*: Enter the email IDs of the users who can monitor provisioning or deprovisioning.
  • Enable Monitored Provisioning for Creates: Select this checkbox to monitor all create operations performed during provisioning. If enabled, monitoring will be triggered whenever a new record is created.
    When enabled, the following field is displayed for further configuration:
    • Monitored Provisioning Create User Threshold*: Enter the minimum number of users that must be queued for creation in a provisioning run for the system to consider it as monitored provisioning. If the number of users to be created is greater than this threshold, the provisioning run will be considered as a monitored run. For example, if you set the threshold to 50, any provisioning run that attempts to create more than 50 users will be treated as a monitored provisioning run.
  • Enable Monitored Provisioning for Updates: Select this checkbox to monitor all update operations performed during provisioning. If enabled, monitoring will be triggered whenever a record is updated.
    When enabled, the following field is displayed for further configuration:
    • Monitored Provisioning Update User Threshold*: Enter the minimum number of users that must be queued for updation in a provisioning run for the system to consider it as monitored provisioning. If the number of users to be updated is greater than this threshold, the provisioning run will be considered as a monitored run. For example, if you set the threshold to 50, any provisioning run that attempts to update more than 50 users will be treated as a monitored provisioning run.
  • Enable Monitored Provisioning for Attribute level Updates: Select this checkbox to monitor all attribute-level updates during provisioning. When enabled, the following fields are displayed for further configuration:
    • Monitored Provisioning Attribute List*: Enter the user attributes that must be evaluated for attribute-level monitored provisioning. For each attribute you select, if the number of users with updates to that attribute exceeds the number configured in the Monitored Provisioning Attribute Update Threshold field, those users will be included in monitored provisioning. Attributes that are not selected, or whose update counts remain below the threshold, will be processed through the standard provisioning flow.
    • Monitored Provisioning Attribute Update Threshold*: Enter the minimum number of users that must have updates to a specific monitored attribute for monitored provisioning to apply at the attribute level. If the number of users to be updated is greater than this threshold, the provisioning run will be considered as a monitored run. For example, if you set the threshold to 50, any provisioning run that attempts to update more than 50 users will be treated as a monitored provisioning run.
  • Enable Monitored Provisioning for Offboarding: Select this checkbox to monitor all delete operations performed during provisioning. If enabled, monitoring will be triggered whenever a record is deleted.
    When enabled, the following field is displayed for further configuration:
    • Monitored Provisioning Deletion User Threshold*: Enter the minimum number of users that must be queued for deletion in a provisioning run for the system to consider it as monitored provisioning. If the number of users to be deleted is greater than this threshold, the provisioning run will be considered as a monitored run. For example, if you set the threshold to 50, any provisioning run that attempts to delete more than 50 users will be treated as a monitored provisioning run.
  • Enable Monitored Provisioning for Email Writebacks: Select this checkbox to monitor all email writeback requests. If enabled, monitoring will be triggered for each email writeback request.
    When enabled, the following field is displayed for further configuration:
    • Monitored Provisioning Email Writeback Threshold*: Enter the minimum number of users that must be queued with email writeback requests in a provisioning run for the system to consider it as monitored provisioning. If the number of users whose email addresses are to be updated is greater than this threshold, the provisioning run will be considered as a monitored run. For example, if you set the threshold to 50, any provisioning run that attempts to writeback emails for more than 50 users will be treated as a monitored provisioning run.

Attribute Map

The Attribute Map tab defines how data is mapped from the source to the target application. The default mapping of attributes is displayed on the page. The source attributes are listed on the left side and the target attributes are listed on the right side of the page.

You can modify only the source attributes as needed, to align with your specific use case.

You can write JavaScript expressions to modify the value of the source attribute and construct a value to be updated in the target attribute. Write the expression directly in the source attribute field or click the Expand icon in the source attribute field to write a long-expression.

All attributes marked with a red asterisk (*) indicates required fields for successful provisioning.

After the source-to-target attribute mapping is completed, click Save Changes at the top-right corner of the page to save the integration.

Mapping Custom Attributes

Click the refresh icon next to the target application name to fetch custom attributes available in Okta under the target application column for mapping.

Additionally, clicking the refresh icon populates the attributes specified in the Custom Attributes to be Created in Okta parameter (in the Okta Attributes section). This allows you to map ADP Workforce Now attributes to the custom attributes that will be created in Okta.

The new custom attributes will be automatically created in Okta during the next integration run, and their data will be synchronized in subsequent runs.

Group Rules

The Group Rules tab allows you to configure how group assignments are managed through this integration. In this tab, you can define conditions that determine how users are assigned to groups in Okta.

Note: This section appears only if you have selected the Enable Okta Group Management checkbox in the Integration Features section.

Perform the following steps to create a group rule:

  1. Click Add New Rule. The following fields are displayed:
  2. Click Choose Attribute to view the list of available attributes that you can use to set conditions.
  3. Search and select the attribute you want to use to define a condition. For example, select Title.
  4. Select the attribute value to define the rule. If multiple values are provided, an OR condition is applied among them. For example, in the following sample condition, a user's title must be either Consultant or Director to satisfy this assignment rule.
  5. In the Okta Groups column, select the groups to which users who satisfy the specified condition should be assigned.
  6. If required to add another condition and assignment for the same selected attribute, click Add Row. You can add any number of rows within a rule.
  7. If required to include another attribute to the group assignment rule, click the + icon next to the selected attribute name. If multiple columns are added for a rule, the AND logic is applied between different columns. For example, in the following sample condition, a user's Title must be either Consultant or Director, and the UserType must be Employee to satisfy this assignment rule. You can use multiple attributes and their corresponding values to define the group assignment rule.
  8. [Optional] If you want to write a JavaScript expression to set the condition, click Switch to Expression.
    Note: When you switch to Expression mode or back to Condition Builder mode, all configured conditions in the rule will be cleared.
    1. Select one of the following based on your requirement:
      • For Conditions Only: Select this if you want to write a JavaScript expression only for the conditions.
      • For Conditions and Assignments: Select this if you want to write expression for both, conditions and assignments.
    2. Click Continue to configure the expression.

    3. Enter the expression based on your selection:

      • If For Conditions Only is selected, enter the expression for the conditions, and then select the group in the Group Name column.
      • If For Conditions and Assignments is selected, enter the expression that defines both the conditions and the group assignment.

      Note: Expressions can use attributes from both the source and destination applications. The configured expressions must always be evaluated to true or false. The prefix SOURCE must be used in an expression to represent an attribute in the source application, and DEST must be used to represent an attribute in the destination application. For example, SOURCE.addresses[].locality, SOURCE.addresses[].country, and DEST.group.displayName.

  9. Click Add New Rule to add another assignment rule, as explained above.

Notes:

  • You can add any number of assignment rules. However, you can see only 5 rules on a single page. If you add more than 5 rules, navigation options will be available at the bottom of the page to navigate to other pages for configuring or viewing the remaining defined rules.
  • You can delete a row by using the delete icon at the right end of the row.

Create Custom Attributes in Okta

You can create custom fields in Okta to include additional ADP Workforce Now Worker attributes in the User object of Okta. You must configure all the custom attributes individually. After the configuration, Aquera automatically populates the fields that you have created in Okta from the Worker resource of ADP Workforce Now. 

Perform the following steps to create custom attributes in Okta:

  1. Sign in to your Okta instance.
  2. Navigate to Directory > Profile Editor and locate the custom identity source integration you've created.
  3. Click Add Attribute to add a new attribute.
  4. Configure the attribute using the same name as ADP Workforce Now in the Variable name field and fill in other fields. 

    Note: You must remove special characters in the name and replace periods with underscores.
    For example, the equivalent name for the attribute primaryAssignment.baseRemuneration.effectiveDate would be primaryAssignment_baseRemuneration_effectiveDate.
  5. Click Save to save the changes or Save and Add Another to create another custom attribute. 

Map Custom Attribute to Okta Profile

Perform the following steps to map the custom attributes to the Okta profile:

  1. Navigate to Mappings > Configure User mappings.
  2. Map the custom attribute to an equivalent attribute in the Okta user profile and click Save Mappings.
  3. If the equivalent attribute is not present in the default Okta user profile, open the default profile from Directory > Profile Editor.

     
  4. Click Add Attribute to add a custom attribute to the Okta default user profile.
  5. Fill in the attribute's display name, description, and set the user permission. Click Save Attribute to save the changes.
  6. Map the custom identity source attribute to the attribute in the Okta default profile and click Save Mappings.

Note: You can repeat these steps to add all employee-related attributes in ADP Workforce Now to Okta.

FAQs

Question 1: Why is the future termination date of a user not being retrieved or synced correctly from ADP?
Answer: If a user’s termination date is set to a future date, the effective date must also be set to that same future termination date in order for the future termination date to appear in the response.
Additionally, GEP begins tracking future termination events only from the date it is enabled.
Termination events that occurred before GEP was configured are typically unavailable, except for those within the previous five days.
For example, if GEP is enabled on November 19, 2025, termination events from November 14, 2025 onward will be available. A user terminated on November 14, 2025, with a future termination date will appear. However, a user terminated before November 14, 2025, with a future termination date will not appear, as their future termination date cannot be retrieved.
