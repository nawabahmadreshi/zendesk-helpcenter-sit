SCIM Filter Expressions in Integrations

← Back to index

Overview

SCIM Filter Expressions are used in the Source SCIM Filter configuration of integrations. SCIM filter expressions follow the SCIM RFC specification.

This guide provides information on the operators that can be provided in the SCIM filter expressions and the functionality of each operator.

Note: You can use JavaScript expressions in all other configuration fields (such as Match Expression, Create Condition, and Update Condition), attribute mappings, tables, and groups when you set up integrations. All the JavaScript operators are supported in the JavaScript expressions of integrations.

SCIM Filter Expressions

Each SCIM Filter expression must contain an attribute name followed by an attribute operator and an attribute value. Multiple expressions may be combined using logical operators. Furthermore, expressions can be grouped using grouping operators. 

Note: The SCIM Filter expression will work only if the connector selected in the integration supports filtering by the specified attribute operators and attribute names. Refer to the Supported Filters section of the respective connector configuration guides for further details.

Attribute Operators

The following table lists the attribute operators that can be used in a SCIM filter expression:

Operator Description Behavior and Example
eq equal

The attribute and operator value must be identical to meet the filter criterion.

Core: userName eq "bjensen"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department eq "Sales"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID eq "Q7DETWGV"
ne not equal

The attribute and operator value must not be identical to meet the filter criterion.

Core: userName ne "bjensen"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department ne "Sales"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID ne "Q7DETWGV"
co contains

The entire operator value must be a substring of the attribute value for a match.

Core: userName co "@domain.com"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department co "Sale"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID co "Q7"
sw starts with

The entire operator value must be a substring matching at the beginning of the attribute value.

Core: userName sw "b"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department sw "S"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID sw "Q"
ew ends with

The entire operator value must be a substring matching at the end of the attribute value.

Core: userName ew "@domain.com"

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department ew "s"

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID ew "V"
pr present
(has value)

If the attribute has a non-empty or non-null value, or if it contains a non-empty node for complex attributes, there is a match.

Core: userName pr

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:department pr

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID pr
gt greater than

If the attribute value is greater than the operator value, there is a match. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTimetypes, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:employeeNumber gt 100

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID gt 100
ge greater than or equal to

If the attribute value is greater than or equal to the operator value, there is a match. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTimetypes, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:employeeNumber ge 100

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID ge 100
lt less than

If the attribute value is less than the operator value, there is a match. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTimetypes, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:employeeNumber lt 100

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID lt 100
le less than or equal to

If the attribute value is less than or equal to the operator value, there is a match. The actual comparison is dependent on the attribute type.

For string attribute types, this is a lexicographical comparison, and for DateTimetypes, it is a chronological comparison. For integer attributes, it is a comparison by numeric value. This operator cannot be used for Boolean and Binary attributes.

Enterprise: urn:ietf:params:scim:schemas:extension:enterprise:2.0:User:employeeNumber le 100

IAN: urn:ietf:params:scim:schemas:ian:2.0:User:employeeID le 100

Logical Operators

The logical operators are used to combine multiple expressions. The following table lists the logical operators that can be used in a SCIM filter expression:

Operator Description Behavior and Example
and Logical AND

The filter is a match only if both the expressions (criteria) are met.

userType ne "Guest" and urn:ietf:params:scim:schemas:ian:2.0:User:employeeID pr
or Logical OR

The filter is a match if either of the expressions (criteria) is met.

userType ne "Guest" or urn:ietf:params:scim:schemas:ian:2.0:User:employeeID gt 100
not Logical NOT

The filter is a match if the expression evaluates to false.

userType eq "Admin" and not (userName co "example.com" or urn:ietf:params:scim:schemas:ian:2.0:User:employeeID le 100)

Note: not takes precedence over and, which takes precedence over or.

Grouping Operators

The following table lists the grouping operators that can be used in a SCIM filter expression:

Operator Description Behavior and Example
() Precedence Grouping

Boolean expressions may be grouped using parentheses to change the standard order of operations, that is, to evaluate logical or operators before logical and operators.

userType eq "Admin" and not (userName co "example.com" or urn:ietf:params:scim:schemas:ian:2.0:User:employeeID le 100)
[] Complex attribute filter grouping

Complex filters are supported where expressions must be applied to the same value of a parent attribute specified immediately before the left square bracket ("["). The expression within square brackets ("[" and "]") must be a valid filter expression based upon sub-attributes of the parent attribute. Nested expressions may be used.

emails[type eq "work" and value co "@example.com"] or ims[type eq "xmpp" and value co "@foo.com"]

Using Data Values in SCIM Filter Expressions

The SCIM filter expression can include optional values. The following table shows how values of different data types can be used in a SCIM filter expression. 

Type of Data Usage of Value in Filter Expressions
Text Use the value within double quotes. For example, "Barbara".
Number Use the value without quotes. For example, 100.
Date Time Use the value within double quotes. For example, "2011-05-13T04:42:34Z".
Boolean Use true or false without any quotes. For example, true.
Null Use null without quotes.
