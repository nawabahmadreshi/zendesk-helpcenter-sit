Overview

This guide covers information on how to set up event orchestration on the Aquera Admin portal. 

Event-based orchestrations run based on the data change in an application such as new user creation, existing user modification, user activation, user addition to a group, user removal from a group, and so on. Event orchestrations require configuration in the application and are initiated by the application outside of the Aquera platform. Whenever an event occurs in the application, the orchestration is triggered.

The following are the 2 types of event orchestrations:

  1. Event Synchronization orchestration: Allows you to configure the required fields to synchronize data from the source application to the target application.
  2. Event Script orchestration: Allows you to write a script to perform the required tasks on single or multiple applications. 

Prerequisites

Ensure that the required applications are added and configured correctly in the Aquera Admin portal before creating the orchestration. Refer to the corresponding configuration guides for details.

Creating the Orchestration

Perform the following steps to create the orchestration:

  1. Navigate to Orchestrations on the Aquera Admin portal.
  2. Click Create Orchestration.
  3. Enter the following details on the Add Orchestration pop-up window:
    • Enter a name for the orchestration.
    • Select the required Event orchestration from the drop-down list.
  4. Click Done. Configure the required fields in the orchestration.
  5. Click Create to create the orchestration.

Setting Up an Event Synchronization Orchestration

Configuration

Enter the following details on the Configuration tab, as required:

Note: The configuration fields vary depending on the type of event orchestration selected.

Basic Details

This section explains the fields to configure the basic details.

Field Description

Source Application

Select the source application from where the data of users must be retrieved. The data of created and modified users are retrieved from this application as part of the orchestration.

Target Application

Select the target application in which the data of created and modified users must be updated as part of the orchestration.

Report Only

Select this checkbox to generate a report that shows the data that would be changed, without actually making those changes. This can be selected for the first instance of orchestration to identify any issues. You can clear this selection after verifying the report and confirming that the result is as expected.

Admin Email List

The administrative email addresses to which notifications on this orchestration have to be sent. Multiple email addresses must be separated by a comma.

Match Expression

The expression that must be used to identify the target record that matches a source record during user imports. This expression can be given only if the target connector supports GET operation. 

The default value is SOURCE[ENT].employeeNumber == DEST[ENT].employeeNumber 
This compares the employeeNumber of the source application with the employeeNumber of the destination application.

If the orchestration does not find a match, the source record will be considered as a new user and the user will be created in the target. If that source record is not a new user, then the expression needs to be modified. You must run the orchestration in report mode to verify this.

Notes:

  • The match expression must be a simple one, with SOURCE on one side and DEST on the other side of the expression.
  • Orchestration will attempt to build a SCIM filter based on the match expression provided by you. If you provide a complex match expression, the expression may not work as only a few connectors can handle complex filter expressions.
  • It is not possible to build a SCIM filter if a function is applied to a DEST attribute. 
    For example, a SCIM filter can be built from SOURCE[ENT].employeeNumber.toLowerCase() == DEST[ENT].employeeNumber. 
    However, SOURCE[ENT].employeeNumber == DEST[ENT].employeeNumber.toLowerCase() will not work as a function is applied to the DEST attribute.
  • It is not possible to build a SCIM filter if the match expression has a parenthesis.

create_condition

The expression that is used to determine whether a user has to be created in the target application. The following are the valid values:

  • SOURCE.active or SOURCE.active==true: Creates users in the target application whose active attribute is set to true in the source application. 
  • true: Creates users in the target application whose active attribute is set to true or false in the source application. 
    Note: To create users whose active attribute value is false, the target application must support the creation of inactive users. 
  • false: Does not create any users in the target application.

update_condition

The expression that is used to determine whether a user has to be updated in the target system. The following are the valid values:

  • SOURCE.active: Updates users in the target application whose active attribute is set to true in the source application. 
  • true: Updates users in the target application whose active attribute is set to true or false in the source application.
  • false: Does not update any users in the target application.
  • SOURCE.active == false: Deactivates users in the target application whose active attribute is set to false in the source application.

Map Manager

Select this checkbox to map users' managers from source to target. The enterprise manager attributes in the user objects are used for mapping.
Note: If the Map Manager checkbox is selected and the manager is deactivated in the source application, then the manager attributes of the users who are mapped to this deactivated manager will be updated as undefined in the target application.

Creation Email Details

This section explains the fields to configure the creation email details. 

Field Description

Send New User's Manager Email

Select this checkbox to send the e-mail notification to the email ID of the user's manager in the source system, when the user is created. This option works only if an email ID of type work is present for the user's manager.

Creation Distribution List

 The email addresses to which an e-mail notification has to be sent when a new user is created in the target application as part of the orchestration. Multiple e-mail addresses must be separated by a comma.

Creation Email Subject

The subject line to be used in the e-mail notification that will be sent when a user is created in the target application.

For example, New Employee {{userName}}

Creation Email Body

The template that has to be used for the body of the notification e-mail, which will be sent when a new user is created in the target application.

To include the source attributes in the e-mail body, the following is a sample template:
name: {{SOURCE.name.givenName}} {{SOURCE.name.familyName}}
email: {{SOURCE.emails[work].value}}
username: {{SOURCE.userName}}
password: {{SOURCE.password}}
address: {{SOURCE.addresses[work].streetAddress}}
manager: {(SOURCE[ENT].manager.displayName}}
department: {{SOURCE[IAN].userDepartment}}

To include the target attributes in the e-mail body, the following is a sample template:
name: {{name.givenName}} {{name.familyName}}
email: {{emails[work].value}}
username: {{userName}}
password: {{password}}
address: {{addresses[work].streetAddress}}
manager: {([ENT].manager.displayName}}
department: {{[IAN].userDepartment}}

Notes:

  • If no attribute mapping is set in the orchestration for any source or target attributes configured in the e-mail body, then the e-mail body will have the value of those attributes as undefined.
  • ENT stands for Enterprise. IAN stands for Integrated Aquera Network.

Advanced Details

This section explains the fields to configure the advanced details. 

Field Description

Enable Group Management

Select this checkbox to allow the management of user groups in the target application. Group management can be done by setting group rules or through attribute mapping. 

Group Management Restriction List

Enter the names of the target user groups to which the group management rules must be applied.

Notes:

1. Multiple groups must be separated by a comma.
For example, Group1,Group-2,Group 3,Group$4.
2. It is recommended to enclose the group name within double quotes if there is a comma in the group name.
For example, "Group,5"
3. Any groups having double quotes in their names are not handled by the orchestration. 
For example, a group name like "Group"6Aquera" is not handled by the orchestration.
4. Group names should not have any leading spaces. 
For example, a group name like " Group7" is not handled by the orchestration.

Phone Number Formatting

The standard format to which users' phone numbers must be converted during the orchestration process. The valid values are:

  • disabled: By default, the conversion of phone numbers to a specific format is disabled.
  • automatic: 
  • E164: The E.164 international telephone numbering format is + <country code> <subscriber number including area code>.
  • INTERNATIONAL: International format
  • NATIONAL,country=US: The format that is being followed in a nation. You can replace US with the appropriate country code.

Notes: The automatic formatting does not work when you write to the ADP application.

To set the international phone number format of +1-123-4567890 during the orchestration, set the field value as INTERNATIONAL,template=+.-...-.......
To set the phone number format of +1 1234567890 during the orchestration, set the field value as INTERNATIONAL,template=+. ..........

Password Length

The length of the random password that will be created when a user is created in the target application. 

Note: There is no minimum or maximum limit on password length. If the user specifies a zero value, an empty string password will be generated. 

Report Logging

The logging mode of orchestration details. This setting controls the level of details to be captured in the orchestration logs. 

The valid values are: minimal, normal, debug, and verbose.

The default value is normal. 

Enter minimal or normal mode to get the basic orchestration details. Enter debug or verbose mode to get a detailed analysis of what is happening in the orchestration.

Mapping of Attributes

You can map the source SCIM attributes to the target SCIM attributes by navigating to the Attribute Map tab. This is required to be performed to synchronize the data from the source application to the target application. You can also write expressions to modify the value of the source attribute before updating the value in the target attribute.

For details, refer to the Building Orchestrations - Attribute Mapping guide.

Adding and Using Tables in Orchestration

Tables define a name-value pair to create custom attributes. Tables are added in orchestrations to create custom source attributes, and POST (sync) them to the target application. Tables are used when an application attribute is empty or particular data is not sent from an application, and you want to update a value for that attribute in another application.

For details on how to add tables and use them in orchestrations, refer to the Building Orchestrations - Tables guide.

Setting Up an Event Script Orchestration

Configuration

Enter the following details on the Configuration tab, as required:

  • Click Add Application and in the Applications drop-down, select the application of which you want to process the events through the orchestration.
  • In the Variable field, enter a variable name that you want to use in the script to refer to the selected application.
    Notes: The variable name should be unique within the orchestration. It can include a combination of letters and numbers. It should always start with a letter. It should not have only numbers. It should not be a JavaScript keyword. It should not include any spaces or special characters. 
  • In the script editor area, enter the JavaScript to be executed for processing the events received from the application. Click the Expand icon, if necessary, to expand the window for entering the JavaScript.

View the Event Detail

Events can be enabled for the applications that support asynchronous notification of specific data changes. When there is a data change in an application, the corresponding event is sent as an XML message to the Aquera orchestration service and the orchestration will update the change in the configured application. 
Perform the following steps to view the event detail:
  1. Navigate to the Event Detail tab in the event orchestration.
  2. Copy the URL. This URL is the location of the Aquera service to which the XML messages sent from the application will be posted. The copied URL must be pasted into the respective application when you configure the web hook/event hook/event listener. 
  3. Click Create Token. A token is generated.
  4. Copy the token. Paste this token into the respective application when you configure the web hook/event hook/event listener.
