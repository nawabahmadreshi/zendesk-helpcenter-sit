Integration Setup

This section outlines how the integration must be configured to push <resource type> data from <source application> to <target application>.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Search for Active Directory / ServiceNow in the search box next to the Explore Templates section.
  5. Click the integration template to display the Create New Integration pop-up window.
  6. Enter a name and a description for the integration.
  7. Select both application connector instances from the dropdown menus.
  8. Click Continue. The integration configuration page is displayed to configure the integration.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Parameters

The Parameters section enables you to customize the integration settings according to your specific use case. It has the following settings:

Integration Features

Enable the following features based on your requirements:

  • Enable user lifecycle management: Select this checkbox to synchronize user data from ServiceNow to Active Directory. When enabled, the Manage Users section will be displayed, allowing you to configure additional integration settings and automate the creation, updating, or deactivation of users in Active Directory based on ServiceNow tickets.
  • Enable ServiceNow writeback: Select this checkbox to enable writeback of user records from Active Directory to ServiceNow. When enabled, the system can create, update, or deactivate user records in ServiceNow based on changes made in Active Directory. In this configuration, Active Directory acts as the source application, and ServiceNow acts as the target application.

General Settings

Configure the following general settings:

  • Enable report-only mode: Select this checkbox to execute the integration in the report-only mode. The report-only mode generates a report to show the data that would be changed without actually making those changes.
  • Enable logging: Select this checkbox to generate logs outlining data changes. When enabled, the Logging section is displayed for further configuration.
  • Enable report generation: Select this checkbox to generate the integration report. When enabled, the Reporting section is displayed for further configuration.
  • Enable notifications: Select this checkbox to enable the integration-related notifications. When enabled, the Notifications section is displayed for further configuration.
  • Enable log insights: Select this checkbox to visualize the integration-related metrics. When enabled, the Log Insights section is displayed for further configuration.

Manage Users

The Manage Users section allows you to configure user integration settings and automate the creation, updating, or deactivation of users in Active Directory based on ServiceNow tickets. It has the following settings:

Correlation Settings

Configure the following correlation field settings:

  • ServiceNow correlation attribute: Select the corresponding unique identifier attribute from ServiceNow that will be used for correlation and user record matching with Active Directory. For example, select Employee number as the correlation attribute in ServiceNow.
  • AD correlation attribute: Select the unique identifier attribute from Active Directory that will be used for correlation and user record matching with ServiceNow. For example, select employeeId as the correlation attribute in Active Directory.

User Provisioning Settings

Configure the following user provisioning settings:

  • Enable user creation: Select this checkbox to create new user records in Active Directory based on requests submitted through the ServiceNow catalog. When enabled, the User Creation Settings section appears for further configuration.
  • Enable user updation: Select this checkbox to update existing user records in Active Directory based on requests submitted through the ServiceNow catalog. When enabled, the User Updation Settings section appears for further configuration.
  • Enable user deactivation: Select this checkbox to deactivate user records in Active Directory based on requests submitted through the ServiceNow catalog. When enabled, the User Deactivation Settings section appears for further configuration.
  • Enable manager mapping: Select this checkbox to map the user's manager. When enabled, the following field will appear for further configuration.
    • Select variable for user manager: Select the ServiceNow catalog variable that stores the user’s manager information.
  • Enable process future hire: Select this checkbox to enable processing users who have a future hire date. When enabled, such users will be created in Active Directory in a disabled state. These accounts can be automatically activated prior to the user's start date based on your configuration. When this option is enabled, additional fields will appear for further configuration.
    • Select variable for user activation date: Select the ServiceNow catalog variable that stores the user’s start date.
    • Activate user before X days from start date: Enter the number of days prior to the user’s start date when the account should be automatically activated in Active Directory. For example, if you enter 3, the integration will activate the user account three days before the configured start date.
  • Enable inactive user scan: Select this checkbox to scan Active Directory for users who have not logged in for a specified number of days. When enabled, the following fields will appear for further configuration.
    • Days since last login: Enter the number of days to look back from the current date to identify Active Directory users who have not logged in. For example, if you enter 90, the integration will identify users who have not logged in within the past 90 days.
    • Select the service catalog for inactive user ticket creation: Choose the service catalog that will be used to create ServiceNow tickets for Active Directory inactive users. When inactive accounts are identified, a ticket will be auto-created under the selected service catalog and assigned to the user’s manager for action. For example, select User Action Requests if you want inactive user tickets to be created in that catalog. 
      Note: The ticket will be auto-created when a user exceeds the threshold specified in Days since last login.
    • Select variable for user from inactive catalog: Choose the ServiceNow catalog variable that identifies the inactive user. For example, select Request For, which stores the inactive user record for which the manager must take action in ServiceNow.
  • Enable inactive user processing: Select this checkbox when a user's manager processes an action ticket in ServiceNow that was created for an identified inactive user. When enabled, the following fields will appear for further configuration.
    • Service catalog: Select the ServiceNow service catalog used for manager user action requests. For example, select User Action Requests to process only user action request tickets.
    • Select variable for user from inactive catalog: Choose the ServiceNow catalog variable that identifies the inactive user. For example, select Request For, which stores the inactive user record for which the manager must take action in ServiceNow.
    • Select variable for action from inactive catalog: Choose the ServiceNow catalog variable that captures the action to be taken for the inactive user. For example, select Action if that variable contains options such as Active, Deactive, or Delete
      Note: The available action values are not predefined by the integration. They are dynamically pulled from the selected ServiceNow catalog variable. The options Active, Deactive, and Delete mentioned above are only examples. The actual values depend entirely on how the selected catalog variable is configured.

      • Select the status to activate the user: Select the status that will be applied to the inactive user account to activate in Active Directory when the manager updates the request in the catalog. The values displayed in this configuration correspond to the options defined in your selected catalog variable.
        Note: This field overrides the status selected by the manager in the action catalog and determines how the account will be updated in Active Directory.

        The following are the available options:

        • Active: Select this option to activate the inactive user account in Active Directory when the manager selects Active in the Action field in the User Action Requests catalog.
        • Deactive: Select this option to activate the inactive user account in Active Directory even when the manager selects Deactive in the Action field in the User Action Requests catalog. In this case, the integration will treat the request as active and update the account accordingly.
        • Delete: Select this option to activate the inactive user account in Active Directory even when the manager selects Delete in the Action field in the User Action Requests catalog. In this case, the integration will treat the request as active and update the account accordingly.

      • Select the status to deactivate the user: Select the status that will be applied to disable the inactive user account in Active Directory without deleting the record. The values displayed in this configuration correspond to the options defined in your selected catalog variable. Once the account is disabled, the user is moved to the designated disabled OU.
        Note: This field overrides the status selected by the manager in the action catalog and determines how the account will be updated in Active Directory.

        The following are the available options:

        • Active: Select this option to disable the inactive user account in Active Directory even when the manager selects Active in the Action field in the User Action Requests catalog. In this case, the integration will treat the request as deactivate and update the account accordingly.
        • Deactive: Select this option to disable the inactive user account in Active Directory when the manager selects Deactive in the Action field in the User Action Requests catalog.
        • Delete: Select this option to disable the inactive user account in Active Directory even when the manager selects delete in the Action field in the User Action Requests catalog. In this case, the integration will treat the request as deactivate and update the account accordingly.

      • Select the status to delete the user: Select the status that will be applied to permanently remove the inactive user account from Active Directory. The values displayed in this configuration correspond to the options defined in your selected catalog variable.
        Note: This field overrides the status selected by the manager in the action catalog and determines how the account will be updated in Active Directory.

        The following are the available options:

        • Active: Select this option to delete the inactive user account in Active Directory even when the manager selects Active in the Action field in the User Action Requests catalog. In this case, the integration will treat the request as a delete and update the account accordingly.
        • Deactive: Select this option to delete the inactive user account in Active Directory even when the manager selects Deactive in the Action field in the User Action Requests catalog. In this case, the integration will treat the request as a delete and update the account accordingly.
        • Delete: Select this option to delete the inactive user account in Active Directory when the manager selects delete in the Action field in the User Action Requests catalog.
    • Auto-deactivation grace period: Enter the number of days to wait for manager approval in ServiceNow before the integration automatically deactivates the user in Active Directory. For example, enter 7 to automatically deactivate the user if the manager does not respond within 7 days. Once the account is deactivated, the user is moved to the designated disabled OU.
    • Enable reactivation through ticket: Select this checkbox to allow inactive users to be reactivated through a ServiceNow ticket created for the user's manager in the User Action Requests service catalog. For example, two requests may be submitted for the same user. One request could be submitted through the Deactivation of User service catalog. Once this request is completed, the user is deactivated in Active Directory and moved to the disabled OU based on your OU Assignment Rules configuration. If the same user has not logged in to Active Directory for the number of days specified in the Days Since Last Login field, a ServiceNow ticket will be created under the User Action Request service catalog and assigned to the user’s manager to review the user’s status. If the manager updates the user’s status to Active, the user will be reactivated in Active Directory through the ticket and moved back to the active users OU.
  • Enable user deletion process: Select this checkbox to automatically delete users from Active Directory after a specified number of days in the disabled state. If the Enable ServiceNow writeback option is enabled, the corresponding user in ServiceNow will also be deactivated when the deletion occurs.
    • Delete disabled users who have been in the 'Disabled' OU for X days: Specify the number of days a user account must remain in the disabled Organizational Unit (OU) in Active Directory before it is deleted. For example, if you enter 30, the integration will delete user accounts that have been disabled in AD for 30 days or more.

The OU Assignment Rules tab allows you to configure how OU assignments for all active users and disabled users are managed through this integration. In this section, you can define the specific OUs that determine how users are assigned.

Attribute Map

The Attribute Map tab defines how data is mapped from the source to the target application. The default mapping of attributes is displayed on the page. The source attributes are listed on the left side, and the target attributes are listed on the right side of the page.

You can modify only the source attributes as needed to align with your specific use case.

You can write JavaScript expressions to modify the value of the source attribute and construct a value to be updated in the target attribute. Write the expression directly in the source attribute field or click the Expand icon in the source attribute field to write a long-expression.

All attributes marked with a red asterisk (*) indicate required fields for successful provisioning.

After the source-to-target attribute mapping is completed, click Save Changes at the top-right corner of the page to save the integration.

Update the Integration to the Latest Version

You can upgrade an integration built from a template to the latest version to access new features and enhancements. 

Perform the following steps to upgrade an integration:

  1. Navigate to the Integrations module on the Aquera Admin portal.
  2. Click the vertical ellipsis icon next to an integration and click View Details. The Details panel is displayed with the following sections:
    • INTEGRATION DETAILS: Displays the integration name, description, creation date, last update date, and the user who created or last updated it.
    • TEMPLATE DETAILS: Displays the current version number and its features under CURRENT VERSION. If a newer version is available, the version number and the changes are shown under NEW VERSION AVAILABLE with an Upgrade button.
  3. Click Upgrade. The Upgrade Integration pop-up window is displayed.
  4. Provide the following details for the new integration to be created after the upgrade:
    • New Integration Name: Enter the name of the new integration.
    • Description: Provide a description for the new integration.
      The current version number and the upgraded version number are displayed.
  5. Click Upgrade. A new integration is created with the same configuration details as the existing one, but with features or enhancements from the upgraded version.
