User Integration Setup Guide - ADP Workforce Now to Active Directory

← Back to index

Overview

Aquera provides a platform that allows you to scale complex data management by automating the integration of users from the ADP Workforce Now application with the Active Directory application.

This guide covers the end-to-end tasks involved in setting up a User Integration to sync users in ADP Workforce Now with Active Directory.

Prerequisites

The following are the prerequisites:

ADP Workforce Now

Perform the following steps for the ADP Workforce Now prerequisite:

  1. Log in to the ADP Marketplace as an ADP Admin.
  2. Purchase Aquera's connector for the ADP Workforce Now® edition from the ADP Marketplace. The connector to be selected from this page must be based on the number of employees (users).
  3. After you purchase the connector app on the ADP Marketplace, you must give appropriate permissions to the app so that the app will have authorized access to your organization's data. Aquera generates a token to make the connection.
    • To authorize Aquera for data access to your ADP instance, navigate to My Apps > Aquera in the Marketplace.
    • For details on the required permissions and how to set them up, refer to the following sections in the ADP Workforce Now Configuration Guide:
      • Customising Permissions in the Workforce Now Portal
      • ADP Workforce Now Permissions

The following screen is displayed in the ADP Marketplace when the authorization is successful:

Tenant Account in the Aquera Admin Portal

You must have a tenant account with admin access in the Aquera Admin portal. Visit https://admin.aquera.io and register to get a tenant account. Contact your Aquera account manager to get the Aquera tenant access. 

Active Directory

Determine the architecture to be followed for setting up the communication between Aquera and the Windows Active Directory Domain Controller. Refer to this Aquera guide for detailed information on the different architectural approaches.

Based on the selected approach, the following procedures may have to be performed:

  • A Windows Server must be ready to deploy the Aquera Agent onto the server where the Domain Controller is present. For instructions on installing the Aquera Agent, refer to the Aquera Agent Installation and Configuration Guide.
  • A firewall must be set up to allow communication with Aquera.
  • The static IP address of Aquera needs to be whitelisted. Contact the Aquera Support team.

To allow Aquera to make updates to Active Directory (for creating, updating, and deleting users), LDAPS communication to Active Directory needs to be set up, which requires a TLS connection. Active Directory Certificate Services must be enabled for this. To install Active Directory Certificate Services, visit the Microsoft website.

Setting Up Aquera Integration

Aquera's integration provides the automatic integration of users from the ADP Workforce Now application to the Active Directory application. To perform this integration, you must set up a User Integration in the Aquera Admin portal.

The following steps are involved in the User Integration setup:

  • Creating Integration
  • Configuring Integration
  • Mapping of Attributes
  • Adding and Using Tables in Integration
  • Scheduling Integration Run

Considerations

The following are the important points to be considered before setting up the User Integration:

  • Define the naming convention format for the User Principal Name (UPN) in Active Directory. This must be mapped to the Active Directory's userName attribute in the Attribute Map. Ensure to provide the userName with a length of less than 20 characters due to the Active Directory limitation for the userName. Otherwise, the user account will not be created in Active Directory.
  • Define the naming convention format for the SAM Account Name in Active Directory. This must be mapped to the Active Directory's IAN attribute sAMAccountName in the Attribute Map.
  • Decide the fields to be populated in the Active Directory. To populate these fields in Active Directory, you must provide attribute maps for the corresponding user attributes of Active Directory in the Attribute Map. 
  • Decide whether the work email is to be retrieved from the ADP Workforce Now application or be generated by the integration to update in Active Directory. This must be mapped to the Active Directory's attribute emails[work].value in the Attribute Map.
  • Define the matching expression to match user records from ADP Workforce Now with the user records in Active Directory. This must be provided as the Match Expression in the Configuration.
  • Define the Organizational Unit (OU) where the users will be created in Active Directory. This must be mapped to the Active Directory's IAN attribute distinguishedName in the Attribute Map.

General Analysis

The following analyses need to be performed before setting up the User Integration:

  • Fill out the Mapping Template attached with this guide for mapping ADP Workforce Now user attributes to Active Directory user attributes. The most common attributes are mapped in the attached template. You can add more attribute maps to the sheet as per your requirement. This sheet can be used as a reference when doing the Attribute Map in the User Integration.
  • JavaScript expressions can be used in the Attribute Map to create User Principal Names and SAM Account Names without duplicates or errors. In the expressions, you can use inbuilt methods such as toNormalizedDiacritics() and toSAMAccountName() for this purpose. You can also use keywords such as %ATTEMPT and %USEQ to avoid duplicates. Refer to the Integration Expressions - Keywords, Methods, and Functions Guide for details.
  • Analyze the existing users in ADP Workforce Now and Active Directory before setting up the integration. You can also analyze the existing groups of users in Active Directory. Refer to the Appendix for details.

Creating a User Integration

Perform the following steps to create a user integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations and click Create Integration. 
  3. Select User integration from the list of integration types.
  4. Enter a name for the integration in the Create New Integration pop-up window.
  5. Click Continue. A new page with the fields and tabs to configure the integration is displayed.
  6. Select the ADP Workforce Now connector as the Source Application.
  7. Select the Active Directory connector as the Target Application.
  8. Click Create to create the integration.

Integration Scenarios

You may need to update the configurations or the attribute mappings for the following scenarios:

Creating a Normal Account for the User in Active Directory

To create a user of type NORMAL_ACCOUNT in Active Directory, set the attributes mapping as follows:

Enter 512 as the source attribute expression and map it to the [IAN].userAccountControl attribute of AD. Set the mapping criterion as Map on Create.

Note: 512 is the decimal value corresponding to the NORMAL_ACCOUNT property flag in Active Directory. For all property flags and the corresponding decimal values, refer to the Microsoft documentation.

Escaping Special Characters in OUs and DNs in the Active Directory

When using the Active Directory connector prior to version 4, it is required to escape any special characters in OUs and DNs in the expressions by using a double-backslash before the special character. For example, if the OU name is #Terminated, mention it in the expression as \\#Terminated. 

DEST[IAN].distinguishedName.split(",")[0] + ",OU=\\#Terminated, DC=local"

Note: For Active Directory connector version 4 onwards, the connector handles escaping the special characters for OUs and DNs used in the integration expressions. Hence, when upgrading to the Active Directory connector version 4 or above, you need to remove any expressions set in the existing integration to escape the special characters.

Prompting for Password Change at First Login to Active Directory

To make it mandatory for the new users in Active Directory to change their password on their first login, map the Active Directory attribute pwdLastSet to 0 with the mapping option of Map on create only. If this mapping is not done and when users sign into their account for the first time, they will not be prompted for password change.

To prompt the existing users for a password reset, set the mapping option as Map on Create and Update.

Creating a Unique Email Address for the Active Directory User

The following is a sample expression that can be used in the attribute mapping to create a unique email ID for a user in Active Directory based on the user's name change in the ADP Workforce Now application.


let attempt = "%ATTEMPT" || 1; (attempt <= SOURCE.name.givenName.length) ? (SOURCE.name.givenName.substring(0,attempt) + SOURCE.name.familyName + "@example.com").replace(/[[]`{};:']/gm, "").toLowerCase().toEmail().toNormalizedDiacritics() : SOURCE.name.givenName + SOURCE.name.familyName + (attempt) + "@example.com").replace(/[[]`{};:']/gm, "").toLowerCase().toEmail().toNormalizedDiacritics()

Populating the Phone Number Value of the ADP User

The following expression can be used in the attribute mapping to populate a phone number value of the ADP Workforce Now user:

  • populates the home phone number value if the home phone number is available. 
  • populates the mobile-home phone number value if the home phone number is not available. 

SOURCE.phoneNumbers[home] && SOURCE.phoneNumbers[home].value || SOURCE.phoneNumbers[mobile-home].value;

Creating Users in Active Directory

 To create users through the Active Directory connector, use one of the following approaches:

  • Use a secure LDAPS connection with port 636:  A password must be provided to create a user.
  • Use a non-SSL connection with port 389: Create an active user without a password and the IAN.userAccountcontrol attribute should be set to 544. In this approach, a password cannot be set when creating the user.

Setting the User Password in Active Directory

Active Directory requires a secure LDAPS connection (port 636) to set or change user passwords. This is an Active Directory’s security requirement. Random generation of passwords when creating new users does not work without the use of LDAPS. Also, setting the password through attribute mapping requires a secure LDAPS connection.

Note: Active Directory does not allow you to update the user password when the connection is non-SSL (port 389).

Active Directory imposes strength requirements on a user password and the requirements vary from one instance to another.

Running the Integration

Perform the following steps to successfully run the integration:

  • Run the integration in Report Only mode and determine if the correct user data is updated in Active Directory. If the user data is not correctly updated, change the Match Expression to correctly match users and/or update the data, for example, employeeNumber, in the Active Directory.
  • Increment the Update Limit (maximum number of user records to be updated in a single run) for each run in Report Only mode to ensure that the user records are updated correctly when the records count is more.
  • Based on your analysis and considerations,
    • Translate the logical maps into attribute mapping
    • Translate the logical OU/container locations into tables and expressions
    • Translate the logical group assignment rules into Group Rules
  • Modify the attribute mapping and repeat the process until the user data is correctly updated.
  • Monitor the integration logs thoroughly to ensure that no errors are encountered. Modify the integration configurations to fix any errors.
  • Once all users are synced, user creations and updates are based on the incremental changes in ADP Workforce Now.
  • ADP Workforce Now does not return an event when a user's manager is removed. This is a limitation of the ADP Workforce Now.  Hence during incremental imports, when a user’s manager is removed from the ADP Workforce Now, the manager attribute values will not be removed from Active Directory even if the Map Manager checkbox is selected. However, during full imports, the user’s manager will be removed from Active Directory.

For details on the actions that can be performed on an integration, refer to the User Actions on Integrations guide. For details on the integration logs, refer to the Integration Logs Guide.

Appendix

This section provides the scripts that may be used by the Customer Support team to analyze the existing users in ADP Workforce Now and Active Directory before setting up the integration. It also provides the script to analyze the existing groups of users in Active Directory.

You can run the scripts using Aquera's integration solution using javascripts. In this integration, select the ADP Workforce Now and Active Directory connectors and set the corresponding variable names. In the script, the variable names must be used to refer to the selected application connectors.

For details on configuring the integration using scripts, refer to the Integration Using Scripts Guide. For details on the functions and methods supported in the script and how to work with the scripts, refer to the Working With Integration Scripts guide.

A few sample scripts are provided here for your reference. The scripts may vary based on your use case and the attributes to be retrieved for analysis.

Analyzing the Users in ADP Workforce Now

You can use the following sample script in integration for generating a report from ADP Workforce Now to analyze the existing users' status, location, department, and so on.

let adp_users = ADP.getUsers();

let i = 0;

log.info('ADP ID,ADP STATUS,ADP EMPNO,ADP GIVENNAME,ADP FAMILYNAME,ADP WORK EMAIL,ADP LOCATION CODE,ADP JOBCODE,ADP DEPT');

for (i = 0; i < adp_users.length; i++) {

    let adp_id = adp_users[i].id;

    let adp_status = adp_users[i][IAN].status;

    let adp_emp = adp_users[i][ENT].employeeNumber;

    let adp_locationcode = adp_users[i][IAN].LocationCode || '';

    let adp_job = adp_users[i][IAN].jobCode || '';

    let adp_dept = adp_users[i][IAN].departmentCode || '';

    let adp_given = adp_users[i].name.givenName;

    let adp_family = adp_users[i].name.familyName;

    let adp_work = adp_users[i].emails && adp_users[i].emails.find(x => x.type == 'work') && adp_users[i].emails.find(x => x.type == 'work').value || '';

    log.info('"' 

    + adp_id + '","' 

    + adp_status + '","'

    + adp_emp + '","'

    + adp_given + '","' 

    + adp_family + '","'

    + adp_work + '","'

    + adp_locationcode + '","'

    + adp_job + '","'

    + adp_dept + '"');

}

Analyzing the Users in Active Directory

You can use the following sample script in Integration for generating a report from Active Directory to analyze the existing users' status, distinguished name, SAM account, and so on.


log.info('AD ID,AD ACTIVE,AD EMPNO,AD GIVENNAME,AD FAMILYNAME,AD USERNAME,AD SAMACCOUNT,AD DN');

for (let user of AD.getUsers()) {

    log.info('"'

    + user.id + '","'

    + user.active + '","'

    + (user[ENT] && user[ENT].employeeNumber || '') + '","'

    + user.name.givenName + '","'

    + user.name.familyName + '","'

    + user.userName + '","'

    + user[IAN].sAMAccountName + '","'

    + user[IAN].distinguishedName.split(',').slice(1).join(',') + '"');

}
