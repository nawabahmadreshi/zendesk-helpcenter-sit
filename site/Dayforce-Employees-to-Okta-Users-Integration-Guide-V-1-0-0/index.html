Overview

The Aquera platform enables your organization to scale complex data management by automating the integration between various applications.

Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

This guide covers the end-to-end tasks in setting up an integration to sync the Dayforce Employees resource to the Okta Users resource.

This integration automates the creation, modification, and deletion of employee data from Dayforce to Okta.

This integration also helps write back the SSO Login, Business Email, Business Phone, and Okta ID to the Dayforce system.

Prerequisites

The following are the prerequisites:

  • Admin Access to Okta: You must have admin access to your Okta instance. You can visit the Okta portal to get access to an Okta instance.
  • Admin Access to Dayforce: You must have admin access to your Dayforce instance.
  • Dayforce Field-Level Access: The specific field-level permissions must be enabled for the service account to manage user information in Dayforce. For more details, refer to the Configuring Field-Level Access in Dayforce section.
  • Enable XaaS access: You must open a support case with Okta to enable XaaS in your Okta instance.
  • Source Priority for Username and Email Update: In case of username and email update for Okta profile, ensure that the username and email attributes in Okta are mastered by the correct system. Refer to the Source Priority section for more details.
  • Tenant Account in the Aquera Admin Portal: You must have a tenant account with admin access in the Aquera Admin portal. Visit https://admin.aquera.io and register to get a tenant account. Contact your Aquera account manager to get the Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/.
  • Configure Application Connectors in the Aquera Admin Portal: You must configure the following application connectors in the Aquera Admin portal to integrate with Okta:
    • Dayforce: Refer to the Dayforce Configuration Guide for details.
    • Okta: Ensure that you configure the Okta (version 11) or Okta XaaS (Anything-as-a-Source) connector in the Aquera Admin portal.

For details on the limitations of Okta's Anything-as-Source (XaaS) solution, refer to the Okta documentation.

Note: The inactive and terminated users will not be integrated from Dayforce to Okta.

Source Priority

If your use case requires updating username and email for Okta profile, ensure that these attributes in Okta are mastered by the correct system. Perform the following steps to configure the source priority:

  1. In Okta Admin Console, navigate to Directory > Profile Editor and select the User Profile.
  2. On the Profile Editor page, scroll down to the Attributes section.
  3. Find username attribute and click the info (i) icon.
  4. Set the Source Priority as follows:
    • Select Inherit from Profile Source if you want Dayforce to be the source of truth to provide and maintain the attribute value.
    • Select Inherit from Okta if the attribute must be managed in Okta and then pushed to Dayforce. In this case, Okta becomes the source of truth.
  5. Repeat the same steps for the email attribute and set the appropriate source priority based on your requirements.

Integration Setup

This section details how the integration must be configured to push data from Dayforce to Okta.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Select Dayforce Employees to Okta Users.
  5. Enter a name and a description for the integration.
  6. Select your Dayforce and Okta connector instances.
  7. Click Continue. The Configuration page is displayed to configure the integration.

Configuration

On the Configuration page, the following sections are displayed:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Parameters

The Parameters section enables you to customize the integration settings according to your specific use case. It has the following configuration parameters:

  • Report Only: Select this checkbox to execute the integration in the report-only mode. The report-only mode generates a report to show the data that would be changed without actually making those changes.
  • Full Sync Configuration: Select a schedule for full sync from Dayforce to Okta. The following options are available:
    • Standard Schedule: Select this to perform a one-time full sync from Dayforce to Okta, followed by incremental data syncing.
    • Custom Schedule: Select this to set the full sync frequency as per your requirement. When selected, the following fields are displayed for further configuration:
      • Full Sync Frequency: Select the frequency of full sync from Dayforce to Okta.
        The following options are available:
        • Every Sync (Full): Select this if you want every sync to be a full sync.
        • Every 6 hours: Select this to execute a full sync after every 6 hours.
        • Every day: Select this to execute a full sync once every day.
        • Every week: Select this to execute a full sync once every week.
        • Custom Frequency: Select this to customize the frequency. When selected, the following field is displayed for further configuration:
          • Days between Full Syncs: Enter the number of days between each full sync.
  • Onboarding and Offboarding Configuration: Select the Onboarding and Offboarding Configuration method. The following options are available:
    • Standard: Select this for standard Onboarding and Offboarding configuration. If selected, the Onboarding Configuration and Offboarding Configuration sections will be displayed with the related parameters. When selected, the following fields are displayed for further configuration:
        • Employees to be Provisioned in Okta: Select the employees to be provisioned in Okta. The following options are available:
          • All Employees: Select this if you want all employees to be provisioned in Okta.
          • All Full Time Employees: Select this if you want all full-time employees to be provisioned in Okta.
            Note: The integration checks the PayClass attribute of employees to identify a full-time employee. 
          • Custom: Select this to configure an expression to identify employees to be provisioned in Okta. When selected, the following field is displayed for further configuration:
            • Custom Expression: Provide an expression to identify employees to be provisioned in Okta.
        • Onboarding Configuration: This section allows you to configure the Onboarding settings. The following fields are available for configuration:
          • Dayforce Attribute to be Considered for Onboarding Date: Select a Dayforce attribute that must be considered to determine the onboarding date. The available options are StartDate and HireDate.
          • Onboarding Date Offset: Select the Onboarding date offset. The following options are available:
            • No Offset: Select this if you do not want to apply any offset to the onboarding date.
            • Offset Before Onboarding Date: Select this to set the onboarding date earlier by a specified number of days.
            • Offset After Onboarding Date: Select this to delay the onboarding date by a specified number of days.
          • Days of Offset: Enter the number of days for offset. This field is available only if you select Offset Before Onboarding Date or Offset After Onboarding Date in the Onboarding Date Offset field.
          • Timezone for Onboarding: Select the timezone to onboard employees. The following options are available:
            • UTC: Select this to onboard employees using Coordinated Universal Time (UTC), regardless of the employee’s local time.
            • Based on Work Location: Select this to onboard employees based on their work location’s timezone.
          • Onboarding Time: Enter the time for the onboarding in 24-hour format (HH:MM). For example, 03:40
        • Offboarding Configuration: This section allows you to configure the Offboarding settings. The following fields are available for configuration:
          • Employment Statuses to Trigger Offboarding: Select one or more employment statuses that must trigger offboarding. The following options are available:
            • Terminated: Select this to trigger offboarding when employment status changes to Terminated.
            • Inactive: Select this to trigger offboarding when employment status changes to Inactive.
            • Leave: Select this to trigger offboarding when employment status changes to Leave.
            • Maternity: Select this to trigger offboarding when employment status changes to Maternity.
            • Paternity: Select this to trigger offboarding when employment status changes to Paternity.
          • Offboarding Date Offset: Select the Offboarding date offset. The following options are available:
            • No Offset: Select this if you do not want to apply any offset to the offboarding date.
            • Offset Before Offboarding Date: Select this to set the offboarding date earlier by a specified number of days.
            • Offset After Offboarding Date: Select this to delay the offboarding date by a specified number of days.

            Note: Currently, the Date Offset works only for the Terminated employment status.

          • Days of Offset: Enter the number of days for offset. This field is available only if you select Offset Before Offboarding Date or Offset After Offboarding Date in the Offboarding Date Offset field.
          • Timezone for Offboarding: Select the timezone to offboard employees. The following options are available:
            • UTC: Select this to offboard employees using Coordinated Universal Time (UTC), regardless of the employee’s local time.
            • Based on Work Location: Select this to offboard employees based on their work location’s timezone.
          • Offboarding Time: Enter the time for the offboarding in 24-hour format (HH:MM). For example, 04:40
          • Enable Custom Offboarding: Select this checkbox to enable the custom offboarding configuration.
            When enabled, the following parameters are displayed:
            • Dayforce Attribute for Offboarding Date: Select a Dayforce attribute to determine the offboarding date.
            • Dayforce Attribute for Offboarding Time: Select a Dayforce attribute to determine the offboarding time.
            • Dayforce Attribute for Offboarding Timezone: Select a Dayforce attribute to determine the timezone for offboarding.

            Note: If Custom Offboarding is enabled and configured along with other Offboarding configurations, the priority is given to the Custom Offboarding configurations.

    • Custom: Select this for custom Onboarding and Offboarding Configuration that allows you to configure the upsert and delete conditions. When selected, the following fields are displayed for further configuration:
        • Upsert Condition: Enter all current employment statuses of employees that should trigger an upsert in Okta. For example, if you enter Active and Pre-start, all Dayforce employees with the Active or Pre-start status will be upserted in Okta.
        • Number of Days Before Effective Employment Status to Upsert: Enter the number of days before the employment statuses configured in the Upsert Condition become effective to trigger an upsert in Okta.
          Note: This field is available only if you select Custom in the Onboarding and Offboarding Configuration field.
        • Delete Condition: Enter all current employment statuses of employees that should trigger a deletion in Okta. For example, if you enter Inactive and Terminated, all Dayforce employees with the Inactive or Terminated status will be deleted in Okta.
          Note: When both Upsert Condition and Delete Condition are configured, the integration gives priority to the Delete Condition first, followed by the Upsert condition. If neither condition is met, no action will be performed.
        • Number of Days Before Effective Employment Status to Delete: Enter the number of days before the employment statuses configured in the Delete Condition become effective to trigger a deletion in Okta.
  • Admin Emails: Enter the email address to which notifications for this integration will be sent, then press Enter. You can add multiple email addresses in this field. For example, ops@example.com, itadmin@example.com.
  • Admin Name: Provide the name to display as the recipient on outgoing emails.
  • Source Employee Number Filter: Enter an employee number to use as a filter. For example, 100078.
  • Save Execution Report to Aquera Console: Select this checkbox to save Integration execution CSV Report to the Aquera Dashboard.
  • Enable Writeback: Select this checkbox to enable writeback of attributes from Okta to Dayforce. When enabled, the following fields are displayed for further configuration:
      • Dayforce Attributes to be Updated: Select the Dayforce attributes that need to be updated. The available options are SSO Login, Business Email, Business Phone, and Okta ID.
        Notes:
        • If a user has an invalid phone number in Okta, the writeback for that specific user will fail with the following error message:
          Invalid phoneNumber / invalid format
        • In Dayforce, if a future effective end date is set for a Business Phone, and the user does not have a Primary Phone in Okta, the integration attempts to update the Business Phone by converting the future effective end date to the current date. Once the effective end date is reached, the Business Phone number value will be removed from Dayforce.
        • If you have selected Okta ID to be updated, you must provide a valid XRefCode value in the XRefCode of Dayforce Custom Attribute for Okta ID field.
      • Source Employee Number Filter for Writeback: Enter an employee number to use as a filter during writeback. For example, 100078.
      • XRefCode of Dayforce Custom Attribute for Okta ID: Enter the unique XRefCode of the custom attribute created in Dayforce to store the Okta ID. For example, EmployeePropertyXrefCode1812.
        Note: This field is available only if you select Okta ID in the Dayforce Attributes to be Updated field.
  • Upsert Limit: Specify the maximum number of records that can be created or updated in each integration run. Once this limit is reached, no additional records will be processed.

Create Custom Attributes in Okta

You can add custom attributes to the App-specific user profile in Okta to include additional Dayforce Employee attributes. You must configure all the custom attributes individually. After the configuration, Aquera automatically populates the fields that you have created in Okta from the Employee resource of Dayforce.

Note: To view Dayforce employee attributes, go to the Data tab of your Dayforce application connector configured in the Aquera Admin Portal. Then navigate to Resources > Employees and click Fetch Records.

Perform the following steps to create custom attributes in Okta:

  1. Sign in to your Okta instance.
  2. Navigate to Directory > Profile Editor and select the App-specific user profile that you've created.
  3. Click Add Attribute to add a new attribute.
  4. Configure the attribute using the same name as Dayforce in the Variable name field and fill in other fields.

    Note: Refer to the Variable Name for Custom Attributes in Okta section for details on the variable name.
  5. Click Save to save the changes or Save and Add Another to create another custom attribute.

While specifying a variable name, you must consider the following points:

  • For non-complex attributes, the variable name must be the same as the Dayforce attribute name. For example, the variable name for NewHireApprovalDate must be NewHireApprovalDate.
  • For complex attributes, the variable name must include a double underscore (__) between each sub-level attribute. For example, the variable name for the attribute country, which is a sub-attribute of EmployeeManagers.contacts must be EmployeeManagers__Contacts__Country.
  • The following table outlines the correct variable naming conventions for handling complex attributes with multiple values:
    Attribute Format and Example
    EmployeeManagers

    Format: EmployeeManagers__{attribute name}

    Example: EmployeeManagers__ManagerFirstName
    Contacts

    Format: Contacts__{attribute name}

    Example: Contacts__ContactNumber

    Note: Only the current active (latest) value will be considered. To consider a value other than the latest one, the variable name must be in the Contacts__{XRefCode value}__{attribute name} format. For example, Contacts__BusinessPhone__ContactNumber.
    EmploymentStatuses

    Format: EmploymentStatuses__{attribute name}

    Example: EmploymentStatuses__PayPolicy

    Note: Only the current active (latest) value will be considered. To consider a value other than the latest one, the variable name must be in the EmploymentStatuses__{XRefCode value}__{attribute name} format.
    Addresses

    Format: Addresses__{attribute name}

    Example: Addresses__Address1

    Note: Only the PrimaryResidence value will be considered. To consider a value other than the PrimaryResidence one, the variable name must be in the Addresses__{XRefCode value}__{attribute name} format. For example, Addresses__Mailing__Address1.
    WorkAssignments

    Format: WorkAssignments__{attribute name}

    Example: WorkAssignments__Position
    Roles

    Format: Roles__{attribute name}

    Example: Roles__EffectiveStart

    Note: Only the current active (latest) value will be considered. To consider a value other than the latest one, the variable name must be in the Roles__{XRefCode value}__{attribute name} format. 
    EmployeeProperties

    Format: EmployeeProperty__{attribute name}

    Example: EmployeeProperty__ShortName

    Notes:

    • Only the current active (latest) value will be considered. To consider a value other than the latest one, the variable name must be in the EmployeeProperty__{XRefCode value}__{attribute name} format.
    • While specifying variable name for EmployeeProperties, ensure that you consider the XRefCode value of EmployeeProperty > XRefCode.
    OrgUnitInfos

    Format: OrgUnitInfos__{attribute name}

    Example: OrgUnitInfos__OrgUnitDetail

    Note: Only the current active (latest) value will be considered. To consider a value other than the latest one, the variable name must be in the OrgUnitInfos__{XRefCode value}__{attribute name} format.
    SSOAccounts

    Format: SSOAccounts__{attribute name}

    Example: SSOAccounts__LoginName

    Note: Only the current active (latest) value will be considered. To consider a value other than the latest one, the variable name must be in the SSOAccounts__{XRefCode value}__{attribute name} format.

Configuring Field-Level Access in Dayforce

Perform the following steps to configure the Field-Level access in Dayforce:

  1. Log in to your Dayforce instance with the admin credentials.
  2. Select the relevant role. For example, Identity API Integration.
  3. Click the Web Services Field-Level Access tab. 
    The Web Services Field-Level Access tab provides a hierarchical view of the web services response types separated by the type of service (RESTful Services and SOAP Services) and the data entities and elements available within each.
  4. To retrieve the organization unit's location details, such as address and contact information, expand Human Resources under the RESTful Services.
  5. Enable the data elements you need for each organization unit from System Admin > Roles > Web Services Field-Level Access > RESTful Services > Human Resources > Employee.
    The following table outlines the permissions that must be enabled in the RESTful Services for each attribute:
    Attribute Field-Level Access

  • EmployeeExportParameters
  • EmployeeExportBulkResponse
  • EmployeeExportBackgroundJobStatus

Map Custom Attribute to Okta Profile

Perform the following steps to map the custom attributes to the Okta profile:

  1. Navigate to Directory > Profile Editor and select the App-specific user profile that you've created.
  2. Click Mappings and select Configure User mappings.
  3. Map the custom attribute in the App-specific user profile to an equivalent attribute in the Okta user profile and click Save Mappings.

    Note: If an equivalent attribute is not present in the Okta user profile, refer to the following section and add custom attributes to the Okta user profile.

Add Custom Attribute to Okta User Profile

Perform the following steps to add a custom attribute to the Okta user profile:

    1. Navigate to Directory > Profile Editor and select the Okta user profile (User default).
    2. Click Add Attribute to add a custom attribute to the Okta default user profile.
    3. Fill in the attribute's display name, description, data type, and set the user permission.
    4. Click Save.
