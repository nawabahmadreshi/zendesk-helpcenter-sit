Dayforce / Okta Integration Guide

← Back to index

This guide applies to the latest public version of the integration (version 3.0.0).

Overview

The Aquera platform allows your organization to scale complex data management by automating data integration across resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

In this integration, any users created, updated, or terminated in Dayforce are automatically updated in Okta based on the defined schedule. The integration also supports the writeback of business email addresses and federated IDs from Dayforce to Okta.

To identify user records across both systems for user sync from Dayforce to Okta, the integration uses the attribute configured in the Source Field for Correlation parameter in Dayforce and matches it to the attribute configured in the Target Field for Correlation parameter in Okta as a correlation attribute.

Key Functions

The Dayforce / Okta Integration performs the following functions:

  • Synchronizes user records from Dayforce to Okta.
  • Creates new users in Okta when they are added in Dayforce.
  • Updates existing Okta users when their information changes in Dayforce.
  • Deactivates or deletes Okta users when they are no longer active in Dayforce.
  • Generates usernames and email IDs for users in Okta based on configured policies.
  • Updates business email addresses and federated ID's from Okta to Dayforce (writeback).
  • Supports both full synchronization and incremental synchronization to process only changed records.

Prerequisites

The following are the prerequisites:

  • Admin Access to Okta: You must have admin access to your Okta instance. You can visit the Okta portal to get access to an Okta instance.
  • Admin Access to Dayforce: You must have admin access to your Dayforce instance.
  • Dayforce Field-Level Access: The specific field-level permissions must be enabled for the service account to manage user information in Dayforce. For more details, refer to the Configuring Field-Level Access in Dayforce section.
  • Enable XaaS access: You must open a support case with Okta to enable XaaS in your Okta instance.
  • Source Priority for Username and Email Update: In case of username and email update for Okta profile, ensure that the username and email attributes in Okta are mastered by the correct system. Refer to the Source Priority section for more details.
  • Aquera Tenant Account: You must have a tenant account in the Aquera Admin portal. Visit https://admin.aquera.io to register for a tenant account. Contact your Aquera account manager to get Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io
  • Configure Application Connectors in the Aquera Admin Portal: You need to configure the following application connectors in the Aquera Admin Portal:
    • Dayforce: Refer to the Dayforce Configuration Guide for configuration details.
    • Okta: Ensure that you configure the Okta (version 11) or Okta XaaS (Anything-as-a-Source) connector in the Aquera Admin portal.

For details on the limitations of Okta's Anything-as-Source (XaaS) solution, refer to the Okta documentation.

Note: The inactive and terminated users will not be integrated from Dayforce to Okta.

Source Priority

If your use case requires updating username and email for Okta profile, ensure that these attributes in Okta are mastered by the correct system. Perform the following steps to configure the source priority:

  1. In Okta Admin Console, navigate to Directory > Profile Editor and select the User Profile.
  2. On the Profile Editor page, scroll down to the Attributes section.
  3. Find username attribute and click the info (i) icon.
  4. Set the Source Priority as follows:
    • Select Inherit from Profile Source if you want Dayforce to be the source of truth to provide and maintain the attribute value.
    • Select Inherit from Okta if the attribute must be managed in Okta and then pushed to Dayforce. In this case, Okta becomes the source of truth.
  5. Repeat the same steps for the email attribute and set the appropriate source priority based on your requirements.

Integration Setup

This section outlines how the integration must be configured to push user-related data from Dayforce to Okta.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Select Dayforce / Okta.
  5. Enter a name and a description for the integration.
  6. Select your Dayforce and Okta connector instances.
  7. Click Continue. The Configuration page is displayed to configure the integration.

The Manage User section enables you to customize the integration settings according to your specific use case. It has the following settings:

  • Okta Correlation Attribute: Select the Okta attribute to distinctly find the associated Dayforce user record. For example, select Employee Number.

User Settings

Configure the following user settings:

Note: All parameters marked with a red asterisk (*) are mandatory.

  • Import Limit: Enter the maximum number of user records to be imported into Okta during each integration execution. Once this limit is reached, no additional records will be processed. For example, if you enter 10, only the first ten user records will be imported into Okta.
  • Source Filter: Select a filter type from the dropdown menu. Based on your selection, only employee records that satisfy the selected condition are imported. The following are the available options:
    • All Employees: Select this option to filter and import all the employees into Okta from Dayforce.
    • All Full-Time Employees: Select this option to filter and import all full-time employees into Okta from Dayforce.
    • Filter by Employee Numbers: Select this option to filter employees by their employee numbers. When selected, the following parameter is displayed:
      • Employee Numbers: Enter the user's employee number that must be imported into Okta from Dayforce.
    • Filter by Employment Statuses: Select this option to filter employees by their employment statuses. When selected, the following parameter is displayed:
      • Employment Statuses: Select the user's employment statuses from the dropdown menu. Only employees with these statuses will be imported into Okta from Dayforce.
  • Dayforce Server Time Zone*: Select the time zone configured for the Dayforce server to ensure accurate date and time processing during synchronization.
  • Properties to Fetch for Dayforce Employees: Select one or more properties from the dropdown menu. Only attributes related to the selected properties are included in the integration.
  • Include Manager's Full Details: Select this checkbox to include the user's manager's full details in the integration.

Deactivate User Settings

Configure the following deactivate user settings:

Note: All parameters marked with a red asterisk (*) are mandatory.

  • Deactivation Limit: Specify the maximum number of user records that can be deactivated from the Okta during a single integration run.
  • Deactivate Condition: Select a condition from the dropdown menu to determine when users are deactivated. The following are the available options:
    • All Employees: Select this option to deactivate all employees from Dayforce in Okta.
    • All Inactive Employees: Select this option to deactivate all inactive employees from Dayforce in Okta.
    • Custom: Select this option to define a custom condition to deactivate an employee from Dayforce in Okta. When enabled, the following field is displayed:
      • Custom Deactivate Condition*: Enter the JavaScript expression to deactivate an employee from Dayforce in Okta.

Time-Based Offboarding Settings

Configure the following time-based offboarding settings:

Note: All parameters marked with a red asterisk (*) are mandatory.

  • Offboarding Date Offset Type: Select an offboarding date offset type from the dropdown menu. The following are the available options:
    • Same Day: Select this option to deactivate the user in Okta on the same day as the user's offboarding date.
    • Deactivate Before Offboarding Date: Select this option to deactivate the user in Okta before the user's actual offboarding date.
    • Deactivate After Onboarding Date: Select this option to deactivate the user in Okta after the user's actual offboarding date.
  • Attribute To Consider For Offboarding Date*: Select an attribute from the dropdown menu whose value will be used as the offboarding date. For example, if Termination Date is selected, its value is used as the user’s offboarding date.
  • Offboarding Time Zone: Select the time zone from the dropdown menu used to evaluate the offboarding date and trigger offboarding actions. The following are the available options:
    • Dayforce Server Timezone: Select this option to use the Dayforce server timezone to determine when offboarding actions are executed.
    • Employee Work Location Timezone: Select this option to use the employee's work location timezone to determine when offboarding actions are executed.
    • Employee Home Location Timezone: Select this option to use the employee's home location timezone to determine when offboarding actions are executed.
    • Custom: Select this option to use the custom timezone to determine when offboarding actions are executed.
  • Offboarding Time Of Day*: Enter the time of day when offboarding actions are triggered based on the configured offboarding date.

FAQs

This section provides answers to the frequently asked questions related to the Dayforce / Okta Integration.

Question 1: Does Aquera support defining percentage-based threshold limits during synchronization?
Answer: Currently, the integration does not support threshold limits by percentage during synchronization. However, it does support threshold limits based on user count, which can be configured in the User Life Cycle section for create, update, and deactivate operations. For details, refer to the User Life Cycle section.

Question 2: Can Aquera generate alerts when a high volume of user records are modified within a sync cycle?
Answer: No, Aquera does not generate alerts for high-volume user record modifications. However, supports sending notification alerts based on the transaction types selected in the Notify Only When Below Transaction Happens parameter in the Notifications section. For details, refer to the Notifications section.

Question 3: Is there a way to require manual approval or confirmation before processing bulk attribute updates?
Answer: Currently, the integration does not support a manual approval or confirmation procedure before processing bulk attribute updates.

Question 4: Can we preview or simulate attribute changes before applying them?
Answer: Yes, you can generate a report that displays all the data that would be changed without actually making those changes. For this, enable the Report-Only Mode parameter in the Global Settings section.
