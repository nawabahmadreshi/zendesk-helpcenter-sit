Active Directory / ServiceNow Integration Guide

← Back to index

This guide applies to the latest public version of the integration (version 4.0.0).

Overview

The Aquera platform allows your organization to scale complex data management by automating data integration across resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

This guide outlines the steps required to set up an integration that synchronizes user data from ServiceNow to Active Directory and enables the writeback process from Active Directory to ServiceNow to support user lifecycle and group management. It also explains how to manage user memberships in Active Directory Security Groups and Distribution Lists (DLs) based on ServiceNow tickets.

To identify user records across both systems, the integration uses the Employee number attribute from ServiceNow and matches it to the Employee ID in the Active Directory as a correlation attribute.

Key Functions

The ServiceNow / Active Directory integrations provide the following functions:

  • User Lifecycle Management: Handles user creation, updates, deactivation, and deletion from ServiceNow to Active Directory. If the Enable ServiceNow writeback feature is enabled, the integration can create, update, or deactivate user records in ServiceNow based on changes made in Active Directory
  • Group Assignment: Assigns users to specific groups in Active Directory based on work location attributes like location, country, state, and city. Group assignments are ignored if the conditions are not met or if work location details are missing.
  • Security Group Management: Automatically adds or removes users from Active Directory security groups based on ServiceNow tickets.
  • Distribution List Management: Automates the creation of new distribution lists and adds users to both new and existing distribution lists based on ServiceNow ticket requests.
  • Group Type Validation: Ensures that requested group types from ServiceNow (Security Group or Distribution List) match the actual group type in Active Directory to prevent processing errors.
  • Automated Ticket Reassignment: Reassigns ServiceNow tickets to a specified group if an integration fails, enabling prompt resolution without manual intervention.

The following are the prerequisites:

Integration Setup

This section describes the configuration required to integrate ServiceNow with Active Directory and enable the writeback process from Active Directory to ServiceNow. It outlines the steps to create the integration, configure integration preferences, and attribute mapping between ServiceNow and Active Directory.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Search for Active Directory / ServiceNow in the search box next to the Explore Templates section.
  5. Click the integration template to display the Create New Integration pop-up window.
  6. Enter a name and a description for the integration.
  7. Select both application connector instances from the dropdown menus.
  8. Click Continue. The integration configuration page is displayed to configure the integration.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Parameters

The Parameters section enables you to customize the integration settings according to your specific use case. It has the following settings:

Integration Features

Enable the following features based on your requirements:

  • Enable user lifecycle management: Select to synchronize user data from ServiceNow to Active Directory. When enabled, the Manage Users section is displayed, allowing you to configure additional integration settings and automate the creation, updating, deactivation, and deletion of users in Active Directory based on ServiceNow tickets.
  • Enable group management: Select to enable group management from ServiceNow to Active Directory. When enabled, the Group Management section is displayed to configure additional integration settings.

General Settings

Configure the following general settings:

  • Enable report-only mode: Select to execute the integration in the report-only mode. The report-only mode generates a report to show the data that would be changed without actually making those changes.
  • Enable logging: Select to generate logs outlining data changes. When enabled, the Logging section is displayed for further configuration.
  • Enable report generation: Select to generate the integration report. When enabled, the Reporting section is displayed for further configuration.
  • Enable notifications: Select to enable the integration-related notifications. When enabled, the Notifications section is displayed for further configuration.
  • Enable log insights: Select to visualize the integration-related metrics. When enabled, the Log Insights section is displayed for further configuration.
  • Enable debug mode: Select to generate logs outlining data changes that would occur for debugging. When enabled, the Debug Mode section is displayed for further configuration.

Correlation Settings

The Correlation Settings section enables you to configure the attributes used to identify and match common user and group records between ServiceNow and Active Directory. These settings ensure that the integration accurately correlates records between the two applications during provisioning, updates, deactivation, group management, and writeback operations.

It has the following settings:

User Correlation Settings

Configure the following settings to identify and match user records between ServiceNow and Active Directory:

Note: This section appears only if you have selected the Enable user lifecycle management checkbox under the Integration Features section in Parameters.

  • ServiceNow correlation attribute: Select an attribute in ServiceNow that matches the selected Active Directory attribute value to identify common users across ServiceNow and Active Directory. For example, select Employee number as the correlation attribute in ServiceNow.
  • AD correlation attribute: Select an attribute in Active Directory that matches the selected ServiceNow attribute value to identify common users across ServiceNow and Active Directory. For example, select employeeId as the correlation attribute in Active Directory.

Group Correlation Settings

Configure the following settings to identify and match group records between ServiceNow and Active Directory:

Note: This section appears only if you have selected the Enable group management checkbox under the Integration Features section in Parameters.

  • ServiceNow correlation attribute: Select an attribute in ServiceNow that matches the selected Active Directory group attribute value to identify common groups across ServiceNow and Active Directory. For example, select Name as the correlation attribute in ServiceNow.
  • AD correlation attribute: Select an attribute in Active Directory that matches the selected ServiceNow group attribute value to identify common groups across ServiceNow and Active Directory. For example, select CN as the correlation attribute in Active Directory.

Manage Users

The Manage Users section allows you to configure user integration settings and automate the creation, updating, or deactivation of users in Active Directory based on ServiceNow tickets. 

Notes:

  • Every user must have an Employee Number. This is the key identifier used by the integration. If it is missing, the system won’t be able to create or manage the user.
  • When a ticket is sent to a manager for approval (for inactive users), it depends on whether that user still exists. If the user gets deleted before the manager takes action, the system won’t be able to process the request. The manager will see a warning that the user no longer exists, and the ticket will remain open.
  • If you update a user’s details (like their name) and, at the same time, try to assign them as a manager to a new user in the same integration run, it may not work. This happens because the system may not recognize the updated details immediately. To avoid this, make such changes in separate runs.

It has the following settings:

User Provisioning Settings

Configure the following user provisioning settings:

  • Enable user creation: Select to create new user records in Active Directory based on requests submitted through the ServiceNow catalog. When enabled, the User Creation Settings on the same page, along with the Naming Policy and Password Policy sections in the left panel, will appear for further configuration.
  • Enable user updation: Select to update existing user records in Active Directory based on requests submitted through the ServiceNow catalog. When enabled, the User Updation Settings section appears for further configuration.
  • Enable user deactivation: Select to deactivate user records in Active Directory based on requests submitted through the ServiceNow catalog. When enabled, the User Deactivation Settings section appears for further configuration.
  • Enable manager mapping: Select to map the user's manager. When enabled, the Variable for user's manager field appears in the User Creation Settings and User Updation Settings sections for further configuration.
  • Enable future hire user activation: Select to activate users with a future hire date. These users will initially be created in Active Directory in a disabled state. When enabled, the Variable for user activation date and Days before onboarding to activation fields appear in the User Creation Settings section for further configuration.
  • Enable inactive user discovery: Select to identify AD users who have not logged in for a specified period. When enabled, the Inactive User Discovery Settings section appears for further configuration:
  • Enable inactive user management: Select to manage AD users who have not logged in for a specified period, based on the manager’s actions in the ServiceNow ticket. When enabled, the Inactive User Management Settings section appears for further configuration.
  • Enable deletion of disabled AD users: Select to automatically delete disabled OU users from Active Directory. If the Enable ServiceNow writeback option is enabled, the corresponding user in ServiceNow will also be deactivated when the deletion occurs. When enabled, the User Deletion Settings section appears for further configuration.
  • Enable ServiceNow writeback: Select to enable writeback of user records from Active Directory to ServiceNow. When enabled, the system can create, update, or deactivate user records in ServiceNow based on changes made in Active Directory. In this configuration, Active Directory acts as the source application, and ServiceNow acts as the target application.

    Note: This field appears only if you have selected the Enable manager mapping checkbox under the User Provisioning Settings section.
  • Variable for user activation date: Select the ServiceNow catalog variable that contains the user’s start date.
    Note: This field appears only if you have selected the Enable future hire user activation checkbox under the User Provisioning Settings section.
  • Days before onboarding to activation: Enter the number of days before the user’s onboarding date when the account should be automatically activated in Active Directory. For example, if you enter 3, the integration will activate the user account three days before the configured start date.
    Note: This field appears only if you have selected the Enable future hire user activation checkbox under the User Provisioning Settings section.
  • RITM status field: Select the status field that contains the status values.

  • Post update RITM status: Select the status from the dropdown to set in the RITM after successful processing. For example, select CLOSED_COMPLETE. When the RITM is processed successfully, its status will be automatically updated to CLOSED_COMPLETE.

Inactive User Discovery Settings

Configure the following settings to identify inactive users:

  • Days since last login: Enter the number of days to look back from the current date to identify Active Directory users who have not logged in. For example, if you enter 90, the integration will identify users who have not logged in within the past 90 days.
  • ServiceNow catalog for inactive user ticket creation: Choose the service catalog that will be used to create ServiceNow tickets for Active Directory inactive users. When inactive accounts are identified, a ticket will be auto-created under the selected service catalog and assigned to the user’s manager for action. For example, select User Action Requests to create inactive user tickets in that catalog. 
    Note: The ticket will be auto-created when a user exceeds the threshold specified in Days since last login.
  • Variable to identify inactive user: Choose the ServiceNow catalog variable that identifies the inactive user. For example, select Request For, which stores the inactive user record for which the manager must take action in ServiceNow.

Inactive User Management Settings

Configure the following settings for inactive user management:

  • Service catalog: Select the ServiceNow service catalog used for manager user action requests. For example, select User Action Requests to process only user action request tickets.
  • Variable to identify inactive user: Choose the ServiceNow catalog variable that identifies the inactive user. For example, select Request For to identify the user record on which the manager took action.
  • Variable to take action for inactive users: Choose the ServiceNow catalog variable that contains the action taken by the manager for the inactive AD user. For example, select Action, which contains the action values applied to inactive users in Active Directory. In this case, the variable may contain values such as Active, Deactive, and Delete.
    • Notes: 
      • The available action values are dynamically pulled from the selected ServiceNow catalog variable. The values Active, Deactive, and Delete mentioned above are only examples. The actual values depend entirely on how the selected catalog variable is configured.
      • The selections in the Action value to keep the user active,  Action value to deactivate the user, and Action value to delete the user fields must be mutually exclusive. If any values overlap, the integration will fail validation.
    • Action value to keep the user active: Select the appropriate action value to apply to the inactive user account in Active Directory based on the manager selection in the action catalog to retain the user's access. For example, select Active if the manager selected Active in the action catalog.
    • Action value to deactivate the user: Select the action value that should be applied to the inactive user account in Active Directory based on the manager selection in the action catalog to deactivate the user. Once deactivated, the account is moved to the designated disabled OU. For example, select Deactive if the manager selected Deactive in the action catalog to deactivate the user.
    • Action value to delete the user: Select the action value that should be applied to the inactive user account in Active Directory based on the manager selection in the action catalog to delete the user. For example, select Delete if the manager selected Delete in the action catalog to delete the user.
  • Auto-deactivation grace period: Enter the number of days to wait for manager approval in ServiceNow before the integration automatically deactivates the user in Active Directory. For example, enter 7 to automatically deactivate the user if the manager does not respond within 7 days. Once the account is deactivated, the user is moved to the designated disabled OU.
  • RITM status field: Select the status field that contains the status values.
  • RITM status to process: Select the status from the dropdown that the RITM must have to be processed. For example, select Open. Only RITMs with this status will be picked for processing.
  • Post update RITM status: Select the status from the dropdown to set in the RITM after successful processing. For example, select CLOSED_COMPLETE. When the RITM is processed successfully, its status will be automatically updated to CLOSED_COMPLETE.

User Deletion Settings

Days disabled before deletion: Enter the number of days a user account must remain in the disabled organizational unit in Active Directory before it is deleted. For example, if you enter 30, the integration will delete user accounts that have been disabled in Active Directory for 30 days or more.

Group Management

The Group Management section allows you to control how users are managed in Active Directory Security Groups and Distribution Lists (DLs) based on ServiceNow tickets. It supports adding and removing users in security groups, as well as creating new distribution lists and adding users to both new and existing distribution lists.

Group Management Settings

Enable the following group management settings based on your requirements:

  • Enable security group management: Select to manage Active Directory security group memberships based on ServiceNow requests. When enabled, the Security Group Management Settings section is displayed for further configuration.
  • Enable distribution list management: Select to create and manage Active Directory distribution lists based on ServiceNow requests. When enabled, the Distribution List Settings section is displayed for further configuration.
    Note: You can create new DLs and add users to existing ones. Users cannot be removed directly from a DL. To remove users, disable Enable group type match, and configure Security group management settings with the DL selected as the group type.
  • Enable group type match: Select to ensure that the requested group type (Security Group or Distribution List) from ServiceNow matches the actual group type in Active Directory during processing. If there is a mismatch in the type, the integration will fail.
  • Enable ticket reassignment for failure: Select to automatically reassign the ticket to the configured group if the requested operation fails during the integration run. When enabled, the Ticket Reassignment section is displayed for further configuration.
  • Enable automated closure of related tasks: Select to automatically close all associated tasks when the parent RITM’s requested operation is successfully completed.

Logging

  • Select log level: Select the desired log detail level from the dropdown menu. The following are the available options:
    • Full: All attribute details are displayed.
    • Include list: Displays only the selected attributes. When selected, the following field is displayed:
      • Custom inclusion list: Enter the attributes to be included in the log.
    • Exclude list: Displays all attributes except the selected ones. When selected, the following field is displayed:
      • Custom exclusion list: Enter the attributes to be excluded from the log.

    • Full: All attribute details are displayed.

      • Custom inclusion list: Enter the attributes to be included in the notification report.
    • Exclude list: Displays all attributes except the selected ones. When selected, the following field is displayed:
      • Custom exclusion list: Enter the attributes to be excluded from the notification report.

Group Assignment Rules

The Group Assignment Rules tab allows you to configure how group assignments are managed through this integration. In this tab, you can define conditions that determine how users are assigned to groups.

Perform the following steps to create a new group rule:

  1. Navigate to the Group Assignment Rules tab. The following fields are displayed:
  2. Select the source table in the Source Table field to fetch all the columns from that table.
  3. Click Choose Attribute to view the list of available ServiceNow attributes that you can use to set conditions.
  4. Search and select the attribute you want to use to define a condition. For example, select Country.
  5. Select the attribute value to define the rule. If multiple values are provided, an OR condition is applied among them. For example, in the following sample condition, a user's country must be either India or the USA to satisfy this assignment rule.
  6. In the Group Name column, select the groups to which users who satisfy the specified condition should be assigned. The list displays all groups from the Active Directory application.
  7. If required to add another condition and assignment for the same selected attribute, click Add Row. You can add any number of rows within a rule.
  8. If required to include another attribute to the group assignment rule, click the + icon next to the selected attribute name. If multiple columns are added for a rule, the AND logic is applied between different columns. For example, in the following sample condition, a user’s country is India, and the state is TN, the user who satisfies both conditions will be assigned to the configured groups. You can use multiple attributes and their corresponding values to define the group assignment rule.
  9. [Optional] If you want to write a JavaScript expression to set the condition, click Switch to Expression.
    Note: When you switch to Expression mode or back to Condition Builder mode, all configured conditions in the rule will be cleared.
    1. Select one of the following based on your requirement:
      • For Conditions Only: Select this if you want to write a JavaScript expression only for the conditions.
      • For Conditions and Assignments: Select this if you want to write an expression for both conditions and assignments.
    2. Click Continue to configure the expression.

    3. Enter the expression based on your selection:

      • If For Conditions Only is selected, enter the expression for the conditions, and then select the group in the Group Name column.
      • If For Conditions and Assignments is selected, enter the expression that defines both the conditions and the group assignment.

      Note: Expressions can use attributes from both the source and destination applications. The configured expressions must always be evaluated to true or false. The prefix SOURCE must be used in an expression to represent an attribute in the source application, and DEST must be used to represent an attribute in the destination application. For example, SOURCE.addresses[].locality, SOURCE.addresses[].country, and DEST.group.displayName.

  10. Click Add New Rule to add another assignment rule, as explained above.
  11. Select the Active Directory attribute that stores the user’s group membership information in the AD user field for the group assignment field.

Notes:

  • You can add any number of assignment rules. However, you can see only 5 rules on a single page. If you add more than 5 rules, navigation options will be available at the bottom of the page to navigate to other pages for configuring or viewing the remaining defined rules.
  • If required, click the Delete icon at the right end of the row to delete a row based on your requirement.

Attribute Map

The Attribute Map tab defines how data is mapped from the source to the target application. The default mapping of attributes is displayed on the page. The source attributes are listed on the left side, and the target attributes are listed on the right side of the page.

You can modify only the source attributes as needed to align with your specific use case.

You can write JavaScript expressions to modify the value of the source attribute and construct a value to be updated in the target attribute. Write the expression directly in the source attribute field or click the Expand icon in the source attribute field to write a long expression.

All attributes marked with a red asterisk (*) indicate required fields for successful provisioning.

After the source-to-target attribute mapping is completed, click Save Changes at the top-right corner of the page to save the integration.

Update the Integration to the Latest Version

You can upgrade an integration built from a template to the latest version to access new features and enhancements. 

Perform the following steps to upgrade an integration:

  1. Navigate to the Integrations module on the Aquera Admin portal.
  2. Click the vertical ellipsis icon next to an integration and click View Details. The Details panel is displayed with the following sections:
    • INTEGRATION DETAILS: Displays the integration name, description, creation date, last update date, and the user who created or last updated it.
    • TEMPLATE DETAILS: Displays the current version number and its features under CURRENT VERSION. If a newer version is available, the version number and the changes are shown under NEW VERSION AVAILABLE with an Upgrade button.
  3. Click Upgrade. The Upgrade Integration pop-up window is displayed.
  4. Provide the following details for the new integration to be created after the upgrade:
    • New Integration Name: Enter the name of the new integration.
    • Description: Provide a description for the new integration.
      The current version number and the upgraded version number are displayed.
  5. Click Upgrade. A new integration is created with the same configuration details as the existing one, but with features or enhancements from the upgraded version.
