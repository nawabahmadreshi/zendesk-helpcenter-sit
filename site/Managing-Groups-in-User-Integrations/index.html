Managing Groups in User Integrations

← Back to index

Overview

This guide provides details about how to manage groups in User Integrations. Managing group membership of users can be facilitated through the Group Rules tab or the Attribute Mapping tab on the integration setup page. 

Group rules can be created using basic conditions or using the full capabilities of JavaScript SCIM extensions. One group rule must be defined for each managed group. If the group rule resolves to true, the user will be included in the group’s membership; otherwise, the user will be excluded from the group. 

Managing Groups 

Groups can be managed using one of the following ways:

  • Setting Group Rules: You can create group rules to manage the target groups.
  • Attribute Mapping:
    • Using Expressions: You can write JavaScript expressions in the attribute mapping to manage the target groups.
    • Using Tables: You can add tables and use them in the attribute mapping expressions to manage the target groups. 

Note:

Only when the checkbox Enable Group Management is selected in the Configuration tab of User Integration, Aquera's integration will be able to manage the groups in the target application through Attribute Mapping or Group Rules.

Setting Group Rules

You can set up rules to manage the groups in the target application. There is no limit to the number of group rules that can be added in User Integration.

Notes: 

  • When the target application does not support groups, the Group Rules tab shows the message Groups are not supported by the target application and the Add Rule button is disabled.
  • When the target application is changed and the selected target application does not support groups, the already existing group rules will be ignored.

Perform the following steps to set a group rule:

  1. Click the Group Rules tab.
  2. Click Add Rule which displays an Add Rule pop-up window.
  3. Enter a name for the rule.
  4. Set the rule condition in the IF section using one of the following methods:
    • Basic Condition
    • Expression
  5. Select the group name from the dropdown group list to which the group rule must be applied in the THEN section.
    Notes:
    • When the group list is being loaded from the target application, this section shows the message Fetching Groups.
    • When there is an error in loading the group list from the target application, you can get any of the following messages after the THEN section:
      • Error while fetching groups from the target application. Unauthorized, Invalid Authentication: This occurs if the authentication token/credentials have not been provided on the connector setup page, or the token/credentials are incorrect.
      • Error while fetching groups from the target application. Bad Gateway: This occurs if the Tenant/Agent details that were provided on the connector setup page are incorrect, or the Agent is not running.
      • Error while fetching groups from the target application. Bad request, The request cannot be fulfilled due to bad syntax: This occurs if the tenant attribute values that were provided on the connector setup page are incorrect.
        Refer to the respective connector configuration guide for further details to fix these errors. 
  6. Click Add at the bottom of the Add Rule window to add the rule.
  7. Close the rule window and click Save Changes at the top right of the integration page to save the group rule to the integration. 

Note: The Else statement is not supported in the Group Rules. However, you can create a new Group Rule to add users to a default group when the users do not match any existing group rules. Refer to the article Setting a Group Rule for Managing Users who do not satisfy any Group Rules for further details.

Using Basic Conditions in Group Rules

Perform the following steps to use a basic condition in a group rule:

  1. Select Use Basic Condition.
  2. Select a source user attribute from the drop-down list in the User Attribute field:
  3. Select an option from the drop-down list in the Condition field:

    The following table explains each of the options:
    Option Description
    Equals Determines if the attribute value equals to a specific value.
    Not Equals Determines if the attribute value is not equal to a specific value.
    Contains Determines if the attribute value contains a specific string.
    Less Than Determines if the attribute value is less than a specific value.
    Greater Than Determines if the attribute value is greater than a specific value.
    Starts With Determines if the attribute value starts with a specific string.
    Ends With Determines if the attribute value ends with a specific string.
  4. Enter the attribute value against which the condition has to be validated in the Attribute Value field.
  5. Enter the group name to which the group rule must be applied in the THEN section.
    The users satisfying the condition will be added to this group.

    Note: If the User Attribute is a string type, ensure to include the Attribute Value in double quotes.

Using Expression in Group Rules

You can write JavaScript expressions to set the rules for updating group-related data. Expressions can use all of the helper functions as well as the user attributes in the source and destination applications. The configured expressions must always be evaluated as true or false.

SOURCE must be used in an expression to represent an attribute in the source application and DEST to represent an attribute in the destination application. For example, SOURCE.userName, SOURCE[ENT].organization, DEST.title, SOURCE[IAN].empStatus, and so on.

For details on the keywords, inbuilt methods, and functions provided by Aquera, refer to the Integration Expressions - Keywords, Methods, and Functions guide.

Provisioning Users to a Single Group

Perform the following steps to use expressions in a group rule and manage a single group:

  1. Select Use Expression.
  2. Enter the condition as an expression in the IF section.
  3. Enter the group name to which the rule has to be applied in the THEN section.
    The users satisfying the condition will be added to the group.

Provisioning Users to Multiple Groups

Perform the following steps to use expressions in a group rule and manage multiple groups:

  1. Select Use Expression.
  2. Enter the condition as an expression in the IF section.
  3. Enter the group names to which the rule has to be applied in the THEN section. 
    The users satisfying the condition will be added to the groups.

Note: If you want to deprovision the users from groups upon termination, include the active attribute in the expression. A separate rule for de-provisioning is not necessary.

For example, you can enter the following expression for de-provisioning:
SOURCE.active && SOURCE[IAN].divisionCode == "TRW"

All the added rules are displayed as a list on the Group Rules page. You can add more rules, and modify or delete an existing rule, as required.

Group Management Functionality in User Integration

The following are the details on how group management works in User Integration version 3:

  • Only when the checkbox Enable Group Management is selected in the configuration of User Integration, the integration process will be able to manage the groups in the target application through Attribute Mapping or Group Rules.

  • When you have set valid group rules, users will be added to the groups specified in the group rules.
    Note: The Group Management Restriction List should be left empty when you have set valid group rules. 

  • The Group Management Restriction List will not be considered if the groups are managed through Group Rules. However, the Group Management Restriction List will be considered only if the groups are managed through Attribute Mapping.

  • When managing groups via attribute mapping, users will be removed from groups, if they are not satisfying the group attribute mapping evaluation and the group is mentioned in the Group Management Restriction List. However, users will be added to groups, if they are satisfying the group attribute mapping evaluation and the group is mentioned in the Group Management Restriction List. 

    Notes:
    • If the Group Management Restriction List is empty, then it is treated the same as if all groups are present in the Group Management Restriction List.
    • The Group Management Restriction List must be filled in accurately when using attribute mapping to avoid unintentional removal of users.
  • When you have set group rules and set groups via attribute mapping, the groups will be managed only by considering the groups in the group rules.

  • When the Group Management Restriction List is empty and you have added a group rule which does not have any groups selected in it, then the users will be removed from all the groups.

  • When the Group Management Restriction List is empty and you have not set any group rules and have not set any attribute mapping for groups, the users will neither be added nor removed from groups. However, users will be retained in their existing groups in the target application. 

  • If the attribute mapping evaluation for groups does not return any result for a particular user's preview, then this user will neither be added nor removed from groups. 

  • When the Report Only checkbox is selected and the Report Mode is selected as normal, debug, or verbose, the integration logs will contain the details of the group names to which the users will be added. 

The following is the major difference in the group management functionality in User Integration version 2 from version 3:

In User Integration version 2, when users are managed via group rules, users will be removed from groups, if they are not included in the Group Rules and the group is present in the Group Management Restriction List. However, users will be added to groups, if they are included in the Group Rules and the group is present in the Group Management Restriction List. If the Group Management Restriction List is empty, then it is treated the same as if all groups are present in the list. The Group Management Restriction List must be filled in accurately to avoid unintentional removal of users.
