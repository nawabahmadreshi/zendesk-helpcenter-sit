Active Directory / Jira Service Management Cloud Integration Guide

← Back to index

Overview

The Aquera platform allows your organization to scale complex data management by automating data integration across resource types. Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

In this integration, any contractors created, updated, or terminated in Jira Service Management Cloud are automatically updated in Active Directory based on the defined schedule.

To identify user records across both systems, the integration uses the attribute configured in the Jira Service Management Correlation Attribute parameter from Jira Service Management Cloud and matches it to the userPrincipalName in Active Directory as a correlation attribute.

Key Functions

The Active Directory to Jira Service Management Cloud Integration performs the following functions:

  • Synchronizes contractor records from Jira Service Management Cloud to Active Directory.
  • Creates new contractors in Active Directory when they are added in Jira Service Management Cloud.
  • Updates existing Active Directory contractors when their information changes in Jira Service Management Cloud.
  • Deactivates or deletes Active Directory contractors when they are no longer active in Jira Service Management Cloud.
  • Supports both full synchronization and incremental synchronization to process only changed records.
  • Generates Common Name (CN), User Principal Name (UPN), Security Account Manager (SAM), and Password for contractors in Active Directory based on configured policies.

Prerequisites

The following are the prerequisites:

  • Aquera Tenant Account: You must have a tenant account in the Aquera Admin portal. Visit https://admin.aquera.io to register for a tenant account. Contact your Aquera account manager to get Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io
  • Configure Application Connectors in the Aquera Admin Portal: You need to configure the following application connectors in the Aquera Admin Portal:
    • Jira Service Management Cloud: Refer to the Jira Service Management Cloud Configuration Guide for configuration details.
    • Active Directory: Refer to the Active Directory (AD) Configuration Guide for configuration details.

Integration Setup

This section outlines how the integration must be configured to push user-related data from Jira Service Management to Active Directory.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Select Active Directory / Jira Service Management Cloud.
  5. Enter a name and a description for the integration.
  6. Select your Jira Service Management and Active Directory connector instances.
  7. Click Continue. The Configuration page is displayed to configure the integration.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Parameters

The Parameters section enables you to customize the integration settings according to your specific use case. It has the following settings:

Integration Scenarios

Enable the following feature based on your requirements:

  • Enable Contractor Management: Select this checkbox to enable contractor provisioning and perform incremental updates. When enabled, the Contractor Management, Naming Policy, and Password Policy sections will be displayed for further configuration.

Execution Settings

Configure the following settings:

  • Enable Report-Only Mode: Select this checkbox to execute the integration in the report-only mode. The report-only mode generates a report to show the data that would be changed without actually making those changes.
  • Enable Full Sync: Select this checkbox to execute the integration in full sync for each integration run.
    If this checkbox is not selected, the integration will run in full sync mode only the first time. After that, it will run in incremental mode for subsequent executions.
  • Enable Logging: Select this checkbox to generate logs outlining data changes. When enabled, the Logging section is displayed for further configuration.
  • Enable Report Generation: Select this checkbox to generate the integration report. When enabled, the Reporting section is displayed for further configuration.
  • Enable Notifications: Select this checkbox to enable integration-related notifications. When enabled, the Notifications section is displayed for further configuration.
  • Enable Log Insights: Select this to visualize the integration-related metrics. When enabled, the Log Insights section is displayed for further configuration.

Contractor Management

The Contract Management section enables you to customize the integration settings for contractor management according to your specific use cases.

Active Directory Settings

Configure the following settings:

  • Active Directory Applications for Provisioning: Select one or more Active Directory applications to which contractors must be provisioned. Only the Active Directory applications configured in your Aquera tenant will be available for selection.
  • Delete Contractors Upon Termination: Select this to delete contractor records from the Active Directory upon the contractor record termination in the Jira Service Management Cloud.
  • Map Manager: Select this to include the contractor's manager information when creating or updating a contractor in the Active Directory.

Jira Service Management Settings

Configure the following settings:

  • Jira Project*: Select a Jira project from which contractors should be provisioned to Active Directory.
  • Work Type*: Select a Jira work type from which contractors should be provisioned to Active Directory.
  • Status*: Select a Jira status for filtering and provision contractors with this status to Active Directory.
  • Jira Status on Successful Provisioning*: Select a Jira status to be set after successful contractor provisioning from Jira Service Management Cloud to Active Directory.
  • Active Directory OU Provisioning Jira Attribute*: Select a Jira attribute that determines the contractor’s provisioning into an OU in Active Directory.
  • Jira Attribute for Manager: Select an attribute that stores the contractor’s manager information in Jira.
  • Jira Service Management Filter: Select this checkbox to filter the Jira tickets by Issue ID or Issue Key. When enabled, the following parameter is displayed:
    • Issue ID or Issue Key*: Enter an Issue ID or Issue Key to filter the Jira tickets.

Provisioning Settings

Configure the following settings:

  • Jira Field to Categorize Tickets*: Select a Jira field that categorizes the ticket. The following parameter values vary based on this selection.
  • Create Conditions*: Select one or more create conditions. Contractors meeting these conditions will be created in Active Directory.
  • Update Conditions*: Select one or more update conditions. Contractors meeting these conditions will be updated in Active Directory.
  • Delete Conditions*: Select one or more delete conditions. Contractors meeting these conditions will be deleted in Active Directory.

Correlation Settings

Configure the following correlation settings:

  • Jira Service Management Correlation Attribute*: Select the Jira attribute to distinctly find the associated Active Directory contractor record. For example, if you select Assignee in this field, the Assignee attribute will be used as a correlation attribute.
  • Active Directory Correlation Attribute*: Select the Active Directory attribute to distinctly find the associated Jira contractor record. For example, if you select userPrincipalName in this field, the userPrincipalName attribute will be used as a correlation attribute.

Provisioning Threshold

Configure the following settings:

  • Import Limit: Enter the maximum number of contractor records to be processed in a single integration run. For example, entering 10 processes only includes the first ten contractor records.
  • Create Limit: Enter the maximum number of contractor records that can be created in a single integration run. For example, entering 7 allows only 7 contractor records to be created in a single integration run.
  • Update Limit: Enter the maximum number of contractor records that can be updated in a single integration run. For example, entering 5 updates only 5 contractor records in a single integration run.
  • Delete Limit: Enter the maximum number of contractor records that can be deleted in a single integration run. For example, entering 10 allows only 10 contractor records to be deleted in a single integration run.

Naming Policy

Configure the following naming policy settings:

Generate Common Name (CN) Settings

  • Enter your prompt: Enter your prompt for generating the common name. For example, enter the following prompt and click Analyse and Preview:

    Using the Ticket context (First Name, Last Name), generate different unique human-readable Common Name (CN). 
    Rules:
    - Prefer  First Name and Last Name.
    - Remove invalid AD characters: , + " \ ; < > |
    - Format: "<First Name> <Last Name>"
    - Ensure it looks like a person’s full display name.
    - Add a numeric suffix only if needed for uniqueness.
    Return minimum 10 unique CN string.

    Preview: Based on the prompt and the default input (First Name: john, Last Name: smith), the generated common name values are John Smith, John Smith1, John Smith2, John Smith3, John Smith4, John Smith5, John Smith6, John Smith7, John Smith8, John Smith9, and John Smith10 as displayed under the Preview section. To view the default name values, turn on the Advanced Preview Options toggle.
    Advanced Preview Options: If you want to test with custom values, turn on the Advanced Preview Options toggle to display the Add Custom Values panel. There, you can enter the first name and last name, and click Refresh Preview. Based on these values and the defined prompt, the generated common name values will appear in the Preview section.

Generate User Principal Name (UPN)

  • Enter your prompt: Enter your prompt for generating the user principal name. For example, enter the following prompt and click Analyse and Preview:

    Using the Ticket context (First Name, Last Name), generate a UPN prefix (without @domain).
    Rules:
    - Otherwise build from preferred names: First Name.Last Name.
    - Remove spaces and invalid AD characters.
    - Ensure the prefix is clean, readable, and unique.
    Return minimum 5 UPN prefix string (do not include any domain).

    Preview: Based on the prompt and the default input (First Name: john, Last Name: smith), the generated common name values are john.smith, johnsmith, jsmith, johns, and j.smith as displayed under the Preview section. To view the default name values, turn on the Advanced Preview Options toggle.
    Advanced Preview Options: If you want to test with custom values, turn on the Advanced Preview Options toggle to display the Add Custom Values panel. There, you can enter the first name and last name, and click Refresh Preview. Based on these values and the defined prompt, the generated common name values will appear in the Preview section.

Generate Security Account Manager (SAM)

  • Enter your prompt: Enter your prompt for generating a unique security account manager for a user in Active Directory. For example, enter the following prompt and click Analyse and Preview:

    Using the worker (First Name, Last Name), generate a unique sAMAccountName.
    Rules:
    - Use only letters and numbers; no spaces or symbols.
    - Keep within the 20-character AD limit.
    - Use a short format such as firstname.lastname.
    - Add a numeric suffix only if needed for uniqueness.
    Return minimum 5 sAMAccountName string.

    Preview: Based on the prompt and the default input (First Name: john, Last Name: smith), the generated common name values are john.smith, joh.smith, j.smith, john.smith1, and john.smith2 as displayed under the Preview section. To view the default name values, turn on the Advanced Preview Options toggle.
    Advanced Preview Options: If you want to test with custom values, turn on the Advanced Preview Options toggle to display the Add Custom Values panel. There, you can enter the first name and last name, and click Refresh Preview. Based on these values and the defined prompt, the generated common name values will appear in the Preview section.

Password Policy

The Password Policy section enables you to generate a password for a user in Active Directory.

Password Policy Settings

Configure the following settings:

  • Password Generation Method: Select a method to generate the contractor's password in Active Directory. The available options are Auto Generate Password and Custom Password Policy.
    • Auto-Generate Password: Select this to automatically generate a password in Active Directory.
    • Custom Password Policy: Select this to create a password according to your preference. Selecting this will display the Custom Password Policy Settings section. 

Custom Password Policy Settings

Configure the following Custom Password Policy settings:

  • Password Complexity Condition: Select one or more conditions to apply when creating a contractor password in Active Directory. The available options are Require Uppercase Letters, Require Lowercase Letters, Require Numbers, and Require Special Chars.
  • Password Special Characters: Select one or more special characters to apply when creating a contractor password in Active Directory. For example, select !, @, and # to allow these characters in the password. This parameter is displayed only if Require Special Chars option is selected in the above parameter.
  • Character Type Condition: Enter a number to determine the password combination. For example, if you enter 2, the password will include any two of the selected character types, such as uppercase letters and lowercase letters.
  • Password Length: Specify the desired length of the generated password. For example, enter 12 to generate a 12-character password instead of the default 8-character length.

Rule Builder

The Rule Builder tab is displayed only when the Enable Contractor Management is enabled in the Parameters section.

This integration allows you to configure and map contractor country locations to OUs in Active Directory. Configure each location and specify its corresponding OU path.

Perform the following procedure to set up a rule:

  1. In the CountryLocation column, click the drop-down to view the list of countries and select the preferred country.
  2. In the OU Path column, click the drop-down to view the list of OU paths and select the preferred OU path to be mapped to the country selected in the CountryLocation column.
    Notes:
    • You can add any number of rows within a rule.
    • You can delete a row by using the delete icon at the right end of the row.

       
  3. [Optional] If you want to write a JavaScript expression to set the condition, click Switch to Expression and perform the following steps:
    Note: When you click Switch to Expression, all the configured conditions of the rule will be cleared.
    1. Select one of the following based on your requirement:
      • For Conditions Only: Select this if you want to write a JavaScript expression only for the conditions. 
      • For Conditions and Assignments: Select this if you want to write expression for both, conditions and assignments.
    2. Click Continue to configure the expression.
      Note: Expressions can use all attributes in the source and destination applications. The configured expressions must always be evaluated to true or false.
      SOURCE must be used in an expression to represent an attribute in the source application and DESTto represent an attribute in the destination application.
      For example, SOURCE.EmployeeNumber, SOURCE.HomeOrganization.ShortName, and DEST.EmployeeNumber.
  4. Click Add New Rule to add another rule, as explained above.
    Note: You can add any number of rules. However, you can see only 5 rules on a single page. If you add more than 5 rules, navigation options will be available at the bottom of the page to navigate to other pages for configuring or viewing the remaining defined rules.

Attribute Map

The Attribute Map tab defines how data is mapped from the source to the target application. The default mapping of attributes is displayed on the page. The source attributes are listed on the left side and the target attributes are listed on the right side of the page.

You can modify only the source attributes as needed, to align with your specific use case.

You can write JavaScript expressions to modify the value of the source attribute and construct a value to be updated in the target attribute. Write the expression directly in the source attribute field or click the Expand icon in the source attribute field to write a long-expression.

All attributes marked with a red asterisk (*) indicates required fields for successful provisioning.

After the source-to-target attribute mapping is completed, click Save Changes at the top-right corner of the page to save the integration.

Tables

Tables can be added to create custom source attributes and POST (sync) them to the target application. Tables are used when an application attribute is empty or a specific data is not sent from the Jira Service Management Cloud application, and you want to update a value for that attribute in Active Directory. Based on your requirement, you can add any number of tables in the integration.

The table setup and functionality are the same as that in the User Integration. Refer to the Adding Tables in Integration section in the Using Tables in Integrations guide for details.
