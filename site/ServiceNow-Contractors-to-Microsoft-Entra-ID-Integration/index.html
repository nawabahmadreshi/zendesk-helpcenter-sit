ServiceNow Contractors to Microsoft Entra ID Integration

← Back to index

Overview

The Aquera platform allows your organization to scale identity and access management by automating integrations across IT service management and identity platforms. Aquera’s integrations facilitate automated lifecycle governance, ensuring consistency, security, and compliance across systems. This enables organizations to streamline contractor access management, enforce expiration policies, and automate approval-driven processes, ultimately improving operational efficiency and reducing security risk.

This integration connects ServiceNow with Microsoft Entra ID to manage contractor identities and their access lifecycle. It uses ServiceNow as the system of record for contractor access approvals and Microsoft Entra ID as the system that enforces contractor access. The integration ensures that contractor access in Entra ID is always controlled by a defined end date and governed by ServiceNow approval workflows.

The integration continuously monitors contractor accounts in Entra ID and evaluates their access end dates. When a contractor is approaching expiration, the integration automatically creates an access extension request in ServiceNow so that approvals can be reviewed and completed before access expires.

ServiceNow ticket status and approval outcomes control whether any changes are applied. The integration extends a contractor’s end date in Microsoft Entra ID only when the associated ServiceNow ticket reaches the configured approved or resolved status. If the ticket is not approved, no extension is applied, ensuring contractor access does not continue beyond the approved timeframe.

The integration ensures that contractor access is time-bound, approval-based, and fully auditable. Every extension applied in Entra ID can be traced back to a corresponding ServiceNow ticket.

The integration supports both full execution and incremental runs. A full execution evaluates all eligible contractor accounts, while incremental runs process only those contractors or ServiceNow tickets whose data has changed since the previous execution.

This guide explains how to configure and set up the integration to synchronize contractor lifecycle and approval outcomes from ServiceNow to Microsoft Entra ID, ensuring contractor access remains secure, approved, and compliant.

Key Functions

The ServiceNow Contractors to Microsoft Entra ID integration provides the following functions:

  • Automatically detects contractor accounts in Microsoft Entra ID that are approaching their configured end date based on the defined expiration threshold.

  • Automatically creates ServiceNow contractor access extension request tickets when a contractor is nearing expiration.

  • Continuously monitors ServiceNow tickets and evaluates their status to determine whether contractor access is approved for extension.

  • Extends the contractor’s end date in Microsoft Entra ID only when a matching ServiceNow ticket reaches an approved or resolved status.

  • Automatically closes the ServiceNow ticket after the contractor’s end date is successfully extended, when auto-close is enabled.

  • Correlates ServiceNow tickets with Entra ID contractor identities using configured correlation attributes to ensure the correct account is updated.

  • Keeps contractor end dates in Entra ID continuously aligned with the latest approved information from ServiceNow.

  • Supports full execution, incremental execution, and report-only mode. In report-only mode, the integration generates reports without creating tickets or updating Entra ID.

  • Generates detailed logs, notifications, and execution reports for contractor expiration detection, ticket creation, access extensions, and integration runs.

Prerequisites

The following are the prerequisites:

  • Tenant Account in the Aquera Admin Portal:
    You must have a tenant account with admin access in the Aquera Admin Portal. Visit https://admin.aquera.io to register for a tenant account. Contact your Aquera account manager to obtain Aquera tenant access.
    Note: If your organization is in the EU region, use https://admin.eu.aquera.io to register.

  • Configure Application Connectors in the Aquera Admin Portal:
    You must configure the following application connectors in the Aquera Admin Portal:

    • ServiceNow: Add and configure the ServiceNow connector in the Aquera Admin Portal with the appropriate API credentials. The connector must have permissions to read, create, and update tickets (such as Incidents or Requests) and access required ticket attributes. Refer to the ServiceNow Configuration Guide for details on connector setup and authentication.

    • Microsoft Entra ID: Add and configure the Microsoft Entra ID connector in the Aquera Admin Portal. This connector is used to read and update contractor user attributes, including contractor end dates. Refer to the Microsoft Entra ID Configuration Guide for details on app registration, connector setup, and authentication.

Integration Setup

This section explains how to set up the ServiceNow Contractors to Microsoft Entra ID integration in the Aquera Admin Portal to manage contractor access expiration and extensions.

Use this integration when you want ServiceNow approvals to control whether a contractor’s End Date is extended in Microsoft Entra ID.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin Portal.
  2. In the left navigation panel, select Integrations.
  3. Click Create Integration.
  4. Select ServiceNow Contractors to Entra ID as the integration template.
  5. Enter a name and description for the integration.
  6. Select the connector instances that the integration will use:
    • ServiceNow
    • Microsoft Entra ID

  7. Click Continue.

    The Configuration page is displayed to complete the integration setup.

Configuration

The Configuration tab is used to review and customize the behavior of the ServiceNow Contractors to Entra ID integration.

The following sections are available under the Configuration tab:

  • Applications
  • Parameters

Applications

The Applications section lists the application connectors used by the integration. It displays the configured source and target systems.

For example, you can confirm that the integration is using the correct ServiceNow production instance and the correct Microsoft Entra ID tenant connector before running the integration.

Parameters

The Parameters section allows you to tailor the integration to a specific use case. It includes configurable options that control how contractor extension tickets are created in ServiceNow and how contractor end dates are updated in Microsoft Entra ID.

Integration Scenarios

Use the Integration Scenarios section to configure the main functional behavior of the integration.

  • Manage Contractor End Date: Select to monitor and manage contractor end dates in Microsoft Entra ID. When enabled, the integration evaluates contractor accounts to determine whether access is nearing expiration and whether an extension request is required.

    For example, if a contractor’s access expires in 15 days and the configured threshold is 30 days, the integration identifies the contractor as eligible for an extension request.

General Settings

Configure the following general settings:

  • Enable Report-Only Mode: Select to generate a report detailing the data changes that would happen, but without applying them. For example, you can review which contractor extension tickets would be created and which end dates would be updated, without making any changes.

  • Enable Logging: Select to generate logs outlining data changes. For example, capture logs for contractor expiration checks, ticket creation, and approval-based updates. Enabling this will enable fields under the Logging section.

  • Enable Report Generation: Select to generate the integration report. For example, generate a report summarizing contractors scanned, tickets created, and extensions applied. Enabling this will enable fields under the Reporting section.

  • Enable Notifications: Select to enable the integration-related notifications.
    For example, send email notifications when the integration completes successfully or fails. Enabling this will enable fields under the Notifications section.

  • Enable Log Insights: Select to visualize the integration-related metrics. For example, view metrics showing how many contractors are nearing expiration or extended. Enabling this will enable fields under the Log Insights section.

  • Enable Incremental Mode: Select to run the integration in incremental mode. For example, process only contractors or ServiceNow tickets that changed since the last run.

  • Enable Debug Mode: Select to generate logs outlining data changes that would occur for debugging. For example, troubleshoot correlation or approval detection logic without impacting production behavior. Enabling this will enable fields under the Debug Mode section.

Contractor Creation

Source Filter Settings

Configure the following source filter settings:

  • Source Filter Key: Select the ServiceNow field used to filter which records are processed by the integration. For example, select the RITM/Ticket number to target a specific request item.
  • Source Filter Value: Enter the value that must match the selected source filter key. For example, enter RITM0011428 to process only that specific ServiceNow request.

Contractor Creation Settings

Configure the following contractor creation settings:

  • ServiceNow Workflow Trigger: Select the ServiceNow workflow mechanism used to initiate contractor identity provisioning requests. For example, select Service Catalog to trigger provisioning from catalog requests.
  • Service Catalog Category: Select the ServiceNow service catalog category used for contractor onboarding requests. For example, select Services so contractor onboarding requests are routed through the Services catalog.
  • ServiceNow Ticket Category: Select the ServiceNow ticket category used to classify contractor onboarding requests. For example, select New Contractor Onboarding12121 to ensure requests follow the correct workflow and SLA.
  • Match Ticket/RITM Statuses: Select the ServiceNow ticket or RITM status that initiates contractor identity provisioning workflows. For example, select Work in Progress and Closed Complete so provisioning starts only when requests reach these stages.
  • Auto Close Ticket After Provisioning: Select to close the ServiceNow ticket automatically after successful contractor identity provisioning in Microsoft Entra ID. For example, close the ServiceNow request automatically once the contractor account is created.

ServiceNow to Entra ID Correlation Settings

Configure the following match expression settings:

  • Source Correlation Attribute: Select the unique identifier attribute from ServiceNow that is used for identity correlation and record matching. For example, select Employee ID or Request ID from ServiceNow to uniquely identify the contractor.
  • Target Correlation Attribute: Select the unique identifier attribute from Microsoft Entra ID used for identity correlation and record matching. For example, select Employee Id to match the ServiceNow contractor record to the correct Entra ID user.

Ticket Closure Settings

Configure the following ticket closure settings:

  • Post-Provisioning Ticket Status: Select the ServiceNow ticket status to transition to after successful contractor identity provisioning in Microsoft Entra ID. For example, select Closed Complete to indicate that the contractor onboarding workflow completed successfully.

Expiry Notification Settings

Configure the following expiry notification settings to control how contractor expiration alerts and extension workflows are triggered.

  • Enable Expiry Notifications: Enable automatic notifications when contractor access is approaching its end date in Microsoft Entra ID. For example, enable this option to notify stakeholders when a contractor’s access is about to expire.
  • Notification Days Before Expiry: Select the number of days before the contractor’s end date when notifications should be sent. For example, select 30 to send notifications 30 days before access expires.
  • Create Contractor Extension Tickets Automatically: Enable automatic creation of ServiceNow extension tickets when contractors approach their end date. For example, enable this option to automatically create a ServiceNow ticket when a contractor has 30 days remaining.
  • Update Contractor End Date: Enable updating the contractor’s end date in Microsoft Entra ID after approval is completed in ServiceNow. For example, enable this option to extend the Entra ID end date once the ServiceNow ticket is approved.
  • Create Tickets X Days Before Contractor Gets Expired: Specify how many days before expiration the extension ticket should be created in ServiceNow. For example, enter 30 to create the extension ticket 30 days before the contractor’s access expires.

Source Filter Settings

Configure the following source filter settings to limit which contractor records are processed during the execution.

  • Source Filter Key for Extension Ticket Creation: Select the Entra ID attribute used to identify which contractor accounts should trigger extension ticket creation. For example, select User Principal Name to target a specific contractor account.
  • Source Filter Value for Extension Ticket Creation: Enter the value that matches the selected filter key to narrow processing to a specific contractor. For example, enter testuser@company.com to process only that contractor.
  • Source Filter Key for End Date Extension: Select the ServiceNow field used to identify which ticket should be used for extending the contractor's end date. For example, select RITM Number / Ticket Number to target a specific ServiceNow request.
  • Source Filter Value for End Date Extension: Enter the ServiceNow ticket value that should be used for the end date extension. For example, enter INC0011042 to extend access based on that ticket.

Contractor Extension Settings

Configure the following contractor extension settings to control how ServiceNow tickets are used to extend contractor access:

  • ServiceNow Workflow Trigger: Select how extension requests are triggered in ServiceNow. For example, select Ticket to use incident-based workflows instead of service catalog requests.
  • ServiceNow Ticket Type: Select the ServiceNow ticket type used for contractor extension requests. For example, select Incident to process access extensions using incident tickets.
  • Match Ticket / RITM Statuses for Extension Ticket Creation: Select the ServiceNow ticket statuses that allow creation of extension tickets. For example, select New, In Progress, and On Hold to allow ticket creation during these stages.
  • End Date Extension Statuses: Select the ServiceNow ticket statuses that allow the contractor’s end date to be extended in Microsoft Entra ID. For example, select Resolved so the end date is extended only after the ticket is approved and completed.
  • Auto-Close Ticket After Extension: Enable automatic closure of the ServiceNow ticket after the contractor end date is successfully updated in Entra ID. For example, enable this option to close the ticket once access is extended.

ServiceNow to Entra ID Correlation Settings

Configure the following correlation settings to ensure the correct contractor account is updated.

  • ServiceNow Correlation Attribute: Select the ServiceNow attribute used to uniquely identify the contractor record. For example, select Employee Number to uniquely identify the contractor in ServiceNow.
  • Entra Correlation Attribute: Select the Microsoft Entra ID attribute used to match the contractor account. For example, select Employee Id to match the ServiceNow contractor to the correct Entra ID user.

Notification Center

Use the Notification Center to configure when and how email notifications are sent for contractor creation, success events, and contractor expiry.

General Notification Settings

Configure the following general notification settings:

  • Send Email Notification On: Select the contractor events that should trigger email notifications. For example, select Contractor Creation, Success Event, and Contractor Expiry Notification to send emails for all major contractor lifecycle events.

Contractor Creation Notifications

Configure the following notification settings for contractor creation events:

  • Primary Recipients: Enter the email addresses of the recipients who should receive contractor creation notifications. For example, enter idm-admin@company.com to notify the identity team when a contractor account is created.
  • CC Recipients: Enter email addresses that should be copied on contractor creation notifications. For example, enter it-support@company.com to keep the support team informed.
  • Email Subject: Enter the content that must be used as the email subject for contractor creation notifications. For example, enter New Contractor Account Created  {{userName}}.
  • User Creation Email Template: Enter the content that must be used as the email body for contractor creation notifications. For example, include placeholders such as {{userType}}, {{creationDate}}, and {{createdBy}} to dynamically populate contractor details.

Success Notifications

Configure the following notification settings for successful integration execution:

  • Primary Recipients: Enter the email addresses of the recipients who should receive success notifications. For example, enter idm-admin@company.com to notify admins when the contractor sync completes successfully.
  • Custom Recipients: Enter additional email addresses to notify when contractor provisioning runs successfully. For example, enter security@company.com to keep the security team informed.
  • CC Recipients: Enter email addresses that should be copied on success notifications. For example, enter audit@company.com for compliance visibility.
  • Email Subject: Enter the content that must be used as the email subject for success notifications. For example, enter Contractor Sync Succeeded {{operationName}}.
  • Success Event Email Template: Enter the content that must be used as the email body for success notifications. For example, include summary details such as records processed, success time, and next steps.

Expiry Email Templates

Configure the following notification settings for contractor expiry alerts:

  • Primary Recipients: Enter the email addresses of the recipients who should receive contractor expiry notifications. For example, enter manager@company.com to notify managers when contractor access is about to expire.
  • CC Recipients: Enter email addresses that should be copied on contractor expiry notifications. For example, enter idm-admin@company.com to ensure visibility for the identity team.
  • Email Subject: Enter the content that must be used as the email subject for contractor expiry notifications. For example, enter Contractor Expiry Alert  {{daysUntilExpiry}} Days Remaining.
  • Contractor Expiry Email Template: Enter the content that must be used as the email body for contractor expiry notifications. For example, include contractor details such as name, employee ID, and end date so reviewers can take timely action.

Naming Policy Settings

Configure how contractor usernames and email addresses are generated.

  • Generate Contractor Principal Name (UPN): Enable automatic generation of a unique User Principal Name for contractors in Microsoft Entra ID. For example, enabling this allows the system to generate a UPN such as barbara.jane based on the configured naming rules.

  • Generate Contractor Principal Name (UPN) Using: Select the method used to generate the contractor UPN. For example, select Use Custom Policy to Generate UPN to define naming logic that produces values like barbara.jane.

  • Generate Email: Enable automatic email address generation for contractors. For example, enabling this allows the system to generate an email address such as barbara.jane@company.com during provisioning.

  • Generate Email Using: Select how the email prefix, the part before the @ symbol, should be generated. For example, selecting UserPrincipalName generates barbara.jane@company.com.

Generate Contractor Principal Name (UPN)

Configure how unique contractor UPNs are created.

  • Unique UPN Generation Pattern: Select the pattern used to generate a unique contractor UPN.
    For example, select Custom to define rules that create barbara.jane.

  • Generate Contractor Principal Name (UPN) (Prompt): Enter the instructions that describe how the UPN should be generated. For example, enter firstName.lastName to generate barbara.jane.

  • Code View: Switch to a code-based view to edit the UPN generation logic directly. For example, adjust logic to handle duplicates when multiple users are named Barbara Jane.

    • Analyse and Preview: Generate and analyze sample UPN values to understand how names will be created. For example, preview values such as barbara.jane and jane.barbara.

  • Preview: Review sample UPN values generated using the current policy. For example, preview barbara.jane before saving the configuration.

    • Advanced Preview Options: Display additional preview details for generated UPN values. For example, view variations like barbara.jane1 when duplicates exist.

  • Entra ID Domain (DNS Name): Enter the primary DNS domain used for your Microsoft Entra ID tenant. This domain is used as the suffix for contractor UPNs and email addresses. For example, enter aqueratest.mail.onmicrosoft.com so a contractor with the generated UPN barbara.jane becomes barbara.jane@aqueratest.mail.onmicrosoft.com.

Email Generation Settings

Configure how email addresses are generated for contractor accounts.

  • Email Domain: Enter the domain that will be used as the suffix for contractor email addresses. For example, enter aqueralabs.com so the email address ends with @aqueralabs.com.
  • Unique Email Generation Pattern: Select the pattern used to generate a unique email address for contractors. For example, select Custom to control how the email prefix is created for barbara.jane@aqueralabs.com.
  • Generate Email (Prompt): Enter instructions that describe how the email address should be generated. For example, enter firstName.lastName ending with @aqueralabs.com to generate barbara.jane@aqueralabs.com.
    • Code View: Switch to a code-based view to edit the email generation logic directly. For example, advanced users can modify the logic used to generate barbara.jane@aqueralabs.com.
    • Analyse and Preview: Generate and review sample email addresses in accordance with the current rules. For example, confirm that barbara.jane@aqueralabs.com is generated as expected.
  • Preview: Review sample email values generated from the configured pattern. For example, preview results such as barbara.jane@aqueralabs.com, jane.barbara@aqueralabs.com, or b.jane@aqueralabs.com.
    • Advanced Preview Options: Display additional preview details for generated email values. For example, review how duplicate email addresses are handled if barbara.jane@aqueralabs.com already exists.

Password Policy Settings

Configure how passwords are generated and enforced for contractor accounts.

Password Generation Settings

  • Password Generation Method: Select how passwords should be generated for contractor accounts.
    For example, select Custom Password Policy to generate passwords using organization-defined rules.

Custom Password Policy

  • Minimum Character Length: Set the minimum number of characters required for contractor passwords. For example, enter 10 to require passwords to contain at least 10 characters.

  • Allowed Special Characters: Select which special characters can be used when generating passwords. For example, select $ and % so generated passwords may include these characters.

  • Password Complexity Requirements: Select the character types that must be included in each password. For example, enable Require Uppercase Letters, Require Lowercase Letters, Require Numbers, and Require Special Characters to ensure strong passwords.

Log Insights

Configure how much detail is captured and displayed in integration logs.

  • Select Log Insight Type: Select the level of detail to include in integration logs. This controls whether logs show only operations, field names, identifiers, or actual data values.
    For example, selecting Employee Number with All Data logs the employee number, operations performed, field names, and all values, including PII and non-PII data.

Available Log Insight Types

  • Minimum: Captures only the type of operations performed, without including any identifiers or field details. For example, logs show that a user was created or updated, but do not include employee numbers or field names.

  • Metadata: Captures operations and field names without including any identifiers or values.
    For example, logs show that the End Date field was updated, but do not show the employee number or actual value.

  • Non-PII (Personally Identifiable Information) Data: Captures operations, field names, and selected non-PII values, without including identifiers. For example, logs show department changes but do not expose employee numbers or personal details.

  • GUID with Non-PII Data: Capture system identifiers (GUID), operations, field names, and selected non-PII values. For example, logs include a unique system ID to trace changes without exposing personal information.

  • Employee Number with All Data: Capture employee number, operations, field names, and all values, including PII and non-PII data. For example, logs show that Employee Number 12345 had the End Date updated from one value to another.
