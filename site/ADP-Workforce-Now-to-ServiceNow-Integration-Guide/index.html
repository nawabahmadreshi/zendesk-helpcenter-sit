ADP Workforce Now to ServiceNow Integration Guide

← Back to index

Overview

The Aquera platform allows organizations to automate data movement and lifecycle processes across business applications, helping teams streamline provisioning, reduce manual effort, and maintain consistency across systems.

The ADP Workforce Now to ServiceNow integration retrieves worker data from ADP Workforce Now, evaluates onboarding and offboarding eligibility, and creates corresponding ServiceNow Service Catalog / RITM-based tickets for downstream fulfilment. The integration also supports destination validation checks, date-offset processing, duplicate-ticket prevention through correlation caching, execution reporting, email notifications, and analytics logging.

Key Features

  • Automated onboarding and offboarding: Creates ServiceNow tickets for new hires and terminated workers based on configured conditions.

  • Service Catalog Item-based ticket creation: Uses ServiceNow Service Catalog Items to generate onboarding and offboarding requests.

  • User validation before ticket creation: Validates user presence in systems such as Okta, Entra ID, and ServiceNow before creating tickets. If the user is not found, ticket creation is deferred until the user becomes available.

  • Date offset support: Allows scheduling of onboarding and offboarding tickets before or after the actual event date based on configured offsets.

  • Duplicate ticket prevention: Prevents duplicate ticket creation by tracking existing requests and allowing recreation only for selected states.

  • Incremental processing: Processes only records that have changed since the last successful run, improving efficiency.

Integration Setup

This section outlines the configuration required to integrate ADP Workforce Now with ServiceNow, enabling seamless transfer of worker data from ADP Workforce Now to ServiceNow.

Create Integration

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations from the left side panel.
  3. Click Create Integration.
  4. Search for ADP Workforce Now / ServiceNow in the search box next to the Explore Templates section.
  5. Click the integration template to display the Create New Integration pop-up window.
  6. Enter a name and a description for the integration.
  7. Select both application connector instances from the dropdown menus.
  8. Click Continue. The integration configuration page is displayed to configure the integration.

Configuration

The following sections are displayed under the Configuration tab:

Applications

The Applications section displays the source and target application connectors for the integration. This section allows you to search and update these application connectors.

Parameters

The Parameters section enables you to customize the integration settings according to your specific use case. It has the following settings:

All attributes marked with a red asterisk (*) indicate required fields.

Integration Features

Configure the following integration features:

  • Enable onboarding ticket generation: Select to create ServiceNow onboarding tickets for new hires in ADP Workforce Now.
    When Enable onboarding ticket generation is selected, the Manage Hires tab appears.
  • Enable offboarding ticket generation: Select to create ServiceNow offboarding tickets for terminated workers in ADP Workforce Now.
    When the Enable offboarding ticket generation is selected, the Manage Terminations tab appears.
  • Enable date offset: Select to enable date offset, allowing records to synchronize based on the specified date offset and time zone.
    When the Enable date offset is selected, the Date Offset tab appears.

Operational Settings

Configure the following integration features:

  • Run in report-only mode: Simulate the integration run and generate results without making any changes to the target system.
  • Enable debug mode: Select to generate logs outlining data changes that would occur for debugging.
  • Run in incremental mode: Run the integration using only records that have changed since the last successful execution.
  • Enable log insights: Select to visualize the integration-related metrics.
  • Enable Logging: Select to generate logs outlining data changes.
  • Enable notifications: Select to enable integration-related notifications.

Manage Hires

This tab enables you to configure onboarding settings. 

Note: The Manage Hires tab appears when you select the Enable onboarding ticket generation checkbox in the Integration Features section of the Parameters tab.

Onboarding Tickets Generation

Configure the following onboarding ticket generation settings:

  • Onboarding condition type: Select the condition to create onboarding tickets in ServiceNow.
    The following options are available in the dropdown:
    • All Workers: Creates onboarding tickets for all workers from ADP Workforce Now.
    • All Active Workers: Creates onboarding tickets only for workers with an active status.
    • All Active Full-Time Workers: Creates onboarding tickets only for active full-time workers.
    • Custom: Allows you to define a custom condition using an expression.
      When the Custom option is selected, the following field is displayed:
      • Custom onboarding condition: Enter the expression to follow when creating onboarding tickets in ServiceNow.
  • ServiceNow trigger for onboarded worker: Creates onboarding tickets using the Service Catalog Item trigger. 
    Note: The integration currently only supports the Service Catalog Item trigger, so this field cannot be configured.
  • ServiceNow catalog item for onboarded worker: Select the Service Catalog Item that will be used to create onboarding tickets in ServiceNow. For example, selecting Onboarding_ADP ensures that onboarding requests are created using that catalog item template.
  • Validate user in Okta before ticket creation: Select to ensure a ServiceNow onboarding ticket is created only after the user is verified in your selected Okta instance. 
    Note: If the worker is not yet found in Okta during the sync, the ticket creation will be deferred until the user is successfully identified in a subsequent run. 
    When you select Validate user in Okta before ticket creation, the Okta instance dropdown appears:
    • Okta instance: Select the Okta instance from the dropdown menu. The worker will be verified in this selected instance.
  • Validate user in Entra ID before ticket creation: Select to ensure a ServiceNow onboarding ticket is created only after the user is verified in your selected Entra ID instance. 
    Note: If the worker is not yet found in Entra ID during the sync, the ticket creation will be deferred until the user is successfully identified in a subsequent run. 
    When you select Validate user in Entra ID before ticket creation, the Entra ID instance dropdown appears:
    • Entra ID instance: Select the Entra ID instance from the dropdown menu. The worker will be verified in this selected instance.
  • Validate user in ServiceNow before ticket creation: Select to ensure a ServiceNow onboarding ticket is created only after the user is verified in ServiceNow. 
    Note: If the worker is not yet found in ServiceNow during the sync, the ticket creation will be deferred until the user is successfully identified in a subsequent run. 
  • Start date for onboarding ticket generation: Specify the date from which the workers' records in the ADP Workforce Now are considered for onboarding ticket creation in ServiceNow.
  • Allowed statuses for duplicate onboarding tickets: Select the RITM states that permit creating a new onboarding ticket when an existing record is already present.

  • Onboarding Subtask Type to Close: Select the onboarding catalog task type under the RITM that should be identified and closed from the dropdown.
    The following are the available:

    • None: No subtask type is selected, and no onboarding subtasks will be automatically closed.
    • SCTASK: Identifies and closes ServiceNow catalog tasks (SCTASK) under the onboarding RITM that match the configured criteria.
      When SCTASK is selected the following fields are displayed: 

      • Onboarding Subtask Short Descriptions: Enter one or more catalog task short descriptions to identify the tasks that should be closed. For example, entering Create User Account, Assign Laptop ensures that only tasks with matching short descriptions are processed.

      • Onboarding Subtask States Eligible for Closure: Select one or more task states that qualify for closure. For example, selecting Open and Work in Progress ensures that only tasks currently in these states are updated.

Correlation Settings

Configure the following correlation settings:

  • ADP Workforce Now correlation attribute: Select an attribute that uniquely identifies a worker in ADP Workforce Now.
  • ServiceNow correlation attribute: Select an attribute in ServiceNow that matches the selected ADP Workforce Now attribute value to identify common users across ADP Workforce Now and ServiceNow.
  • Okta correlation attribute: Select an attribute in Okta that matches the selected ADP Workforce Now attribute value to identify common users across ADP Workforce Now and Okta. This field is required only when you have selected the Validate user in Okta before ticket creation checkbox.
  • Entra ID correlation attribute: Select an attribute in Entra ID that matches the selected ADP Workforce Now attribute value to identify common users across ADP Workforce Now and Entra ID. This field is required only when you have selected the Validate user in Entra ID before ticket creation checkbox.

Manage Terminations

This tab enables you to configure offboarding settings. 

Note: The Manage Terminations tab appears when you select the Enable offboarding ticket generation checkbox in the Integration Features section of the Parameters tab.

Offboarding Tickets Generation

Configure the following settings for offboarding ticket generation:

  • Offboarding condition type: Select the condition type to create offboarding tickets in ServiceNow.
    The following options are available: 
    • Basic: Allows you to select predefined worker statuses for offboarding.
      When you select Basic, the following field is displayed: 
      • Offboarding conditions: Select one or more worker conditions to determine which workers are marked for offboarding. For example, you can select All Terminated Workers, All Deceased Workers, or All Retired Workers.
    • Custom: Allows you to define a custom condition using an expression.
      When you select Custom, the following field is displayed: 
      • Custom offboarding condition: Enter the expression to follow when creating offboarding tickets in ServiceNow.
  • ServiceNow trigger for offboarded worker: Creates offboarding tickets using the Service Catalog Item trigger.
    Note: The integration currently supports only the Service Catalog Item trigger, so this field cannot be configured.
  • ServiceNow catalog item for offboarded worker: Select a catalog item from the dropdown menu. This catalog item will be added for the offboarded worker.
  • Retrieve future termination date from ADP Workforce Now: Select to retrieve the upcoming termination dates for a worker or workers from ADP Workforce Now.
  • Start date for offboarding ticket generation: Specify the date from which the workers' records in the ADP Workforce Now are considered for offboarding ticket creation in ServiceNow.
  • Allowed statuses for duplicate offboarding tickets: Select the Request Item states that permit creating a new offboarding ticket when an existing record is already present.

  • Offboarding Subtask Type to Close After Termination: Select the offboarding catalog task type under the request item that should be identified and closed from the dropdown.
    The following options are available:

    • None: No subtask type is selected, and no offboarding subtasks will be automatically closed.

    • SCTASK: Identifies and closes ServiceNow catalog tasks (SCTASK) under the offboarding RITM that match the configured criteria.
      When SCTASK is selected, the following fields are displayed:

      • Offboarding Subtask Short Descriptions: Enter one or more catalog task short descriptions to identify the tasks that should be closed. For example, entering Disable User Account, Revoke Access ensures that only tasks with matching short descriptions are processed.

      • Offboarding Subtask States Eligible for Closure: Select one or more task states that qualify for closure. For example, selecting Open and Work in Progress ensures that only tasks currently in these states are updated.

      • Validate User Termination in Okta Before Closing ServiceNow Tasks: Select to ensure that the user is confirmed as terminated in Okta before automatically closing associated ServiceNow catalog tasks.

      • Validate User Termination in Entra ID Before Closing ServiceNow Tasks: Select to ensure that the user is confirmed as terminated in Entra ID before automatically closing associated ServiceNow catalog tasks. When enabled, the system verifies the user’s termination status before closing tasks to prevent premature completion.

Correlation Settings

Configure the following correlation settings for offboarding ticket creation:

  • ADP Workforce Now Correlation Attribute: Select an attribute that uniquely identifies a user in ADP Workforce Now.
  • ServiceNow Correlation Attribute: Select an attribute in ServiceNow that matches the selected source attribute value to identify common users across ADP Workforce Now and ServiceNow.
  • Okta correlation attribute: Select an attribute in Okta that matches the selected ADP Workforce Now attribute value to identify common users across ADP Workforce Now and Okta. This field is required only when you have selected the Validate user in Okta before ticket creation checkbox.
  • Entra ID correlation attribute: Select an attribute in Entra ID that matches the selected ADP Workforce Now attribute value to identify common users across ADP Workforce Now and Entra ID. This field is required only when you have selected the Validate user in Entra ID before ticket creation checkbox.

Additional Instances

This section allows you to configure additional identity provider instances for validation during onboarding and offboarding.

Configure the following settings:

  • Okta instance: Select the Okta application instance to be used for user validation. For example, selecting Okta_exa ensures that users are validated against that Okta environment.
  • Entra ID instance: Select the Entra ID application instance to be used for user validation. For example, selecting Office365_exa ensures that users are validated against that Entra ID environment.

Note: These instances are used when validation options are enabled to verify whether a user exists or is terminated before processing tickets.

Date Offset

This tab enables you to configure date offset settings. All attributes marked with a red asterisk (*) indicate required fields.

Note: The Date Offset tab appears when you select the Enable date offset checkbox in the Integration Features section of the Parameters tab.

Onboarding Date Offset

Configure the following date offset settings for onboarding:

  • Onboarding date offset type*: Select an option from the dropdown menu that determines the onboarding date offset.

    • No Offset: Creates onboarding tickets on the actual onboarding date.

    • Offset Before Onboarding Date: Creates onboarding tickets a specified number of days before the onboarding date.
      When Offset Before Onboarding Date is selected, the following field is displayed:

      • Days of offset before the onboarding date: Specify the number of days before the onboarding date to create an onboarding ticket in ServiceNow.

    • Offset After Onboarding Date: Creates onboarding tickets a specified number of days after the onboarding date.
      When Offset After Onboarding Date is selected, the following field is displayed:

      • Days of offset from the onboarding date: Specify the number of days after the onboarding date to create an onboarding ticket in ServiceNow.

  • Onboarding offset time zone priority: Select the options from the dropdown menu in an order that determines the time zone for onboarding ticket creation in ServiceNow.
    Note: If the first selected time zone is unavailable, the system evaluates the next option in the list.
  • Onboarding offset time*: Select the time at which the onboarding ticket is created based on the calculated offset.

Offboarding Date Offset

Configure the following date offset settings for offboarding:

  • Offboarding date offset type*: Select an option from the dropdown menu that determines the offboarding date offset.
    The following are the available options:

    • No Offset: Creates offboarding tickets on the actual offboarding date without any delay or advance.

    • Offset Before Offboarding Date: Creates offboarding tickets a specified number of days before the offboarding date.
      When Offset Before Offboarding Date is selected, the following field is displayed:

      • Days of offset before offboarding*: Specify the number of days before the offboarding date to create an offboarding ticket in ServiceNow.

    • Offset After Offboarding Date: Creates offboarding tickets a specified number of days after the offboarding date.
      When Offset After Offboarding Date is selected, the following field is displayed:

      • Days of offset after offboarding*: Specify the number of days after the offboarding date to create an offboarding ticket in ServiceNow.
  • Offboarding offset time zone priority: Select the options from the dropdown menu in an order that determines the time zone for offboarding ticket creation in ServiceNow.
  • Offboarding offset time*: Select the time at which the offboarding ticket is created based on the calculated offset.

Log Insights Settings

This section allows you to configure analytics and dashboard visibility for integration activity and performance metrics.

Configure the following Log Insights settings:

  • Enter Dashboard Name: Enter the dashboard name where integration analytics will be displayed. For example, entering ADPWFN creates or updates a dashboard with that name to track execution metrics.
  • Select Inisights Type: Select the type of data to include in insights logs. For example, choosing Employee Number with All Data includes employee number, operations, field names, and all values (PII and non-PII).
  • Select insights Level: Select the level of detail to include in Log Insights.
    The following options are available in the dropdown:
    • Include List: Includes only the attributes specified in the Custom Inclusion List field.
      If Include List is selected, the following field is displayed:
      • Custom Inclusion List: Enter the attributes that should be included in Log Insights. For example, entering firstName, lastName, and email ensures only these fields are captured in logs.
    • Exclude list: Includes all attribute details except those specified in the Custom Exclusion List field.
      • Custom Exclusion List: Enter the attributes that should be excluded from Log Insights. For example, entering salary, bonus, and SSN ensures these fields are not captured in logs.
  • Enter PII Fields: Enter the fields that contain personally identifiable information (PII) to be included in analytics logs. For example, adding dateOfBirth or SSN includes these fields in the logs based on the selected log insights type.
  • Enter GUID Fields: Enter the fields that contain globally unique identifiers to ensure they are handled correctly in analytics logs. For example, adding workerId ensures the value is treated as a system identifier.
  • Enter EmployeeId Fields: Enter the fields that represent employee identification values to be displayed consistently in analytics. For example, adding associateId ensures the employee identifier is included in analytics reports.

      If Exclude List is selected, the following field is displayed:
      • Custom Exclusion List: Enter the attributes that should be excluded from logs. For example, entering salary, and SSN ensures these fields are not captured.
  • Enter PII Fields: Enter the fields that contain personally identifiable information (PII) to be included in logs. For example, adding dateOfBirth or SSN includes these fields based on the selected log type.
  • Enter GUID Fields: Enter the fields that contain globally unique identifiers to ensure they are handled correctly in logs. For example, adding workerId ensures the value is treated as a system identifier.

Attribute Map

The ADP Workforce Now to ServiceNow integration maps the ADP Workforce Now and ServiceNow attributes. You can modify only the source attributes as needed to align with your specific use case.

You can write JavaScript expressions to modify the value of the source attribute and construct a value to be updated in the target attribute. Write the expression directly in the source attribute field or click the Expand icon in the source attribute field to write a long-expression.

All attributes marked with a red asterisk (*) indicate required fields for successful provisioning.

After the source-to-target attribute mapping is completed, click Save Changes at the top-right corner of the page to save the integration.

Update the Integration to the Latest Version

You can upgrade an integration built from a template to the latest version to access new features and enhancements. 

Perform the following steps to upgrade an integration:

  1. Navigate to the Integrations module on the Aquera Admin portal.
  2. Click the vertical ellipsis icon next to an integration and click View Details. The Details panel is displayed with the following sections:
    • INTEGRATION DETAILS: Displays the integration name, description, creation date, last update date, and the user who created or last updated it.
    • TEMPLATE DETAILS: Displays the current version number and its features under CURRENT VERSION. If a newer version is available, the version number and the changes are shown under NEW VERSION AVAILABLE with an Upgrade button.
  3. Click Upgrade. The Upgrade Integration pop-up window is displayed.
  4. Provide the following details for the new integration to be created after the upgrade:
    • New Integration Name: Enter the name of the new integration.
    • Description: Provide a description for the new integration.
      The current version number and the upgraded version number are displayed.
  5. Click Upgrade. A new integration is created with the same configuration details as the existing one, but with features or enhancements from the upgraded version.
