Any Application to Active Directory Integration Setup Guide

Overview

Aquera provides an innovative platform that allows your organization to scale complex data management by automating data synchronization from any application to Active Directory.

Aquera's integrations facilitate the automatic provisioning, ensuring consistency across systems. It allows organizations to streamline processes, dynamically scale resources, and automate repetitive tasks, ultimately improving efficiency in managing intricate IT infrastructures and applications.

In this integration, every record newly created or modified in any application is updated in the Active Directory according to the schedule.

This guide covers details on how to set up the Any App to AD integration to synchronize data from any application to the Active Directory. 

Prerequisites

The following are the prerequisites:

  • Aquera Tenant Account: You must have a tenant account in the Aquera Admin portal. Visit https://admin.aquera.io and register to get a tenant account. Contact your Aquera account manager to get the Aquera tenant access.
    Note: If your organization is in the EU region, use the following URL to register: https://admin.eu.aquera.io/

  • Configure Applications in the Aquera Admin Portal: 
    • Source Connector: Ensure you add and configure the required connector in the Aquera Admin portal. Refer to the Aquera Help Center guides for details on configuring the respective connector.

    • Active Directory Connector: For Any App to AD integration, ensure you add and configure the Active Directory V8 connector in the Aquera Admin portal. Refer to the Active Directory Version 8 Configuration Guide for details.

Integration Setup

Aquera's Any App to AD integration automatically synchronizes data from the configured source application to Active Directory.

Perform the following steps to create the integration:

  1. Log in to the Aquera Admin portal.
  2. Navigate to Integrations on the left panel.
  3. Click Create Integration.
  4. Select Any App to AD.
  5. Enter a name and a description for the integration.
  6. Click Continue. A new page with the fields and tabs to configure the integration is displayed.
  7. Select your required connector in the Source Application dropdown field.
  8. Select the Active Directory Version8 connector in the Target Application dropdown field.
  9. You can update the configurations as per your requirements. For details on the configurations, refer to the User Integration Setup Guide.
  10. Click Create to create Any App to AD integration.

User Naming Attributes

In Active Directory, the user naming attributes are User Principal Name (UPN), SAM Account Name (SAM), Common Name (CN), and Mail. These attributes are used to identify user objects for security purposes.

  • UPN: The User Principal Name attribute is the logon name for the user that is used to log on to a domain. The userPrincipalName must be unique in the forest. By convention, UPN should map to the user's email address. A UPN consists of a prefix (the user account name) and a suffix (a DNS domain name). The prefix is joined with the suffix using the @ symbol. The format of UPN is username@DomainName. For example, barbarajensen@aquera.com

  • SAM: The SAM Account Name attribute is the logon name used to support clients and servers running earlier versions of Windows. This name must be less than 20 characters. The SAM Account Name must be unique within the domain. The format of the SAM Account Name is DomainName\username. For example, Aquera\barbarajensen

  • CN: The Common Name attribute is the name of an object in the directory. This will be the object's relative distinguished name (RDN). The Common Name must be unique in the container or the organizational unit. For instance, you can specify the first name, followed by a space, and the last name as the common name of a user. For example, Barbara Jensen 

  • Mail: The Mail attribute is the user's email address, often set to match the UPN. The mail should be unique within the same domain to avoid email delivery issues. The mail attribute typically follows a standard email formatting. For example, barbarajensen@aquera.com

A user object in AD requires unique names for UPN, SAM, CN, and Work Mail attributes. Aquera's Any App to AD integration enables you to generate unique names for UPN, SAM, CN, and Work Mail of an AD user. The following are the available configurations:

  • Set User Naming Attributes: Select the following checkboxes based on your use case:
    • On Create: The attributes UPN, SAM, CN, and Work Mail will be updated in Active Directory only during a user create operation. 
    • On Update: The attributes UPN, SAM, CN, and Work Mail will be updated in Active Directory only during a user update operation. 

  • UPN suffix: Set a domain name for the UPN suffix. For example, aquera.com

    You can also set a UPN suffix based on a condition. Click the Conditional UPN Suffix to assign a domain name for the UPN suffix based on the condition evaluations.


  • User Naming Attributes Setup: Select the attributes that should be mutually dependent with UPN. In the Define name format fields, you can write JavaScript expressions for UPN, SAM, CN, and Work Mail which the integration evaluates to generate unique names. If the evaluated values are not already assigned to any of the existing users in the Active Directory, the integration assigns the values to these unique attributes when an AD user is created or updated.
    You can click Add Format to add multiple JavaScript expressions to create a different value for the user naming attributes in case the evaluated values are already assigned to existing users.

    Note: The suffix of the UPN will be extracted from the UPN suffix field. However, the suffix for the Work Mail must be specified within the JavaScript expression in the Work Mail field.

The following is a scenario explaining how the unique names are generated:  

Consider you have selected UPN, SAM, and Work Mail as mutually dependent. This means that while creating or updating a user, the values evaluated as per the defined format for UPN, SAM, and Work Mail must be available. If the defined format values are already used for existing users, the integration checks for the next defined format. In this scenario, CN is an independent attribute.

Suppose you have given the following JavaScript expressions for UPN, SAM, Work Mail, and CN. When a new user with the name Barbara Jane Jensen is to be created in AD, the integration evaluates the JavaScript expressions to assign values for the UPN, SAM, Work Mail, and CN attributes of the user. 



The following are the values generated from the defined expressions for the new user Barbara Jane Jensen:

UPN, SAM, and Work Mail formats:

  1. UPN: Barbara.Jensen, SAM: Barbara.Jane.Jensen, Work Mail: Barbara.Jensen
  2. UPN: Barbara.Jensen1, SAM: Barbara.Jane.Jensen1, Work Mail: Barbara.Jensen1
  3. UPN: Barbara.Jensen2, SAM: Barbara.Jane.Jensen2, Work Mail: Barbara.Jensen2

CN formats:

  1. CN: Barbara Jensen1
  2. CN: Barbara Jensen2
  3. CN: Barbara Jensen3

The integration checks whether a unique value is available for all the mutually dependent attributes, to be assigned to the new user. Even if one of those mutually dependent attribute values is not available (that means, the value is already assigned to a user), then the integration evaluates the next set of formats/expressions defined for the mutually dependent attributes. 

The integration evaluates the format 1, followed by the format 2, and so on. If the values of a defined format have already been assigned to existing users, the integration will look at the next format to evaluate the values.

For format 1, if the UPN, SAM, and Work Mail values are not assigned to any user, then this format will be used for the new user. In this case, the UPN value of Barbara.Jensen, the SAM value of Barbara.Jane.Jensen, and the Work Mail value of Barbara.Jensen will be assigned to the new user Barbara Jane Jensen.

For format 1, if the UPN and Work Mail values are already assigned to an existing user, the integration checks for the next format (which is format 2) for all the 3 mutually dependent attributes, even if one of those attribute values is available. This means UPN, Work Mail, and SAM values for a defined format must be available to be assigned for a new user creation. For format 2, if the UPN, Work Mail, and SAM values are not already assigned to any user, then this format will be used for the new user creation. In this case, the UPN value of Barbara.Jensen1, the SAM value of Barbara.Jane.Jensen1, and the Work Mail value of Barbara.Jensen1 will be assigned to the new user Barbara Jane Jensen.

The integration next evaluates the formats for the CN. Suppose the CN values of format 1 and format 2 are already used for existing users and format 3 is not used by any user, the integration assigns the format 3 value which is Barbara Jensen3 for the CN of the new user Barbara Jane Jensen.

OU Assignment

In Active Directory (AD), an Organizational Unit (OU) is a container object that organizes and manages resources such as users, groups, computers, and other OUs within a domain. Each OU has a distinguished name that specifies its location in the Active Directory hierarchy. An organization can create OUs based on departments, geographic locations, or roles. It is highly recommended to place user objects in OUs to help with better management, policy application, and administrative delegation.

Aquera's Any App to AD integration allows you to define rules to assign users to Active Directory OUs. The following are the available configurations:

  • Expression: You can write JavaScript expressions in the Enter Expression field to define OU rules that place users in the respective OUs when the user satisfies the expression provided.
  • Priority: The priority indicates the priority in which the OU rules will be evaluated. If the user satisfies an OU rule, the user is placed in that OU. However, if the user does not satisfy an OU rule, the next OU rule will be evaluated. You can drag and drop the icon next to the priority number to change the order of the priority of the OU rules.
  • Assign On: In the Select Map dropdown list, select one of the following mapping options based on your use case:
    • Create: The OU and the DN attributes of the AD user will be updated during the user create operation.
    • Update: The OU and the DN attributes of the AD user will be updated only during the user update operation. 
    • Create & Update: The OU and the DN attributes of the AD user will be updated during the user create and update operations.
  • Organizational Units: In the Select OU dropdown list, select one of the organizational units where the user will be placed based on the expression evaluation. To ensure all the required OUs are listed in this OU list, ensure you have configured the base DN in the Active Directory connector's Aquera Tenant Attribute User Base Distinguished Names.
  • Display only OUs that contain users: When you select this checkbox, the Select OU dropdown list will contain only the OUs that contain users.
  • New Rule: You can click New Rule to add more rules. 
  • Assign Default OU: If a user does not satisfy any of the defined OU rules, the user will be placed in the default OU selected in the Select Default OU dropdown list.
  • Refresh OU List: When the configuration of the User Base Distinguished Names is updated in the AD connector or when the OU names are updated in the AD application, it is required to refresh the OU list displayed in the Select OU dropdown list. To accomplish this, click the Refresh OU List button to retrieve the most recent OU list.

In Active Directory, each object has a Distinguished Name (DN), which serves as a unique identifier within the directory. A DN describes the fully qualified path to an entry within the Active Directory hierarchy. A DN is made up of attribute=value pairs, separated by commas. Example of a user DN:
CN=Barbara Jensen,OU=Sales,DC=example,DC=com

A user DN is formed from the combination of the CN and the OU assigned to the user. For example, if the CN for the user is Barbara Jensen3 and the OU where the user will be placed is OU=Admins,DC=example,DC=com, then the DN for the user will be:
CN=Barbara Jensen3,OU=Admins,DC=example,DC=com
